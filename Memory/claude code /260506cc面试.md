## Q1：CC 为什么选择中心化星型拓扑架构？

**你的回答：** 架构简单，不需要处理 Agent 间的消息路由、共享状态等；对话天然一进一出，只有一个 Agent 对最终回复负责；CC 是工具密集 + 顺序推理 + 需要把控质量，中心 Agent 负责推理，分派子 Agent 执行。

**点评：** 抓住了"简单"和"一进一出"两个核心驱动力。但漏了两点：①**安全是第一驱动力**——中心化 = 单一审计点，保证所有行为在用户可监督范围，权限链条不断裂；②**没有引用理论数据**——Google 论文实验结论：独立多体错误放大 17.2 倍，中心化只有 4.4 倍；去中心化通信开销 O(D·N²)，coding 低熵场景完全不划算。面试中抛具体数字说服力立刻不同。`⚠️ 不完整`

## Q3：解释"工具调用 vs 移交"和"仅共享最终结果 vs 共享完整推理链"

**你的回答：** 不知道。

**正确答案：**
- **工具调用模式**：主 Agent 把子 Agent 当作工具调用，传参进→等返回值，自己保留控制权继续决策。**移交模式**：主 Agent 交出控制权自己退出，子 Agent 完成后直接回复用户。CC 和 OpenClaw 都选工具调用，主 Agent 永远是唯一对最终回复负责的人。
- **仅共享最终结果**：子 Agent 在自己的私有上下文（Boris 称 "uncorrelated context windows"）完成全部推理，只把结论返回。好处：①隐私隔离——子 Agent 中间数据不泄露；②控制上下文膨胀。**共享完整推理链**：每一步 tool call + thinking 全传给父 Agent，协作强但上下文快速爆满。`❌ 未掌握`

## Q6：extractMemories / Session Memory / AutoDream 的分工和优先级？

**你的回答：** extractMemories 是主力（每轮结束后触发，有频率限制）；Session Memory 保存当前会话结构化笔记，在内存中不持久化；AutoDream 有时间/会话数限制，较重量，会精读最近会话判断 memory 是否重复需修改。优先级：extractMemories 最高。

**点评：** extractMemories 时机和主力角色对，AutoDream 三个门条件对。**但 Session Memory 不是"在内存中不持久化"——它是写入磁盘的结构化笔记文件**，三大用途就是证明：`/resume` 恢复会话、压缩时当现成摘要来源、`/skillify` 转 SKILL.md。它持久化，只是生命周期限于当前会话。**漏了**：extractMemories 的执行方式——Fork Agent 后台执行 + 严格权限沙箱（只能读文件 + 只写 memory 目录，不能 bash/MCP/子代理）。`⚠️ 有错误`

## Q7：Push 模式 vs Pull 模式记忆加载，差异和风险？

**你的回答：** Push 像秘书主动推，缺点是成本高，有时记忆不能及时推送（小模型异步获取，最快下次推理才用到，甚至下次用户发言才用，那时相关性存疑）。Pull 作为工具调用，整体时长变长，风险是模型不知道什么时候该去查记忆。Push 模式更合理——"反正我给你了，用不用你自己看"。

**点评：** Pull 风险说得很好——"模型不知道自己不知道什么"（例子：用户说过讨厌 forEach，模型没想到去查，直接写了 forEach）。**但 Push 机制你理解有误**：文档写得很清楚，Push 是**同步的**——每轮对话开始前，系统用 Sonnet 先扫记忆，挑出最相关的直接塞进上下文。主模型开始思考时，相关记忆已经在眼前了，**不存在异步延迟问题**。Push 的真正风险是：①固定占用上下文空间（不管用不用得上）；②选了不相关记忆可能带偏模型。`⚠️ 机制理解有误`

## Q8：CC 5 层压缩 vs OpenClaw 1 层精压，Proteus 偏向哪种？

**你的回答：** 偏向 CC 设计。分层压缩够细，大概率触发不了完整压缩。CC 采用渐进式加载，工具结果缓存起来需要时再加载，上下文报警往往是工具返回结果导致的。

**点评：** 分层设计和渐进式披露说对了。**但完全没提 OpenClaw 的优势**：①保留最近 ~20K tokens 原文不动（CC Full Compact 全部丢弃只留摘要，丢失细微上下文）；②Pre-Compaction Memory Flush（压缩前提醒 Agent 写记忆，CC 没有）；③实现简单行为可预测。回答应该两面对比。`⚠️ 不完整`

## Q9（追问）：Microcompact 为什么区分缓存过期和有效两种情况？

**你的回答：** 主要是推理成本考量，TTL 过期了清旧结果无所谓，服务端都要重新计算 KV Cache。

**点评：** 只说对一半。关键在于 Anthropic Prompt Cache 机制——缓存的是客户端发过去的**消息前缀**，如果客户端改了对话历史中任何旧内容，前缀变了 → 缓存失效 → 重新处理全部 tokens → **更贵更慢**。所以分两种情况：
- **缓存已过期（>60min）**：直接改客户端消息，把旧结果替换为 `[Old tool result content cleared]`。反正缓存没了，改了没损失。
- **缓存仍有效（≤60min）**：不碰客户端消息，发 `cache_edits` 指令让服务端在缓存副本里删——**前缀不变，缓存继续有效，省钱**。这才是 Microcompact 的精妙之处。`⚠️ 核心动机没说透`
