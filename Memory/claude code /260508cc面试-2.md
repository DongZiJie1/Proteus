**你的回答：** 不清楚。

## Q5（精妙机制·Microcompact cache_edits 第二次）

**你的回答：** 不知道，请求详细解释。

**正确答案：**

**背景：** Anthropic Prompt Cache 缓存的是客户端发过去的**消息前缀**（TTL 约 60 分钟）。如果客户端修改了对话历史中的任何旧内容，前缀就变了 → 缓存失效 → 更贵更慢。

**两种策略（按优先级尝试）：**

1. **缓存已过期（>60min）**：直接改客户端消息，把旧工具结果替换为 `[Old tool result content cleared]`，只保留最近 5 条。缓存已经自然过期，改了没损失。

2. **缓存仍有效（≤60min）——精妙之处**：客户端消息**完全不改**，而是额外发一条 `cache_edits` 指令，告诉服务端"在你缓存的副本里删掉这些工具结果"。**前缀不变，缓存继续有效，省钱。** 客户端的本地消息数组保持完整（用于 session 持久化、resume 等），但模型实际看到的上下文已经被缩减了。

```
传统做法：改客户端消息 → 前缀变了 → 缓存失效 → 贵
cache_edits：不改客户端消息 → 前缀不变 → 缓存有效 → 省
```

**精妙在哪：** 这是**协议级操作**——在不破坏缓存的前提下，让服务端"假装"旧内容不存在。客户端和"模型看到的上下文"出现了分歧，但这个分歧是刻意设计的：本地保留完整历史用于持久化，服务端只看到精简版本用于推理。两全其美。`⚠️ 第二次仍未掌握`

## Q7（新题·记忆写入三种机制）

**你的回答：** 第一种是用户主动说记住，agent 主动往记忆里写东西；第二种是每几次交流有一个 agent 异步查看会话中有没有值得记忆的东西；第三种是 AutoDream，异步执行，比较重。

**点评：** 方向对，但有两处关键错误：

**第一种（extractMemories）**——不是"用户主动说记住"才触发。它是**每次完整查询循环结束后**自动由 Fork Agent 后台执行的。用户不需要说"记住"，Agent 自动从最近几条对话中提取 user/feedback/project/reference 四类事实写入 `~/.claude/memory/*.md` + MEMORY.md 索引。

**第二种（Session Memory）**——不是"每几次交流"，是按 **token 增量**触发：对话 token 超 10K 后启动，之后每 5K tokens 更新一次。写入目标不是持久记忆，而是**当前会话的结构化笔记文件**，生命周期限于当前会话。

**第三种（AutoDream）**——对了。三个触发门：>24h 且 >5 个会话且获取文件锁。四个阶段：Orient → Gather → Consolidate → Prune。

三种机制都是 **Fork Agent 后台执行**，都受严格权限沙箱约束——只能读代码文件，只能写记忆目录，不能执行 bash、不能调 MCP、不能启动子代理。类比：extractMemories 是"上课记笔记"，Session Memory 是"写会议纪要"，AutoDream 是"周末整理笔记本"。`⚠️ 第一二种有误`

## Q8（追问·extractMemories 权限沙箱）

**你的回答：** 因为这个工具的功能就是写记忆，给了它其他的操作目录，模型可能产生幻觉去瞎搞。

**点评：** 方向接近，但核心点没抓住。两个原因：

**第一，最小权限原则。** 它的职责就是读对话 → 提取事实 → 写记忆文件。只需要"读文件 + 写 memory 目录"两种能力，多余的权限不该给。这不是因为幻觉，而是安全工程的基本原则。

**第二，也是更重要的——它是后台自主运行的，没有 human-in-the-loop。** 主 Agent 执行 bash 时用户能看到、能审批（权限弹框），但 extractMemories 是 Fork Agent 在后台静默执行的，用户看不到它的中间步骤。如果它能执行 bash，出了问题用户根本来不及阻止。CC 的安全设计核心是"人在回路中"，后台 Agent 绕过了这个机制，所以必须用沙箱物理约束。`⚠️ 方向对但没说到点上——核心是"后台自主运行 + 无 human-in-the-loop"`

## Q9（新题·AI 分类器防注入设计）

**你的回答：** 不知道。

**正确答案：** **防 Prompt Injection 攻击。**

攻击场景：恶意代码文件中嵌入 prompt injection 指令，诱导主 AI 输出特定文本来说服分类器"这个操作是安全的"。如果分类器能看到主 AI 的文本输出，攻击者就能通过 文件内容 → 主 AI → 分类器 这条链路注入恶意指令。

CC 的解决方案：分类器只看"做了什么"（工具调用记录），不看主 AI "说了什么"（文本输出）。**从信息流源头切断注入路径**——即使主 AI 被诱导输出了恶意文本，分类器也看不到，无法被说服。

另外，分类器还有自己**独立的系统提示词**，读取 CLAUDE.md 作为用户意图上下文，与主 AI 的提示体系完全隔离。两层隔离：信息流隔离 + 提示词隔离。`❌ 未掌握`

## Q10（综合·从 CC 借鉴什么）

**你的回答：** 1.记忆分类 2.记忆写入的方式 3.工具权限分类 4.压缩分成细节 5.AutoDream

**点评：** 题目说挑最重要三点，给了五个。面试时说五点会显得没有优先级判断。

**最值得讲的三点：**

1. **记忆分类**（user/feedback/project/reference）——直接解决"记忆什么都往里塞"的问题，分类后召回精准度高
2. **五层梯度压缩**——直接能映射到 Proteus。大部分情况下前两层（不需要 LLM 调用）就能压下来，Full Compact 是中低频事件，用工程复杂度换成本效率
3. **四层权限管道**——这是项目目前没有的维度，加进去能提升项目深度。面试官问"Agent 安全怎么做"时有东西讲

**为什么不选另外两个：**
- 记忆写入三种机制 → 和记忆分类有重叠，面试时可以合并讲
- AutoDream → 太细了，面试官一般不会追问到这个粒度，被问到再说

另外，每个点要能说出**具体怎么落地到 Proteus**，不能只说"借鉴了 CC"。比如压缩：可以说"替换 CC 的全量压缩为 OpenClaw 的精压策略——保留最近 20K tokens 原文 + 分块摘要合并"，这样面试官会觉得做了独立思考。`⚠️ 能说出五个说明理解到位，但面试要精选 + 说清落地方式`
