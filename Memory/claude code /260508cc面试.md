## Q1（温故·高熵/低熵）

**你的回答：** 低熵=路径相对较少，高熵=路径不确定、信息分散。去中心化适合高熵因为多 Agent 并发探索不同路径。

**点评：** 核心定义抓住了。缺：①论文数据（独立多体错误放大 17.2x，中心化 4.4x）；②协调税 O(D·N²) 在低熵不划算；③没举具体例子。`⚠️ 有进步但不完整`

## Q2（温故·工具调用/移交 + 消息传递）

**你的回答：** 工具调用=主 Agent 把子 Agent 当工具传参等返回值；移交=交出控制权子 Agent 直接回复用户。CC/OpenClaw 都选工具调用，因为对话一进一出。仅共享最终结果=只拿结论不拿中间推理链。

**点评：** 从上次"未掌握"到基本正确。可补：Boris 术语"uncorrelated context windows"、两大好处（隐私隔离+控制上下文膨胀）。`✅ 基本正确`

## Q3（安全·四层权限管道）

**你的回答：** 第一层规则过滤（用户意志直接表达），第二层工具自检（.git 例子好），第三层模式兜底（用户选择风险偏好），第四层 AI 分类器（安全工具白名单+工作目录内编辑快速路径）。四层原因：确定性从高到低递进。

**点评：** 从上次"完全漏掉"到框架清晰。缺：Bash AST 解析器、bypass-immune、两阶段分类器（64 token→4096 token）、防注入设计、瑞士奶酪模型/Fail Closed 术语。`✅ 良好`

## Q4（纠错检验·Session Memory + Push 模式）

**Session Memory：** 你仍误认为是"整个会话记录（tool call/推理/消息）"——那是 messages[]。正确：后台 Fork Agent 维护的结构化笔记文件（任务描述/状态/文件/错误/经验），写入磁盘，用于 resume/压缩/skillify。

**Push 模式：** 已纠正——不是异步，是每轮对话前 Sonnet 同步完成筛选并注入。`SM ⚠️ 仍需纠错 / Push ✅ 已纠正`

---

## Q5（精妙机制·Microcompact cache_edits）

**你的回答：** 缓存过期直接改消息；缓存有效通过 cache_edits 调用服务端修改 KV cache。

**点评：** 核心逻辑说反了。cache_edits 不是"修改 KV cache"，而是不修改客户端消息前缀，发指令让服务端在缓存副本里删——前缀不变缓存有效。你理解成"在修改提示词前提下享受缓存"，恰好相反。`⚠️ 核心机制仍需理解`

**你的回答：** CC 梯度递进用低成本先避免全量压缩；OpenClaw 保留最近 20K + 分块摘要合并。Proteus：5 层框架 + 替换全量压缩为 OpenClaw 精压。
