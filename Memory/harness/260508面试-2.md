# Harness Engineering 面试 Q&A 记录 — 2026-05-08（第二轮）

## Q1：请完整描述 Anthropic Harness 架构中，一个 Coding Agent 从创建到销毁的完整生命周期。

**你的回答：** 不知道。面试官讲解后回答追问——追问 1：外部状态文件是全局状态，上下文窗口是当前原子任务的执行情况；追问 2：Handoff 在销毁前写是因为销毁后上下文没了，Handoff 保证任务状态完整正确，避免下一个 Agent 重复思考甚至出现不一致。

**追问 1 点评：** "全局 vs 当前任务"的方向对，但漏了更本质的区别——**持久性**。上下文窗口是易失的 RAM，Agent 销毁就没了，容量有限还会被垃圾信息填满（Context Rot）。外部状态文件是持久存储，不随 Agent 生亡而消失，容量不受限，信息质量可控。

**追问 2 点评：** 答得好。补充一个角度：Handoff 记录的不只是"做了什么"，还有"遇到了什么坑、为什么这样选择"。这种**决策上下文**是代码本身无法表达的。

**正确答案：**
Coding Loop 六步生命周期：
1. **创建全新 Agent** — 上下文窗口是空的，这是 feature 不是 bug
2. **读取外部状态文件** — feature_list.json、progress.md、代码库当前状态，不依赖前序 Agent 的上下文
3. **选择一个明确的原子任务** — 从任务清单里挑一个未完成的小任务
4. **只实现这一个任务** — 全部注意力只放在这个小任务上
5. **确定性验证** — 跑测试、lint、类型检查，不通过继续修
6. **更新状态 + 写 Handoff + git commit → Agent 销毁** — 验证通过后更新 progress.md、写交接文件、git commit，然后 Agent 退出销毁

外部状态文件 vs 上下文窗口的本质区别：持久性。上下文是易失 RAM，状态文件是持久存储。Handoff 记录决策上下文（坑、选择理由），代码本身无法表达这些。`❌ 初始未掌握，讲解后理解到位`

## Q2：ClawCode 的 Initializer 和 Anthropic 的 Initializer 有什么不同？为什么它要建一个 Python 中间层？

**你的回答：** 不知道。面试官讲解后表示理解了。

**正确答案：**

**核心区别：起点不同。**
- Anthropic Initializer 起点是**需求文档/PRD**，读需求、拆 feature list、初始化项目骨架
- ClawCode Initializer 起点是**已有的 51 万行 TypeScript 代码库**，扫描源码把系统拆解成机器可查的清单

**ClawCode Initializer 三件事：**
1. 用字符串匹配自动提取清单（207 个命令、184 个工具），不需要 AI 介入
2. 生成结构化 JSON 快照（仓库结构、命令清单、工具清单、29 个子系统元数据）——这是"有什么"
3. 建 Python 中间层（5 个可运行文件建模动态行为）——这是"怎么运转的"

**Python 中间层四样东西：**
- 1:1 结构映射表（TS→Python 完整映射）
- 29 个占位包（每个子系统的元数据，告诉 Agent 原始系统有哪些模块）
- 5 个关键架构模型（bootstrap_graph.py 启动流程、command_graph.py 命令分类、query_engine.py turn-loop 行为、setup.py 环境发现、runtime.py 路由算法）
- 可运行的审核 CLI（供 Agent 和人随时验证中间层的正确性）

**为什么需要 Python 中间层：** JSON 只有静态结构（"有什么文件"），Python 中间层建模动态行为（"系统怎么运转"）。Agent 先查这个轻量沙盘就能快速定位，不需在 51 万行代码里大海捞针。`❌ 未掌握`

## Q3：ClawCode 的验证方式和 Anthropic 有什么不同？为什么不能直接用 Anthropic 那套？

**你的回答：** 不知道。面试官讲解后表示理解了。

**正确答案：**

Anthropic 是从零写新项目，验证方式直接——写测试，跑通就行。

ClawCode 是**重写**，验证标准不是"代码能不能跑"，而是**"Rust 版的行为和原始 TypeScript 版是否一致"**。如果每次调真实 API 测试，贵、慢、结果不确定。

**ClawCode 验证体系三样东西：**
1. **Mock Anthropic 服务** — 本地 mock 服务模拟 API 响应，返回确定性结果
2. **10 个场景脚本** — 覆盖核心工具链路（streaming_text、read_file_roundtrip、write_file_denied 等），每个场景通过 `parity_refs` 指向 PARITY.md 中的具体条目，测试和文档一一对应
3. **双向验证工具（run_mock_parity_diff.py）** — 检查场景脚本引用的条目在 PARITY.md 中是否存在 + 跑 cargo test 输出通过状态，防止文档和代码漂移

`❌ 未掌握`

## Q4："原子任务 + 上下文刷新"和"确定性验证轨道"这两个原则之间是什么关系？

**你的回答：** 原子任务完成之后才能进行确定性验证，两者是先后关系，缺少任何一个都不能保证稳定性。

**点评：** 方向对但太浅。说的只是"先后顺序"，没说出**互相支撑的逻辑**。`⚠️ 部分正确`

**正确答案：**

不是简单的先后关系，而是**互相依赖**：

- **原子任务 → 确定性验证：** 原子任务缩小验证范围。如果一个 Agent 一口气做 10 个功能，测试挂了不知道哪个出了问题。只做一个小任务，验证不通过，原因就在这一个小任务里，定位成本极低。**任务越小，验证越精准。**

- **确定性验证 → 原子任务：** 验证通过保证任务**真正完成了**，不是 Agent 自己说的"我觉得完成了"。下一个 Agent 接手时信任上一步结果，不需要重新检查。**没有验证，原子任务的交接就是空中楼阁。**

## Q5：Harness 的外部状态有哪三种具体存储形式？分别存什么？

**你的回答（初始）：** JSONL 文件、MySQL、Redis。

**点评：** 说的是通用存储技术，不是 Harness 工程里具体的外部状态形式。面试官问的是"在 Harness 架构里，项目状态具体以什么形式存在"，不是"用什么数据库"。`❌ 答偏`

**提示后补充：** 三种形式合在一起才是最完美的，不是三选一。

**正确答案（三种互补的外部状态形式）：**

1. **任务清单文件（feature_list.json）** — 存"要做什么"。任务名称、描述、验证标准、状态（待做/进行中/完成）
2. **进度文件（progress.md / PARITY.md）** — 存"做到哪了"。已完成任务、覆盖率、commit hash、实现深度（strong/good/moderate/stub）
3. **Git 历史和 commit** — 存"做了什么、怎么做的"。代码变更 + commit message，是最可靠的证据
4. **Handoff 交接文件** — 存"遇到什么坑、为什么这样选"。决策上下文，防止下一个 Agent 重复踩坑

这些都是文件系统上的明文文件，不是数据库——Agent 天然会读写文件，轻量、透明、可审计。三种形式互补，缺任何一个新 Agent 接手时都会有信息盲区。

## Q6：新模型出来了，Harness 需要改什么、不需要改什么？

**你的回答（初始）：** Harness 和模型无关，可以让弱模型达到强模型+普通 Harness 的能力。不知道需要改什么。

**点评：** 说对了"不需要改什么"（架构层），但没回答"需要改什么"。`⚠️ 部分正确`

**正确答案：**

**不需要改的（Harness 架构层）：**
- 任务清单结构、进度文件格式、Git 工作流
- 确定性验证轨道（测试、lint、类型检查）
- Agent 生命周期（创建→读状态→执行→验证→交接→销毁）
- Human-in-the-loop 断点

**需要改的（和模型行为特性挂钩的部分）：**
1. **Prompt 模板和系统提示词** — 不同模型对指令的理解方式不同
2. **任务粒度** — 强模型可以接更大的原子任务，弱模型需要拆得更细
3. **工具描述和 JSON Schema** — 不同模型对工具参数的理解能力不同
4. **验证容忍度** — 弱模型通过率可能更低，需要调整重试策略
5. **上下文预算分配** — 不同模型上下文窗口大小、token 价格不同

**一句话总结：** Harness 的"骨架"（架构、验证、状态管理）是模型无关的，但"皮肤"（prompt、粒度、工具描述、预算）要跟着模型换。理想状态是这些参数可以配置化，换模型时只改配置不改代码。

## Q7：0.99^10 ≈ ? 这个数学规律对 Harness 设计有什么启示？

**你的回答：** 10 步后准确率逐步降低。最重要的是系统需要为 LLM 的预测兜底，能不要 LLM 实现的就用代码解决，尽量将每一步可以达到 100% 的任务都准确达到 100%，能用代码解决的就不让 LLM 思考。

**点评：** 设计思路非常好，甚至比标准答案更有实践价值。准确抓住了 Harness 的核心设计哲学——把 LLM 的不确定性压缩到尽可能小的范围。`✅ 优秀`

**补充数学：** 0.99^10 ≈ 0.904（90.4%），0.99^100 ≈ 0.366（36.6%）。

**完整启示：** 能用代码 100% 判断的事情，绝不交给 LLM 判断：
- 验证：测试通过/不通过，是代码判断
- 任务选择：从清单里挑下一个未完成任务，是代码逻辑
- 状态更新：progress.md 写什么，是模板化输出
- Git commit：提交什么、格式，是约定好的

LLM 只负责一件事：**写代码实现具体功能。** 每把一个环节从"LLM 判断"变成"代码判断"，整体成功率就往 100% 靠一步。

## Q8：Harness Engineering 和传统的 CI/CD 有什么区别？

**你的回答：** 不知道。

**正确答案：**

**表面很像：** 都有流水线、测试门禁、自动化执行。本质区别在于**流水线里跑的是什么**。

| | **传统 CI/CD** | **Harness Engineering** |
|---|---|---|
| 流水线里跑的 | 确定性代码，同样代码跑 100 次结果一样 | LLM 的不确定输出，同样 prompt 跑 100 次结果可能不同 |
| 流水线本身 | 静态的，yaml 写好步骤固定 | 动态的，Agent 根据当前状态自主决定下一步 |
| 主体 | 人是主体，CI/CD 只是自动执行人定义好的流程 | AI 是主体，环境是约束 |

**一句话区分：** CI/CD 是**确定性流水线跑确定性代码**；Harness 是**确定性约束跑不确定 Agent**。

**再深一层：** CI/CD 里测试挂了，你修代码重新提交，流水线本身不需要改。Harness 里 Agent 反复失败，你可能需要调整 Harness 本身——任务粒度太粗？验证标准太严？上下文预算不够？**Harness 本身也需要迭代优化**，这是 CI/CD 不需要做的。`❌ 未掌握`

# 总结

## 比上轮进步的

- Anthropic Coding Loop 六步生命周期，听完能理解本质（状态外部化、上下文丢弃是 feature）
- 原子任务和确定性验证的关系，理解了互相依赖的逻辑
- "能用代码解决就不让 LLM 思考"这个设计思路非常成熟，是加分项
- 外部状态形式纠正后答对了（三种互补，不是三选一）

| 原子任务与确定性验证的关系只说了先后，没说互相依赖 | Q4 | 中 |
| 新模型替换时 Harness 需要改什么不知道 | Q6 | 中 |
