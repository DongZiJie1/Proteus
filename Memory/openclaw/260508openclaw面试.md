## Q2（架构·三层拆解）

**你的回答：** 触发层 + 网关层 + Agent 层，触发层是打通各种通道，网关层接收前面传来的事件，转换协议给到 Agent 层，Agent 层是执行 Agent Loop 逻辑。关键边界我不知道。

**点评：** 三层答对了。两个问题：

**触发层描述不准确。** 触发层不是"打通通道"，而是**产生事件的五种来源** — Messages、Heartbeats、Crons、Hooks、Webhooks。通道接入是 Gateway 的 Channel Manager 干的，不是触发层。

**Gateway 关键边界：Gateway 不做推理，不调 LLM，不执行工具。** 它只负责"消息到了该给谁"和"回复该送回哪里"。这条线很重要 — 很多人会把 Gateway 当成智能路由，但它其实只是个协议转换 + 消息分发的中间件，零智能。

| 层 | 核心职责 |
|----|---------|
| **触发层** | 五种事件源（Messages / Heartbeats / Crons / Hooks / Webhooks）不断产生事件 |
| **网关层** | 协议转换、认证鉴权、会话路由、插件加载、设备配对 |
| **Agent 层** | 推理（Think）、工具执行（Act）、记忆（Remember） |

`⚠️ 三层对了，触发层和 Gateway 边界需补`

## Q3（记忆·四层架构）

**你的回答：** 第一层：Agent 的身份与规则；第二层：完整对话历史存档；第三层：当前上下文；第四层：语义搜索引擎。第一层免疫压缩，最近的 2000 tokens 免疫压缩。

**点评：** 四层答对了，每一层的描述也到位。第一层免疫压缩也对了。

但有一处混淆：**"最近 20000 tokens 免疫压缩"不是第一层的特性，而是第三层（Context Window）里 Compaction 的机制。** 压缩时把最近约 20,000 tokens（不是 2,000）划为"保留区"原封不动，更早的消息才被压缩。

两个概念别串：
- **第一层免疫 Compaction** → Bootstrap Files 每次会话从磁盘重读，不在对话历史里
- **保留区原封不动** → Context Window 内的 Compaction 策略，属于第三层

| 层 | 本质 | 类比 |
|----|------|------|
| **Bootstrap Files** | 工作空间里的 Markdown 文件，每次会话从磁盘读取注入上下文 | 硬盘 |
| **Session Transcript** | 完整对话历史，JSONL 格式逐条追加写入磁盘 | 文件系统日志 |
| **Context Window** | 模型当前能"看到"的所有内容，满了触发压缩 | RAM |
| **Retrieval Index** | 为记忆文件建立的搜索索引，Agent 通过工具按需检索 | 搜索引擎 |

`⚠️ 四层对了，保留区概念混到了第一层，数字记错（20K 不是 2K）`

## Q4（压缩机制·高频考点）

**你的回答：** 上下文分成最近的 20000 tokens 和过去（除去第一层内容），把过去分成块，每一块都摘要，然后摘要聚合在一起再摘要。压缩前记忆冲刷指的是压缩前，用另一个 LLM 来判断这轮上下文里有没有值得记忆的东西。

**点评：**

**压缩流程**方向对了，但细节不够：
1. "过去分成块" — 默认分 2 块，块大小会根据消息平均长度**自适应调整**，同时预留 20% 安全裕量防止 token 估算不准
2. 摘要有**严格保留要求** — 必须包含：进行中的任务状态、用户最后的请求、决策理由、TODO 和约束条件、所有不透明标识符（UUID、URL、文件路径等必须原样保留，由 `identifierPolicy: "strict"` 控制）
3. 压缩后模型看到的结构：`[压缩摘要] + [保留区原文]`

**记忆冲刷描述有误。** 不是"用另一个 LLM 判断"，而是**系统静默触发一轮 Agent 回合**，提醒当前 Agent 把重要信息写入 `memory/YYYY-MM-DD.md`，用户不可见（`NO_REPLY`）。每个压缩周期只触发一次。

**完整 Compaction 流程：**
1. 划分保留区（最近 ~20K tokens）和压缩区
2. 分割压缩区为若干块（默认 2 块，自适应调整）
3. 逐块用 LLM 生成结构化摘要（失败重试 3 次）
4. 多块摘要合并为一个连贯总摘要
5. 写入 JSONL，包含摘要文本和 `firstKeptEntryId`

**必须保留的内容：**
- 进行中的任务及其状态
- 批量操作的进度
- 用户最后请求的内容
- 已做出的决策及其理由
- TODO、开放问题和约束条件
- 所有不透明标识符（UUID、URL、文件路径等）

`⚠️ 流程对了但缺保留策略细节，冲刷机制理解有误`

## Q5（Agent 架构·高频考点）

**你的回答：** 中心化架构，因为交互方式一问一答就决定了和用户对接的 Agent 最好一直都是同一个，这可以大大降低系统复杂度，子 Agent 是作为工具调用的形式创建的。

**点评：** 架构类型和理由都说得很好，"一进一出"的核心逻辑抓住了。补充两个细节：

**子 Agent 创建的具体实现：** `sessions_spawn` 被注册为一个普通工具，和 `exec`、`web_search` 地位相同。LLM 自主判断是否 spawn，依据只有系统提示词里的一句话：*"If a task is more complex or takes longer, spawn a sub-agent."* 没有复杂的拆分规则，"复杂"和"耗时长"的边界完全交给模型自己理解。

**两种 runtime：**
- **subagent** — 轻量模式，同一宿主进程运行，继承父 Agent 的 workspace，适合常规任务
- **acp** — 重量级模式，独立进程运行，可对接 Codex 等外部 CLI，支持持久化 session 和恢复

**为什么选 Centralized：** 对话本身的结构就是中心化的 — 有确定的入口（用户发消息），有确定的出口（给用户一条完整的回复），中间的执行可以分发，但责任链不能断。如果换成 Decentralized，多个 Agent P2P 协商，最终谁来发回复没有确定答案。

`✅ 核心逻辑正确，缺实现细节`

## Q6（通信机制）

**你的回答：** 只有结论，子 Agent 本质是一个工具，主 Agent 调用它就是为了拿它的结果，拿它的中间值反而会污染上下文。

**点评：** "只有结论"对了，"污染上下文"的理由也对。但漏了一个关键机制：

**结果传递是异步 push，不是同步返回。** 具体流程：
1. 子 Agent 完成任务
2. `subagent-announce.ts` 把结果以**"用户消息"形式**注入父 Agent 的消息队列
3. 父 Agent 在下一轮 LLM 推理中感知到完成事件
4. 父 Agent 等所有期待的子 Agent 都 announce 后，才输出最终回复

这意味着 spawn 时立刻返回 `accepted`（非阻塞），父 Agent 不轮询、不阻塞线程，等消息触发即可。源码注释写得很明确：*"After spawning children, do NOT call sessions_list, sessions_history, exec sleep, or any polling tool. Wait for completion events to arrive as user messages."*

`⚠️ 结论对了，通信机制（push-based announce）没说`

## Q7（对比题·CC vs OpenClaw）

**你的回答：** RAG 本质上是向量 + 关键词，如果不是海量数据的话，其实直接用 LLM 来判断更好。但是我觉得二者结合一下更好。用 LLM 去记忆里搜数据，用 RAG 去过去的上下文找到相关信息。它们还有不同的检索方式，一个是 push 主动，一个 pull，记忆检索用模型判断是否需要使用这个工具。

**点评：** 你的观点有一定道理，但面试要答的是文档里的设计决策，不是自己的看法。

**CC vs OpenClaw 的两个根本分歧：**

| 分歧 | Claude Code | OpenClaw |
|------|-------------|----------|
| **检索方法** | Agentic Search — 模型自主调 glob + grep，不建索引 | RAG — SQLite 本地索引，向量语义 + BM25 混合检索 |
| **加载模式** | Push — 每轮前 Sonnet 同步扫描，主动塞进上下文 | Pull — memory_search 作为工具暴露，模型自己决定何时查 |

**CC 拒绝 RAG 的原因：** Boris 说过早期试过向量数据库 + embedding，但**代码修改后索引不同步、权限管控复杂**，最终结论是 agentic search 全面优于 RAG。原话：*"agentic search just outperformed everything, and when I say agentic search, this is a fancy word for glob and grep."*

不是"LLM 判断更好"这么简单，而是**代码场景下索引同步是硬伤**。你说的"二者结合"可以作为个人观点，但面试时先准确答出两个系统的设计差异，再提个人观点会更好。

`⚠️ 混入个人观点，核心分歧没答清楚，CC 拒绝 RAG 的原因没答`

## Q8（追问·子 Agent 限制）

**你的回答：** 不能，首先如果还可以继续生成子 Agent，这很可能产生无限套娃的行为，为了解决这个问题修改架构是得不偿失的。它们防止的方法是不给生成子 Agent 的工具。

**点评：** "物理移除工具"这个点抓住了，但结论太绝对。**不是完全不能，是取决于配置。**

**两层防护机制：**

**第一层 — 硬性参数检查：**
- `maxSpawnDepth` — 默认为 1，即子 Agent 默认不能再 spawn
- `maxChildrenPerAgent` — 单个 session 最多 5 个活跃子 Agent
- 任一条件不满足，`sessions_spawn` 返回 `forbidden`

**第二层 — 物理移除（叶子节点）：**

| 深度 | 角色 | sessions_spawn |
|------|------|----------------|
| 0（main） | main | ✅ |
| 1（maxDepth≥2） | orchestrator | ✅ |
| 1（maxDepth=1） | leaf | ❌ 工具不存在 |
| 2+ | leaf | ❌ 工具不存在 |

如果把 `maxSpawnDepth` 设成 2，子 Agent 是可以继续 spawn 的。默认配置下不能，但不是架构层面的硬限制。

`⚠️ 结论对了但不精确，两层防护机制需要补充`

## Q9（安全·现实风险）

**你的回答：** 不知道。

**正确答案：**

Cisco 安全团队发现 **26% 的可用 skill 包含至少一种安全漏洞**，称其为 "security nightmare"。主要风险四类：

1. **Prompt Injection** — 恶意指令嵌入邮件/文档/网页，Agent 读取后被诱导执行非预期操作
2. **恶意 Skill / 插件** — 第三方 skill 安装后拥有 Agent 全部能力（Shell、文件、网络）
3. **凭证泄露** — Agent 可访问环境变量、SSH key、API token，被劫持后可能外泄
4. **命令误解** — "清理这个目录"可能被理解为删掉你不想删的文件，`rm -rf` 不会问确认

**官方建议：** 用隔离账号运行、用次要设备、限制 skill 数量、监控日志。官方也承认**没有完美的安全 setup**。

本质矛盾：**能力越强，风险越大** — 它拥有真实的系统权限，运行真实的命令，访问真实的文件。

`❌ 未掌握`

**你的回答：** 1. 压缩策略保留最近一部分 token；2. 使用 RAG 去搜索过往对话记录；3. 接入各大社交渠道。

**点评：** 三个点都能用，但优先级判断有问题。面试说这三点会显得"挑了容易的"。

**更值得讲的三点：**

2. **Bootstrap Files 机制** — 你没提。SOUL.md / AGENTS.md / USER.md 这套设计直接解决了"Agent 身份和规则每次会话都要重设"的问题，Proteus 可以直接抄，比自己设计一套要成熟
3. **五种触发源** — 你只说了"接入渠道"，但 OpenClaw 真正的亮点不是渠道多，而是 **Heartbeats / Crons / Hooks / Webhooks 让 Agent 能主动行动**，不只是被动等消息

**为什么不选 RAG 搜索过往对话：** Proteus 定位类似 CC，CC 明确拒绝 RAG 的原因（索引不同步）对 Proteus 同样适用。面试官追问"你为什么不直接用 agentic search"会很被动。

`⚠️ 能说出来，但优先级判断和落地方式都需要加强`
