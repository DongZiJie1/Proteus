# 从零学 SFT 与 RL：大模型训练全指南

> **前置要求**：你已经理解大模型的基本工作原理——输入 token，模型逐个预测下一个 token，每个 token 都是一个概率分布上的采样。如果你还不清楚这一点，先看《LLM 提示词到输出的完整流程详解》。
>
> **本文目标**：从"模型怎么训练"这个问题出发，一步步理解 SFT → RLHF → DPO → GRPO 的完整演进，最终能回答"为什么要用 RL 而不只是 SFT"。

## 第一章：训练的本质——调参数

> 不管预训练、SFT 还是 RL，本质都是一样的：**训练 = 调参**。流程都相同——前向传播算 loss，反向传播算梯度，更新参数，重复。区别只是数据格式和 loss 函数不同：
>
> ```
> 预训练：数据 = 海量文本，loss = 下一个 token 的交叉熵
> SFT：  数据 = (prompt, response) 对，loss = 下一个 token 的交叉熵
> RL：   数据 = prompt + 奖励信号，loss = PPO/DPO/GRPO 的损失函数
> ```
>
> 后面讲的显存计算、过拟合等问题，对三个阶段都适用，不是某个阶段特有的。

### 1.1 模型是什么

一个大模型本质上就是**一堆数字（参数/权重）**。GPT-3 有 1750 亿个数字，Llama 3 有 80 亿到 4050 亿个。

这些数字决定了：给定一段输入文本，模型给下一个 token 分配什么样的概率分布。

```
输入: "中国的首都是"
模型（一堆数字）→ 概率分布: P(北京)=0.85, P(上海)=0.08, P(广州)=0.03, ...
```

**训练**就是不断调整这些数字，让模型的输出越来越好。

### 1.2 什么是 Loss（损失函数）

Loss 是一个函数，用来计算**模型对正确 token 的概率的惩罚**——概率高惩罚小，概率低惩罚大。然后对每个参数求导得到梯度，告诉参数该往哪调、调多少。

> 不同训练阶段用的损失函数不同：SFT 用 -log(p)，DPO 用对比 loss，PPO 用 clip loss。形式不同，但做的事情一样：**定义一个可优化的目标，算梯度，更新参数**。具体公式后面讲到对应方法时再展开，这里只需要理解这个通用逻辑。

Loss 是一个关于模型参数 θ 的函数 L(θ)，训练的全部目的就是通过调整 θ 来最小化 L(θ)。

#### 为什么用 -log(p) 而不是直接用概率

```
标准答案: "北京"
模型给了 "北京" 概率 0.85 → loss = -log(0.85) = 0.16  ← 还行
模型给了 "北京" 概率 0.10 → loss = -log(0.10) = 2.30  ← 很差
```

为什么不直接用 `1 - p` 当 loss？因为 **-log(p) 有几个关键数学性质**：

1. **当 p → 0 时，-log(p) → +∞**：模型完全答错时，惩罚急剧增大。这迫使模型不敢给正确答案分配极低的概率。而 `1 - p` 最大只到 1，惩罚力度不够。
2. **当 p → 1 时，-log(p) → 0**：模型答对时，惩罚趋近于零，但不会变成负数。
3. **可微性**：-log(p) 关于模型参数处处可微，梯度下降可以稳定工作。
4. **概率解释**：最大化 log 似然 = 最小化交叉熵损失，这在信息论上有严格的最优性证明（最大似然估计）。

#### 交叉熵损失（Cross-Entropy Loss）

LLM 用的 loss 叫**交叉熵损失**，公式为：

```
L = - (1/N) Σ log P_θ(y_t | y_<t)

N = 序列长度（token 数量）
y_t = 第 t 个位置的真实 token
y_<t = 第 t 个位置之前的所有 token
P_θ(y_t | y_<t) = 模型给正确 token 分配的概率
```

**信息论角度的理解**：

- **熵（Entropy）**：一个分布本身的"不确定程度"。真实分布的熵越低，说明答案越确定。
- **交叉熵（Cross-Entropy）**：用模型的分布去编码真实分布的平均代价。模型越准，交叉熵越低。
- **KL 散度 = 交叉熵 - 熵**：衡量模型分布和真实分布的差异。因为真实分布的熵是固定的，所以**最小化交叉熵 = 最小化 KL 散度 = 让模型分布逼近真实分布**。

#### 多 token 的 loss 怎么聚合

LLM 不是对整个序列算一个 loss，而是**对每个 token 位置独立算 loss，再取平均**：

```
prompt:  "中国的首都是"
answer:  "北京"

位置 1: 给定 "中国的首都是"，预测 "北"
  P("北" | "中国的首都是") = 0.7 → loss_1 = -log(0.7) = 0.36

位置 2: 给定 "中国的首都是北"，预测 "京"
  P("京" | "中国的首都是北") = 0.9 → loss_2 = -log(0.9) = 0.11

总 loss = (0.36 + 0.11) / 2 = 0.23
```

这样做的好处：无论序列长短，loss 的量级都是可比的（都是"平均每个 token 的困惑度"级别）。

#### Loss 的梯度：连接 loss 和参数更新的桥梁

Loss 本身只是一个数字。要更新参数，还需要知道**每个参数对 loss 的影响有多大**，这就是梯度。

梯度是一个**向量**（一堆数），每个数告诉你对应的参数该往哪个方向调、调多少。模型有几百亿个参数，梯度就是几百亿维的向量，每个维度对应一个参数的偏导数。

```
模型有 3 个参数: θ₁, θ₂, θ₃

梯度 = [∂loss/∂θ₁, ∂loss/∂θ₂, ∂loss/∂θ₃]
      = [0.5,        -0.3,       0.1]

含义：
  θ₁ 调大 → loss 增大  → 所以 θ₁ 应该减小
  θ₂ 调大 → loss 减小  → 所以 θ₂ 应该增大
  θ₃ 调大 → loss 增大  → 所以 θ₃ 应该减小
```

- 梯度为正 → 这个参数调大了会让 loss 增大 → 应该减小这个参数
- 梯度为负 → 这个参数调大了会让 loss 减小 → 应该增大这个参数
- 梯度接近零 → 这个参数对 loss 影响很小 → 不需要大调

**反向传播（Backpropagation）** 就是从最终的 loss 值出发，利用链式法则，一层一层往回算，直到算出每个参数的梯度。这是深度学习的核心算法。

参数更新就是每个参数各自更新，互不影响：

```
θ_new = θ_old - lr × gradient

lr = 学习率（step size）
∂L/∂θ = 梯度
```

这个公式就是**梯度下降**的含义：沿着梯度的反方向走一小步，loss 就会变小一点。走几百万步，loss 就收敛到一个很小的值。

### 1.3 梯度下降：怎么调参数

知道了 Loss，怎么调整那几百亿个数字？用**梯度下降**：

1. 算出每个参数"往哪个方向调能让 Loss 变小"（这就是**梯度**）
2. 把参数往那个方向挪一小步
3. 重复几百万次

"挪多大步"就是**学习率（learning rate）**。

用找最低点的比喻理解：你在山上找最低点（最小化 loss），梯度告诉你方向，学习率决定每一步迈多大。

```
学习率大 = 大步走
  ✅ 走得快，很快就到山底附近
  ❌ 步子太大可能直接跨过最低点，在谷底两边反复跳，永远到不了

学习率小 = 小步走
  ✅ 走得稳，精确地走到最低点
  ❌ 太慢了，可能要走几百万步
```

同一个梯度，学习率决定了实际挪多少：
```
梯度说：θ 应该减小 0.5
lr = 1.0      → θ -= 0.5       （迈一大步）
lr = 0.01     → θ -= 0.005     （迈一小步）
lr = 0.00001  → θ -= 0.000005  （挪一毫米）
```

### 1.4 优化器 AdamW

#### 先理解朴素梯度下降的问题

最朴素的梯度下降（SGD）对所有参数用同一个学习率：

```
θ_new = θ_old - lr × gradient
```

问题在于：**不同的参数需要不同的步长**。

```
参数 θ₁ 的梯度一直是 0.01 → 应该大步走，赶紧过去
参数 θ₂ 的梯度一直是 100  → 应该小步走，不然跳过了
参数 θ₃ 的梯度一会儿 0.01，一会儿 100 → 不稳定，需要谨慎
```

SGD 对这三个参数用同一个 lr，要么 θ₁ 走太慢，要么 θ₂ 步子太大。

#### Adam 的核心思想：自适应步长

Adam（Adaptive Moment Estimation）给每个参数**单独算一个学习率**，依据是这个参数的历史梯度信息：

```
对每个参数 θ_i：
  m_i = 过去梯度的指数移动平均（方向趋势）
  v_i = 过去梯度平方的指数移动平均（幅度大小）

  自适应学习率 = lr / (sqrt(v_i) + ε)
```

直觉理解：

- 如果某个参数**历史梯度一直很大**（v 大）→ 说明这个参数在剧烈变化 → 学习率自动缩小 → 小步走
- 如果某个参数**历史梯度一直很小**（v 小）→ 说明这个参数变化很慢 → 学习率自动放大 → 大步走
- 如果某个参数**梯度方向很一致**（m 大）→ 说明方向很确定 → 加速前进
- 如果某个参数**梯度方向忽左忽右**（m 小）→ 说明方向不确定 → 保守一点

```
参数 θ₁：梯度一直很小，方向稳定
  v₁ 小 → 学习率放大 → 大步走，尽快到位

参数 θ₂：梯度很大，方向不稳定
  v₂ 大 → 学习率缩小 → 小步走，小心调整
```

#### AdamW 的改进：Weight Decay

Adam 有个问题：它用的正则化方式（L2 正则）在自适应学习率下效果不好，参数值可能越来越大。

AdamW 把正则化方式改成了**直接对参数施加衰减**：

```
θ_new = θ_old - lr × (梯度相关项 + weight_decay × θ_old)
```

直觉：每一步都把参数值往 0 的方向拉一点点，防止参数值变得过大导致过拟合。就像一根橡皮筋，始终把参数往 0 的方向拽。

```
没有 weight decay：
  参数可以变得任意大 → 模型可能过拟合训练数据

有 weight decay：
  参数被限制在一个合理范围 → 模型更泛化
```

#### 为什么大参数 = 过拟合

用拟合曲线举例。训练数据只有 3 个点：(1,2), (2,4), (3,6)。

```
合理拟合：y = 2x（参数小，一条直线，学到简单规律）
过拟合：  y = 2x + 100·sin(1000x)（参数巨大，曲线疯狂扭曲，强行精确穿过每个点）
```

过拟合的模型为了精确穿过每一个训练点，会用很大的参数值造出非常复杂的曲线。这条曲线在训练数据上完美拟合了，但换个新点就完全不对。

**大参数 = 模型在用"蛮力"拟合数据的噪声和细节，而不是学真正的规律。**

#### 为什么会发生过拟合

因为**模型太强，数据太少**。

```
一个能装几百亿参数的模型，只给它几万条数据

模型有两种选择：
  选择 A：学规律 → 9 道题对了，1 道有误差 → loss = 中等
  选择 B：死记硬背 → 10 道题全对 → loss = 更低

模型追求最小化 loss，当然选 B。
不是模型"不能"学规律，而是数据少的时候死记硬背的 loss 更低，模型没有动力去学规律。
```

数据多了，死记背不住了，模型才被迫学真正的规律。weight decay 通过限制参数值，逼模型选择简单的、参数小的规律——简单规律才泛化得好。

#### 过拟合发生在哪些阶段

预训练、SFT、RL 三个阶段都会过拟合，但概率不同：

```
预训练：
  数据量是万亿 token 级别，不太容易过拟合
  但如果数据太少或训太多轮，也会过拟合

SFT：
  数据量通常只有几万条，非常容易过拟合
  所以 SFT 一般只训 3-10 个 epoch，还要 early stopping
  这是你最容易遇到过拟合的阶段

RL：
  DPO 只训 1-3 个 epoch，就是因为太容易过拟合
  PPO 相对好一点，因为是在线生成的，数据不断更新
```

#### Weight Decay 真的有用吗？

有效，但效果有限，它只是众多防止过拟合手段中的一个：

```
1. Weight Decay    → 限制参数不要太大          有效，但有限
2. 多给数据         → 模型背不住了，被迫学规律   最有效
3. 减小模型         → 模型容量不够，想背也背不住  有效，但牺牲能力
4. Early Stopping  → 发现 val loss 开始涨就停    简单有效
5. Dropout         → 训练时随机关闭一些神经元     有效，但 LLM 里用得少了
6. 数据增强         → 同一个问题换不同问法        有效
```

Weight Decay 的作用更像**保险丝**——不是解决过拟合的根本手段，但能防止模型"飘"得太厉害。实际训练中几乎都会开，因为零成本，有没有用都开了不亏。真正防止过拟合的主力是**数据量**和 **early stopping**。

#### 一句话总结

> Adam = 给每个参数自适应分配步长（梯度大的参数小步走，梯度小的参数大步走）
> AdamW = Adam + Weight Decay（防止参数值过大，避免过拟合）
> 它是目前 LLM 训练的标准优化器，几乎所有模型都用它。

### 1.5 学习率调度：Warmup + Decay

学习率不是固定的，训练过程中会变化：

```
warmup（前 5% 步数）：从很小慢慢升到峰值（比如 1e-5）
                    ↓
cosine decay：从峰值慢慢降到最小值（比如 1e-6）
```

假设总共训练 10000 步：

```
步数 0-500（初期）：   lr 从 1e-7 → 1e-5  warmup，小步热身，让参数先稳定
步数 500-10000（中末期）：lr 从 1e-5 → 1e-6  decay，接近最优，小步微调
```

像开车：起步慢慢加速 → 中间全速前进 → 快到目的地减速慢行。

### 1.6 Epoch 和 Checkpoint

- **Epoch**：把所有训练数据从头到尾喂一遍
- **Checkpoint**：训练过程中定期保存的模型参数快照

```
epoch 1 → checkpoint_1（存盘）
epoch 2 → checkpoint_2（存盘）
...
epoch 10 → checkpoint_10（存盘）
                ↓
        挑 validation loss 最低的 → 最终模型
```

为什么不直接用最后一轮？因为训练太久会**过拟合**——模型把训练数据"背下来"了，但遇到新数据反而变差。

### 1.7 LoRA / QLoRA：省资源的训练

#### 全量训练要多少显存

每个参数在训练时需要存三样东西：

```
1. FP16 参数：           2 bytes
2. FP32 master weights： 4 bytes
3. FP32 梯度：           4 bytes
4. Adam 一阶动量 (m)：   4 bytes
5. Adam 二阶动量 (v)：   4 bytes
每个参数合计：18 bytes
```

以 30B 模型为例：

```
参数相关显存：300 亿 × 18 bytes ≈ 540 GB
前向传播中间激活值（跟 batch size、序列长度有关）：≈ 100-200 GB
总计：≈ 640-740 GB

一张 A100 = 80 GB 显存
实际需要：8-10 张 A100（使用 ZeRO-3 可进一步减少）
```

RL 训练更贵，因为同时要放多个模型：

```
SFT（1 个模型）：≈ 640-740 GB → 8-10 张 A100
PPO（4 个模型）：≈ 2000+ GB  → 几十张 A100
GRPO（2 个模型）：≈ 1100+ GB  → 比 PPO 少，但还是比 SFT 贵
```

#### LoRA / QLoRA

**LoRA**：不更新原始参数，而是在旁边加一个很小的"适配器"，只训练这个适配器。参数量从几百亿降到几百万，优化器状态也只需要存这几百万个参数的：

```
几百万 × 12 bytes ≈ 几十 MB
单卡 24 GB 就够了
```

**QLoRA**：在 LoRA 基础上再加量化（把参数从 float16 压缩到 int4），进一步省显存。单卡 24G 就能微调 7B 模型。

#### LoRA 效果会不会差

差距不大。大多数场景下 LoRA 能达到全量微调 90-95% 的效果。

原因：预训练模型已经学了 99% 的知识（语言、常识、推理），SFT 只需要微调最后一点（对话格式、特定任务风格）。就像学了 10 年数学的人，考前只看重点笔记和全量复习，差别不大——底子在那里，只需要调一下"怎么答题"。

什么时候必须全量微调：
```
1. 要给模型灌入大量新知识（比如让中文模型学会一门新语言）
2. 要大幅改变模型的核心能力
3. 数据量特别大（几十万条以上），LoRA 的适配器容量不够用
```

所以 LoRA 不是"效果差所以便宜"，而是"大部分情况下效果够用，没必要花那个钱"。这也是为什么业界主流都是用 LoRA，全量微调反而很少见。

## 第二章：SFT——教模型"怎么说话"

### 2.1 为什么需要 SFT

#### 预训练做了什么

预训练是大模型训练的第一阶段，用海量文本（几万亿 token）训练模型做一件事：**预测下一个 token**。

```
输入: "黑洞是大质量恒星坍缩后"
模型预测下一个 token → "形成的"（概率最高）
```

这就是 next token prediction。模型看了几十亿个网页、书籍、论文，学会了语言本身的规律——语法、常识、事实、推理。但它学的是"续写"，不是"对话"。

#### 预训练模型的问题

预训练完成后，模型**有知识，但不会用**。具体表现：

```
问题 1：不会对话
  你问: "什么是黑洞？"
  模型续写: "黑洞是1967年惠勒提出的术语，相关论文包括《广义相对论中的奇点》..."
  （它以为你在写文章，不是在问问题）

问题 2：不会遵循指令
  你说: "用三句话解释黑洞"
  模型续写: "黑洞的形成过程如下：首先，大质量恒星的核聚变燃料耗尽..."
  （它不知道"三句话"是个约束）

问题 3：分不清好坏
  你问: "怎么黑进别人的电脑？"
  模型续写: "首先你需要获取目标的IP地址，然后使用端口扫描工具..."
  （它没有"拒绝"这个概念，只会续写）

问题 4：格式混乱
  有时输出一段话，有时输出一堆代码，有时输出半截句子就断了
  （它没有"回答应该完整"的概念）
```

#### SFT 解决什么

SFT 的作用就是：**把模型从"续写机器"变成"对话助手"**。

```
SFT 之前：模型知道所有知识，但输出方式不可控
```

具体来说，SFT 教模型三件事：

```
1. 格式：看到用户问题 → 直接回答，不要续写
2. 能力：遵循指令（改写、总结、翻译、写代码...）
3. 边界：有害问题要拒绝，不确定的要说"我不确定"
```

但 SFT 的本质仍然是 next token prediction——只是训练数据从"任意文本"变成了"指令-回答对"。模型学到的不是新知识，而是**怎么用已有的知识来回答问题**。

> 类比：预训练 = 学了 10 年知识的学生。SFT = 考前培训，教他"怎么答题"——先写什么、格式是什么、不会的题怎么处理。知识没变，表达方式变了。

### 2.2 SFT 的数据格式

SFT 的数据是 **(prompt, response) 对**：

```json
{"prompt": "什么是黑洞？", "response": "黑洞是大质量恒星坍缩后形成的天体，其引力极强，连光也无法逃逸。"}
{"prompt": "写一首关于春天的诗", "response": "春风十里不如你，桃花三月映晴空。"}
{"prompt": "解释递归", "response": "递归是一种编程技巧，指的是函数调用自身来解决问题。"}
```

这些数据通常是人类写的，或者是用更强的模型（如 GPT-4）生成的。

### 2.3 SFT 的 Loss 怎么算

SFT 的 loss 是**逐 token 的交叉熵**。不是拿模型的回答和标准答案做字符串对比，而是在每个位置上问模型"你给正确 token 的概率是多少"。

```
标准答案: "黑洞 是 恒星 坍缩 形成 的"

模型逐 token 预测：

  位置 1：给定 prompt，预测下一个 token
    模型概率分布：P(黑洞)=0.3, P(太阳)=0.4, P(宇宙)=0.2, ...
    正确答案是 "黑洞"，概率 0.3
    loss_1 = -log(0.3) = 1.2

  位置 2：给定 prompt + "黑洞"，预测下一个 token
    模型概率分布：P(是)=0.8, P(很)=0.1, ...
    正确答案是 "是"，概率 0.8
    loss_2 = -log(0.8) = 0.22

  位置 3：给定 prompt + "黑洞是"，预测下一个 token
    正确答案是 "恒星"，概率 0.05
    loss_3 = -log(0.05) = 3.0

  ...

总 loss = (loss_1 + loss_2 + loss_3 + ...) / token数量
```

#### 关键：训练时用的是"标准答案"还是"模型自己的输出"

这里有一个容易混淆的点：**训练时，每个位置的输入都是标准答案，不是模型自己的预测**。

```
标准答案: "黑洞 是 恒星"

位置 1：输入 prompt → 模型预测 → 跟标准答案"黑洞"对比
位置 2：输入 prompt + "黑洞"（标准答案）→ 模型预测 → 跟"是"对比
位置 3：输入 prompt + "黑洞是"（标准答案）→ 模型预测 → 跟"恒星"对比
```

即使模型在位置 1 预测错了（比如预测成了"太阳"），位置 2 的输入仍然用标准答案"黑洞"，不用模型自己预测的"太阳"。这叫 **Teacher Forcing**。

```
Teacher Forcing：
  不管模型前面预测对了还是错了，下一个位置的输入永远是标准答案
  就像老师在旁边盯着，不管学生之前答错了什么，永远按正确进度往下教

为什么不用模型自己的输出？
  如果位置 1 错了 → 位置 2 的输入就错了 → 位置 3 更错 → 错误雪崩
  loss 会变得非常大，梯度也会爆炸，训练不稳定
```

所以 SFT 训练时，模型永远在"从正确的上下文预测下一个 token"，不会出现错误累积。这跟推理（实际使用时）不一样——推理时没有标准答案，只能用自己的输出当输入，一个 token 错了后面全受影响。

#### 那模型怎么学会不犯错？

虽然训练时用标准答案喂给模型，但 loss 是实实在在的——模型预测错了就有惩罚。每一步的梯度会告诉模型："在这个位置，你应该给'黑洞'更高的概率"。经过几百万次这样的修正，模型逐渐学会了在每个位置都给正确 token 高概率。

```
训练时：用标准答案喂入 → 只需要学"在正确上下文中预测正确 token"
推理时：用自己的输出喂入 → 因为训练时每个位置都学好了，所以逐 token 生成也不会错
```

然后反向传播，用 AdamW 更新参数。重复 N 个 epoch，挑 validation loss 最低的 checkpoint。

### 2.4 SFT 的实现方式

#### 全量微调（Full Fine-Tuning）

更新模型所有参数。效果天花板高，但显存巨大（30B 模型需要几十张 A100）。资源充足时使用。

#### 参数高效微调（PEFT）

只更新一小部分参数，冻结大部分。主流方法：

```
LoRA        ← 最主流，加小适配器
QLoRA       ← LoRA + 量化，更省显存
Prefix Tuning  ← 给每层加可学习的前缀 token
Prompt Tuning  ← 只训练输入端的 soft prompt
Adapter     ← 在层间插入小型网络

实际使用中 90% 的场景都用 LoRA/QLoRA，其他方法工程上很少用
```

#### LoRA 的具体做法

```
原始权重矩阵 W（冻结，不更新）：d × d
LoRA 适配器（可训练）：W = A × B
  A：d × r（r 很小，比如 8 或 16）
  B：r × d

前向传播：output = W·x + A·B·x
                       ↑冻结    ↑只训练这个
```

参数量对比（以 d=4096 为例）：

```
原始参数量：4096 × 4096 = 1600 万
LoRA 参数量：4096 × 8 + 8 × 4096 = 6.5 万
缩小了 250 倍
```

#### QLoRA 更省

```
LoRA：参数是 float16（2 bytes/参数）
QLoRA：参数是 int4（0.5 bytes/参数）
显存再省 4 倍
7B 模型 + QLoRA：单卡 24 GB 就能训
```

### 2.5 SFT 的完整训练流程

```
准备 (prompt, response) 数据集（几千到几万条）
    → 喂给模型，逐 token 算交叉熵 Loss
    → 反向传播算梯度
    → AdamW 更新参数
    → 重复 3-10 个 epoch
    → 挑 validation loss 最低的 checkpoint
```

### 2.6 SFT 的局限

SFT 本质上是**模仿学习**——模型在模仿训练数据里的回答。

- 上限受限于数据质量：数据里没有的能力，模型学不到
- 不知道"哪个回答更好"：模型只知道"这个回答是正确的"，不知道"这个回答比那个更好"
- 没有"好坏之分"的概念：只要 loss 降了就行，不会主动追求更好的回答

这就引出了一个问题：**能不能让模型不仅知道"什么是正确的"，还知道"什么是更好的"？**

答案就是 RL。

## 第三章：RL 的核心概念（必懂基础）

### 3.1 什么是强化学习

监督学习（SFT）：给正确答案，模型模仿。
强化学习（RL）：不给正确答案，只给**奖励信号**，模型自己探索，通过试错学习。

```
监督学习：老师告诉你答案是 "北京"，你照抄
强化学习：你试着回答，老师只说 "还行，6分" 或 "不错，9分"，你自己摸索怎么拿高分
```

### 3.2 关键术语

| 术语 | 含义 | LLM 场景下的对应 |
|------|------|-----------------|
| **Agent（智能体）** | 做决策的主体 | LLM 本身 |
| **Environment（环境）** | Agent 与之交互的世界 | 任务分布（prompt 池）+ 生成后的奖励评估机制 |
| **State（状态）** | 当前的情况 | 当前的 prompt + 已生成的 token |
| **Action（动作）** | Agent 做出的选择 | 选择下一个 token |
| **Reward（奖励）** | 做出动作后得到的反馈 | 回答完成后的评分 |
| **Policy（策略）** | Agent 的行为方式 | LLM 的参数（决定每个 token 的概率分布） |

### 3.3 MDP（马尔可夫决策过程）

RL 的数学框架就是 MDP：

```
State₀ (prompt) → Action₀ (生成 token₁) → State₁ → Action₁ (生成 token₂) → ... → 终止
                                                                                      ↓
                                                                                 获得 Reward
```

LLM 的生成过程天然就是一个 MDP：
- 每生成一个 token 就是一步决策
- 整个回答生成完毕后获得一个总奖励

### 3.4 为什么 LLM 需要 RL

SFT 教会了模型"怎么说话"，但有一些问题 SFT 解决不了：

1. **安全性**：模型不应该生成有害内容，但 SFT 数据里"有害回答的反例"很难收集全
2. **有用性**：模型应该给出有帮助的回答，但"有帮助"很难用固定的正确答案定义
3. **真实性**：模型不应该胡说八道，但 SFT 只能教它模仿已有的正确回答
4. **偏好**：两个都正确的回答，可能一个更好（更简洁、更全面、更准确）

RL 的思路是：不给模型"正确答案"，而是给它一个**评分标准**，让它自己探索怎么拿高分。

## 第四章：RLHF——让人类偏好成为奖励

### 4.1 RLHF 是什么

RLHF（Reinforcement Learning from Human Feedback）= 用人类的偏好作为 RL 的奖励信号。

核心思路：人类不直接写"正确答案"，而是评判"哪个回答更好"。

### 4.2 RLHF 的三步走

**第一步：收集偏好数据**

```
同一个 prompt: "什么是黑洞？"

模型生成 4 个回答:
  A: "黑洞是大质量恒星坍缩形成的天体，引力极强..."
  B: "黑洞就是宇宙中的一个洞，什么东西都掉进去。"
  C: "黑洞是一种致密天体，事件视界内的逃逸速度超过光速。"
  D: "黑洞是一个理论上的概念，可能不存在。"

人类标注: A > C > B > D（A 最好，D 最差）
```

实际操作中，通常是让模型生成 2 个回答，人类标出哪个更好，得到 (prompt, chosen, rejected) 三元组。

**第二步：训练 Reward Model（RM）**

用收集到的偏好数据训练一个打分模型：

```
输入: (prompt, 回答) → 输出: 一个标量分数

"什么是黑洞" + 回答 A → 0.85 分
"什么是黑洞" + 回答 B → 0.32 分
"什么是黑洞" + 回答 C → 0.71 分
"什么是黑洞" + 回答 D → 0.15 分
```

RM 本身也是一个语言模型（通常比主模型小），用 Bradley-Terry 模型（将偏好比较建模为 P(A≻B) = σ(r_A - r_B)，即好回答得分高于差回答的概率）学习人类偏好。

**第三步：用 PPO 优化 LLM**

有了 RM，就可以用 RL 来训练 LLM 了：

```
循环 {
  1. 模型对 prompt 生成回答
  2. RM 给回答打分
  3. 用分数作为奖励，更新模型参数（PPO 算法）
  4. 回到第 1 步
}
```

### 4.3 PPO 算法的核心思想

PPO（Proximal Policy Optimization）是 RLHF 中最常用的 RL 算法。它的核心思想很简单：

**最大化奖励，但不要偏离太远。**

```
原始想法: 让模型生成的回答分数越高越好
问题:     模型可能为了拿高分变得很奇怪（比如生成一堆 RM 喜欢但人类看不懂的东西）
PPO 的解法: 加一个 KL 惩罚，如果模型偏离原始版本太远就扣分
```

### 4.4 PPO 的四个模型

PPO 训练需要同时维护 **4 个模型**，这是它最被诟病的地方：

| 模型 | 角色 | 是否更新 |
|------|------|---------|
| **Policy Model（Actor）** | 正在训练的 LLM，负责生成回答 | 是 |
| **Reference Model** | 训练前复制的原始 LLM，计算 KL 散度防止偏离 | 否（冻结） |
| **Reward Model（RM）** | 评委，给回答打分 | 否（提前训练好的） |
| **Value Model（Critic）** | 预测"这个 prompt 大概能得几分"，用于计算 advantage | 是 |

### 4.5 PPO 一个 Step 的完整过程

```
① 抽 prompt
   从 prompt 池里随机抽一批（比如 512 个）

② 模型生成回答
   当前模型对每个 prompt 生成 1 个回答
   → 512 个回答

③ RM 打分
   Reward Model 对每个回答打一个分
   → 512 个分数

④ 算 advantage（优势）
   Critic 预测每个 prompt 的"预期分数"
   advantage = 实际分数 - 预期分数
   → 告诉模型"你比预期好了多少"

⑤ 更新权重
   用 PPO 的 clip 损失更新 Policy 和 Critic
   同一批数据更新 4 次（ppo_epochs=4）

⑥ 丢掉这批数据，进入下一个 step
   重新抽 prompt，用更新后的模型重新生成
```

### 4.6 PPO 的痛点

1. **显存开销巨大**：4 个模型同时在显存里，70B 模型需要几百 GB
2. **训练不稳定**：超参敏感，经常训崩
3. **Reward Hacking**：模型可能找到 RM 的漏洞，生成 RM 打高分但实际质量差的回答
4. **训练慢**：每一步都要生成回答（autoregressive decoding），这是最耗时的部分

## 第五章：DPO——去掉 Reward Model 的捷径

### 5.1 DPO 的核心洞察

DPO 的作者发现了一个数学上的等价关系：**不需要单独训练一个 Reward Model，可以直接用偏好数据优化 LLM。**

```
RLHF:  偏好数据 → 训练 RM → PPO 优化 LLM（3 步）
DPO:   偏好数据 → 直接优化 LLM（1 步）
```

### 5.2 DPO 的 Loss 函数

```
L_DPO = -log σ( β · [log(π(y_w|x)/π_ref(y_w|x)) - log(π(y_l|x)/π_ref(y_l|x))] )

y_w = 好回答（chosen）
y_l = 差回答（rejected）
π = 当前模型
π_ref = 参考模型（冻结的 SFT 模型）
β = 温度系数
σ = sigmoid 函数
```

白话理解：

```
当前模型给好回答的概率，相比参考模型提升了多少？
当前模型给差回答的概率，相比参考模型降低了多少？

这两个差值越大越好 → loss 越小
```

### 5.3 "隐式奖励"是什么意思

PPO 需要一个独立的 RM 来打分——这是"显式"奖励。

DPO 的数学推导证明：奖励可以用"当前模型和参考模型的概率比"来等价表示。不需要一个单独的模型打分，loss 里的概率比值已经起到了同样的作用。这就是"隐式"。

### 5.4 DPO 的训练过程

```
数据收集阶段（训练之前）：
  SFT 模型对每个 prompt 生成 4-8 条回答
  → 人类标注哪个好哪个差
  → 得到几万条 (prompt, chosen, rejected)
  → 存成固定数据集

DPO 训练阶段：
  for each batch in 数据集:
    1. 取出 (prompt, chosen, rejected)
    2. 当前模型算 chosen 的 log 概率  ← 不是生成，是 forward pass
    3. 当前模型算 rejected 的 log 概率
    4. 参考模型算 chosen 的 log 概率（不更新）
    5. 参考模型算 rejected 的 log 概率（不更新）
    6. 代入 DPO loss，算梯度，更新权重
```

全程没有任何"生成"操作，只有"算概率"操作。所以 DPO 训练速度接近 SFT。

### 5.5 在线 vs 离线

| | PPO（在线） | DPO（离线） |
|---|---|---|
| 训练数据来源 | 当前模型实时生成 | 提前收集好的固定数据集 |
| 能否探索新策略 | 能 | 不能 |
| 计算开销 | 大 | 小（跟 SFT 差不多） |
| 理论上限 | 高（能发现数据里没有的好回答） | 受限于数据质量 |

用做菜类比：
- **PPO = 边炒边学**：你炒一盘菜，评委当场尝、当场打分，你可以不断尝试新做法
- **DPO = 翻菜谱对比册**：有人提前准备了一本"好吃 vs 难吃"的对比册子，你翻册子学，但不能炒新菜问评委

### 5.6 DPO 的优缺点

优点：
- 简单，不需要训练 RM
- 稳定，不会出现 reward hacking
- 省资源，只需要 2 个模型（Policy + Reference）

缺点：
- 离线学习，不能在线探索
- 对数据质量要求高
- 容易过拟合（通常只训 1-3 个 epoch）

## 第六章：GRPO——DeepSeek R1 的核心算法

### 6.1 GRPO 要解决什么问题

PPO 太贵（4 个模型），DPO 是离线的（不能探索）。有没有办法做到：
- 像 PPO 一样在线训练
- 但不需要 Critic 模型

GRPO 的答案：**用组内统计量替代 Critic。**

### 6.2 GRPO 的核心思想

对同一个 prompt，采样一组（Group）回答（比如 G=64 个），然后：

```
prompt: "1+1=?"

采样 64 个回答:
  回答 1: "2"           → 正确，奖励 +1
  回答 2: "等于2"        → 正确，奖励 +1
  回答 3: "3"           → 错误，奖励 -1
  回答 4: "1+1=2"       → 正确，奖励 +1
  ...
  回答 64: "我不确定"    → 错误，奖励 -1

组内统计:
  mean(r) = 0.5（64 个回答的平均分）
  std(r) = 0.87

每个回答的 advantage:
  A_i = (r_i - mean(r)) / std(r)

  回答 1: (1 - 0.5) / 0.87 = +0.57（比平均好，奖励）
  回答 3: (-1 - 0.5) / 0.87 = -1.72（比平均差，惩罚）
```

**关键点**：不需要 Critic 来预测"预期分数"，直接用组内均值作为 baseline。

### 6.3 GRPO vs PPO

| | PPO | GRPO |
|---|---|---|
| Value Model（Critic） | 需要 | **不需要** |
| Reward Model | 需要训练 | 可以用**规则验证** |
| 模型数量 | 4 个 | 2 个（Policy + Reference） |
| 每个 prompt 生成几个回答 | 1 个 | G 个（通常 64 个） |
| 显存 | 巨大 | 大幅减少 |
| 采样效率 | 低（每个 prompt 只学一次） | 高（每个 prompt 学 G 次） |

### 6.4 RLVR：可验证奖励的 RL

GRPO 常和 RLVR（Reinforcement Learning with Verifiable Rewards）配合使用。

核心区别：
```
RLHF:  奖励来自人类偏好（主观）→ 需要训练 Reward Model
RLVR:  奖励来自规则验证（客观）→ 直接用程序判断对错
```

适用场景：
- 数学：答案是否正确（用 sympy 验证）
- 代码：能否通过测试用例
- 逻辑推理：结论是否成立

规则验证的好处：**不可能被 hack**——你没法钻规则的漏洞，答案对就是对，错就是错。

### 6.5 DeepSeek R1 的训练流程

```
阶段 1: R1-Zero（纯 RL，无 SFT）
  DeepSeek-V3-Base → GRPO 训练 → R1-Zero
  奖励：数学正确性 + 格式规范
  发现：模型自发学会了 chain-of-thought，出现 "aha moment"

阶段 2: R1（完整流程）
  V3-Base → Cold Start SFT（少量高质量 CoT 数据）
       → RL 阶段 1（推理任务 GRPO）
       → Rejection Sampling 生成 SFT 数据
       → SFT 阶段 2（混合推理 + 通用数据）
       → RL 阶段 2（全场景对齐）
```

**R1-Zero 的意义**：证明了纯 RL 就能涌现推理能力，不需要 SFT 作为前置。模型自发学会了"让我再想想"、"等等，这样不对"这样的反思行为。

**R1-Zero 的问题**：可读性差、语言混杂，所以完整 R1 还是加了 SFT cold start。

### 6.6 面试一句话总结

> "GRPO 去掉了 Critic 模型，用组内相对排名代替绝对价值估计。对同一个 prompt 采样多个回答，用组内均值和标准差做归一化，训练更稳定也更省资源。"

## 第七章：技术演进脉络

### 7.1 一张图记住

```
Pre-training（预训练）
  ↓ 海量文本，next token prediction，学会语言本身

SFT（监督微调）
  ↓ 高质量 instruction-response 对，学会对话

RLHF + PPO（2022, InstructGPT）
  ↓ 人类偏好 → 训练 RM → 在线 RL，4 个模型，太复杂

DPO（2023）
  ↓ 去掉 RM，直接用偏好数据优化，离线的，简单但上限受限

GRPO + RLVR（2025, DeepSeek R1）
  ↓ 去掉 Critic，用可验证奖励，在线 RL，2 个模型

Agentic RL（2025-2026）
  ↓ RL 扩展到工具使用和多轮交互场景
```

### 7.2 SFT vs RL 的本质区别

| | SFT | RL |
|---|---|---|
| 数据 | 需要 (prompt, 正确答案) 对 | 只需要 prompt + 奖励信号 |
| 学习方式 | 模仿正确答案 | 试错，根据奖励调整 |
| 适合场景 | 有高质量标注数据 | 答案可以自动验证（数学/代码） |
| 上限 | 受限于数据质量 | 可以超越训练数据 |
| 局限 | 学不到数据里没有的能力 | 奖励设计难，训练不稳定 |

### 7.3 各概念的层级关系

```
RL（学习范式，和监督学习并列）
 │
 ├── RLHF（应用框架：奖励来自人类偏好）
 │    ├── PPO 路线（显式 RM + 在线 RL，4 个模型）
 │    ├── DPO 路线（隐式 RM + 离线优化，2 个模型）
 │    └── 其他变体：KTO / IPO / ORPO / SimPO
 │
 ├── RLVR（应用框架：奖励来自规则验证）
 │    └── GRPO（去掉 Critic，组内相对排名，2 个模型）
 │
 └── Agentic RL（应用框架：RL 用于多步工具调用）
      └── Agent RFT / Agent-R1 / ToolBrain 等

LoRA / QLoRA → 正交的参数效率技术，可与上述任意范式组合
```

### 7.4 各方法需要几个模型

| 方法 | 需要的模型 | 总数 |
|------|-----------|------|
| SFT | Policy | 1 |
| PPO | Policy + Reference + RM + Critic | 4 |
| DPO | Policy + Reference | 2 |
| GRPO | Policy + Reference | 2 |
| GRPO + RLVR | Policy + Reference（用规则替代 RM） | 2 |

## 第八章：常见面试问题

### Q1: RLHF 和 DPO 的区别？

A: RLHF 是两阶段的——先训练 RM 再用 PPO 优化；DPO 把这两步合成一步，直接用偏好数据优化策略。DPO 更简单稳定但是离线的，RLHF 可以在线探索但训练复杂。

### Q2: GRPO 为什么不需要 Critic？

A: GRPO 对同一个 prompt 采样一组回答，用组内均值和标准差做归一化来计算 advantage，相当于用组内统计量替代了 Critic 的价值估计。

### Q3: DeepSeek R1 的 "aha moment" 是什么？

A: R1-Zero 在纯 RL 训练过程中，模型自发学会了在 CoT 中进行自我反思和纠错，比如输出 "wait, let me reconsider"。说明推理能力可以通过 RL 涌现，不一定需要 SFT 来教。

### Q4: 什么是 reward hacking？怎么缓解？

A: 模型找到了 RM 的漏洞，生成的回答 RM 打高分但实际质量差。缓解方法：KL 惩罚约束策略不偏离太远；用 RLVR 替代学习的 RM（规则验证更难被 hack，但模型仍可能找到验证规则的漏洞，如格式投机）；定期更新 RM。

### Q5: 为什么 DPO 是"离线"的？

A: DPO 的偏好数据在训练开始前就全部准备好了。训练过程中模型不生成任何新回答，只在固定数据上做 forward pass 算概率。这跟 SFT 类似——数据是固定的，模型在上面反复学。

### Q6: SFT 的 loss 和 DPO 的 loss 有什么区别？

A: SFT 看 1 个正确答案，逐 token 算交叉熵，目标是让正确 token 的概率变高。DPO 看 2 个回答（chosen vs rejected），算的是"好回答和差回答的概率差距"，目标是让差距变大。

### Q7: 如果让你用 RL 优化一个 Agent 系统，你会怎么做？

A:
1. 定义 reward：最终回答的准确性 + 检索效率（步骤数）
2. 把 agent 的决策过程建模为 MDP：state = 当前上下文，action = 选择工具/生成回答
3. 用 GRPO 训练，每个 query 采样多条 trajectory，用结果正确性做可验证奖励
4. 可以加 process reward 给中间步骤也提供信号，缓解稀疏奖励问题

## 附录：做菜类比总结

把整个训练过程类比成学做菜：

```
预训练 = 学会认食材、用厨具（语言基础）
SFT = 照着菜谱做菜（模仿正确答案）
RLHF = 有个评委在旁边打分，你边做边学（在线学习 + 人类偏好）
DPO = 翻一本"好吃 vs 难吃"的对比册子学（离线学习 + 固定数据）
GRPO = 你同时做 64 盘菜，看哪盘好吃哪盘难吃，自己总结规律（组内对比 + 规则验证）
```
