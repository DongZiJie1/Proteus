# Proteus 记忆系统设计

## 1. 设计目标

记忆系统是 Agent 的"大脑"，统一管理所有层级的记忆。类比人类认知：

- **上下文窗口** = 工作记忆（当前正在处理的信息，容量有限）
- **会话历史** = 短期记忆（本次对话的完整记录，会话结束清空）
- **记忆文件** = 长期记忆（跨会话沉淀的知识，持久存储）

三者不是独立系统，而是同一套记忆体系的不同层级。上下文是最活跃的表层，记忆文件是最深的底层，autodream 是连接两者的沉淀机制。

## 2. 记忆架构

```
┌─────────────────────────────────────────────────────────────┐
│                      上下文窗口（工作记忆）                    │
│                                                             │
│  ┌───────────────┐  ┌───────────────┐  ┌─────────────────┐  │
│  │ 系统提示词     │  │ MEMORY.md 索引 │  │ 用户指令         │  │
│  └───────────────┘  └───────────────┘  └─────────────────┘  │
│  ┌───────────────┐  ┌───────────────┐  ┌─────────────────┐  │
│  │ 对话历史(近期) │  │ 工具调用结果   │  │ 读取的记忆文件   │  │
│  └───────────────┘  └───────────────┘  └─────────────────┘  │
│                                                             │
│  ↑ 信息不断涌入                      容量有限，需要压缩管理 ↓ │
└─────────────────────────────────────────────────────────────┘
                            │
                            │ autodream（定期沉淀）
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                    长期记忆（持久存储）                        │
│                                                             │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐    │
│  │  user    │  │ feedback │  │ project  │  │reference │    │
│  │ 用户画像 │  │ 行为反馈 │  │ 项目动态 │  │ 外部资源 │    │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘    │
│                                                             │
│  按文件存储，MEMORY.md 为索引，每次会话自动加载索引            │
└─────────────────────────────────────────────────────────────┘
                            │
                            │ 会话历史归档
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                    对话历史（JSONL 归档）                      │
│                                                             │
│  完整对话记录，JSONL 格式逐条存储，支持检索、回放、/insights   │
└─────────────────────────────────────────────────────────────┘
```

## 3. 上下文管理（工作记忆）

上下文窗口是 Agent 每次调用 LLM 时实际发送的内容，容量有限，需要精心管理。

### 3.1 上下文组成

| 组成部分 | 是否始终加载 | 说明 |
|----------|:----------:|------|
| 系统提示词 | 是 | Agent 角色定义、行为规范 |
| MEMORY.md 索引 | 是 | 记忆目录摘要，200 行以内 |
| 用户指令 | 是 | 用户当前消息 |
| 相关记忆文件 | 按需 | 根据话题从索引判断是否需要读取 |
| 对话历史 | 滚动保留 | 最近 N 轮完整对话 |
| 工具调用结果 | 最近 | 最近一次工具执行结果 |
| 工具结果摘要 | 历史 | 早期工具结果替换为一句话摘要 |

### 3.2 上下文压缩策略

当上下文接近窗口上限时，按以下顺序压缩：

1. **工具结果摘要化** — 旧的工具结果（如截图、长文本）替换为一句话描述
2. **对话历史摘要化** — 早期对话替换为 LLM 生成的摘要
3. **记忆文件卸载** — 不相关的记忆文件从上下文移除（索引仍在）
4. **截断最早内容** — 丢弃最早的消息（最后手段）

压缩前触发 `pre_compact` Hook，压缩后触发 `post_compact` Hook。

### 3.3 工具结果的临时性

工具执行结果（截图、文件内容、命令输出等）是临时的上下文内容：

- 当前轮次：完整保留在上下文中
- 后续轮次：保留文本摘要，大附件（图片）替换为描述
- 会话结束后：原始结果不保留，有价值的信息通过 autodream 沉淀到记忆

## 4. 长期记忆（持久存储）

### 4.1 记忆存储结构

```
~/.proteus/
├── config.yaml                    # 全局配置
├── memory/                        # 全局记忆（跨项目共享）
│   ├── user_preferences.md
│   └── ...
└── projects/
    └── {project_hash}/
        ├── memory/                # 项目记忆
        │   ├── MEMORY.md          # 记忆索引（每次会话自动加载）
        │   ├── user_role.md
        │   ├── feedback_testing.md
        │   ├── project_auth_rewrite.md
        │   └── ...
        └── conversations/         # 对话历史归档（JSONL）
            ├── 2026-04-28_143000.jsonl
            ├── 2026-04-28_150000.jsonl
            └── ...
```

### 4.2 记忆类型

#### user — 用户画像

描述用户的角色、目标、责任、知识背景。帮助 Agent 针对不同用户调整沟通方式和建议深度。

**何时记录：**
- 用户提到自己的职业/角色
- 用户表达技术偏好或经验水平
- 用户说明自己的工作职责

**示例：**

```
---
name: user-role
description: 用户是后端开发，主攻 Go，前端经验少
type: user
---

用户是后端开发工程师，5 年 Go 经验，对 React 不太熟悉。
涉及前端的解释用后端类比会更有效。
```

#### feedback — 行为反馈

用户对 Agent 行为的纠正或肯定。这是最重要的记忆类型——避免重复犯错，也避免丢掉已验证的好做法。

**何时记录：**
- 用户纠正 Agent 的做法（"不要这样做"、"别用那个"）
- 用户肯定 Agent 的做法（"对，就是这样"、"这个方案不错"）
- 用户表达对某种风格的偏好

**结构：** 规则本身 → **Why:**（原因）→ **How to apply:**（何时/如何应用）

**示例：**

```
---
name: feedback-no-mocks
description: 集成测试必须用真实数据库，不能 mock
type: feedback
---

集成测试必须连真实数据库，不要 mock 数据库层。

**Why:** 上个季度 mock 测试全过了但线上迁移挂了，mock 和真实行为不一致。

**How to apply:** 写测试时默认用测试数据库，不要用 unittest.mock 做数据库层 mock。
```

#### project — 项目动态

项目中正在进行的工作、目标、决策、事件。这类记忆时效性强，需要保持更新。

**何时记录：**
- 用户提到谁在做什么、为什么做、什么时候完成
- 项目有新的约束或决策
- 出现紧急事件或变更

**结构：** 事实 → **Why:**（动机）→ **How to apply:**（如何影响建议）

**示例：**

```
---
name: project-auth-migration
description: 认证中间件正在重写，原因是合规要求
type: project
---

认证中间件正在重写，旧版已标记废弃。

**Why:** 法务指出旧中间件的 session token 存储方式不符合新的合规要求。

**How to apply:** 涉及认证相关的建议优先考虑合规性，不要为了易用性牺牲安全性。
```

#### reference — 外部资源

外部系统和资源的位置指针。当用户提到某个外部系统时，Agent 知道去哪里找。

**何时记录：**
- 用户提到某个工具/平台的位置或用途
- 项目使用外部系统进行管理

**示例：**

```
---
name: ref-pipeline-bugs
description: Pipeline 相关 bug 在 Linear 的 INGEST 项目中跟踪
type: reference
---

Pipeline 相关的 bug 在 Linear 的 INGEST 项目中跟踪。
```

### 4.3 对话历史（JSONL 归档）

每次会话的完整对话记录以 JSONL 格式持久化存储，一行一条消息。

**存储路径：** `~/.proteus/projects/{project_hash}/conversations/YYYY-MM-DD_HHMMSS.jsonl`

**每行 JSON 结构：**

```json
{
  "role": "user",
  "content": "帮我看看桌面有什么文件",
  "timestamp": "2026-04-28T14:30:05Z",
  "channel": "cli",
  "attachments": [],
  "tool_calls": [],
  "tool_results": []
}
```

```json
{
  "role": "assistant",
  "content": "桌面有以下文件...",
  "timestamp": "2026-04-28T14:30:08Z",
  "channel": "cli",
  "tool_calls": [{"id": "tc_001", "name": "list_files", "arguments": {"path": "~/Desktop"}}],
  "tool_results": [{"tool_call_id": "tc_001", "content": "report.pdf\nnotes.txt", "is_error": false}]
}
```

**用途：**
- `/history` — 列出历史会话，选择某个会话查看详情
- `/insights` — 分析历史会话，提取使用模式、高频话题、工具偏好、token 消耗趋势
- 会话回放 — 加载历史会话恢复上下文，继续之前的对话
- autodream 输入 — 自动沉淀机制从历史会话中提取记忆

**与记忆的区别：**
- 对话历史是原始记录，忠实地保存每一句话和每一次工具调用
- 记忆是从对话历史中提炼的结构化知识
- 对话历史可以很大，记忆必须精简
- 对话历史不加载到上下文，记忆索引始终加载

### 4.4 记忆索引（MEMORY.md）

`MEMORY.md` 是记忆目录的索引文件，每次会话启动时自动加载到上下文中。

**格式要求：**
- 每条记忆一行，格式：`- [标题](文件名.md) — 一句话摘要`
- 不超过 200 行（超出截断）
- 按主题组织，不按时间排列
- 索引只放指针，不放内容

**示例：**

```markdown
- [用户角色](user_role.md) — 后端开发，Go 为主，前端经验少
- [测试偏好](feedback_testing.md) — 集成测试必须用真实数据库，不要 mock
- [认证重写](project_auth_rewrite.md) — 认证中间件重写中，合规驱动
- [Linear INGEST](ref_pipeline_bugs.md) — Pipeline bug 在 Linear INGEST 项目
```

## 5. Autodream（自动沉淀机制）

Autodream 是记忆系统的"反刍"过程——Agent 在后台定期回顾近期对话，从中提炼有价值的信息沉淀为长期记忆。

### 5.1 设计原理

人类不是在经历的当下就形成长期记忆的，而是在事后（尤其是休息时）对经历进行整理和巩固。Autodream 模拟这个过程：

- **不做实时判断** — 不在每轮对话中判断是否要写记忆（会打断主流程）
- **定期批量回顾** — 每隔 N 轮对话或会话结束时，一次性回顾近期内容
- **异步独立执行** — 用独立的 LLM 调用做记忆决策，不占用主 Agent 的上下文

### 5.2 触发时机

| 触发条件 | 说明 |
|----------|------|
| 每 N 轮对话 | 可配置，默认每 10 轮触发一次 |
| 会话结束时 | 会话关闭前对本次对话做最终回顾 |
| 用户显式触发 | 用户说"整理一下记忆"、"记住刚才的讨论" |
| 上下文压缩时 | 压缩上下文前先做一次 autodream，避免有价值信息随压缩丢失 |

### 5.3 执行流程

```
触发 autodream
    │
    ▼
收集输入：最近 N 轮对话 + 工具调用记录
    │
    ▼
调用 LLM（独立上下文，不占用主 Agent 窗口）
    │
    ▼
LLM 输出决策：
    ├── 无值得记忆的内容 → 结束
    ├── 写入新记忆 → 创建记忆文件 + 更新 MEMORY.md
    ├── 更新已有记忆 → 修改对应记忆文件
    └── 验证已有记忆 → 标记过时的 project 记忆
    │
    ▼
更新 MEMORY.md 索引
```

### 5.4 Autodream 提示词

```
你是一个记忆管理器。你的任务是回顾最近的对话，决定是否有值得长期记住的信息。

回顾以下对话内容，判断是否有：

1. 关于用户的新信息（角色、偏好、知识背景）→ type: user
2. 用户对 Agent 行为的纠正或肯定 → type: feedback
3. 项目的新动态、决策、约束 → type: project
4. 新的外部系统或资源位置 → type: reference

对于每条有价值的信息，输出：
- action: create / update / validate / delete
- type: 记忆类型
- name: 记忆文件名
- content: 记忆内容（含 frontmatter）
- index_entry: MEMORY.md 中的一行索引

如果没有什么值得记住的，输出 {"memories": []}

注意：
- 不要记录可以从代码或 git 中获取的信息
- 不要记录临时性的操作细节
- feedback 类型要包含 Why 和 How to apply
- project 类型要标注时间
- 检查是否有与现有记忆冲突的内容，如有则输出 update
```

### 5.5 异步执行

- Autodream 在后台异步执行，不阻塞用户交互
- 如果 autodream 正在执行时用户发来新消息，autodream 继续完成但结果延迟应用
- autodream 执行期间的对话仍在正常进行，结果在下一次需要时生效
- autodream 失败不影响主流程，仅记录日志

## 6. 记忆生命周期

### 6.1 写入触发

| 触发方式 | 场景 | 记忆类型 |
|----------|------|----------|
| **用户显式** | "记住 XXX"、"以后不要 XXX" | 按内容判断 |
| **实时检测** | 用户纠正/肯定 Agent 做法 | feedback |
| **实时检测** | 用户提到角色/偏好/职责 | user |
| **Autodream** | 每 N 轮对话定期回顾 | 按内容判断 |
| **Autodream** | 会话结束前最终回顾 | 按内容判断 |
| **Autodream** | 上下文压缩前回顾 | 按内容判断 |

实时检测 vs Autodream 的分工：
- **实时检测**只捕获明确的信号（显式指令、强语气纠正/肯定）
- **Autodream**负责捕获隐含的信息（对话中自然流露的偏好、项目细节等）
- 两者互补，不重复——Autodream 会检查实时检测已写入的记忆，避免重复

### 6.2 读取策略

1. 会话启动 → 加载 `MEMORY.md` 索引到上下文
2. 用户发消息 → 根据消息内容判断哪些记忆相关 → 读取对应记忆文件
3. Agent 做决策时 → 参考相关记忆调整行为

**相关性判断：**
- 消息提到测试 → 读取 feedback_testing.md
- 消息涉及认证模块 → 读取 project_auth_rewrite.md
- 消息问"bug 在哪" → 读取 ref_pipeline_bugs.md

### 6.3 更新与删除

- 记忆过时或错误时，更新或删除对应文件，同步更新 MEMORY.md 索引
- 用户说"忘掉 XXX"时，找到并删除对应记忆
- project 类记忆衰减快，autodream 会定期 validate，过时的标记或删除
- 写入新记忆前先检查是否有可更新的旧记忆，避免重复

## 7. 不该存入记忆的内容

- 代码模式、架构、文件结构 — 读代码就能知道
- Git 历史、最近改动 — git log 就能查
- 调试方案、修复步骤 — 修完就完了，代码里有答案
- 已在配置文件中记录的内容 — 不要重复
- 正在进行的临时任务 — 用 Task 管理，不是记忆
- 一次性的工具调用结果 — 上下文中临时保留即可

## 8. 安全与隐私

- 记忆文件存储在本地 `~/.proteus/` 目录，不上传到任何服务
- 记忆文件不纳入项目 git 仓库（加入 .gitignore）
- 不记录密码、token、API key 等敏感信息
- 用户可随时查看、编辑、删除任何记忆
- autodream 的 LLM 调用内容不包含敏感配置信息
