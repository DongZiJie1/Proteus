### 1.1 大纲注入 Prompt 的 Token 爆炸问题

**面试官原话：**
"你说把文档大纲 JSON 树注入 Agent 的 Prompt。好，一份 500 页的技术手册，大纲可能有 200+ 个节点。你把这棵树序列化成文本塞进 Prompt，光大纲就吃掉 3000-5000 token。如果用户同时关联了 5 份文档呢？光大纲就 15000-25000 token 了，还没算 System Prompt、Action 历史、工具定义。你的小模型上下文窗口才多大？你是怎么控制 Prompt 总长度的？大纲是全量注入还是按需加载？"

**问题本质：** 考你有没有真正在生产环境跑过这套方案。大纲注入是你的核心创新，但 Token 预算管理是绕不开的工程问题。

**回答思路：**

1. 单文档场景：大纲不是全量注入的，而是根据标题总数做自适应裁剪。我在注入前会统计大纲树的总标题节点数：
   - **标题数 < 100**：全量注入完整大纲树（经序列化压缩后通常 800-1500 token，完全可控）。Agent 可以直接看到所有层级，一步定位到叶子章节。
   - **标题数 ≥ 100**：裁剪所有叶子节点，只保留父级标题（即有子节点的标题）。这样 200+ 节点的大纲裁剪后通常只剩 40-60 个父标题，Token 消耗降到 500-800。
   - 这个 100 的阈值是实测调出来的——100 以下全量注入时大纲 Token 占比不超过 Prompt 总量的 15%，对小模型的上下文窗口压力可控；100 以上如果全量注入，大纲本身就可能吃掉 2000-3000 token，挤占工具返回和历史记录的空间。
   - 关键设计：**Agent 不需要理解层级结构，只需要输出它认为和问题相关的标题行号（可以一次输出多个）**。系统侧拿到这些行号后，自动展开每个标题到其所有叶子节点，收集叶子节点集合后做去重（比如 Agent 同时选了一级标题 A 和 A 下面的某个子标题，两者展开后叶子必然有重叠，系统自动去重），最终用去重后的叶子节点集合去 `read_section` 拿内容。这样做的好处是：LLM 的决策负担从"精确选择正确层级的正确章节"降到了"粗筛哪些标题可能相关"，对小模型尤其友好——它不需要分析缩进层级的含义，只需要看标题文本判断相关性，系统侧负责精确展开和去重。
2. 大纲序列化的 Token 优化——行号索引替代章节名输出：
   - 大纲注入时，每个标题前加 `# + 行号` 前缀（如 `#12 3.2 消防设计要求`）。Agent 在工具调用时只需输出行号（如 `read_section(line=12)`），而不是输出完整的章节标题文本。
   - 三个收益：① **省 Token**——输出一个数字比输出一段中文标题省得多，Agent 跑 5 轮每轮都要输出工具调用参数，累计节省可观；② **防幻觉**——小模型输出长标题时经常漏字、多字、改字（比如把"消防设计要求"输出成"消防设计规范"），用行号是精确匹配，不存在这个问题；③ **解析确定性**——行号是整数，正则一提就出来，不需要做模糊匹配。
   - 兜底机制：正常路径通过行号精确查找对应标题内容；如果模型没有按预期输出行号而是输出了标题文本（小模型偶尔会这样），系统会 fallback 到标题文本模糊匹配（去除空格和标点后做子串匹配），仍然能定位到正确章节。两条路径都走不通才报格式错误进入纠错流程。
3. 多文档场景：根本不走"把多份大纲塞进同一个 Prompt"的路线。我的架构是多 Agent 并行——Planner 根据用户问题和文档摘要做文档级分配，每份文档由一个独立的 Reader Agent 处理，各自只注入自己那份文档的大纲。5 份文档 = 5 个 Reader，每个 Reader 的 Prompt 里只有 1 份大纲，Token 压力和单文档场景一样。
4. 单文档内的大纲序列化做了压缩：去掉冗余字段，只保留 `id`、`title`、`children` 三个字段，用缩进文本而非完整 JSON 格式。
5. 所以面试官假设的"5 份文档大纲 15000-25000 token 塞进一个 Prompt"这个前提在我的架构里不成立。

**参考话术：**
"这个问题要分两个层面回答。单文档场景下，大纲不是无脑全量塞的，我做了基于标题数量的自适应裁剪。注入前先统计大纲树的总节点数——如果标题数小于 100，全量注入，经过序列化压缩后通常也就 800-1500 token，完全可控。如果标题数大于等于 100，比如您说的 200+ 节点的大型技术手册，我会裁剪掉所有叶子节点，只保留父级标题。200 个节点裁剪后通常只剩 40-60 个父标题，Token 消耗降到 500-800。100 这个阈值是实测调出来的，100 以下全量注入时大纲 Token 占比不超过 Prompt 总量的 15%，对小模型窗口压力可控。

这里有一个关键的设计决策：Agent 不需要理解大纲的层级结构，它只需要输出它认为和问题相关的标题行号，而且可以一次输出多个。系统侧拿到这些行号后，自动展开每个标题到所有叶子节点，做去重——比如 Agent 同时选了一级标题 A 和 A 下面的某个子标题，两者展开后叶子必然有重叠，系统自动去掉重复的——然后用去重后的叶子集合去拿内容精读。这样 LLM 的决策负担从'精确选择正确层级的正确章节'降到了'粗筛哪些标题可能相关'，对小模型尤其友好。它不需要分析缩进层级的含义，只需要看标题文本判断相关性，精确展开和去重是系统侧的确定性逻辑，不依赖 LLM。

另外大纲的序列化格式我也做了针对性优化。每个标题前加一个行号前缀，比如'#12 3.2 消防设计要求'。Agent 在调用工具时只需要输出行号——`read_section(lines=[12, 25, 38])`，而不是输出完整的章节标题文本。这个设计有三个收益：第一省 Token，输出几个数字比输出几段中文标题省得多，Agent 跑 5 轮每轮都有工具调用参数，累计节省可观；第二防幻觉，小模型输出长标题时经常漏字多字，比如把'消防设计要求'输出成'消防设计规范'，用行号就是精确匹配不存在这个问题；第三解析确定性高，行号是整数正则一提就出来。当然我也做了兜底——如果模型没按预期输出行号而是输出了标题文本，系统会 fallback 到标题文本模糊匹配，去除空格标点后做子串匹配，仍然能定位到正确章节。两条路径都走不通才进入格式纠错流程。

但更关键的是多文档场景——您说的 5 份文档同时关联，我的架构根本不会把 5 份大纲塞进同一个 Prompt。我用的是多 Agent 并行架构：Planner 先做文档级分配，每份相关文档分配一个独立的 Reader Agent，各自只处理自己那份文档的大纲。5 份文档就是 5 个 Reader 并行跑，每个 Reader 的 Prompt 里只有 1 份大纲，Token 压力和单文档完全一样。最后由 Synthesizer 汇总所有 Reader 的提取结果。所以您假设的'光大纲就 15000-25000 token'这个前提在我的架构里不成立——我是通过任务分解把多文档问题拆成了多个单文档问题，而不是在一个 Prompt 里硬塞。"

### 1.2 Java 调 Python 的跨语言架构开销

**面试官原话：**
"你说核心 Agent Loop 用 Java 原生实现，但 MinerU 是 Python 的，vLLM 也是 Python 的，ES 客户端 Java 倒是有。那你的 Java Agent 怎么调 vLLM？HTTP？gRPC？每一轮 Agent 决策都要发一次 HTTP 请求到 Python 侧的 vLLM 服务？你知道 HTTP 的连接建立、序列化反序列化、网络往返，每次至少加 5-10ms 的 overhead 吧？Agent 跑 5 轮就是 50ms 的纯通信开销。你为什么不直接用 Python 写 Agent，省掉这层跨语言调用？你说的'避免跨语言调用开销'到底避免了什么？"

**问题本质：** 你简历里说"避免引入 Python 框架带来的跨语言调用开销"，但实际上你还是在跨语言调用。面试官在检验你的说法是否自洽。

**回答思路：**

1. 澄清"避免跨语言开销"指的是避免在 Agent 编排层引入 Python 进程——如果用 LangChain，Java 微服务需要调 Python Agent 服务，Python Agent 再调 vLLM，多了一跳。
2. 现在的架构是 Java Agent 直接通过 HTTP 调 vLLM 的 OpenAI 兼容 API，少了 Python Agent 这个中间层。
3. vLLM 的 HTTP 调用延迟（5-10ms overhead）相对于 LLM 推理本身的 800ms-1.2s 来说可以忽略不计。
4. 真正的收益不是省那几毫秒，而是：状态管理在 Java 侧，和公司的服务治理体系（限流、熔断、链路追踪）天然集成，运维不需要额外维护一个 Python 微服务。

**参考话术：**
"好问题，我需要澄清一下。我说的'避免跨语言开销'不是说完全不跨语言——vLLM 是 Python 服务，我当然要通过 HTTP 调它。我避免的是在 Agent 编排层再套一个 Python 进程。如果用 LangChain，链路是 Java 业务服务 → Python Agent 服务 → vLLM，三跳。现在是 Java Agent 直接 → vLLM，两跳，少了一个中间层。而且 vLLM 的 HTTP 调用 overhead 大概 5-10ms，相对于模型推理本身的 800ms 完全可以忽略。真正的核心收益是运维层面的：Agent 的状态管理、异常处理、链路追踪全在 Java 微服务体系内，不需要额外维护一个 Python 服务的部署、监控和扩缩容。我们团队的 Java 基建很成熟，限流熔断都是现成的，如果引入 Python Agent 服务，这些都要重新搭一套。"

### 1.3 为什么不用长上下文模型直接塞全文？

**面试官原话：**
"现在 Gemini 有 1M context window，Claude 有 200K，国产模型也有 128K 的。你的文档才 50K token，直接塞进去让模型读全文回答不就行了？你搞这么复杂的大纲导航、多轮检索、状态机，是不是过度工程化了？Long Context 模型的 Lost in the Middle 问题现在也在改善，你的方案相比直接塞全文的优势到底在哪？用数据说话。"

**问题本质：** 这是对你整个架构存在价值的根本性质疑。面试官想看你能不能从成本、准确率、多文档场景三个维度给出有说服力的回答。

**回答思路：**

1. 先用实测数据打脸"1M 够用"的假设：我实际测试过，拿我们的目标 PDF（500 页工程标准）直接上传 Gemini 2.5 Pro，结果直接报错"文件太大无法处理"。面试官说的"1M context window"是 token 数，但一份 500 页的中文 PDF 经过 tokenize 后实际 token 量远超预期——中文字符的 token 效率比英文低得多，加上表格、公式、页眉页脚等结构化内容，一份 500 页中文文档可能达到 3M-5M token，远超 1M 窗口。所以"直接塞全文"这个前提在我们的文档规模下根本不成立。
2. 即使文档小到能塞进去，成本问题也绕不开：50K token 全文塞进去，每次问答的 Input Token 成本是固定的高。我的方案平均只消耗 20K token（降低 60%），在高 QPS 场景下成本差距是数量级的。
3. 准确率：Lost in the Middle 不是"在改善"，而是"在特定模型上有所缓解"。小模型（0.5B-7B）的长上下文能力远不如大模型，我的系统要兼容小模型场景。
4. 多文档：用户问题可能涉及 5-10 份文档，单份就可能 3M-5M token，多份加起来任何模型都吃不下。我的方案通过大纲导航精准定位，不受文档总量限制。
5. 可解释性：全文塞入是黑盒，我的方案每一步检索都有 Trace，出了问题能精确定位是哪个环节错了。

**参考话术：**
"这个问题我不光理论分析过，还实测过。我拿我们的目标文档——一份 500 页的中文工程标准——直接上传 Gemini 2.5 Pro，结果直接报错文件太大无法处理。面试官说的 1M context window 是 token 数，但中文的 token 效率比英文低得多，一个中文字可能占 2-3 个 token，加上表格、公式、页眉页脚这些结构化内容，一份 500 页中文 PDF 实际 token 量可能到 3M-5M，远超 1M 窗口。所以'直接塞全文'这个前提在我们的文档规模下根本不成立，不是我不想用，是用不了。

退一步说，即使文档小到能塞进去，还有三个问题。第一，成本。我们日均处理上千次问答，每次 50K token 的 Input 成本是不可接受的，我的方案平均 20K token，成本降 60%，年化下来省的钱是六位数。第二，我们要兼容小模型。0.5B 的模型上下文窗口本身就有限，而且小模型的长上下文注意力衰减比大模型严重得多，Lost in the Middle 在小模型上几乎是必现的。第三，多文档场景。用户经常问跨文档的对比问题，涉及 3-5 份文档，单份就可能 3M-5M token，多份加起来任何模型都吃不下。第四，可解释性。全文塞入后模型的推理过程是黑盒，出了错你不知道它到底看了哪段。我的方案每一步都有 Trace，运维能精确回溯。所以这不是过度工程化，而是'直接塞全文'这条路在我们的场景下根本走不通，Agentic 架构是被逼出来的务实选择。"

### 1.4 砍掉 Rerank 层——你凭什么挑战业界标配？

**面试官原话：**
"Rerank 在 RAG 里是标配，几乎所有生产级 RAG 系统都有这一层。你说砍掉了，还说砍掉后效果更好。这个结论反直觉。你是怎么得出这个结论的？有没有做过严格的消融实验——同样的测试集，有 rerank 和没 rerank 的对比数据？还是你只是觉得'理论上不需要'就砍了？"

**问题本质：** 面试官在检验你是否有独立思考能力和数据驱动的决策习惯，还是人云亦云或者拍脑袋。

**回答思路：**

1. 结论来自 bad case 分析 + 消融实验，不是拍脑袋。
2. Bad case 分析：排查了老方案 50+ 个错误 case，发现 70% 的错误发生在召回阶段（相关内容根本没被召回），不是排序阶段。Rerank 只能对已召回的内容重排序，对召回阶段的问题无能为力——garbage in garbage out。
3. 消融实验：在同一个 200 条评测集上，对比四种配置：① 老方案（有 rerank）65%；② 老方案去掉 rerank 62%（只降了 3 个点，说明 rerank 的边际贡献很小）；③ 新方案（无 rerank）90%+；④ 新方案加回 rerank 89%（反而略降，因为 rerank 的延迟增加导致 Agent 可用的轮次预算减少）。
4. 核心论据：cross-encoder 是 pointwise 打分，不考虑多 chunk 间的互补性；开源 BGE-Reranker 在垂直领域泛化差；延迟代价（200-300ms）换来的精度提升 ROI 极低。

**参考话术：**
"不是拍脑袋砍的，是数据驱动的决策。我先做了 bad case 分析——排查了 50 多个老方案的错误 case，发现 70% 的错误根因在召回阶段，相关内容根本没被召回，rerank 再牛也排不出来。然后做了消融实验：老方案有 rerank 65%，去掉 rerank 62%，只降了 3 个点——说明 rerank 在我们场景里的边际贡献很小。新方案无 rerank 90%+，加回 rerank 反而 89%，因为 rerank 的 200ms 延迟挤占了 Agent 的轮次预算。核心问题是：cross-encoder 是 pointwise 打分，不考虑多 chunk 间的互补性和信息完整性，排到第一的 chunk 未必是回答问题最需要的那个。而且 BGE-Reranker 在我们的垂直领域（工程标准、合同条款）实测效果打折严重。与其在碎片召回后加一层补救，不如从源头解决——大纲驱动全局感知 + 按需精读完整章节。"

### 2.1 状态机的具体状态和转移条件

**面试官原话：**
"你反复提到'状态机驱动'。好，请你现在在白板上画出你的状态机。有几个状态？每个状态之间的转移条件是什么？哪些转移是确定性的，哪些依赖 LLM 的输出？状态机的 State 对象在内存里长什么样？是个 Java class？有哪些字段？序列化了吗？如果 Agent 跑到一半 JVM 挂了，状态能恢复吗？"

**问题本质：** 考你是真的实现了状态机，还是只是用"状态机"这个词来包装一个普通的 while 循环。

**回答思路：**

先正面回应：我说的"状态机驱动"不是指一个独立的有限状态机（FSM），而是指 Agent Loop 中每轮精确维护和注入的 State 对象。代码层面就是一个 while 循环，但关键区别在于：我不是简单地把历史 messages 扔给 LLM，而是在每轮 Prompt 中显式注入经过筛选和压缩的结构化状态快照。

State 对象是一个 Java POJO，核心字段：
- `actionHistory`（List）：每轮 Agent 的决策和工具调用记录，但不是原始对话流水，而是压缩后的摘要（"第 2 轮：读取了第 3.2 节，提取了 XX 信息"）
- `visitedNodeIds`（Set）：已读章节 ID 集合，防止 Agent 重复读同一个章节。工具层做硬拦截——如果 LLM 要求读一个已经在集合里的章节，直接返回"你已经读过这个章节，内容摘要如下：[之前的摘要]"，不实际执行读取。
- `usedKeywords`（Set）：已用搜索关键词集合，防止 Agent 用同样的词反复搜索。新关键词和已用关键词做 embedding 相似度比较，相似度 > 0.85 的视为语义重复，拦截并提示 LLM"这个关键词与你之前用过的 XX 语义重复"。
- `currentStep`（int）：当前轮次，用于判断是否达到最大轮次（15 轮）
- `cumulativeTokens`（long）：累计 Token 消耗，用于 Token 预算控制。当达到窗口容量约 80% 时触发上下文压缩——完整历史落盘，LLM 摘要中间过程后重建精简 messages，但 visitedNodeIds 和 usedKeywords 独立于压缩逻辑单独维护，不受影响。
- `outlineContext`（String）：当前注入的大纲片段（经粗筛裁剪后的相关子树，不是完整大纲）

每轮循环的逻辑：组装 Prompt（System Prompt + 工具定义 + 大纲 + State 快照 + 本轮问题）→ 调 LLM → 解析输出（工具调用 or 直接回答 or 格式异常）→ 执行工具 → 更新 State → 下一轮。格式异常时走纠错逻辑（把错误信息写入下一轮 Prompt 让 LLM 重新输出）。

这跟纯粹的"把所有历史对话塞进 context"的 Agent Loop 有本质区别：我的 State 是经过筛选和压缩的，LLM 每轮看到的是精炼后的决策依据，而不是原始的对话流水。`visitedNodeIds` 和 `usedKeywords` 这两个集合是防止 Agent 原地打转的关键——LLM 能明确看到"这些章节我已经读过了，这些关键词我已经搜过了"，从而做出不同的决策。

关于持久化：当前没有做。单次 Agent Session 最多 15 轮，耗时 10-30 秒，JVM 挂掉的概率极低，用户重试即可。如果未来要支持长时间任务，State 可以序列化到 Redis，字段都是可序列化的基础类型。

**参考话术：**
"我先澄清一下——我说的'状态机驱动'不是指一个教科书式的有限状态机，而是指 Agent Loop 中每轮精确维护的 State 对象。代码层面就是一个 while 循环，但关键在于我怎么管理状态。State 是一个 Java POJO，核心字段有 actionHistory、visitedNodeIds、usedKeywords、currentStep、cumulativeTokens、outlineContext。每轮循环我会把 State 压缩成结构化快照注入 Prompt——不是把原始对话历史全塞进去，而是精炼后的决策依据。比如 actionHistory 不是原始的 messages，而是压缩摘要：'第 2 轮读了 3.2 节，提取了 XX'。visitedNodeIds 和 usedKeywords 这两个集合是防止 Agent 原地打转的关键——LLM 能明确看到哪些章节已经读过、哪些关键词已经搜过，从而做出不同的决策。这跟那种把所有历史对话塞进 context 的 Agent Loop 有本质区别：我的状态是经过筛选和压缩的，Token 消耗可控，而且 LLM 的决策质量更高，因为它看到的是结构化的决策依据而不是流水账。持久化目前没做，单次 Session 最多 15 轮、10-30 秒完成，JVM 崩溃概率极低，重试成本可接受。"

### 2.2 read_section 的实现细节

**面试官原话：**
"你的 `read_section` 工具说'基于大纲 JSON 树精确定位章节，返回完整章节内容'。具体怎么定位的？大纲树的节点 ID 和 ES 里存储的文档内容是怎么关联的？你是把整篇文档按章节切好分别存的，还是存的全文然后运行时按偏移量截取？如果一个章节有 5000 token，超过了小模型的处理能力怎么办？"

**问题本质：** 考你对核心工具的实现深度。大纲树和实际文档内容之间的映射关系是整个方案的关键工程细节。

**回答思路：**

1. 文档解析后，按章节切分存储。每个章节是一个独立的文档单元，存入 ES 时带上 `doc_id`、`section_id`（对应大纲树节点 ID）、`parent_section_id`、`content`、`page_range` 等字段。
2. 大纲树的每个节点有唯一 `section_id`，和 ES 中的存储单元一一对应。`read_section` 就是拿 `section_id` 去 ES 做精确查询，O(1) 复杂度。
3. 如果章节过长（超过阈值，比如 3000 token），在解析阶段就会被拆成子节点。大纲树上会体现为该章节下多了几个"Part 1"、"Part 2"的子节点。Agent 可以选择读取其中一个 Part，而不是整个章节。
4. `read_section` 还接受一个 `goal` 参数——Agent 描述它想从这个章节获取什么信息。精读工具内部有一个独立的 LLM 调用，它接收主 LLM 给的阶段目标，读完章节后输出三样东西：① 针对阶段目标的回答；② 回答对应的关键证据（原文引用，带页码和段落定位）；③ 给主 LLM 的下一步执行建议（比如"建议继续查看第 4.2 节免责条款"）。这样主 LLM 每轮看到的不是原始文档内容，而是经过提炼的结构化信息，既省 token 又提高决策质量。

**参考话术：**
"具体实现是这样的：MinerU 解析完文档后，我在 ETL 管道中按大纲树的节点做章节级切分。每个章节作为一个独立单元存入 ES，带上 `section_id`（和大纲树节点 ID 一一对应）、`doc_id`、`content`、`page_range` 等字段。所以 `read_section` 的实现很简单——拿 `section_id` 去 ES 做精确查询，直接返回该章节的完整内容。对于过长的章节，我在解析阶段就做了预处理：超过 3000 token 的章节会被拆成多个子节点挂在大纲树上，Agent 可以选择性地读取。另外 `read_section` 还有一个 `goal` 参数，Agent 会描述它想获取什么信息，工具内部会做一次轻量级的相关性过滤，只返回和 goal 匹配的段落，避免把整个长章节无脑塞进上下文。"

### 2.3 search_within_doc 和 read_section 的决策边界

**面试官原话：**
"你有 `search_within_doc` 做文档内搜索，也有 `read_section` 做章节精读。Agent 怎么决定用哪个？如果 Agent 每次都选 `read_section` 一个一个章节读过去，那和全文阅读有什么区别？如果每次都选 `search_within_doc`，那和传统 RAG 的 Chunk 检索有什么区别？你是怎么引导 Agent 合理使用这两个工具的？靠 Prompt 工程？还是有硬约束？"

**问题本质：** 工具链设计的核心矛盾——工具太多，Agent 可能选错；工具太少，能力不够。面试官想看你怎么平衡。

**回答思路：**

1. 两个工具的定位不同：`search_within_doc` 是"我知道要找什么关键词，但不知道在哪个章节"；`read_section` 是"我知道在哪个章节，但需要完整阅读理解"。
2. 引导策略是 Prompt 工程 + 工具描述。每个工具的 description 里明确写了适用场景。比如 `search_within_doc` 的描述是"当你需要在文档中搜索特定关键词或概念时使用"，`read_section` 的描述是"当你已经通过大纲定位到目标章节，需要完整阅读该章节内容时使用"。
3. 没有硬约束限制 Agent 只能用某个工具，因为硬约束会降低灵活性。但有软约束：如果 Agent 连续 3 次使用 `read_section` 读不同章节但都没找到有用信息，状态机会在下一轮 Prompt 中提示"建议尝试使用 search_within_doc 换一种检索策略"。
4. 实际观察：Agent 通常的行为模式是先看大纲 → `read_section` 读最可能的章节 → 如果不够再用 `search_within_doc` 补充搜索。这和人类的阅读习惯一致。

**参考话术：**
"这两个工具的定位是互补的。`search_within_doc` 解决的是'我知道要找什么但不知道在哪'的问题，`read_section` 解决的是'我知道在哪但需要完整理解'的问题。引导 Agent 选择主要靠工具描述——每个工具的 description 里明确写了适用场景和使用时机。我没有加硬约束，因为实际场景太多样了，硬约束反而会限制 Agent 的灵活性。但我有软引导：如果 Agent 连续多次用同一个工具但效果不好，状态机会在下一轮 Prompt 里建议它换一种策略。实际跑下来，Agent 的行为模式很符合直觉——先看大纲定位章节，read_section 精读，信息不够再 search_within_doc 补充。这和人类翻书的习惯是一致的。"

### 3.1 "65% → 90%" 的评测方法论

**面试官原话：**
"你说复杂多跳问题准确率从 65% 提升到 90%+。这个数字是怎么算出来的？你的测试集有多大？多少条数据？谁标注的？标注标准是什么？'准确'的定义是什么——完全匹配还是语义等价？如果是语义等价，谁来判断？LLM 判断的话，LLM 本身就有偏差。你有没有做过 A/B 测试？还是只是在离线测试集上跑的？线上真实用户的反馈和这个数字一致吗？"

**问题本质：** 简历上的数字是最容易被拷打的。面试官想确认你不是随便编的，而是有严谨的评测方法论。

**回答思路：**

1. 测试集：约 200 条问答对，覆盖单跳事实、多跳推理、跨文档对比三类。其中多跳推理约 80 条。
2. 标注：由业务方（产品经理 + 领域专家）人工标注标准答案。标注标准是"答案包含所有关键信息点"，不要求措辞完全一致。
3. 评测方式：自动评测 + 人工抽检。自动评测用 LLM-as-Judge（GPT-4 级别模型判断回答是否覆盖了标准答案的所有关键信息点），人工抽检覆盖 30% 的 case，校准 LLM 评测的偏差。
4. 65% 是老方案在同一测试集上的表现，90% 是新方案的表现，对比条件一致。
5. 没有做线上 A/B 测试（因为是 B 端产品，用户量有限），但有线上用户的"点踩"反馈数据作为辅助验证。

**参考话术：**
"这个数字的来源是这样的。我们构建了一个约 200 条的评测集，其中多跳推理类问题约 80 条，由业务方的领域专家人工标注标准答案。评测标准是'关键信息点覆盖率'——回答需要包含标准答案中的所有关键信息点，不要求措辞一致。自动评测用的是 LLM-as-Judge，用 GPT-4 级别的模型做裁判，同时人工抽检 30% 的 case 来校准 LLM 评测的偏差。65% 和 90% 都是在同一个测试集上、相同评测标准下跑出来的，对比条件一致。坦白说，200 条的测试集规模不算大，但对于 B 端产品来说，这已经覆盖了我们核心客户的主要问题类型。线上没有做 A/B 测试，但我们有用户的'点踩'反馈数据，新方案上线后点踩率下降了约 50%，和离线评测的趋势一致。"

### 3.2 Token 消耗降低 60% 的计算口径

**面试官原话：**
"Token 消耗降低 60%，这个对比基准是什么？老方案是一次性把所有 Chunk 塞进 LLM，新方案是多轮检索。你算的是单次 LLM 调用的 Token，还是整个 Session 的总 Token？新方案虽然每轮 Token 少，但它跑了 3-5 轮啊。3 轮 × 20K token = 60K token，可能比老方案一次性塞 50K 还多。你是怎么算的？"

**问题本质：** 面试官怀疑你的 Token 节省数据是"选择性统计"。

**回答思路：**

1. 对比口径是整个 Session 的总 Token 消耗（Input + Output 全算）。
2. 老方案的 Token 消耗：多路召回的 Chunk（通常 10-15 个 Chunk × 500 token = 5000-7500 token）+ System Prompt + 用户问题 + LLM 输出，单次约 15K-20K token。但老方案对复杂问题会拆分子问题，每个子问题走一次完整链路，3 个子问题就是 45K-60K token。
3. 新方案的 Token 消耗：大纲（500-800 token）+ 状态历史（逐轮增长）+ 工具返回的精确片段（通常 1000-2000 token/轮）。平均 3 轮，总计约 15K-25K token。
4. 关键差异：新方案每轮只读取精确相关的章节/片段，而老方案每次都要把所有召回的 Chunk 全塞进去（包括大量噪音 Chunk）。

**参考话术：**
"对比口径是整个 Session 的总 Token，Input 和 Output 全算。老方案看起来是'一次调用'，但对复杂问题它会拆分子问题，每个子问题走一次完整的'召回 Chunk → 塞进 LLM'链路。3 个子问题，每次塞 10 个 Chunk，总 Token 轻松到 50K-60K。而且老方案的 Chunk 里有大量噪音——10 个召回的 Chunk 可能只有 3 个真正相关，剩下 7 个是浪费。新方案虽然跑 3-5 轮，但每轮只读取精确定位的章节片段，没有噪音 Chunk 的浪费。平均下来总 Token 在 15K-25K，确实比老方案低 60% 左右。这个数据是在评测集上统计的平均值，不是挑的最好 case。"

## 五、并发与生产环境类：能扛住真实流量吗？

---

### 5.1 Agent 的并发模型

**面试官原话：**
"你的 Agent 是有状态的——每个 Session 都维护一个 State 对象。那 100 个用户同时提问，你就有 100 个 State 对象在内存里。每个 State 里还有 Action 历史、大纲片段，内存占用不小。你的 Java 服务是怎么处理并发的？每个 Agent Session 是一个线程？线程池多大？如果 vLLM 响应慢了，线程全被阻塞怎么办？你用了异步非阻塞吗？CompletableFuture？WebFlux？"

**问题本质：** 考你的 Java 并发编程能力和对生产环境资源管理的理解。

**回答思路：**

1. 先把问题拉回正轨：Session 内存根本不是问题。单个 State 对象几十 KB，1000 个并发也就几十 MB，对 JVM 来说不值一提。面试官问内存占用，直接一句话带过。
2. 真正的并发瓶颈只有一个：LLM 推理的吞吐量。我们用的是本地部署的 vLLM，并发能力完全取决于 GPU 配置——几张卡、batch size 多大、模型多大。Java 侧的并发控制就是配合 vLLM 的处理能力做限流，没什么复杂的。
3. Java 侧用线程池 + CompletableFuture 做异步编排，调 vLLM 的 HTTP 请求是异步非阻塞的，线程不会阻塞等待。
4. 超时兜底：每次 vLLM 调用设 60s 超时，超时熔断。

**参考话术：**
"Session 内存完全不是问题——单个 State 对象几十 KB，就算 1000 个并发也就几十 MB，对 JVM 来说可以忽略。并发的真正瓶颈只有一个：LLM 推理的吞吐量。我们是本地部署的 vLLM，并发能力取决于 GPU 配置——几张卡、batch size、模型大小。Java 侧就是配合 vLLM 的处理能力做限流，用线程池 + CompletableFuture 异步编排，调 vLLM 的 HTTP 请求是非阻塞的，线程发出请求后立即释放。每次调用设 60 秒超时做熔断兜底。说白了，Java 侧的并发处理没什么花活，瓶颈在 GPU 侧，Java 侧只要不阻塞、不泄漏就行。"

---

### 5.2 多轮 Agent 的流式输出体验

**面试官原话：**
"传统 RAG 可以做流式输出——LLM 一边生成一边往前端推。你的 Agent 要跑 3-5 轮，每轮都要等 LLM 决策 + 工具执行，用户要干等 5-10 秒才能看到第一个字？这个用户体验能接受吗？你有没有做中间状态的反馈？比如告诉用户'正在搜索第 3 章...'？"

**问题本质：** Agentic RAG 的天然劣势——多轮决策导致首字延迟（TTFT）远高于传统 RAG。面试官想看你有没有意识到这个问题并做了优化。

**回答思路：**

1. 确实存在这个问题，用户体验是 Agentic RAG 的已知短板。
2. 我的解决方案是中间状态流式推送。Agent 每轮决策时，通过 SSE（Server-Sent Events）向前端推送当前状态：
   - "正在分析文档大纲..."
   - "正在阅读《XX报告》第 3.2 节..."
   - "正在搜索关键词'营收增长'..."
   - "信息收集完成，正在生成回答..."
3. 最终回答阶段用流式输出（vLLM 支持 streaming），用户能看到答案逐字生成。
4. 整体体验类似于 Perplexity 的搜索——用户能看到 Agent 的"思考过程"，心理等待感大幅降低。

**参考话术：**
"这个问题确实存在，而且是 Agentic RAG 相比传统 RAG 的天然劣势。我的处理方式是借鉴 Perplexity 的交互模式——通过 SSE 向前端实时推送 Agent 的中间状态。用户不是干等 10 秒看到一个结果，而是能看到'正在分析文档大纲'、'正在阅读第 3.2 节'、'正在搜索关键词 XXX'这样的实时进度。最终生成回答时走 vLLM 的 streaming 输出，答案逐字出现。实际体验下来，用户反馈是'虽然总时间比以前长，但能看到系统在做什么，感觉更可靠'。心理等待感和实际等待时间是两回事。"

### 6.1 Agent 选错工具或选错章节怎么办？

**面试官原话：**
"你的 Agent 看了大纲，决定去读第 3 章。结果第 3 章和问题完全无关，答案其实在第 7 章。这一轮就浪费了。更糟的是，Agent 读了第 3 章之后，基于错误的信息做出了错误的判断，认为'答案就是 XXX'，直接输出了错误答案。你的系统怎么防止这种'自信的错误'？你的质量评估环节能检测出来吗？"

**问题本质：** Agent 的决策是不确定的，选错章节是必然会发生的事。面试官想看你的容错设计。

**回答思路：**

1. 选错章节本身不是致命问题——状态机会记录这次访问，Agent 在下一轮可以选择其他章节。最大 15 轮的预算给了 Agent 纠错空间。
2. "自信的错误"是更严重的问题。我的防御是质量评估环节：Agent 生成初步回答后，会有一个评估步骤判断"回答是否完整且有依据"。评估的 Prompt 会要求 LLM 检查回答中的每个关键信息点是否能在已读取的内容中找到出处。如果找不到出处，判定为"信息不充分"，继续检索。
3. 但坦白说，如果 Agent 基于错误信息生成了一个"看起来合理"的回答，评估环节也可能被骗过。这是 LLM 系统的固有局限。我的兜底是 Trace 系统——每个回答都附带引用来源（具体到章节和页码），用户可以点击查看原文验证。

**参考话术：**
"选错章节是必然会发生的，这不是 bug，是 Agent 系统的常态。我的设计思路是给 Agent 足够的纠错空间——15 轮预算意味着即使前几轮选错了，还有机会找到正确信息。更关键的防御是质量评估环节：Agent 生成回答后，评估 Prompt 会要求检查每个信息点是否有出处。如果回答中包含'无中生有'的信息，评估会判定为不充分，触发继续检索。但我要坦诚说，如果 Agent 基于错误信息生成了一个'看起来很合理'的回答，评估环节也可能被骗过——这是 LLM 系统的固有局限，不是我的架构能完全解决的。我的兜底是每个回答都附带引用来源，精确到章节和页码，用户可以点击查看原文。这不是技术上的完美解决方案，但在工程上是务实的。"

### 6.2 文档更新后大纲树和索引的一致性

**面试官原话：**
"用户上传了一份文档，你解析了大纲树，建了索引。一周后用户更新了这份文档——加了两个章节，删了一个章节，改了一些内容。你怎么处理？是全量重新解析还是增量更新？如果全量重新解析，解析期间用户查询怎么办？用旧索引还是等新索引？如果增量更新，你怎么知道哪些章节变了？"

**问题本质：** 数据一致性问题，考你对生产环境中数据更新场景的思考。

**回答思路：**

1. 文档更新必须重新上传，触发全量重新解析，不做增量更新。原因：文档更新可能涉及章节重排、标题修改、内容交叉引用变化，增量更新的复杂度远高于全量重解析，而且容易引入一致性 bug。
2. 解析速度不是问题：实测 500 页的文档，MinerU 解析 + 大纲构建 + ES 索引全流程也就几分钟。这个耗时对用户来说完全可接受——上传一份新文档等几分钟很正常。
3. 解析是异步的，用户上传后不需要干等，解析完成后通知即可。
4. 不需要"双版本切换"这种复杂机制——文档更新本身就是用户主动行为，用户知道自己在更新文档，等几分钟解析完再查询是自然的交互流程。

**参考话术：**
"全量重新解析，不做增量。增量更新的一致性保障太复杂，投入产出比不划算。而且解析速度根本不是瓶颈——我们实测过，500 页的文档从 MinerU 解析到大纲构建到 ES 索引，全流程也就几分钟。文档更新是用户主动上传的行为，等几分钟解析完再查询，这个体验完全可接受，不需要搞什么双版本切换的复杂机制。"

### 6.3 Synthesizer 反馈闭环的具体工程实现

**面试官原话：**
"你说 Synthesizer 能自动发现维度缺口并驱动 Reader 补充。这个听起来很美好，但我想知道具体怎么实现的。维度是预定义的还是动态发现的？Synthesizer 怎么知道 Reader A 缺了'喷淋系统'这个维度？它是用 LLM 做的还是规则匹配？Reader 收到补充请求后，怎么高效地补充而不是从头读一遍？"

**问题本质：** 反馈闭环是项目二的核心亮点，面试官想确认你不是纸上谈兵。

**回答思路：**

1. 维度是动态发现的，不是预定义的。因为不同问题的对比维度完全不同。
2. Synthesizer 通过 `compare_dimension` 工具让 LLM 遍历所有 Reader 的提取结果，收集全部出现过的对比维度，然后逐文档检查覆盖情况，生成"维度 × 文档"矩阵。
3. 对于缺口，Synthesizer 通过 `request_supplementary` 工具发送定向补充请求——具体的自然语言描述，不是泛泛的"再读一遍"。
4. Reader 收到补充请求后不需要从头读——状态机中保留了已读章节集合和大纲信息，LLM 可以直接判断"这个信息可能在 4.3 节，我之前没读过"，精准定位后读取。如果之前已经读过相关章节但没提取该维度，Reader 从已有上下文中直接提炼，无需重复读取。

**参考话术：**
"维度是动态发现的。Synthesizer 拿到所有 Reader 的结构化提取结果后，通过 `compare_dimension` 工具让 LLM 遍历所有提取项，收集出现过的全部对比维度——比如 Reader A 提取了'耐火等级'和'疏散通道宽度'，Reader B 提取了'耐火等级'和'喷淋系统规格'，LLM 会发现全集是三个维度，然后标记 A 缺'喷淋系统'、B 缺'疏散通道'。

对于每个缺口，Synthesizer 通过 `request_supplementary` 工具向对应 Reader 发送定向请求，内容是具体的自然语言描述，比如'请补充提取文档 A 中关于喷淋系统的要求'。消息通过 MessageBus 写入 Reader 的收件箱。

Reader 收到后不需要从头读文档——它的状态机里保留了已读章节集合和大纲。LLM 可以直接判断'喷淋系统可能在第 7.3 节自动灭火系统，我之前没读过这个章节'，精准定位后读取并提交补充结果。如果之前已经读过相关章节但没提取该维度的信息，Reader 会从已有上下文中直接提炼，无需重复读取。这就是为什么状态机中独立维护已读章节集合很重要——它让补充提取变得高效。"

### 7.1 vLLM 的 Prefix Caching 在你的场景下的实际收益

**面试官原话：**
"你简历里写了 vLLM。你的 Agent 每轮决策都要调 vLLM，而且每轮的 Prompt 都包含相同的 System Prompt 和工具定义。你有没有利用 vLLM 的 Prefix Caching 来优化这部分重复计算？如果用了，缓存命中率是多少？如果没用，为什么？"

**问题本质：** 考你对 vLLM 的理解深度，以及是否真正做过推理层的优化。

**回答思路：**

（结合你已有的 vLLM Prefix Cache 文档）

1. 用了，而且收益显著。Agent 每轮 Prompt 的前缀部分（System Prompt + 工具定义 + 大纲）在同一个 Session 内是高度重复的。
2. 开启 `--enable-prefix-caching` 后，同一 Session 内后续轮次的 Prefill 只需要计算新增的部分（本轮的 Action 历史和工具返回结果），前缀部分直接复用 KV Cache。
3. 实际命中率约 75-80%（因为每轮 Prompt 的后半部分——Action 历史——是变化的，但前半部分——System Prompt + 工具定义——是固定的）。
4. TTFT 从 800ms 降到约 300ms，对多轮 Agent 的总延迟优化非常明显（5 轮 × 500ms 节省 = 2.5 秒）。
5. 踩过的坑：一开始发现命中率只有 60%，排查后发现是 Prompt 模板里有动态时间戳字段，导致前缀不一致。把时间戳挪到 Prompt 末尾后命中率提升到 75%+。

**参考话术：**
"用了，而且这是我做的一个重要优化。Agent 每轮 Prompt 的前缀——System Prompt、工具定义、大纲——在同一个 Session 内是不变的，天然适合 Prefix Caching。开启后，后续轮次的 Prefill 只需要计算新增的 Action 历史部分，前缀直接复用 KV Cache。实际命中率稳定在 75% 以上，TTFT 从 800ms 降到 300ms 左右。5 轮 Agent 下来，总延迟节省了 2-3 秒，体验提升很明显。踩过一个坑：一开始命中率只有 60%，排查发现是 Prompt 模板里有个动态时间戳字段，每次请求都不一样，导致前缀 hash 不匹配。把时间戳挪到 Prompt 末尾后命中率就上来了。这个经验让我意识到，Prefix Caching 的收益高度依赖 Prompt 结构的设计——动态内容必须后置。"

### 7.2 search_within_doc 的检索实现——向量还是关键词？

**面试官原话：**
"你的 `search_within_doc` 用的是什么检索方式？纯向量？纯 BM25？还是混合？你用的什么 Embedding 模型？通用模型还是领域微调过的？你的客户涉及金融、法律、科研，这些领域的专业术语通用 Embedding 模型能 cover 吗？"

**问题本质：** 向量检索是 RAG 的基础能力，面试官想看你在这个基础环节上的功力。

**回答思路：**

1. `search_within_doc` 用的是 ES 的全文检索（BM25），不是向量检索。原因：这个工具的定位是"在已定位文档内部搜索特定关键词"，BM25 对精确关键词匹配的效果比向量检索好，而且延迟更低。
2. 向量检索主要用在文档级筛选（top_doc）和老方案遗留的多路召回中。Embedding 模型用的是 bge-large-zh-v1.5（中文场景），没有做领域微调。
3. 原因：我们的文档类型跨行业，微调一个领域的 Embedding 可能在其他领域退化。通用模型 + BM25 混合检索在实测中效果已经足够。
4. 坦白说，在新架构下向量检索的重要性降低了——Agent 更多依赖大纲导航 + `read_section` 精读，`search_within_doc` 是补充手段，而且它走的是 BM25 不是向量。

**参考话术：**
"`search_within_doc` 用的是 ES 的 BM25 全文检索，不是向量检索。这个工具的定位是'在已定位文档内部搜索特定关键词'，BM25 对精确关键词匹配的效果比向量好，延迟也更低。向量检索主要用在文档级筛选阶段，Embedding 模型用的 bge-large-zh-v1.5，没有做领域微调——我们客户跨行业，微调某个领域可能在其他领域退化。坦白说，在新架构下向量检索的重要性降低了。Agent 主要靠大纲导航 + read_section 精读来获取信息，search_within_doc 更多是补充手段。这也是为什么我没有在 Embedding 微调上投入太多精力——投入产出比不如优化 Agent 的决策逻辑。"

**参考话术：**
"有几个点。第一，知识图谱从一开始就不应该引入，投入产出比不划算。第二，评测体系应该更早建立，用评测驱动开发。第三，SFT 微调应该更早启动。我后来做了，效果不错，但如果从上线第一天就开始积累轨迹数据，能省掉中间很多格式纠错的补丁代码。但最大的遗憾是没有做 Agentic RL。SFT 的上限是训练数据——模型只能模仿，不能超越。如果加上 GRPO，用答案正确性作为可验证奖励，再加步骤数惩罚，模型可以自己探索出更高效的检索策略，比如发现某些问题 2 步就够不需要走 4 步。这是 SFT 做不到的。没做的原因是 RL 的工程落地比 SFT 复杂得多——要搭交互环境、设计奖励函数、做多轨迹采样，当时精力不够。但这是最有价值的下一步优化方向。"

## 九、模型压缩与微调类：你能让小模型顶大模型吗？

### 9.1 用微调让小模型替代 235B 主模型

**面试官原话：**
"你说主 LLM 用的是 235B 的大模型。235B 的推理成本有多高你知道吗？一张 H100 跑 235B 都费劲，你们公司有多少卡？如果我告诉你，现在有团队用 7B 模型微调后在特定任务上达到了 GPT-4 级别的效果，你觉得你的场景能做到吗？你有没有想过用微调把 235B 换成 7B 或者 32B？如果做了，推理成本能降多少？你打算怎么做？"

**问题本质：** 考你对模型压缩、知识蒸馏、任务特化微调的理解，以及能不能把这个技术方向和你的实际业务场景结合起来讲清楚。这是一个能把面试拉升一个档次的话题。

**回答思路：**

1. 先承认问题的真实性：235B 的推理成本确实是生产环境的痛点，单次推理延迟高、GPU 资源占用大，是我们系统的主要成本来源。

2. 为什么小模型在通用任务上不行，但在特化任务上可以：
   - 我的 Agent 的核心决策任务其实是有限的：看大纲选章节、判断工具调用格式、评估信息是否充分。这些任务的输入输出空间是收敛的，不需要 235B 的通用能力。
   - 235B 的大部分参数是在"做通用任务"，我的场景只需要它的一小部分能力。

3. 具体方案——三步走：
   - **Step 1：数据蒸馏**。用 235B 模型在生产流量上跑，收集高质量的 `(Prompt, 决策轨迹)` 对。重点收集：工具选择正确、格式规范、最终回答准确的 trajectory。大约需要 5000-10000 条。
   - **Step 2：SFT 微调**。用蒸馏数据对 7B/32B 模型做 SFT，让小模型学会在 Agent 场景下的决策格式和工具调用规范。这一步解决"格式遵循"和"基础决策能力"。
   - **Step 3：GRPO + RLVR 强化学习**。定义可验证奖励：最终回答准确性（LLM-as-Judge 或关键信息点覆盖率）+ 工具调用效率（步骤数惩罚）。用 GRPO 在线优化，让模型学会更高效的检索策略。这一步解决"决策质量"，让小模型超越 SFT 的上限。

4. 预期效果与成本收益：
   - 32B 模型经过任务特化微调，在 Agent 决策任务上达到 235B 的 90%+ 效果是合理预期（参考 DeepSeek R1 的蒸馏结论）。
   - 推理成本：235B 需要 4-8 张 H100，32B 只需要 1 张，成本降低 4-8 倍。7B 更极端，1 张消费级 GPU 就能跑，成本降低 10-20 倍。
   - 延迟：235B 单次推理 800ms-1.2s，32B 约 200-400ms，7B 约 50-100ms。Agent 跑 5 轮，总延迟从 5-6 秒降到 1-2 秒，用户体验质变。

5. 关键风险和边界：
   - 小模型的泛化能力弱——遇到训练分布之外的问题（比如新类型文档、罕见问题模式）可能退化严重。需要持续收集 Bad Case 补充训练数据。
   - 奖励设计是难点：如果奖励只看最终准确性，模型可能学会"走捷径"（比如直接猜答案而不检索）。需要加过程奖励（process reward）惩罚跳过检索步骤的行为。
   - 7B 达到 235B 效果是理想情况，更现实的目标是 32B 达到 235B 的 85-90%，这在工程上是可落地的。

**参考话术：**
"这个问题我认真想过。235B 的推理成本确实是我们系统的主要瓶颈——延迟高、GPU 占用大。但我的 Agent 的核心决策任务其实是收敛的：看大纲选章节、判断工具调用、评估信息充分性。这些任务的输入输出空间远比通用对话小，不需要 235B 的全部能力。

我的思路是三步走。第一步，用 235B 在生产流量上跑，收集高质量的决策轨迹数据——工具选择正确、格式规范、最终回答准确的 trajectory，大概需要 5000-10000 条。第二步，用这些数据对 32B 模型做 SFT，让它学会 Agent 场景下的决策格式和工具调用规范。第三步，用 GRPO + 可验证奖励做强化学习——奖励信号是最终回答的准确性加上步骤效率惩罚，让模型学会更高效的检索策略。

预期效果：32B 经过任务特化微调，在我们的评测集上达到 235B 的 85-90% 是合理预期，参考 DeepSeek R1 的蒸馏结论。成本收益很直接——235B 需要 4-8 张 H100，32B 只需要 1 张，推理延迟从 800ms 降到 200-400ms，Agent 5 轮下来总延迟从 5 秒降到 1-2 秒，用户体验是质变。

当然有风险：小模型泛化能力弱，遇到训练分布外的问题会退化；奖励设计不好模型会走捷径。但这些都是工程上可以迭代解决的问题，不是根本性障碍。"

### 9.2 蒸馏数据的质量控制

**面试官原话：**
"你说用 235B 跑生产流量收集蒸馏数据。但生产流量里有大量低质量的 trajectory——用户问了奇怪的问题，Agent 跑偏了，最终回答是错的。你把这些垃圾数据也喂给小模型，不是在教它学坏吗？你怎么筛选高质量的 trajectory？筛选标准是什么？筛完还剩多少数据？够不够训练？"

**问题本质：** 数据质量是微调效果的决定性因素。面试官想看你有没有想清楚数据工程这一层。

**回答思路：**

1. 筛选标准分两层：
   - **结果层**：最终回答通过 LLM-as-Judge 评分，只保留"准确且完整"的 trajectory。这一步大概过滤掉 30-40% 的数据。
   - **过程层**：trajectory 中的工具调用格式合法（JSON 可解析）、没有出现 ERROR_RECOVERY 状态（说明 Agent 没有走弯路）、步骤数在合理范围内（3-5 步，过多说明 Agent 在乱试）。

2. 筛完数据量的估算：假设日均 1000 次问答，高质量 trajectory 占 60%，一个月能积累约 18000 条。对于 SFT 来说 5000-10000 条已经足够，数据量不是瓶颈。

3. 额外的数据增强：对于稀缺的"多跳推理"类问题，可以用 235B 主动生成合成数据——给定文档和复杂问题，让 235B 生成完整的推理轨迹，人工抽检后加入训练集。

4. 数据多样性：要确保训练数据覆盖不同文档类型、不同问题类型，避免模型只会处理某一类问题。

**参考话术：**
"数据质量是微调效果的关键，这个我想得比较细。筛选分两层：结果层用 LLM-as-Judge 过滤掉最终回答不准确的 trajectory，大概过滤掉 30-40%；过程层过滤掉工具调用格式异常、触发了 ERROR_RECOVERY、或者步骤数异常多的 trajectory——这些说明 Agent 在那次推理里走了弯路，不是好的示范。筛完之后，日均 1000 次问答，一个月能积累 15000-18000 条高质量数据，SFT 用 5000-10000 条就够了，数据量不是瓶颈。对于稀缺的多跳推理类问题，可以让 235B 主动生成合成 trajectory 来补充，人工抽检后加入训练集。"

### 9.3 SFT 训练的关键细节——Observation Masking 和训练轨迹

**面试官原话：**
"你说用 trajectory 数据做 SFT。Agent 的 trajectory 里有模型自己的决策（think + tool_call），也有工具返回的结果（observation）。你训练的时候，observation 部分算 loss 吗？如果算，你不是在教模型背诵搜索结果？如果不算，你怎么实现的？另外，生产日志里的 trajectory 可能有 10 步，但其中 5 步是走弯路的，你怎么处理？"

**问题本质：** 考你对 Agent SFT 的理解深度——不是简单地把数据扔进去训，而是要理解哪些 token 该学、哪些不该学，以及训练轨迹的质量问题。

**回答思路：**

1. Observation 不算 loss，这是 Agent SFT 的标准做法（FireAct, 2023 最早提出）。tool_response 是外部环境返回的，不是模型的决策。如果让模型对这部分也算 loss，等于在教它"背诵搜索结果"，没有意义，反而干扰它学习"什么时候该搜、搜什么"。
2. 实现方式：把 tool_response 对应 token 的 label 设为 -100，PyTorch 的 CrossEntropyLoss 内置 ignore_index=-100，自动跳过这些位置不算梯度。如果用 LLaMA-Factory，role=tool 的 turn 自动 mask，不需要手动处理。
3. 训练轨迹 ≠ 生产日志原始轨迹。生产日志里的 trajectory 可能有冗余步骤（搜了但没用上、读了但信息不相关）。处理方式有两种：
   - 简单方案：只保留最终回答正确的 trajectory，不做路径压缩。冗余步骤虽然不是最优的，但至少不是错的，模型学到的是"能答对的路径"。
   - 进阶方案：对 trajectory 做回溯分析，标记每一步的 observation 是否出现在最终回答的推理链中，没贡献的步骤删掉。或者同一个 question 采样多条轨迹，只保留答案正确且步骤最少的那条。
4. 如果生产日志量不够（SFT 至少要几千条），可以用合成数据补量——参考 DocDancer 的 Exploration-then-Synthesis 方案，用强模型在文档上跑探索生成 QA 对，再通过 rejection sampling 生成精简的解题轨迹作为训练数据。

**参考话术：**
"Observation 不算 loss，这是 Agent SFT 的标准做法。tool_response 是环境返回的，不是模型决策，如果算 loss 就是在教模型背搜索结果，没意义。实现上很简单——把 tool_response 对应 token 的 label 设为 -100，PyTorch 的 CrossEntropyLoss 自动跳过。用 LLaMA-Factory 的话 role=tool 的 turn 自动 mask。

关于训练轨迹质量，生产日志里的 trajectory 确实有冗余步骤。我的处理是分两步：第一步只保留最终回答正确的 trajectory，过滤掉答错的；第二步对保留的 trajectory 做回溯分析，标记每步 observation 是否对最终回答有贡献，没贡献的删掉。这样训练数据教的是'最高效的解题路径'而不只是'能答对的路径'。如果生产日志量不够，我会参考 DocDancer 论文的合成方案——用强模型在文档上跑探索自动生成 QA 对，再通过 rejection sampling 产出精简轨迹来补量。生产日志保证分布真实性，合成数据保证覆盖度，双轨并行。"

### 10.1 Reader 容错——崩溃了怎么办？

**面试官原话：**
"你说多个 Reader 并行跑，每个 Reader 内部是一个完整的 Agent Loop。如果其中一个 Reader 崩了——API 超时、格式错误连续 3 次、或者 Python 线程直接挂了——其他 Reader 怎么办？Synthesizer 怎么办？整个任务是不是就废了？"

**问题本质：** 分布式系统的容错能力。多 Agent 系统里一个节点挂了，整体是否还能工作。

**回答思路：**

1. Reader 崩溃不会阻塞其他 Reader——每个 Reader 在独立线程中运行，互不影响。
2. Reader 内部有重试机制：API 超时自动重试（最多 3 次），格式错误走 ERROR_RECOVERY 状态纠错。
3. 如果重试后仍然失败，Reader 标记自己的 task 状态为 `failed`（写入 TaskBoard 的 JSON 文件），并通知 Planner。
4. Synthesizer 启动时检查所有 task 状态：如果有 Reader 失败，Synthesizer 会在对比表格中标注"文档 X 的信息未能提取，原因：Reader 执行失败"，而不是整个任务失败。用户拿到的是一个不完整但有明确标注的结果，比什么都没有好。
5. TaskBoard 的状态持久化支持故障恢复：如果 Reader 崩溃后线程被重新拉起，它可以从 task 文件中读取上次的进度（已读章节、已提取结果），从断点继续而不是从头开始。

**参考话术：**
"Reader 崩溃不会拖垮整个系统。每个 Reader 在独立线程里跑，一个挂了其他的完全不受影响。Reader 内部有重试机制——API 超时重试 3 次，格式错误走纠错状态。如果重试后仍然失败，Reader 把自己的 task 标记为 failed 写入 TaskBoard。Synthesizer 启动时会检查所有 task 状态，如果有 Reader 失败了，它会在对比表格中标注'文档 X 的信息未能提取'，而不是整个任务报错。用户拿到的是一个不完整但有明确标注的结果。另外 TaskBoard 的状态持久化支持断点恢复——如果 Reader 崩溃后被重新拉起，它可以从 task 文件读取上次进度，从断点继续。"

### 10.2 项目二的多 Agent 编排在 Java 里怎么实现的？

**面试官原话：**
"项目二有 Planner、多个 Reader、Synthesizer，还有 MessageBus 做通信。这套多 Agent 编排你用 Java 怎么实现的？Python 生态里有 LangGraph、CrewAI 这些现成框架，Java 生态里多 Agent 框架几乎没有。你是自己从零写的编排层？线程模型是什么样的？Reader 之间怎么通信？Planner 怎么等所有 Reader 完成？"

**问题本质：** Java 生态里多 Agent 框架确实稀缺，面试官想看你是否有能力从零搭建编排层，以及你的并发模型设计是否合理。

**回答思路：**

1. 是自己写的编排层，没有用第三方多 Agent 框架。Java 生态里确实没有成熟的多 Agent 框架，但多 Agent 编排的本质就是并发任务调度 + 消息通信，Java 的并发原语（线程池、CompletableFuture、BlockingQueue）完全够用。
2. 线程模型：Planner 在主线程运行，分析完任务后为每个 Reader 提交一个异步任务到线程池（ExecutorService）。每个 Reader 是一个独立的 Callable/Runnable，内部跑完整的 Agent Loop。Planner 通过 CompletableFuture.allOf() 等待所有 Reader 完成，然后启动 Synthesizer。
3. Reader 之间不需要直接通信——它们是独立并行的，各自处理各自的文档。通信只发生在 Synthesizer 和 Reader 之间（补充提取的反馈闭环），通过 TaskBoard（共享的任务状态对象）实现，用 ConcurrentHashMap 保证线程安全。
4. MessageBus 的实现很轻量：就是一个基于 BlockingQueue 的消息队列，Synthesizer 往里写补充请求，Reader 从里面读。不需要 Kafka 这种重量级方案，因为都在同一个 JVM 进程内。
5. 选择 Java 而不是 Python 的原因和项目一一致：和公司微服务体系集成、复用项目一的 Agent Loop 核心代码（State 管理、工具链、LLM 调用层）。项目二的 Reader 内部的 Agent Loop 和项目一的单文档 Agent 是同一套代码，只是外面套了一层多 Agent 编排。

**参考话术：**
"编排层是自己写的，Java 生态里确实没有成熟的多 Agent 框架，但说实话也不需要——多 Agent 编排的本质就是并发任务调度加消息通信，Java 的并发原语完全够用。线程模型很直接：Planner 在主线程跑，分析完任务后为每个 Reader 提交异步任务到线程池，每个 Reader 是一个独立的 Callable，内部跑完整的 Agent Loop。Planner 通过 CompletableFuture.allOf() 等所有 Reader 完成，然后启动 Synthesizer。Reader 之间不需要通信，它们各自独立处理各自的文档。通信只发生在 Synthesizer 和 Reader 之间——补充提取时 Synthesizer 往 BlockingQueue 里写请求，Reader 从里面读，都在同一个 JVM 内，不需要 Kafka 这种重量级方案。选 Java 的核心原因是复用——项目二的 Reader 内部的 Agent Loop 和项目一的单文档 Agent 是同一套代码，State 管理、工具链、LLM 调用层全部复用，只是外面套了一层多 Agent 编排。这也是技术栈统一的好处。"

2. 对比维度分两块：正文段落（人工看解析结果是否一致）和表格结构（行列数是否正确、单元格内容是否对齐）。MinerU 的表格还原准确率大概 85%，DeepDoc 大概 60%，差距主要在无边框表格和合并单元格上。
3. 成本方面，阿里云按页计费日均万页成本不低，MinerU 私有化部署后只有 GPU 算力成本，边际成本降了 70%+。
4. 样本量不算大，但按文档复杂度分了四档（纯文本、单栏带表格、双栏排版、混合排版），重点关注"在哪些类型上差距大"，不是追求统计显著性——这不是发论文，是做工程决策。

**参考话术：**
"不是拍脑袋决定的。我们从生产文档里按复杂度分四档各挑了十几份，同一批文档分别走 DeepDoc 和 MinerU，人工逐份对比。表格还原准确率 MinerU 大概 85%，DeepDoc 大概 60%，差距主要在无边框表格和合并单元格。成本方面，阿里云按页计费，MinerU 私有化部署后边际成本降了 70%+。样本量不算大，但复杂文档那一档的差距大到不需要统计检验就能看出来。"

### 11.2 BBox 聚类大纲生成的实现与边界

**面试官原话：**
"你说 BBox 高度容差聚类来区分 H1/H2/H3，具体怎么做的？聚类算法是什么？容差阈值怎么定的？"

**问题本质：** 是真的自己实现了，还是只是知道概念。

**回答思路：**

1. 不是通用聚类算法（K-Means 之类的），因为标题层级数未知且样本量小。做法是：提取所有 title 元素的 BBox 高度，按高度降序排列，相邻高度差小于容差阈值的归为同一组，高度最大的组映射为 H1，次大 H2，以此类推。
2. 容差阈值是相对值——同一文档内最大标题高度的 10%-15%。不同 PDF 的渲染 DPI 不同，绝对像素值没有可比性。
3. 标题折行的情况通过 MinerU 的 `middle.json` 里 lines 数量做归一化：实际高度 = BBox 高度 / lines 数量。
4. 边界情况：如果 H1 和 H2 字号差异很小（只差 1pt），BBox 高度无法区分，聚类后只得到一个层级，退化为"平铺大纲"——所有标题作为同级节点。Agent 拿到平铺大纲仍然可以导航，只是失去层级信息，定位效率下降但不完全失效。
5. 无标题文档（扫描件、纯文本）：降级到非 Agentic 架构走老方案多路检索，或者按页码构建虚拟章节（每 3-5 页一个虚拟章节），如果 Agent 在虚拟大纲上表现明显差就自动降级。

**参考话术：**
"不是 K-Means，样本量太小不适合。做法是提取所有 title 的 BBox 高度，按高度降序排列，相邻高度差小于容差阈值的归为同一组。容差是相对值——最大标题高度的 10%-15%，因为不同 PDF 的 DPI 不同，绝对像素值没有可比性。折行标题通过 lines 数量归一化。如果字号差异太小聚类失败，退化为平铺大纲——所有标题同级，Agent 仍然能导航只是效率降低。无标题文档直接降级到老方案的多路检索，或者按页码构建虚拟章节试一下，不行就降级。"

### 12.1 Reader 之间的 P2P 通信——循环引用怎么办？

**面试官原话：**
"Reader 之间的 P2P 通信，你怎么保证不会出现循环引用？A 问 B，B 又问 A？"

**问题本质：** 分布式系统中的死锁/活锁意识。

**回答思路：**

1. 循环引用不会导致死锁——每个节点都是异步的。A 等 B 的时候还在干活或处理其他收件箱消息，B 的 subAgent 等 A 的时候也是独立线程，不存在互相阻塞。
2. 但调用链变长需要控制。防护机制：每个跨文档请求带 `depth` 字段，A.main → B.sub 是 depth=1，B.sub → A.sub 是 depth=2。depth >= 2 时硬拦截，不允许再发起跨文档请求——对方的 subAgent 只能在自己文档内尽力回答。
3. 两层嵌套已经覆盖了绝大多数实际场景（A 引用 B 的标准，B 的标准里又引用 A 的定义），三层以上收益极低但复杂度翻倍。
4. 实测跑了几十组文档，depth=2 的情况偶尔出现（约 5%），depth 被拦截的情况从未发生过。

**参考话术：**
"循环引用在我的设计里不会导致死锁。走一遍调用链：A 问 B，B 起 subAgent 处理，subAgent 发现需要 A 的信息又问 A，A 起 subAgent 回答。整个链路每个节点都是异步的，不存在互相阻塞。但调用链变长需要控制——每个跨文档请求带 depth 字段，depth >= 2 时硬拦截，subAgent 只能在自己文档内尽力回答。两层嵌套覆盖了绝大多数场景，实测 depth=2 的情况只有 5%，被拦截的从未发生过。"

### 12.2 子 Agent 的上下文隔离——为什么不直接在主 Loop 里处理？

**面试官原话：**
"子 Agent 处理跨文档请求，为什么不直接在 Reader B 的主 Loop 里处理？"

**问题本质：** 对上下文污染问题的理解深度。

**回答思路：**

1. 如果直接在 B 的主 Loop 里处理 A 的请求，A 请求的内容和 B 自己的提取目标会混在同一个 messages 上下文里。LLM 后续轮次可能混淆——把为 A 查到的信息当成自己的提取结果，或者被 A 的请求干扰偏离原本的 extraction_goal。
2. 子 Agent 的隔离方式：启动独立 Agent 实例，全新 messages 列表，只注入 B 文档的大纲和 A 的具体请求。跑完后只把最终结果通过 MessageBus 回复 A，中间推理过程全部丢弃。B 的主 Loop 完全不知道这件事发生过。
3. 代价是多了一个 Agent 实例的 LLM 调用开销（通常 2-3 轮），但换来的是主 Loop 上下文的纯净性。长文档场景下，上下文纯净比省几次 LLM 调用重要得多。

**参考话术：**
"如果在 B 的主 Loop 里处理 A 的请求，两个不同目标的内容会混在同一个 messages 里，LLM 后续轮次可能混淆——把为 A 查到的信息当成自己的提取结果。子 Agent 是完全隔离的：独立 messages、只注入 B 的大纲和 A 的请求，跑完后只回传结果，中间过程丢弃。B 的主 Loop 完全不受影响。代价是多 2-3 轮 LLM 调用，但换来的是上下文纯净性，长文档场景下这个 trade-off 很值。"

### 12.3 跨文档请求的等待机制和超时处理

**面试官原话：**
"Reader A 发出跨文档请求后是怎么等待回复的？如果回复一直不来怎么办？"

**问题本质：** 异步通信的等待机制和超时处理。

**回答思路：**

1. Reader A 发出请求后不阻塞，继续阅读自己文档的其他章节。每轮 Agent Loop 开头检查收件箱——有回复就注入 messages 继续分析，没有就继续做自己的事。
2. 当 A 无其他章节可读时，进入轮询等待模式：每隔 2 秒检查收件箱，带 60 秒超时兜底。超时后放弃等待，在提取结果中标注"跨文档引用未获取到，相关信息可能不完整"。
3. 大多数情况下，A 等跨文档回复的同时还在读自己的其他章节，等它读完自己的部分，跨文档回复通常也到了。
4. 还有一种更极端的情况：Reader A 后续的执行路径全部依赖跨文档回复。比如文档 A 是设计方案，第 3 节写"消防设计按照 B 标准执行"，Reader A 发现后面的第 4 节"消防验收"、第 5 节"设备清单"都依赖 B 标准的具体要求才能判断是否合规——这时候即使还有其他章节没读，读了也没意义，因为不知道 B 标准要求什么。
   - 这种情况下，决策 LLM 会判断"当前信息不足以继续，需要等待 B 的回复"，输出一个等待动作而不是硬找其他章节读。这是 ReAct "走一步看一步"的正常行为——"这一步是等"也是一个合理的决策。
   - 然后直接进入轮询等待模式，和上面第 2 点一样的机制。
   - 超时兜底：60 秒还没收到回复，Reader A 基于已有信息给出不完整的提取结果，标注"后续分析依赖文档 B 的消防标准要求，未获取到"。这个信息会传递到汇总 Agent，汇总 Agent 的反馈闭环可以在汇总阶段针对性地要求 Reader B 补充这部分信息，所以即使跨文档请求超时了，最终结果也不一定有缺失。

**参考话术：**
"A 发出请求后不阻塞，继续读自己的其他章节。每轮 Loop 开头检查收件箱，有回复就用，没有就继续干活。大多数情况下 A 读完自己的部分时回复也到了。

但有一种更极端的情况——A 后续所有的执行路径都依赖跨文档回复。比如文档 A 是设计方案，第 3 节说'消防设计按照 B 标准执行'，后面的验收标准、设备清单全都依赖 B 标准的具体要求，这时候即使还有其他章节没读，读了也没意义。这种情况下决策 LLM 会判断'当前信息不足以继续'，输出一个等待动作——ReAct 走一步看一步，'这一步是等'也是合理的决策。然后进入轮询等待，60 秒超时兜底。超时后 A 基于已有信息提交不完整的结果，标注'后续分析依赖文档 B 的 XX 信息，未获取到'。这个标注会传递到汇总 Agent，汇总 Agent 的反馈闭环可以在汇总阶段针对性地要求 Reader B 补充，所以即使跨文档请求超时了，最终结果也不一定有缺失。"

### 13.1 小模型格式纠错的具体实现和数据

**面试官原话：**
"你说小模型格式错误的自动恢复率 95%+，这个数据怎么来的？剩下 5% 怎么处理？纠错反馈具体怎么注入的？"

**问题本质：** 数据是否可信，异常处理是否完整。

**回答思路：**

1. 统计方法：上线后跑了两周生产日志，约 2000 次格式异常，1900+ 次在 1-2 轮纠错后恢复，恢复率约 95%。
2. 剩下 5%：约 3% 是连续 3 轮都输出错误格式（超过最大重试次数），降级返回兜底回答 + 记录错误日志；约 2% 是合法 JSON 但语义错误（参数填错），在工具执行层报错后触发重试。
3. 纠错信息包含三部分：① 你上一轮的输出是什么（原文截取前 200 字符）；② 哪里出了问题（具体错误描述）；③ 正确的格式示例（从工具定义中提取）。第 3 部分是关键——给 LLM 一个正确示例比只告诉它"你错了"有效得多。

"上线后跑了两周生产日志统计的，约 2000 次格式异常，95% 在 1-2 轮纠错后恢复。剩下 5% 里，3% 是连续 3 轮都错直接降级返回兜底回答，2% 是合法 JSON 但参数填错在工具执行层报错后重试。纠错信息不是简单地说'你错了'，而是三部分：你上一轮输出了什么、哪里出了问题、正确格式长什么样。第三部分最关键——给 LLM 一个正确示例比只告诉它错了有效得多。"

### 13.2 Agent 死循环的具体模式和防护

**面试官原话：**
"你的 Agent Loop 最多跑多少轮？有没有遇到过 Agent 陷入死循环的情况？"

**问题本质：** Agent 系统的可控性和安全边界。

**回答思路：**

1. 单 Agent 最多 15 轮，多 Agent 的 Reader 也是 15 轮，汇总 Agent 最多 1 轮（逻辑上一轮就够，前面 14.5 有详细论证）。
2. 遇到过两种死循环模式：
   - 乒乓循环：Agent 在两个章节之间反复跳。解决方案：visitedNodeIds 集合 + 工具层硬拦截（已读章节直接返回之前的摘要，不实际执行读取）。
   - 关键词穷举：Agent 用完有意义的关键词后生成无意义的细化变体（"违约" → "违约金" → "违约责任" → "违约条款"……）。解决方案：usedKeywords 集合 + **子串包含关系判断**——新关键词如果是已用关键词的超集（包含已用关键词作为子串），直接拦截。比如已经搜过"违约"，再搜"违约金"就会被拦住，因为 BM25 搜"违约"能命中的结果，搜"违约金"一定也能命中，后者只是更窄的子集，不会带来新信息。这比语义去重更精确、更轻量，不需要过 embedding 模型。
3. 还有一种对称的情况：Agent 连续精读章节但都没有有用信息——大纲上看着相关，读完发现不对。这时候 Agent 容易陷入"反复选章节但都扑空"的循环。我的处理是**精读连续失败时自动降级到关键词搜索**：
   - 触发条件：连续 2 次 `read_section` 的精读 LLM 返回 `not_relevant`（即精读 LLM 明确判定"本章节未涉及与当前目标相关的内容"），系统自动触发降级，不依赖决策 LLM 自己判断"有没有用"——精读 LLM 的判断比决策 LLM 更可靠，因为它刚读完原文，决策 LLM 只看到精读摘要。
   - 降级流程：① 用一个轻量 LLM 基于用户原始问题 + 已经读过但无用的章节标题列表做查询改写——"这些章节都不对，换个方向想想该搜什么关键词"，避开已经试过的方向；② 改写后的关键词直接走 `search_within_doc`（BM25 全文检索），取 top 结果；③ 如果搜索有命中，按章节归并后交给决策 LLM 选择精读。
   - 和第 2 点的关键词穷举防护不冲突：降级生成的关键词同样要过 usedKeywords 的子串包含检查，不会重复搜已经搜过的词。
   - 和第 4 点的向量检索降级形成完整降级链：精读连续失败 → 降级到关键词搜索 → 如果关键词搜索也连续失败 → 再降级到向量检索。三层 fallback，每层解决不同类型的"找不到"问题：精读失败说明大纲定位偏了（换检索方式），搜索失败说明关键词不对（换检索模态）。
   - 状态记录：写进 actionHistory 的是一条合并记录（"精读连续未命中，降级关键词搜索，关键词：XX，命中章节：YY"），visitedNodeIds 和 usedKeywords 照常更新。
4. 还有一种更棘手的情况：大纲看不出该去哪读，关键词搜索也连续扑空。这时候 Agent 容易陷入"反复换关键词但都搜不到"的死循环。我的处理是**自动降级到向量检索**：
   - 触发条件：连续 2 次 `search_within_doc` 返回空结果或命中数极低，系统自动触发降级流程，不需要决策 LLM 参与判断。
   - 降级流程：① 用一个轻量 LLM 基于用户原始问题 + 之前失败的关键词列表做查询改写（避开已经试过的方向）；② 改写后的 query 做 embedding 向量检索，取 top-5 相关 chunk；③ top-5 chunk 按所在章节去重（多个 chunk 落在同一章节只算一个），去重后如果超过 2 个章节只取前 2 个；④ 直接调用 `read_section` 精读这些章节，不需要决策 LLM 再做一次"选哪个章节"的决策。
   - 状态记录：虽然这一轮实际执行了 1 次 search + 1 次查询改写 + 1 次向量检索 + 1-2 次 read 精读，但写进 actionHistory 的是一条合并记录（"关键词搜索连续失败，降级向量检索，命中章节 X/Y，精读结果：..."），不是多条独立记录——决策 LLM 下一轮看到的是一个完整的降级流程结果，不会困惑。visitedNodeIds 和 usedKeywords 照常更新。
   - 本质上这是"大纲导航失败时 fallback 到传统 RAG 的向量检索，但检索到的不是碎片 chunk 而是 chunk 所在的完整章节"——比纯传统 RAG 好（拿到完整上下文），比纯大纲导航鲁棒（多了一条退路）。
5. 这些问题在小模型上更严重，大模型的决策稳定性好很多。

**参考话术：**
"单 Agent 最多 15 轮，硬限制，汇总 Agent 最多 1 轮。遇到过两种死循环：一是乒乓循环——在两个章节之间反复跳，靠 visitedNodeIds 集合加工具层硬拦截解决，已读章节直接返回之前的摘要不实际执行。二是关键词穷举——用完有意义的词后生成无意义的细化变体，比如搜过'违约'又搜'违约金'。我的处理不是语义去重，而是更简单直接的子串包含判断——新关键词如果包含已用关键词作为子串，直接拦截。因为 BM25 搜'违约'能命中的结果，搜'违约金'一定也能命中，后者只是更窄的子集，不会带来新信息。这比过 embedding 模型做语义相似度更精确也更轻量。

还有一种情况是反过来的：Agent 通过大纲选章节去精读，但连续两次精读 LLM 都返回'本章节未涉及相关内容'——大纲上看着相关，读完发现不对。这时候系统自动降级到关键词搜索。触发条件是精读 LLM 连续 2 次返回 not_relevant，注意是精读 LLM 判断的，不是决策 LLM 自己觉得没用——精读 LLM 刚读完原文，判断比只看摘要的决策 LLM 可靠得多。降级流程是：用一个轻量 LLM 基于用户问题加上已经读过但无用的章节标题列表做查询改写——'这些章节都不对，换个方向想想该搜什么'，改写后的关键词走 search_within_doc 做 BM25 检索。生成的关键词同样要过 usedKeywords 的子串包含检查，不会重复搜。

这和前面的向量检索降级形成了一条完整的降级链：精读连续失败说明大纲定位偏了，降级到关键词搜索换检索方式；关键词搜索也连续失败说明关键词不对，再降级到向量检索换检索模态。三层 fallback，每层解决不同类型的'找不到'问题。

还有一种更棘手的情况：大纲看不出该去哪读，关键词搜索也连续扑空。这时候我做了自动降级到向量检索的机制——连续 2 次 search 返回空结果时自动触发，不需要决策 LLM 判断。流程是：先用一个轻量 LLM 基于用户问题加上之前失败的关键词列表做查询改写，避开已经试过的方向；改写后做向量检索取 top-5 chunk，按章节去重后只保留前 2 个章节，直接调 read 工具精读。虽然这一轮实际跑了 search、改写、向量检索、精读好几个动作，但写进 actionHistory 的是一条合并记录——决策 LLM 下一轮看到的是一个完整的降级结果，不会困惑。本质上就是大纲导航失败时 fallback 到传统 RAG 的向量检索，但拿到的不是碎片 chunk 而是 chunk 所在的完整章节，比纯传统 RAG 好，比纯大纲导航鲁棒。"

### 13.3 read_section 精读隔离——为什么不让原文进决策 LLM？

**面试官原话：**
"你的 `read_section` 工具内部还调了一次 LLM 做精读，这不是多了一次 LLM 调用吗？为什么不直接把章节原文返回给决策 LLM，让它自己读？多一次调用意味着多一次延迟和 token 消耗，你觉得值吗？"

**问题本质：** 考你对 Agent 架构中"决策 LLM 上下文纯净性"的理解——为什么要在工具层做一次信息提炼，而不是把原始内容直接塞给决策 LLM。

**回答思路：**

1. 核心设计意图：**原文从头到尾不进入决策 LLM 的 messages**。决策 LLM 要的只是两样东西：① 这个章节里跟当前阶段目标相关的关键信息（精读结果）；② 下一步应该往哪个方向继续探查（精读 LLM 给出的建议）。原始章节内容对决策 LLM 来说是噪音——一个 3000 token 的章节里可能只有 200 token 跟当前问题相关，剩下 2800 token 全是干扰。
2. 如果把原文直接塞给决策 LLM，会有三个问题：
   - **上下文污染**：Agent 跑 5 轮，每轮读一个章节，5 个章节的原文全堆在 messages 里就是 15000 token 的噪音。决策 LLM 在这堆噪音里做下一步决策，注意力被稀释，决策质量必然下降。
   - **Token 预算被挤占**：原文占的空间越多，留给状态快照、Action 历史、大纲的空间就越少，Agent 能跑的轮次也越少。
   - **Lost in the Middle 被放大**：原文塞在 messages 中间，正好是 LLM 注意力最弱的位置。
3. 精读 LLM 的输出是结构化的三部分：① 针对阶段目标的回答（精炼后通常 200-500 token）；② 关键证据引用（原文片段 + 页码定位，供最终回答时引用）；③ 下一步探查建议（比如"建议继续查看第 4.2 节免责条款"）。决策 LLM 拿到的是干净的、跟当前目标直接相关的信息，而不是一大段需要自己从中筛选的原文。
4. 多出来的一次 LLM 调用的成本是值得的：精读用的可以是更小的模型（因为任务简单——给定章节和目标，提取相关信息），延迟 200-300ms；换来的是决策 LLM 每轮 messages 里只有精炼后的几百 token 而不是几千 token 的原文，整个 Session 的总 Token 反而更少，决策质量也更高。
5. 这个设计本质上是把"阅读理解"和"策略决策"两个能力解耦了——精读 LLM 负责"读懂这段话"，决策 LLM 负责"下一步该干什么"。各司其职，互不干扰。

**参考话术：**
"这是一个有意为之的架构决策——原文从头到尾不进入决策 LLM 的 messages。决策 LLM 要的只是两样东西：这个章节里跟当前目标相关的关键信息，以及下一步该往哪个方向继续探查。原始章节内容对它来说是噪音——一个 3000 token 的章节可能只有 200 token 跟当前问题相关，剩下全是干扰。如果把原文直接塞进去，Agent 跑 5 轮就是 15000 token 的噪音堆在 messages 里，决策质量必然下降，Token 预算也被挤占，Lost in the Middle 效应还会被放大。精读 LLM 的输出是结构化的：针对阶段目标的精炼回答、关键证据引用带页码、下一步探查建议。决策 LLM 拿到的是干净的、直接可用的信息。多出来的一次 LLM 调用延迟大概 200-300ms，精读任务简单可以用更小的模型，但换来的是决策 LLM 每轮只看几百 token 的精读结果而不是几千 token 的原文，整个 Session 的总 Token 反而更少。本质上是把'阅读理解'和'策略决策'两个能力解耦了，各司其职。"

### 14.1 Lost in the Middle 问题——你的方案怎么缓解的？

**面试官原话：**
"你提到 Lost in the Middle 问题，能具体解释一下吗？你的方案是怎么缓解的？"

**问题本质：** 对 LLM 局限性的学术理解 + 工程解法。

**回答思路：**

1. Lost in the Middle 是 2023 年 Stanford 论文提出的现象：LLM 输入很长时，对开头和结尾关注度高，中间部分注意力显著下降。
2. 项目一：大纲驱动 + 按需阅读，Agent 每轮只读 1-2 个章节，context 里的文档内容始终是精炼的、直接相关的，不会出现"中间塞了一堆无关内容"。
3. 项目二：每个 Reader 只负责一篇文档，Synthesizer 拿到的是结构化提取结果（几百 tokens），不是原始文档。通过分治把每个 Agent 的 context 长度控制在安全范围内。
4. 本质上不是"解决"Lost in the Middle，而是通过架构设计"绕过"它——让每个 LLM 调用的 context 都足够短。

**参考话术：**
"我的方案不是解决 Lost in the Middle，而是绕过它。项目一通过大纲驱动按需阅读，Agent 每轮只读 1-2 个章节，context 始终精炼。项目二通过分治，每个 Reader 只处理一篇文档，Synthesizer 拿到的是结构化提取结果不是原始文档。每个 LLM 调用的 context 都足够短，短到 Lost in the Middle 效应不显著。"

### 14.2 不相关问题的处理——系统会怎么表现？

**面试官原话：**
"如果用户问的问题跟所有文档都不相关怎么办？你的系统会怎么表现？"

**问题本质：** 边界情况处理，系统的鲁棒性。

**回答思路：**

1. 文档级过滤拦住：top_doc 检索没有返回相关文档（相似度低于阈值），直接回复"未找到相关文档"，不启动 Agent Loop。
2. 过了过滤但实际不相关：Agent 读大纲发现没有相关章节，1-2 轮后选择"文档未涉及"，返回兜底回答。
3. 最怕的是文档看起来相关但实际回答不了（比如用户问 2024 版标准，文档是 2020 版）。Agent 可能"硬答"——从文档里找看起来相关的内容拼凑回答。缓解措施是最终回答附带引用来源（章节和页码），让用户自行判断。

**参考话术：**
"分三种情况。第一种，文档级过滤就拦住了，直接回复未找到相关文档。第二种，过了过滤但 Agent 读大纲发现没相关章节，1-2 轮后声明文档未涉及。最怕第三种——文档看起来相关但实际回答不了，比如版本不对。Agent 可能硬答，这是 LLM 幻觉的变体，没有完美解法。我的缓解是每个回答附带引用来源精确到章节和页码，让用户自行验证。"

### 14.3 长上下文模型会淘汰 Agentic RAG 吗？

**面试官原话：**
"你觉得 Agentic RAG 这个方向，未来会被更长的上下文窗口淘汰吗？比如 Gemini 已经支持 1M tokens 了。"

**问题本质：** 技术判断力和前瞻性。

**回答思路：**

1. 短期不会，长期部分会。窗口变长解决的是"装得下"，但解决不了三个问题：
   - 成本：1M tokens 单次调用成本是 10K 的 100 倍，高频场景不可接受。
   - Lost in the Middle 不会消失：窗口越长中间注意力衰减越严重。
   - 可控性和可解释性：全量塞入是黑盒，Agentic RAG 每一步都有记录。
2. 如果窗口足够长且注意力问题被解决，简单问题确实可以直接塞入。Agentic RAG 的价值会收敛到复杂多跳推理和跨文档分析。
3. 这也是做项目二（多 Agent）的原因——即使单文档问题被长窗口解决了，跨文档对比分析仍然需要分治和协作。

**参考话术：**
"短期不会，长期部分会。窗口变长解决的是'装得下'，但解决不了成本（1M tokens 单次调用成本是 10K 的 100 倍）、Lost in the Middle（窗口越长中间注意力衰减越严重）、可解释性（全量塞入是黑盒）。如果注意力问题被解决，简单问题确实可以直接塞入，Agentic RAG 的价值会收敛到复杂多跳推理和跨文档分析。这也是我做项目二的原因——跨文档对比分析需要分治和协作，这个需求不会因为窗口变长而消失。"

### 14.4 Planner 的 DAG 依赖检测

**面试官原话：**
"Planner 怎么判断任务之间有没有依赖关系？DAG 依赖是怎么标记的？"

**问题本质：** 对 Planner 设计细节的理解。

**回答思路：**

1. 绝大多数场景下各文档的提取任务互相独立，`depends_on` 字段为空，全部并行启动。
2. 少数存在依赖的场景（如"先总结 A 的结论，再用该结论评估 B"），Planner 的 Prompt 要求它分析任务间逻辑依赖，在 task B 的 `depends_on` 里标记 task A 的 ID。系统先启动无依赖 task，等依赖 task 完成后再启动后续 task。
3. DAG 依赖检测完全依赖 LLM 理解能力，不是硬编码规则。明确的顺序依赖（"先……再……"）判断准确率很高，隐含依赖可能漏判。漏判时 Synthesizer 的反馈闭环可以兜底。

**参考话术：**
"绝大多数场景下任务互相独立，全部并行。少数有依赖的场景，Planner 通过 LLM 分析任务间逻辑依赖，在 depends_on 字段里标记。明确的顺序依赖判断准确率很高，隐含依赖可能漏判——但 Synthesizer 的反馈闭环可以兜底，发现缺口后要求补充。"

### 14.5 Synthesizer 反馈闭环为什么最多一轮？

**面试官原话：**
"汇合agent 的反馈闭环你说最多一轮，为什么不多轮？如果一轮补充后还有缺口呢？"

**问题本质：** 收敛控制的设计决策。

**回答思路：**

1. 一轮不是"硬限制"，而是逻辑上就是充分的。Synthesizer 汇总时，手里已经有了所有 Reader 的完整提取结果——哪些对比维度存在、哪些文档缺了哪个维度，在这一刻就全部清楚了。补充请求发出去后，Reader 要么找到了补上，要么确认文档里没有（标注 not_found）。不管哪种结果，都不会产生新的对比维度——新维度只能从文档内容中发现，而 Reader 第一轮提取时已经把自己文档里有的维度都提出来了。所以补充一轮后如果还有空缺，100% 是文档中确实不存在该信息，再问一遍也不会凭空出现。
2. 实测数据印证了这个逻辑：一轮补充后对比表格的完整度从 60% 到 95%，剩余 5% 经人工检查全部是文档中确实不存在的信息。第二轮补充的边际收益是零，不是"接近零"。
3. 唯一的理论例外：Reader 补充时意外发现了一个全新维度（之前所有 Reader 都没提到的）。但这种概率极低——如果某个维度在所有文档的第一轮提取中都没出现，说明它要么不重要，要么藏得很深。即使出现了，它只影响一个文档的一个维度，不值得为此再跑一轮全量补充（成本是所有有空缺的 Reader 重新启动 Agent Loop）。
4. 区分"文档中确实没有"和"Reader 没找到"：补充请求的 Prompt 要求 Reader 输出 `found` 或 `not_found`，如果声明 not_found 但只检查了 1 个章节，Synthesizer 会质疑要求扩大搜索范围。通过要求列出已检查章节列表做事后审计。这个机制保证了一轮补充的质量——不是随便搜一下就说没有。

**参考话术：**
"一轮不是拍脑袋的硬限制，而是逻辑上就是充分的。Synthesizer 汇总时，所有 Reader 的提取结果已经全部到位了，哪些维度存在、哪些文档缺了什么，这一刻就全部清楚了。补充请求发出去后，Reader 要么找到了补上，要么确认文档里没有。不管哪种结果，都不会产生新的对比维度——新维度只能从文档内容中发现，而 Reader 第一轮提取时已经把自己文档里有的维度都提出来了。所以补充一轮后如果还有空缺，100% 是文档中确实不存在该信息，再问一遍也不会凭空出现。实测也印证了这个逻辑——一轮补充后对比表格完整度从 60% 到 95%，剩余 5% 全部是文档中确实没有的信息。区分'确实没有'和'没找到'靠 Reader 的显式声明——必须输出 found 或 not_found，如果声明 not_found 但只查了 1 个章节，Synthesizer 会质疑要求扩大搜索范围。这个机制保证了一轮补充的质量，不是随便搜一下就说没有。"
