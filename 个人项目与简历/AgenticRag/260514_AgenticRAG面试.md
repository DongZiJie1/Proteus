## Q1：砍掉 Rerank 层

**你的回答：** 砍掉 Rerank 的核心逻辑是长文档多跳推理场景下，Rerank 无论 pointwise 还是 listwise 都无法解决碎片化检索的根问题——chunk 间逻辑链断裂和召回阶段缺失。用 lost in the middle 反驳 listwise 方案（RankGPT），用"杀/人"例子说明 chunk 切分边界不可控。替代方案是大纲驱动全局感知 + 按需精读完整章节。

**追问过程：**
- Q: pointwise vs listwise（RankGPT）本质区别？→ A: 用 lost in the middle 反驳，但数字估算有误（40 chunk × 5000 字符 ≠ 200K token）→ ⚠️
- Q: 5000 字符 chunk 为何还切碎逻辑结构？→ A: "杀/人"切割边界例子 → ✅ 有说服力
- Q: 选章节 vs 选 chunk 在 LLM 认知负担上有什么区别？→ A: 标题天然摘要 + 向量匹配对多跳问题无效 → ✅ 核心论据清晰
- Q: 项目一有子 Agent 吗？→ A: 混淆了项目二的多 Agent 架构（说 read_section 用子 Agent），后纠正为"工具内嵌小模型" → ⚠️ 架构边界模糊
- Q: read_section 小模型细节（模型、参数、部署）？→ A: 32B 外部 API，"叫小模型"，JSON 四字段输出（摘要/线索/证据/下一步建议），2-3 秒延迟 → ⚠️ 32B 称为"小模型"易被挑战
- Q: "下一步建议"字段与 Agent 决策边界？→ A: 被追问后承认可能是冗余设计，决策边界没想透 → ⚠️ 架构瑕疵暴露
- 知识边界：第 5 层——架构决策的表述一致性、模块边界清晰度、以及被挑战后的自我反思能力

**点评：** 砍 Rerank 的方向判断正确且有实战数据支撑（65%→90%+）。三个真实问题：
1. **表述混淆项目一和项目二的架构元素**——read_section 是工具调用而非子 Agent，"子 Agent 不污染主上下文"是项目二的 P2P 跨文档引用设计，面试中这种混淆会严重损耗信任；
2. **32B 称为"小模型"会在正式面试中被追问翻车**——行业共识 32B 是中等模型，尤其是在你已经接入外部 API 大模型做决策的前提下，这个说法自相矛盾；
3. **read_section 的"下一步建议"字段与 Agent 规划 LLM 存在决策边界模糊**——工具提供建议 vs Agent 做规划，边界在哪？你在被追问后也承认了没想透，这是真实的架构设计瑕疵，而非表述问题。

**正确/完整答案：** Rerank 的根本问题不是 pointwise vs listwise 的选型，而是无论什么 Rerank 方案都依赖上游召回质量。核心创新在于把"文档理解"从检索阶段前移到解析阶段（大纲预计算），用结构化的文档元信息替代基于语义相似度的碎片召回——改变的是 Agent 获取信息的范式（从"搜碎片"到"翻目录"）。

**评级：** `⚠️ 方向对但细节不足`
