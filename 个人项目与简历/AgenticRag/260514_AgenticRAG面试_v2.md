## Q3：read_section "下一步建议"字段与决策边界（上午弱项重考）

**你的回答：** 精读 LLM 能看到原文中的交叉引用（"详见第 4.2 节"），决策 LLM 看不到原文所以不知道这个线索。返回的是事实陈述（"原文引用了第 4.2 节"），不是决策建议（"建议读第 4.2 节"）。决策 LLM 根据这个事实自己判断。

**追问过程：**
- Q: 精读 LLM 给"建议"还是决策 LLM 做规划？→ A: 信息差（精读 LLM 看原文 vs 决策 LLM 看大纲）决定了有价值 → ⚠️ "建议"一词仍模糊
- Q: 具体是事实陈述还是决策建议？→ A: 前者，只报告原文中存在的事实引用 → ✅ 边界清晰

**点评：** 比上午"没想透"进步明显。上午回答是"可能是冗余设计，决策边界没想透"，本轮明确了信息差（原文交叉引用 vs 大纲看不到）是保留该字段的正当理由，事实陈述 vs 决策建议的区分给出了清晰的架构边界。

**评级：** `✅ 基本正确`

## Q5：Chunk 评分文档内排名衰减公式 0.5^(k-1)

**你的回答：** 0.5 指数衰减参考论文，核心目的是防止超长文档靠 Chunk 数量作弊。排序衰减是数学手段，与语言和 Chunk 大小无关。BM25 排第 5 就是第 5，不需要救回——最终目标是文档总得分排名取 top-5，跨 5 个文档以上有兜底措施。

**追问过程：**
- Q: 为什么指数衰减、底数 0.5？论文场景一致吗？→ A: 参考论文，本质是排序衰减 → ✅ 设计意图清晰
- Q: BM25 误排怎么办？→ A: 排第 5 就是第 5，最终看文档排名非 Chunk 排名 → ✅ 逻辑自洽

**点评：** 公式来源（论文）记不清可接受。设计意图——防止超长文档作弊、最终输出是文档级排序而非 Chunk 级排序——讲清楚了。

**评级：** `✅ 基本正确`

## Q12：知识图谱反思——为什么"一开始就不该引入"

**你的回答：** KG 引入动机是补充 Chunk 间逻辑关联，在一些场景确实能找到向量/关键词找不到的 Chunk。两个致命问题：(1) 实体消歧不行（"苹果"手机 vs "苹果"水果），(2) 构建成本以每 5 chunk/次 LLM 调用线性增长，长文档不可持续。短文档效果好但不需要这么重的工具。测试集标注成本高，需标注查询路径/相关文档/目标章节。

**追问过程：**
- Q: KG 具体失败原因？→ A: 同名实体消歧 + 关系抽取不准 + 构建成本线性增长 → ✅ 三点覆盖
- Q: 同名实体场景？→ A: "苹果"手机 vs "苹果"水果 → ✅ 经典案例

**点评：** 坦诚架构决策失误并准确分析根因，面试加分项。测试集标注的具体数量未给出——如果被追问"到底多少条"需要数字。

**评级：** `✅ 基本正确`

## Q14：Java 原生 Loop vs LangChain AgentExecutor

**你的回答：** Java 原生实现的核心价值在于对 Prompt/Loop 逻辑/占位符替换的百分之百控制，不引入框架依赖。状态管理可以解决，错误恢复不需要（因为是"无状态一次 RAG 行为"），可观测性依靠 Trace 完整查看每次 LLM 思考过程。

**追问过程：**
- Q: 你说"不需要错误恢复"，但你的格式纠错闭环就是错误恢复——怎么解释？→ A: Trace 日志可回放重跑 → ⚠️ 把"日志记录"等同于"状态持久化"
- Q: 与 LangChain 的 callback 异常时状态丢失对比？→ A: 通过 Trace 找到日志和原始 Prompt 即可重跑 → ⚠️ 没区分"调试回放"和"自动恢复"

**点评：** Java 原生实现的优势（完全控制、无框架约束）讲得对。但两个问题：(1) 称 AGentic RAG 为"无状态一次行为"与你自己设计的 State 对象矛盾——15 轮 Loop 的每轮决策都依赖 State，这是有状态的多轮决策过程；(2) 没说清楚 State 对象相对于 LangChain callback 的核心优势——State 是结构化数据可序列化、可精确还原到任意轮次，而 LangChain callback 是事件流不易重建快照。

**评级：** `⚠️ 方向对但细节不足`
