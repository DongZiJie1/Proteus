# 企业级长文档智能问答系统（Agentic RAG 架构重构）

**技术栈：** Java, Python, MinerU, Elasticsearch, 知识图谱, LLM (32B 推理模型 + 7B 模型), vLLM, OpenTelemetry

**项目规模：** 核心代码 5000+ 行，3-4 个月主导架构重构从 0 到 1 落地。系统由文档解析层（Python/MinerU）+ Agent 编排层（Java 原生实现）+ 多粒度工具链三层组成

---

## 项目概述

公司原有 RAG 系统采用业界标准的"Chunk 切分 + 向量/关键词/知识图谱多路召回 + 重排序"线性管道架构。在长文档（5000+ tokens）和复杂多跳推理问题场景下，准确率仅 65%。

我在排查 bad case 时意识到根因：传统 RAG 模拟的是一个"没有目录、只能在碎纸堆里靠关键词翻找"的人——Chunk 切分把文档打碎成几百个片段，向量检索在碎片里做相似度匹配，模型拿到的是几段脱离上下文的片段，表头和表格分离、条款和引用断裂。但真人查一份 200 页的合同，绝不会这样做：一定是先翻目录建立全局认知，根据问题判断该看哪个章节，读完后评估信息是否充分，不够再去读下一个相关章节。

这个"模拟真人查资料认知过程"的思路，直接驱动了我将系统从线性管道向 Agentic 架构重构的方向——从"预计算索引（离线智能）"转向**"运行时认知（Runtime Cognition）"**。

**系统从设计之初就支持多文档同时检索**——用户输入的是问题 + 知识库编号，而非单份文档。一个知识库可能包含几十到上百份文档。进入 Agent Loop 之前，有一层**基于文档摘要的智能文档定位**，将候选范围从"整个知识库"收缩到"几份最可能相关的文档"，然后 Agent 在这些文档的章节间交叉跳转阅读。这与项目二的多 Agent 跨文档分析不同——项目一是单个 Agent 在多文档间自主导航，项目二是有多个 Agent 分工协作。

**架构上分为三层：**

- **文档解析层（Python/MinerU）：** 将 PDF/Word 等非结构化文档解析为结构化的大纲 JSON 树 + 章节文本 + 图片引用。采用视觉模型管线（DocLayout-YOLO 版面检测），自研后处理模块解决 MinerU 标题层级不分的问题
- **Agent 编排层（Java 原生实现）：** 状态驱动的 Agent Loop，State 对象维护已读章节、已用关键词、Action 历史、轮数/token 消耗等细粒度状态，每轮从 State 实时序列化快照注入 Prompt 驱动 LLM 决策
- **多粒度工具链：** `search_within_doc`（文档内关键词检索）、`read_section`（基于大纲精确定位章节）、`search_kg`（知识图谱遍历）、`top_doc`（全局文档检索），解耦 Search 与 Read

开发过程中遇到的核心技术挑战包括：MinerU 标题层级不分导致大纲树错误、7B 模型频繁输出格式异常、上下文窗口爆炸导致 Agent 无响应。通过自研 BBox 后处理模块、多层格式纠错闭环、状态机熔断机制逐一解决。

**归属：** 我独立完成了：① 文档解析层的选型论证与 MinerU 迁移（含 BBox 后处理模块）；② Agent Loop 的 Java 原生实现与 State 对象设计；③ 大纲驱动方案的设计与行号索引优化；④ 多粒度工具链的 Search/Read 解耦设计；⑤ 7B 模型异常处理与纠错闭环；⑥ 砍掉 Rerank 层的论证与决策。

## 文档解析层：从 RAGFlow + 阿里云迁移到 MinerU

### 原有方案的问题

项目初期解析层采用 RAGFlow 内置 DeepDoc 解析 + 阿里云文档智能（Document Mind）的组合方案。随着业务规模扩大，暴露三个结构性问题：

1. **RAGFlow DeepDoc 框架耦合：** DeepDoc 与 RAGFlow 框架深度绑定，解析中间产物（BBox 坐标、标题元素、表格结构）无法精细控制，二次开发成本极高。它是面向终端用户的产品，不是面向工程师的可编排组件
2. **阿里云文档智能合规风险 + 成本陡增：** 涉密文档出网是合规红线，且按页计费在日均万页量级下成本线性增长，不可持续
3. **两套引擎并行维护的代价：** 双倍异常处理和不一致的输出格式，下游 Agent 的 Prompt 依赖稳定结构化输入时，这种不一致被放大为检索质量波动

### 选型与迁移

核心选型诉求：私有化部署（合规）、模块化可编排（Agentic RAG 精细控制需求）、多模态元素高质量结构化输出（为 VLM 扩展预留空间）。

横向压测 MinerU、Unstructured.io、Marker 等开源方案后收敛到 MinerU：

- 插图精确裁剪保存、公式还原（UniMERNet）、表格识别（StructEqTable）上工程完成度更高
- 输出的 `文本 + 裁剪图片 + BBox 坐标` 富结构数据为多模态 RAG 留足扩展空间
- 直接走视觉模型管线（DocLayout-YOLO 版面检测），保证复杂排版（双栏、无边框表格、公式密集）的解析质量

### 解决 MinerU 的工程缺陷：标题层级不分

MinerU 上线后发现一个严重影响下游 Agent 的问题：`text_level` 字段无法区分标题层级深度，实测经常把所有标题打成 level 1。如果大纲树的 H1/H2/H3 层级全是错的，Agent 的章节导航就失去了意义。

根因分析：MinerU 的标题检测依赖版面检测模型输出的元素类型标签，但模型对"标题的层级"（一级标题 vs 二级标题）区分能力很弱，尤其在文档排版不规范时（标题字号差异小、标题折行等）。

**自研 BBox 后处理模块：** 基于标题元素 BBox 坐标高度做容差聚类——同一层级的标题通常渲染字号相近，BBox 高度也相近。具体做法：

```
MinerU 输出 → 提取所有标题元素的 BBox height →
  1. 按 height 降序排列，相邻 height 差值 < 容差阈值（2pt）的归为同一簇
  2. 簇按高度降序映射为 H1/H2/H3/...
  3. 输出修正后的大纲 JSON 树
```

这个后处理模块不依赖 MinerU 的 `text_level` 字段，完全基于版面几何信息重建层级，在标题格式规范的文档上准确率 95%+。

### 灰度切换机制

迁移过程设计了灰度切换：同一批文档同时走新旧两条管线，用 ROUGE-L + 人工抽检对比解析质量，确认 MinerU 在核心文档类型上达到或超过原方案精度后逐步切流。解析层设计为异步事件驱动微服务，解耦用户请求，视觉管线的高延迟不影响用户体验。

## 进入 Agent Loop 之前：基于文档摘要的智能文档定位

传统 RAG 的检索范围是"整个知识库的全量 Chunk"——用户问一个问题，系统在几万甚至几十万个 Chunk 里做向量相似度匹配。当知识库有上百份文档时，召回质量急剧下降：向量检索返回的是"最相似的 100 个 Chunk"，但这 100 个 Chunk 可能来自 50 份不同的文档，每份文档只有 2 个命中片段。Agent 看这 100 个碎片时完全没有文档级的全局感知。

我在 Agent Loop 之前加了一层**文档级预定位**，将候选范围从"整个知识库"收缩到"几份最相关的文档"。这个定位不是简单的向量匹配，而是通过一次 LLM 调用 + 文档摘要 + ES 全文搜索 + 自定义 Chunk 评分机制完成的。

### 文档摘要的两级设计

每份文档在上传时预计算两级摘要：

- **详细摘要（~500 字）：** 覆盖文档的主要章节、核心论点、关键数据。库中文件少时使用，LLM 可以基于详细摘要做出更精准的判断
- **简短摘要（~50 字）：** 一句话概述文档内容。库中文件多（如 100+ 份文档）时使用，防止摘要总长度撑爆上下文窗口

摘要随文档一起索引化存储，检索时按库内文档数量动态选择摘要层级。

### 三路分流的预判断

进入 Agent Loop 之前，系统用一次 LLM 调用分析用户问题 + 所有文档摘要，判断问题类型，走三条路径之一：

```
用户问题 + 知识库编号
  │
  ▼
抓取库中所有文档的摘要（文档多→简短摘要，文档少→详细摘要）
  │
  ▼
LLM 分析：用户问题 + 所有摘要 → 判断问题类型
  │
  ├── 路径A：总览类问题（"这个知识库讲的什么""有哪些主要内容"）
  │     → 直接生成回答，不进 Agent Loop
  │
  ├── 路径B：用户指定了文档（"XX标准里的消防要求""第三份合同怎么说的"）
  │     → 输出目标文档 ID，直接进入 Agent Loop 精读该文档
  │
  └── 路径C（最常见）：未指定文档，需要从摘要中提取关键词
        → LLM 输出关键词列表 → 进入 Chunk 检索与评分
```

大多数情况走路径 C：LLM 分析摘要后输出与问题相关的关键词列表，然后进入下一步。

### Chunk 检索与评分：关键词加权 + 文档内衰减

拿到关键词后，去 ES 做全文搜索（BM25），取分值最高的前 100 个 Chunk。这里有一个重要的**过滤步骤**：ES 的 BM25 分值是"部分匹配"的，一个 Chunk 可能被召回但实际不含任何关键词（只是相关词被匹配了）。系统在拿到 100 个 Chunk 后，先排除掉不含任何关键词的 Chunk。

然后做 **Chunk → 文档的 Map 操作**：将 Chunk 按所属文档归类。

接下来是核心的评分逻辑：

**第一步：关键词加权。** 每个 Chunk 的原始 ES 分数乘以一个关键词系数：

```
chunk_score = es_score × 1.2^n
```

其中 n 是该 Chunk 包含的去重关键词数量。包含 3 个关键词的 Chunk 得分是只含 1 个关键词的 Chunk 的 1.44 倍。这保证了**关键词密度高的 Chunk 排名大幅靠前**。

**第二步：文档内排名衰减。** 为防止超长文档作弊——假设一份 200 页的文档有 30 个 Chunk 命中了关键词，另一份 20 页的文档只有 5 个 Chunk 命中，但后者可能才是真正相关的——系统在每个文档内部对 Chunk 按得分排序，然后对排序后的 Chunk 施加指数衰减：

```
排名第 1 的 Chunk：score × 1.0  （不衰减）
排名第 2 的 Chunk：score × 0.5
排名第 3 的 Chunk：score × 0.25
排名第 k 的 Chunk：score × (0.5)^(k-1)
```

其效果是：一个文档被命中的 Chunk 再多，只有排名前几的 Chunk 对最终文档相关性贡献大，后面的 Chunk 得分被指数级压扁。一个真正相关的文档（Chunk 虽少但都排在前面）不会被一份"碰巧大范围命中关键词"的超长文档淹没。

### 进入 Agent Loop

经过上述评分后，系统按文档总得分排序，取排名前几的候选文档。进入 Agent Loop 之前，有一条快速判断：如果候选文档总 token 数低于阈值（20,000 token），直接全文注入 LLM 生成回答，跳过 Agent Loop。否则，Agent 拿到的是"经过预筛选的几份候选文档 + 各自的大纲 JSON 树"，而非"整个知识库的所有 Chunk"。这就从源头上解决了传统 RAG"大海捞针"的问题——Agent 面对的不是几百个碎片，而是几份有目录的、结构清晰的文档。

这个文档定位步骤与 Agent Loop 本身是分离的——定位只做一次（在 Loop 之前），Agent 在 Loop 内不再调用 `top_doc` 重新选择文档（除非 Observation 显示当前文档完全不相关）。这样 Agent 的上下文里看到的一直是聚焦的几份文档，不会在大量文档间跳来跳去。

## 大纲驱动的文档理解方案（核心创新）

### 问题：盲猜关键词

老架构的检索逻辑是：用户提问 → LLM 在未见文档的情况下盲猜关键词 → 用关键词去 ES/向量库搜 Chunk。这有两个致命缺陷：

- LLM 不知道文档里有什么，只能凭经验猜。猜偏了，后续召回的 Chunk 全是噪声
- 即使关键词猜对了，召回的 Chunk 是脱离文档逻辑结构的碎片，表头和表格可能来自不同的 Chunk，Agent 看到的是一堆无序片段

### 方案：让 Agent 先看到目录，按需加载章节

基于 MinerU 解析层输出的结构化数据，在文档上传时提取标题层级结构，生成**文档大纲 JSON 树**。对于无标题结构的文档（扫描件、纯文本），按页码将固定段落数组成"虚拟章节"，确保所有文档都具备可导航的大纲。

这个设计的本质和一个 Skill 系统的加载机制完全一致——Agent 不会在启动时把 Skill 的所有 reference 文件和 prompt 模板全塞进上下文，而是先看到 Skill 文件夹的目录结构（`skill.md` + `reference/` + `prompts/`），知道"有哪些资源可用"，然后在执行过程中按需加载需要的文件。类比到文档阅读：

```
Skill 系统                         文档阅读系统
──────────────────────────       ──────────────────────────
skill.md  (行为定义)              用户问题 (任务定义)
reference/ 目录 (资源清单)         大纲 JSON 树 (章节目录)
按需加载 reference 文件            按需 read_section 加载章节
已加载的文件不再重复加载            visitedNodeIds 拦截已读章节
```

Agent 在决策时看到大纲 JSON 树——像看 Skill 的 reference 目录一样，知道文档有几章、每章讲什么。然后通过 `read_section` 按需加载需要的章节，就像 Skill 按需加载 reference 文件。**从未读过的章节不会出现在 Agent 的上下文里，就像从未加载过的 reference 文件不占用上下文一样。** 从根本上消除了传统 RAG"把所有 Chunk 一次性塞给 LLM"的上下文污染问题。

```
传统 RAG:  用户提问 → LLM 盲猜关键词 → 碎片检索 → 重排序 → 生成回答
Agentic:   用户提问 → Agent 看到大纲树 → 定位相关章节 → 精读完整章节
                         ↑
              文档上传时预计算生成
```

### 大纲全量注入

无论文档有多少标题节点，大纲 JSON 树全量注入 Agent Prompt。一份 200 页的工程标准文档可能有 200+ 个标题节点，但每个标题仅占一行文本 + 行号前缀，实际 Token 占比可控（通常 500-800 token）。

Agent 通过行号前缀定位相关标题，系统侧自动去重后精读对应章节。

### 行号索引优化

大纲序列化时，每个标题前加 `#行号` 前缀：

```
#1 1 总则
#5 1.1 适用范围
#12 3.2 消防设计要求
#18 3.3 疏散通道规范
```

Agent 工具调用时只需输出行号列表（如 `[12, 18]`）而非完整章节标题文本。三个好处：省 Token（一个数字 vs 几十个字的标题文本）；防止弱模型输出标题时漏字/改字导致匹配失败；行号为整数，解析确定性高，不存在模糊匹配问题。

兜底机制：当 Agent 输出标题文本而非行号时，系统 fallback 到模糊子串匹配，保障鲁棒性。

## 状态驱动的 Agent Loop（Java 原生实现）

### 为什么用 Java 原生实现

Python Agent 框架的抽象层在此场景下反而成为定制障碍：

- 需要在工具执行层面做拦截（如 `read_section` 返回前检查 `visitedNodeIds`，已读则直接返回摘要而非实际执行），通用框架的 callback 机制粒度不够
- 7B 模型的格式纠错需要在每轮解析环节插入自定义逻辑，通用框架的 parser 封装后难以修改
- 核心 Agent 服务需要与公司现有的服务治理（限流、熔断、链路追踪）天然集成，Java 微服务体系原生支持

### State 对象设计

基于 Java 原生实现完整的状态驱动 Agent Loop。代码层面是一个 while 循环 + 每轮的结构化输出解析，核心在于每轮循环中对 State 对象的精确维护和注入。State 对象是一个 Java POJO：

```java
public class AgentState {
    Set<String> visitedNodeIds;      // 已读章节集合，工具层硬拦截防重复阅读
    Set<String> usedKeywords;        // 已用关键词集合，配合语义去重防穷举
    List<ActionRecord> actionHistory; // Action 执行历史及结果摘要
    int currentStep;                 // 当前轮数，配合最大轮次硬限制
    long cumulativeTokens;           // 累计 token 消耗，超限触发熔断
    OutlineContext outlineContext;   // 当前注入的文档大纲 JSON 树
    String finalAnswer;              // 最终回答
}
```

每轮 Prompt 构建时，从 State 对象实时序列化出结构化状态快照注入 Prompt：

```
你已阅读章节：[1.1, 3.2, 5.4]
已用关键词：[消防设计, 耐火等级, 疏散通道]
当前第 4/15 轮，累计消耗 18,500 tokens
```

这跟纯粹的"对话历史堆叠"式 Agent Loop 有本质区别——LLM 每轮看到的是经过筛选和压缩的决策依据，而不是原始对话流水。

### 三层防御机制

在状态机设计阶段预判了 Agent 无限循环的风险，三层防御是状态机的一等公民设计：

1. **Max Steps（最大轮次）：** 硬限制 15 轮，超限强制跳转到 `GENERATE_ANSWER`，基于已收集到的信息尽力回答，而非返回空结果。用户至少拿到"部分答案"，体验远好于"未找到"

2. **Visited Memory（访问记忆）：** `visitedNodeIds` + `usedKey words` 注入每轮 Prompt。如果 Agent 尝试调用 `read_section` 读取已访问章节，Tool 层直接拦截返回 `"该章节已读取，请选择其他章节或尝试回答"`，零 Token 消耗。`usedKeywords` 通过字符串包含或相等匹配判断重复，防止关键词穷举

3. **Cost Guard（成本卫士）：** 每轮循环入口累加 Token 消耗，单次 Session 上限 50k tokens。超限触发熔断，状态机跳转终态输出当前最优答案

## 多粒度工具链：解耦 Search 与 Read

传统 RAG 把 Search 和 Read 混在一起——检索返回 Chunk 后就当"读完"了。但实际上，Search 告诉你"哪里可能相关"，Read 才是"真正理解内容"。两者应该解耦。

工具链分为两层——外层（Loop 之前的文档定位）和内层（Loop 内的章节导航）：

**外层：文档定位层（Loop 之前，只执行一次）**

| 步骤 | 职责 | 实现 |
|------|------|------|
| 摘要提取 | 获取知识库中所有文档的摘要（两级自适应） | 入库时预计算，检索时读取 |
| LLM 预判断 | 三路分流（总览/指定文档/关键词提取） | 一次 LLM 调用，不进 Loop |
| ES 全文搜索 | 用关键词在知识库内全文检索 | ES BM25，返回 Top 100 Chunk |
| Chunk 评分 | 关键词加权 + 文档内排名衰减 | 自定义评分算法，见上文 |

**内层：Agent Loop 内可用工具**

| 工具 | 职责 | 实现 |
|------|------|------|
| `search_within_doc` | 在已定位文档内部关键词搜索 | ES 文档内检索，返回精确片段 |
| `read_section` | 基于大纲 JSON 树精确定位章节并返回完整内容 | 大纲 JSON 树定位 + 章节文本提取 |
| `search_kg` | 知识图谱遍历 | 图谱查询接口 |

关键设计：外层文档定位把"大海捞针"变成"在几本书里找答案"，内层 Agent Loop 在聚焦的文档集合内做章节级导航。Search 是热路径（P99 < 100ms），走 ES BM25 + 向量检索，不涉及 LLM 推理。Read 是冷路径，读取的完整章节内容被注入 Agent 上下文，LLM 在下一轮 Thought 中综合判断信息充分性。

**我在完成这套方案后，发现两篇近期论文从学术界独立验证了相同的技术路线：**

- **DocDancer（北大 & 腾讯，arXiv:2601.05163，2026 年 1 月）**：同样将文档问答建模为 agentic information-seeking 问题，采用 MinerU 解析 + Search/Read 双工具 + ReAct Agent Loop 的架构，在 MMLongBench-Doc 和 DocBench 上超越了传统 RAG 和多 Agent 方案
- **AgenticRAG: Agentic Retrieval for Enterprise Knowledge Bases（微软，arXiv:2605.05538，2026 年 5 月）**：同样验证了 Agentic 检索范式在企业知识库场景下的有效性，与传统 RAG 管道相比取得了显著提升

这说明"将文档问答从线性管道重构为 Agentic 运行时认知"这个技术方向是经过学术界双重验证的。我的工作在此基础上进一步延伸：扩展到多 Agent 多文档交叉分析场景、在状态机中维护细粒度状态（`visitedNodeIds`/`usedKeywords`）防止重复探索、实现 7B 模型场景下的格式纠错闭环（上述论文依赖 GPT-4o 等强模型）。

## Agent 核心能力

### 1. 大纲感知与智能导航

Agent 在每轮决策时看到的是文档大纲 JSON 树 + 已读章节标记 + 已用关键词历史，而非孤立的 Chunk 列表。它像人翻目录一样知道：

- 文档有几章，每章大概讲什么（从标题文本推断）
- 哪些章节已经读过了，不需要重复读
- 哪些章节还没读，但标题看起来可能相关

### 2. 按需精读：像 Skill 加载一样，只加载需要的章节

这是整个系统最核心的设计理念。`read_section` 不是一个普通的检索工具——它的行为模式和 Skill 系统的文件加载机制完全一致：

**Skill 是怎么加载的？** Agent 启动时只加载 `skill.md`（行为定义），看到 reference 目录下的文件列表，知道有哪些资源可用。但不会把所有 reference 文件全部塞进上下文——什么时候需要，什么时候 `Read` 对应的文件。已经读过的不会重复加载，不需要的永远不会出现在上下文里。

**`read_section` 就是文档版的 Skill 加载：**

- **大纲 JSON 树 = reference 目录的文件列表。** Agent 在 Prompt 中看到章节结构，知道文档有哪些章节、各自讲什么，但不加载任何章节正文
- **`read_section(doc_id, section_ids, goal)` = 按需 Read reference 文件。** Agent 判断信息不足时，指定要加载的目标章节和提取目标。系统返回完整章节内容，同时更新 `visitedNodeIds` 防止重复加载
- **已读章节硬拦截 = Skill 的"已加载、不再重复"。** Tool 层在 `read_section` 执行前检查 `visitedNodeIds`，已读的章节直接返回摘要，零额外 Token 消耗
- **没被触发的章节不占上下文 = 没被加载的 reference 文件不占上下文。** 一份 200 页的文档 Agent 最终可能只读 3-4 个章节，其他几十个章节的内容从未进入过上下文窗口

这与传统 RAG 的"一次性召回 top-k Chunk 全塞给 LLM"完全不同：

```
传统 RAG：  用户提问 → 召回 20 个 Chunk → 全塞进 Prompt → LLM 在碎片堆里找答案
本方案：    用户提问 → 看到大纲（章节目录）→ read_section 加载第 3 章
                      → 读完发现还需要第 7 章 → read_section 加载第 7 章
                      → 信息充分 → 回答
           ↑
          每一步只加载当前需要的信息，上下文始终精简
```

Agent 的每一步阅读都在动态修正后续方向——读了一个章节后发现里面的引用指向另一个章节，下轮直接定位那个章节。这不需要 Planner 提前拆分好子问题，是 ReAct Loop 自然支持的能力。

### 3. 多文档联合检索（进入 Loop 之前的文档定位）

系统从设计之初就支持一个知识库内多文档同时检索。关键不是"能搜多份文档"，而是**怎么在进入 Agent Loop 之前把范围从整个知识库收缩到几份候选文档**。

文档定位层（详见上文"进入 Agent Loop 之前"章节）的核心逻辑：

- 通过两级文档摘要（详细 500 字 / 简短 50 字，按库内文档数量自适应选择）让 LLM 在未见全文的情况下判断文档相关性
- 三路分流：总览问题直接回答，指定文档直接锁定，通用问题走关键词提取 → ES 检索 → Chunk 评分
- Chunk 评分采用关键词加权（1.2^n）+ 文档内排名衰减（0.5^(k-1)），防止超长文档作弊

定位后 Agent 拿到的是几份候选文档 + 各自的大纲 JSON 树，在 Loop 内交叉跳转阅读。Agent 在 Loop 内不再调用 `top_doc` 重新选文档，上下文保持聚焦。

### 4. 多模型协作与格式纠错

系统采用双模型协作架构：**32B 推理模型（Qwen3-32B）** 作为 Agent 的决策 LLM（负责 Thought/Action 结构化输出），**7B 模型（Qwen3-7B）** 用于 `read_section` 的章节精读和信息提取。两者均通过 vLLM 部署为 OpenAI 兼容 API，Java Agent 层通过 HTTP 统一调用。

**部署拓扑：**

```
┌──────────────────────────────────────────────┐
│            Java Agent Loop（编排层）           │
│      HTTP (OpenAI 兼容 API)                   │
└──────┬───────────────────────┬───────────────┘
       │                       │
 http://h100:8000        http://4090:8001
       │                       │
┌──────▼──────────────┐  ┌──────▼──────────────┐
│     H100 80GB       │  │    RTX 4090 24GB    │
│  Qwen3-32B FP8      │  │  Qwen3-7B FP8       │
│                     │  │                     │
│  vLLM 参数：         │  │  vLLM 参数：         │
│  gpu_mem_util: 0.95 │  │  gpu_mem_util: 0.95 │
│  max_model_len: 20K │  │  max_model_len: 8K  │
│  enable_prefix_cach │  │  enable_prefix_cach │
│   = true            │  │   = true            │
│                     │  │                     │
│  决策 LLM           │  │  精读 LLM           │
│  P99: 2-3s          │  │  P99: 0.3s          │
└─────────────────────┘  └─────────────────────┘
```

**H100 80GB 显存分配（Qwen3-32B FP8）：**

Qwen3-32B 模型参数约 32B，架构为 64 层 Transformer，hidden_size=5120，GQA（8 KV heads，head_dim=128）。

| 项目 | 大小 | 计算方式 |
|------|------|---------|
| 模型权重（FP8） | ~34 GB | 32B × 1 byte + embedding/lm_head 等非量化层 |
| vLLM 运行时开销 | ~2 GB | 调度器、CUDA context、预留缓冲 |
| 可用 KV Cache | ~40 GB | 80 × 0.95 - 34 - 2 |
| KV Cache 每 token | ~128 KB | 2 × 64 layers × 8 KV heads × 128 dim × 1 byte |
| 20K token 单序列 KV Cache | ~2.56 GB | 20000 × 128 KB |
| 理论最大并发（20K 序列） | ~15 | 40 / 2.56 |

对比：如果选 FP16，模型权重占 64 GB，可用 KV Cache 仅 10 GB，最多支撑 2 个 20K 并发序列——所以 FP8 是必要的选择。H100 有原生 FP8 Tensor Core 支持，vLLM 通过 `quantization="fp8"` 参数直接启用，推理吞吐约为 FP16 的 1.8-2 倍。

生产环境中 Agent Loop 实际并发远低于 15 的理论上限：同一 Session 内大量请求共享 prefix cache（前缀 KV Cache 复用），不同 Session 才占用独立 KV Cache 空间。

**RTX 4090 24GB 显存分配（Qwen3-7B FP8）：**

Qwen3-7B 约 7.6B 参数，28 层 Transformer，hidden_size=3584，GQA（4 KV heads，head_dim=128）。

| 项目 | 大小 | 计算方式 |
|------|------|---------|
| 模型权重（FP8） | ~9 GB | 7.6B × 1 byte + 非量化层 |
| vLLM 运行时开销 | ~1.5 GB | |
| 可用 KV Cache | ~12.3 GB | 24 × 0.95 - 9 - 1.5 |
| KV Cache 每 token | ~28 KB | 2 × 28 layers × 4 KV heads × 128 dim × 1 byte |
| 8K token 单序列 KV Cache | ~0.23 GB | 8000 × 28 KB |

精读 LLM 输入通常为单章节（几百到几千 token），输出 200-500 token 结构化摘要，RTX 4090 24GB 绰绰有余。精读任务选择 4090 而非第二张 H100 是刻意的——不需要 H100 级别的显存带宽和算力，4090 性价比高一个数量级。

**vLLM Prefix Caching：** 开启 automatic prefix caching。同一 Session 内每轮 Prompt 的 System Prompt + 工具定义 + 大纲前缀高度重复（约 2200 token），后续轮次 Prefill 直接复用 KV Cache。踩过一个坑：Prompt 模板中动态时间戳置于前缀位置导致 hash 不匹配，命中率下降，将动态内容后置到 Prompt 末尾后恢复正常。

7B 模型的指令遵循能力相对较弱，在章节阅读和结构化信息提取时频繁出现格式异常（非 JSON 输出、字段缺失、输出截断）。设计了完整的异常捕获与纠错闭环：

```
LLM 输出 → 标准 JSON 解析
  ├── 成功 → 正常执行
  └── 失败 → 正则提取兜底
        ├── 成功 → 正常执行
        └── 失败 → 错误信息写入 State，LLM 下轮自我修正
                    ├── 修正成功 → 正常执行
                    └── 连续 3 轮失败 → 熔断，跳转 GENERATE_ANSWER，
                         基于已收集信息尽力回答，标记 Bad Case
```

工具名匹配：7B 模型输出工具名时可能格式不规范，Tool 路由层做精确字符串匹配。格式异常时走通用纠错闭环（正则提取兜底 → LLM 自修复），不依赖模糊匹配。

### 6. 行为纠偏：Search/Read 双向强制机制

Agent 在 Loop 中可能出现两种行为失衡：**只搜不读**（反复换关键词搜索但从不精读章节）和**只读不搜**（连续精读不同章节但都没找到有用信息）。两者对称但根因不同——前者是 Agent 在搜索结果中迷失，后者是大纲定位偏移。单独靠 Prompt 引导或轮次熔断不够精细，设计了双向硬 override 机制：

**方向一：连续搜索 → 强制精读。** 触发条件：已经连续 2 次调用 `search_within_doc`，第 3 轮如果 LLM 仍输出 `search_within_doc`，系统跳过 LLM 决策环节，从前面两轮搜索结果中取 ES 分数最高的章节，硬 override 为 `read_section` 调用。不做阈值检查——即使搜索结果质量不高，读一章没用的内容最多浪费一轮（精读 LLM 返回 `not_relevant` 写入 State），远比让 Agent 在搜索循环里出不来安全。这个 override 不是"替 Agent 决定"，而是"搜了两轮该有的结果已经有了，强制推进到阅读"。

**方向二：连续精读失败 → 强制搜索。** 触发条件：连续 2 次 `read_section` 的精读 LLM 返回 `not_relevant`（即明确判定该章节不包含与目标相关的信息）。注意判断标准来自精读 LLM 而非决策 LLM——精读 LLM 刚读完原文，判断"这章有没有用"比只看摘要的决策 LLM 可靠。系统自动降级到 `search_within_doc`，用轻量 LLM 基于用户原始问题 + 已读但无用的章节标题列表做查询改写——"这些章节都不对，换个方向想想该搜什么"，改写后的关键词走 BM25 检索。

两个方向对称但不等价：

```
只搜不读：连续 2 次 search 后，第 3 次若仍是 search → 硬 override 为 read（从前两轮搜索结果取章节）
只读不搜：连续 2 次 not_relevant → 自动降级到 search（查询改写换方向）
```

这比纯粹的轮次/Token 熔断精细得多——是**行为层面的纠偏**，在 Agent 陷入无效模式时就介入，而不是等资源耗尽才兜底。配合已读章节拦截（`visitedNodeIds`）和关键词去重（`usedKeywords`），四条栅栏让 Agent 在搜索和阅读之间保持动态平衡。

**架构要点：**

- **文档定位层与 Agent Loop 分离**——定位只做一次（Loop 之前），将候选范围从"整个知识库"收缩到"几份最相关的文档"。Agent 在 Loop 内不重新选文档，上下文保持聚焦
- **摘要分级设计**——文档多时用简短摘要（~50 字）防上下文爆炸，文档少时用详细摘要（~500 字）做精准判断
- **Chunk 评分双重机制**——关键词加权（1.2^n）让高密度 Chunk 突出 + 文档内排名衰减（0.5^(k-1)）防超长文档作弊
- **文档解析层异步解耦**——MinerU 解析走异步队列（RabbitMQ），Upload 接口立刻返回，视觉管线高延迟不影响用户体验
- **Agent 编排层纯 Java 实现**——与公司微服务体系无缝集成，限流/熔断/链路追踪天然可用
- **Search 与 Read 解耦**——Search 是热路径（< 100ms，ES 直接返回），Read 是冷路径（LLM 决策后按需读取）
- **大纲预计算 + 全量注入**——大纲树随文档一起索引化存储，检索时全量注入 Agent Prompt
- **State 对象驱动决策**——不是对话历史堆叠，每轮看到的是经过筛选和压缩的决策依据

另一个关键：Agent 每一步阅读都在动态修正后续方向——读了一个章节后发现里面的引用指向另一个章节，下轮直接定位那个章节。不需要 Planner 提前拆分好子问题。老架构的子任务之间无信息传递，一步拆错全链路失败；Agentic 架构中每一步的 Observation 都在修正下一步的 Thought。

## 核心技术挑战

### 挑战一：砍掉 Rerank 层的决策

老架构在多路召回后接了一层 cross-encoder rerank（BGE-Reranker）。上线初期看指标涨了几个点，但我在深入排查 bad case 后判断：rerank 在我们的场景里是在给上游的烂召回擦屁股，不是一个值得保留的环节。

四个论据：

① **rerank 解决不了召回阶段的根本问题**——如果相关内容在 top-k 召回里根本没出现（Chunk 切碎了、关键词盲猜偏了），rerank 再牛也排不出来，garbage in garbage out

② **延迟代价不值**——cross-encoder 是 query-doc pair 逐对打分，O(n) 复杂度，召回 50 个 chunk 过一遍 rerank 延迟直接翻倍

③ **pointwise 打分逻辑不适用于多章节综合推理**——rerank 只能给单个 chunk 打相关性分，不考虑文档间的互补性和信息完整性，排到第一的 chunk 未必是回答问题最需要的那个

④ **开源 rerank 模型泛化存疑**——BGE-Reranker 在 BEIR 上跑分好看，但在垂直领域（工程标准、合同条款）实测效果打折严重

替代思路：与其在碎片召回后加一层重排序补救，不如从源头解决问题——大纲驱动让 Agent 具备文档全局感知能力，按需精读完整章节。实测砍掉 rerank 后，配合新架构的大纲粗筛 + 章节精读机制，准确率反而从 65% 提升至 90%+，同时省掉了 rerank 模型的推理延迟和 GPU 资源。

**不过面试时补充一句**：如果面试官问"什么场景下 rerank 还是有用的"——事实型精确查询（"XX 公司 2024 年营收多少"）、延迟预算充足且上游已优化到位时，rerank 作为最后一层 polish 有边际价值。但在长文档多跳推理场景，投入产出比远不如优化 query 理解和文档导航策略。

### 挑战二：MinerU 标题层级不分

详见上文"文档解析层"章节。核心思路：不依赖 MinerU 的 `text_level` 字段，自研基于 BBox 坐标高度容差聚类的后处理模块，通过版面几何信息重建大纲树层级。

### 挑战三：7B 模型格式异常

详见上文"多模型协作与格式纠错"章节。核心思路：多层解析兜底（标准 JSON → 正则提取 → LLM 自修复）+ 错误反馈写入状态机 + 工具名模糊匹配 + 连续 3 轮熔断。

### 挑战四：上下文窗口爆炸

上线初期 Agent 在处理多文档长章节时突然无响应。排查发现根因与商机智能体的上下文爆炸同源——但这次不是 JSON 噪音，而是 Agent 在多轮循环中把所有读取的章节原文全部堆在 messages 里，读到第 8-9 轮时上下文窗口被打爆。

解决方案在 State 设计中内置了双重防护：① 已读章节拦截（`visitedNodeIds`），读了的不再读，从源头杜绝重复内容进入上下文；② 最大轮次硬限制（15 轮）+ Token 上限熔断（50k），超限强制跳转 `GENERATE_ANSWER`。由于 `read_section` 返回的是精读 LLM 的结构化摘要而非完整章节原文，每轮新增上下文量被严格控制，上下文膨胀从源头被消除。

## 成果总结

- 复杂多跳推理问题的回答准确率从 **65%** 提升至 **90%+**
- 通过"大纲粗筛 + 按需阅读"替代全量 Chunk 检索，Token 平均消耗降低 **60%**

## 面试深度问答

### Q1: 为什么用 Java 原生实现 Agent Loop 而不是 Python Agent 框架？

A: 三个原因。① Python Agent 框架的抽象层在此场景下成为定制障碍——我需要在 `read_section` 执行前检查 `visitedNodeIds` 做硬拦截，通用框架的 callback 机制做不到这个粒度；② 7B 模型的格式纠错需要在每轮结构化输出解析环节插入自定义逻辑，通用框架的 parser 是封装好的，改不动；③ 核心 Agent 服务跑在 Java 微服务体系里，和公司现有的限流/熔断/链路追踪天然集成，引入 Python Agent 框架反而增加跨语言调用和运维复杂度。通用框架适合快速原型，但产品级系统需要更细粒度的控制。

### Q2: 大纲驱动方案，如果文档本身没有标题结构怎么办？

A: 这是 ETL 管道的标准流程，不是 edge case 补丁。每份文档解析完都会经过大纲质量校验——检查树的深度和节点密度。如果发现解析失败（OCR 乱码、无标题结构、深度 < 2），系统自动降级为"物理页码模式"：按页码将内容切分为虚拟章节，页与页之间保留一定的文本重叠率，防止关键信息在分页边界处被截断。对 Agent 来说，它依然可以通过大纲进行翻页和查找，只是粒度从"章节"变成了"页"。另外，生产环境中发现绝大多数无标题文档（90%+）其实是 1000 字以内的简单 txt 文件，本身就无需章节导航，Agent 直接全量阅读即可，所以这个问题在实际场景中远比听起来轻量。

### Q3: 大纲全量注入，200+ 标题会不会撑爆上下文窗口？

A: 不会。每个标题仅占一行文本 + 行号前缀，200+ 节点的完整大纲 JSON 树通常只需 500-800 token。对于 32k/128k 的上下文窗口，这个占比完全可控。且行号索引让 Agent 只需输出数字即可定位章节，不消耗额外 Token 去输出标题文本。

### Q4: 你的 Agent 怎么保证不会在两个章节之间反复横跳？

A: 三重保障，逐级升级：① `visitedNodeIds` 在 Prompt 中明文告诉 LLM"你已经读过这些了"，绝大部分情况下 LLM 会自觉避开；② 如果 LLM 还是尝试 `read_section` 读已读章节，Tool 层硬拦截，返回"该章节已读取"，零 Token 消耗；③ 极端情况下——硬拦截之后 LLM 仍然执着地尝试再读同一章节——系统直接从大纲 JSON 树中删除该章节节点。章节从大纲中消失后，LLM 在后续轮次中根本看不到这个选项，自然无法再尝试。配合轮次上限（15 轮）和 Token 上限（50k）的最终熔断，反复横跳在工程上被彻底杜绝。

### Q5: 怎么评估 Agent 的回答质量？有标注数据集吗？

A: 有三套评估数据源。① **客户测试集**——我们和国内一家头部船舶企业有合作，为其提供知识库超长文档问答服务，对方的核心文档（船舶设计规范、工艺流程手册）单份 PDF 超过 500 页。对方给了明确的标注测试集，作为核心评估基准；② **公开 RAG 评测集**——用于横向对比业界方案；③ **公司内部产品文档测试集**——覆盖我们自身的产品手册和运维文档。每次改 Agent Loop 逻辑后三套测试集全跑一遍，确保不退化。另外用户纠错反馈也会自动积累——前端点踩按钮触发 Trace ID 推送，通过 Trace ID 回放分析完整决策轨迹。

### Q6: Agent 的每轮延迟大概是多少？用户能接受吗？

A: 系统从两个层面做了流式输出，大大减轻用户等待焦虑：① **Thought 级流式**——Agent 每轮推理过程实时流式输出，用户可以看到"正在分析大纲…→ 定位到第 3 章 → 发现需要补充第 7 章"的完整思考轨迹，而非对着一片空白等 8-12 秒；② **最终答案流式**——Answer 生成同样是 token 级流式输出。Agent 每轮 Thought + Action 的实际延迟约 2s-3s（32B 模型 vLLM 推理），Search 本身 < 100ms（ES 直接返回）。流式输出使得用户感知延迟远低于实际总延迟——就像翻书一样，看到内容在持续产出，等待焦虑大幅降低。Search 热路径上不涉及 LLM，这是延迟可控的关键。

### Q7: 你的方案跟学术界论文是什么关系？

A: 我是在完成这套方案后，才陆续看到 DocDancer（北大 & 腾讯，2026 年 1 月）和 AgenticRAG（微软，2026 年 5 月）两篇论文。两篇论文从学术界独立验证了相同的技术路线——MinerU 解析 + Search/Read 双工具 + ReAct Agent Loop。这说明"将文档问答从线性管道重构为 Agentic 运行时认知"这个方向是对的。我的工作在此基础上延伸了四点：① 多文档同时支持——系统输入是知识库而非单文档，通过文档摘要预定位 + Chunk 评分机制（关键词加权 + 文档内衰减）在 Loop 之前收缩候选范围；② 多 Agent 多文档交叉分析（论文只处理单文档）；③ 状态机中维护细粒度状态（`visitedNodeIds`/`usedKeywords`）防重复探索；④ 7B 模型格式纠错闭环（论文依赖 GPT-4o 等强模型，实测 7B 模型格式异常频发）。

### Q8: Chunk 评分为什么要加文档内排名衰减？直接用原始分数不行吗？

A: 预防超长文档作弊。假设知识库有一份 200 页的工程标准（几百个 Chunk）和一份 20 页的合同（几十个 Chunk）。用户问的是合同条款问题，但关键词"违约""赔偿"在工程标准的某些章节里也出现了——只是上下文完全不相关。如果不做文档内衰减，工程标准因为 Chunk 数量多、命中绝对数量高，总得分会显著高于合同，Agent 就会拿到错误的候选文档。加了文档内排名衰减（排名第 k 的 Chunk 得分 × 0.5^(k-1)）后，一份文档只有排名前几的 Chunk 对最终得分贡献大，后面的大量弱相关 Chunk 得分被指数级压扁。真正相关的文档（Chunk 少但都排在前列）不会被"碰巧大范围命中关键词"的长文档淹没。这是一个非常 cheap 但有效的 trick——不需要训练模型，不需要调整 ES 参数，就是几行评分代码。

### Q9: 现在 Gemini 有 1M context window，直接塞全文不就行了？你的方案是不是过度工程化了？

A: 这个问题我不光理论分析过，还实测过。我拿目标文档——一份 500 页的中文船舶设计规范——直接上传 Gemini 2.5 Pro，结果直接报错文件太大无法处理。1M context window 是 token 数，但中文的 token 效率比英文低得多，加上表格、公式等结构化内容，一份 500 页中文 PDF 实际 token 量可能到 3M-5M，远超 1M 窗口。所以"直接塞全文"在我们文档规模下根本不成立。

退一步说，即使文档小到能塞进去，还有三个问题。第一，成本——我们日均处理上千次问答，每次几十万 token 的 Input 成本不可接受，我的方案平均总 Token 仅需老方案的 40%。第二，Lost in the Middle——窗口越长中间注意力衰减越严重，7B/32B 模型的长上下文能力远不如大模型，我们恰好用的是这个量级的模型。第三，多文档场景——用户经常涉及 3-5 份文档，单份 3M-5M token，多份加起来任何模型都吃不下。我的方案通过大纲导航精准定位，不受文档总量限制。

### Q10: 你说用 Java 原生实现 Agent Loop，但 MinerU 和 vLLM 都是 Python 的，Java 怎么调它们？这不是跨语言调用吗？

A: 我说的"避免跨语言开销"不是说不跨语言——vLLM 是 Python 服务，当然通过 HTTP 调它（OpenAI 兼容 API）。我避免的是在 Agent 编排层再套一个 Python 进程。如果用 LangChain，链路是 Java 业务服务 → Python Agent 服务 → vLLM，三跳。现在是 Java Agent 直接 → vLLM，两跳，少了一个中间层。HTTP 调用 overhead 大概 5-10ms，相对于 32B 模型推理的 2s-3s 完全可以忽略。真正的核心收益是运维层面：Agent 的状态管理、异常处理、链路追踪全在 Java 微服务体系内，限流熔断都是现成的，不需要额外维护一个 Python 服务的部署和监控。

### Q11: Token 消耗降低 60%，对比基准是什么？新方案跑了 3-5 轮，总 Token 真的比老方案少吗？

A: 对比口径是整个 Session 的总 Token（Input + Output 全算）。老方案看起来是"一次调用"，但对复杂问题会拆分子问题，每个子问题走一次完整链路——3 个子问题，每次塞 10 个 Chunk（其中 7 个是噪音），总 Token 轻松到 50K-60K。新方案虽然跑 3-5 轮，但每轮只读取精确定位的章节片段，没有噪音 Chunk 的浪费。大纲 500-800 token + 每轮精读几百 token 结构化结果 + 状态快照，平均总 Token 在 15K-25K。关键差异不是"单次调用谁省"，而是新方案没有把大量噪音 Chunk 反复塞进上下文。

### Q12: vLLM 的 Prefix Caching 在你的多轮 Agent 场景下实际收益如何？

A: 收益显著。Agent 每轮 Prompt 的前缀——System Prompt、工具定义、大纲——在同一 Session 内高度重复。开启 prefix caching 后，后续轮次的 Prefill 只需计算新增的 Action 历史部分，前缀直接复用 KV Cache，命中率稳定在 75% 以上。踩过一个坑：一开始命中率只有 60%，排查发现是 Prompt 模板里有动态时间戳字段导致前缀 hash 不匹配，把时间戳挪到 Prompt 末尾后命中率就上来了——动态内容必须后置才能最大化 prefix cache 收益。

### Q13: 如果重新做一次，你会改什么？

A: 几个点。第一，知识图谱从一开始就不应该引入，投入产出比不划算，省下来的精力投到评测体系建设上。第二，SFT 微调应该更早启动——从上线第一天就该记录完整 Agent 轨迹，积累数据做 SFT，中间那些格式纠错的补丁代码都可以省掉。第三，最大的遗憾是没有做 Agentic RL——用 GRPO + 可验证奖励（答案正确性 + 步骤效率惩罚），让模型自己探索出更高效的检索策略。SFT 的上限是训练数据质量，模型只能模仿不能超越；RL 可以让模型发现"其实不需要 4 步，2 步就够"。没做的原因是 RL 的工程落地比 SFT 复杂得多，当时精力不够，但这是最有价值的下一步。

### Q14: Agent 看了大纲选第 3 章，结果答案在第 7 章，选错了会不会输出错误答案？

A: 选错章节是必然会发生的，不是 bug，是 Agent 系统的常态。防御分两层。第一，给 Agent 足够的纠错空间——15 轮预算意味着即使前几轮选错了，还有机会找到正确信息。第二，更关键的是精读 LLM 的质量判断——`read_section` 内部调用精读 LLM 时，会判定该章节是否真正包含与目标相关的信息，如果明确判定 not_relevant，这个信号会写入 State 引导决策 LLM 换方向。连续 2 次精读返回 not_relevant 后，系统自动降级到关键词搜索——"大纲定位偏了，换一种检索方式"。坦白说，如果 Agent 基于错误信息生成了"看起来很合理"的回答，评估环节也可能被骗过，这是 LLM 系统的固有局限。我的兜底是每个回答附带引用来源，精确到章节和页码，用户可点击验证原文。

### Q15: 你的 `read_section` 内部还调了一次 LLM 做精读，这不是多了一次调用吗？为什么不直接把章节原文返回给决策 LLM 让它自己读？

A: 这是一个有意为之的架构决策——**原文从头到尾不进入决策 LLM 的 messages**。决策 LLM 要的只是两样东西：这个章节里跟当前目标相关的关键信息，以及下一步该往哪个方向继续探查。一个 3000 token 的章节可能只有 200 token 跟当前问题相关，剩下的全是干扰。如果把原文直接塞进去，Agent 跑 5 轮就是 15000 token 的噪音堆在 messages 里，决策质量必然下降，Token 预算被挤占，Lost in the Middle 效应还会被放大——原文恰好塞在 messages 中间位置，是 LLM 注意力最弱的区域。

精读 LLM（用 7B 模型）的输出是结构化的三部分：① 针对阶段目标的精炼回答（通常 200-500 token）；② 关键证据引用（原文片段 + 页码定位）；③ 下一步探查建议（如"建议继续查看第 4.2 节"）。决策 LLM 拿到的是干净的、直接可用的信息，而不是一大段需要自己从中筛选的原文。多出来的一次 LLM 调用延迟 200-300ms，但换来的是决策 LLM 每轮只看几百 token 的精读结果而非几千 token 的原文，整个 Session 的总 Token 反而更少。本质上是把"阅读理解"和"策略决策"两个能力解耦了——精读 LLM 负责读懂这段话，决策 LLM 负责下一步该干什么，各司其职互不干扰。
