# 个人看法



### 核心逻辑缺陷：检索

假设 embedding（包含知识图谱、关键词检索）可以找到所有相似的片段，如果关键信息在 Chunk A，但解题线索在 Chunk B（比如表格在第 5 页，表头在第 4 页），你的 Chunk 切分直接切断了上下文。你捞到了数据，但丢了逻辑。

**具体失败场景：** 用户问"2024年Q3营收同比增长多少？"，Q3 营收数字在第 8 页的表格 Chunk 里，但"同比"的基准（2023年Q3）在第 3 页的另一个表格 Chunk 里。向量检索大概率只命中其中一个，LLM 拿到单边数据要么幻觉编造，要么拒答。

Embedding 本质是找和查询语义最相似的句子。当 chunk 很长（几百 token）、查询很短（十几个 token）时，向量空间中的余弦相似度被稀释，匹配度普遍偏低，top-k 召回的质量不稳定。

关键词检索的问题更根本：系统在没看过文档的情况下，靠 LLM 凭空生成关键词去检索。这本质上是用一个缺乏上下文的模型去猜测文档里的用词，一旦文档的专业术语或表述方式和 LLM 的猜测不一致（比如文档写的是"营业收入"，LLM 生成的关键词是"营收"），检索直接偏移。而且这个方案针对一次查询只检索一次，没有纠错机会。

知识图谱在我们的场景下问题也很明显：文档中同一实体的不同表述（如"腾讯控股""腾讯""Tencent"）无法自动对齐，导致图谱中出现大量重复节点；跨文档的关系抽取准确率低，经常出现张冠李戴的错误关系边。



### 决策机制缺陷：伪 Agent

遇到复杂问题，老方案用"Decomposition（分解）"。这个方案假设问题是可以被**预先完美拆解**的。现实世界的复杂问题往往是**非凸（Non-convex）**的——你必须试探一步，根据返回的结果才知道下一步该怎么走。

**具体失败场景：** 用户问"对比A公司和B公司近三年的盈利能力，分析差异原因"。分解器可能拆成"A公司近三年利润"和"B公司近三年利润"两个子问题。但真正的差异原因可能藏在成本结构、市场份额变化等维度里，这些维度只有在看到利润数据之后才知道该去追问。分解器在第一步就把问题框死了，后续不会根据中间结果动态调整检索方向。

更关键的是，子问题之间是并发或顺序执行后直接汇总，**子任务之间没有信息传递**。子任务 A 检索到的结果可能直接推翻子任务 B 的前提假设，但架构上它们互相不知道对方的结果，只在最后做一次粗暴的汇总生成。这不是 Agent，这是一条道走到黑的线性流水线。



### 数据结构缺陷：碎片化

Chunk + 知识图谱。你把完整的文档打碎成了玻璃渣（Chunks），然后试图用胶水（KG）把它们粘回去。但碎片化带来的信息损失是不可逆的——文档内的段落递进关系、章节间的逻辑依赖、表格与上下文的引用关系，在切分的瞬间就丢失了，KG 只能补回一小部分显式的实体关系，隐式的逻辑链条无法恢复。



### 本质问题

老方案专注于在文档上传时"预计算索引"（向量化、知识图谱、文档摘要等），把所有的智能前置到了离线阶段。真正的 Agent 方案应该是**"运行时认知（Runtime Cognition）"**——在用户提问时，Agent 能够主动地、多轮地与文档交互，根据每一步的检索结果动态决定下一步的行动。



### 关于已有优化措施的不足

老方案并非没有做优化，但这些优化本质上都是在错误的架构上打补丁：

- **带上下 chunk：** 检索出相关 chunk 后自动拼接上下文 chunk。问题在于这是无差别的——如果命中的 chunk 本身就是噪音，带上它的上下游只会成倍放大噪音，导致 LLM 的注意力被稀释，幻觉率反而上升。
- **假设问（HyDE）：** 上传文档时对每个 chunk 生成假设问题，用假设问的向量来匹配用户查询。这在简单事实类问题上有效，但对于需要跨 chunk 推理的复杂问题，单个 chunk 生成的假设问根本覆盖不到跨文档的组合查询。
- **监督者判断幻觉：** 输出前有一个 LLM 判断是否幻觉、是否完整回答。但监督者本身也是 LLM，它判断幻觉的依据还是那些不完整的 chunk。如果源头的检索就是错的，监督者大概率也会被误导——它没有能力回溯去重新检索，只能基于已有的错误上下文做判断。



# gemini看法

### 一、 检索逻辑缺陷：不仅是"玻璃渣"，更是"上下文毒药" (Context Poisoning)

你精准指出了 Chunk 切分导致上下文丢失（如表头与表格分离）和 Embedding 的短视问题。但你遗漏了生产环境中更致命的连锁反应。

- **表象缺陷：** 丢了逻辑，匹配度低。
- **架构师视角的深层灾难：**
  - **信噪比失衡（Signal-to-Noise Ratio）：** 你在补充说明中提到"检索出相关 chunk 之后会自动带上 上下 chunk"。这是极其粗暴的创可贴方案。如果一个 Chunk 是噪音，带上它的上下游只会成倍放大噪音，导致 LLM 在生成时发生严重的"注意力稀释（Attention Dilution）"，直接推高幻觉率。
  - **多源信息对齐灾难：** 当你同时使用关键词、向量、甚至知识图谱（KG）进行多路召回后，粗暴地丢给重排序（Reranking）。**Edge Case（边界情况）：** 如果向量召回了一个事实错误但语义极度相似的陈旧数据，而关键词召回了最新但语义稍弱的数据，你的 Reranker 依据什么权重来做决策？如果缺乏时间衰减因子和版本控制，系统必然输出过时信息。
  - **前置关键词生成的伪逻辑：** "在没看过文档时生成关键词"不仅是正确的质疑，更是架构上的死穴。这意味着你用一个缺乏上下文的 LLM 调用去瞎猜，这不仅浪费 Token，更会产生**检索偏移（Retrieval Drift）**，导致南辕北辙。

### 二、 决策机制缺陷：缺乏状态机的"自杀式单向流"

你提到"老方案假设问题是可以被预先完美拆解的"，并且批评了这种线性逻辑。一针见血，但你没有给出故障恢复（Failure Recovery）的推演。

- **表象缺陷：** 伪 Agent，一步错步步错（Garbage In, Garbage Out）。
- **架构师视角的深层灾难：**
  - **死锁与无限循环（Deadlock & Infinite Loops）：** 真正的复杂非凸问题，往往需要基于第一次检索的结果来决定第二次的查询动作。你的系统如果在"循环遍历或并发执行子问题"时，某个子问题持续召回失败，你的架构图里**没有任何退出机制或降级策略**。系统是会一直死循环耗尽 Token？还是直接抛出超时异常（Timeout）？
  - **并发灾难与 Latency（延迟）失控：** 假设一个问题被拆解为 5 个子问题并发执行，每个子问题走一次完整的"检索 -> 生成"链路。P99 Latency 将直接由最慢的那个子节点决定。如果遇到网络抖动或大模型限流（Rate Limit），用户的直观体验就是系统彻底卡死。
  - **全局状态丢失：** 子问题的并发执行剥夺了相互之间的上下文感知。子任务 A 的结果可能推翻子任务 B 的假设，但你们的架构只在最后做"汇总生成"，这在逻辑图上是断裂的。

### 三、 被动检索

当前采用线性的 Pipeline 架构（检索 -> 打包 Context -> 喂给 LLM）。系统一次性将被动检索到的 Chunk 推送给 LLM，Agent 缺乏二次获取信息的主动权。 LLM 被迫基于不完整信息产生幻觉（Hallucination）或直接拒答，导致任务失败。
