### 怎么处理图片检索？

**问题描述：** "你的文档里有大量的图表、流程图、架构图。用户问'系统架构是什么样的'，答案可能就在一张图里。你的纯文本检索链路怎么处理这种情况？图片又不能被 Embedding。"

**解决方案：**

我的方案分两层处理：

1. **解析阶段（MinerU 侧）：** MinerU 的视觉模型管线在做版面检测时，会把图片元素精确裁剪出来，同时输出对应的 BBox 坐标和在文档中的位置信息（属于哪个章节、在哪一页）。我在 ETL 管道中把这些裁剪后的图片存入 OSS，并在大纲 JSON 树的对应节点上挂载图片引用（`image_refs` 字段）。

2. **检索阶段（Agent 侧）：** 当 Agent 通过 `read_section` 工具读取某个章节时，返回的内容不仅包含文本，还会附带该章节下所有图片的引用链接和 MinerU 输出的图片上下文描述（即图片前后的文本段落）。Agent 可以基于这些上下文描述判断图片是否与用户问题相关。如果需要深度理解图片内容（比如读取图表中的数据），我们预留了 VLM（视觉大模型）的扩展接口——将裁剪后的高质量图片直接送入多模态模型做图片问答。

3. **兜底策略：** 对于 MinerU 无法识别的嵌入式图片（如某些 PDF 中图片被编码为矢量路径），我在解析后处理中加了一层基于页面渲染的截图 fallback：对整页做渲染截图，再用简单的区域检测把非文本区域裁出来作为"疑似图片"挂载到对应页码节点上。精度不如 MinerU 原生裁剪，但保证不丢图。

> **面试加分点：** 这个设计的核心思路是**"图片不参与检索，但图片的上下文参与检索"**。用户搜到了相关章节，图片自然就跟着出来了。这比单独做图片 Embedding（CLIP 之类的）更工程化、更可控。

### 多租户数据隔离？

采用物理隔离，mysql采用数据库隔离，es采用索引隔离，索引由租户的名字和模型的名字决定。

我设计了一套 **Context Propagation (上下文透传)** 机制：

1. **入口层：** API Gateway 解析 Token，提取 `tenant_id`，注入到当前线程的上下文（如 Java `ThreadLocal` 或 Python `ContextVar`）。
2. **Agent 编排层：** 我修改了 LangChain/LangGraph 的 BaseTool 定义。所有的 Tool (Retriever, GraphQuery, WebSearch) 执行时，会自动从上下文中'吸取' `tenant_id`。
3. **安全卡点：** 这样，即使 LLM 产生了幻觉，试图查询'所有人的数据'，底层的 Tool 代码会强制将其重定向到当前 `tenant_id` 对应的物理库中。**我们在代码层面实现了'逻辑上的无感知'，但在执行层面是'强制隔离'的。**

> 面试官的反击：每个租户一个库，如果有 10,000 个小客户怎么办？你的 ES 会因为 Shard 太多炸掉的？

对于 VIP 客户，承诺物理隔离，收费更贵；对于免费客户，采用逻辑隔离（同一个 ES 索引内通过 `tenant_id` 字段做 filter），这是商业与成本的平衡。

### 如何保障外部调用的安全性？（只要有一个签约客户写了一个死循环脚本不断调用你的 API，或者一次性上传了 10GB 的文档进行 RAG 索引，整个系统的 GPU 显存和数据库 IO 就会被耗尽，导致其他所有 VIP 客户服务不可用（Downtime））

###### **核心战略：读写分离 + 优先级队列 (Read/Write Split & Priority Queuing)**

RAG 系统有一个显著特征：**"解析/索引文档（Write）"是重资源、高延迟的；"回答问题（Read）"是轻资源、低延迟的。** 绝不能让"写操作"阻塞了"读操作"。

###### **第一道防线：网关层限流（Redis + Lua 脚本）**

限制每个租户每秒请求的次数和 Token 的消耗量。

###### **第二道防线：读写分离 + 异步队列 (Python/RabbitMQ)**

**回答问题**：依旧是同步 HTTP 调用，直接等待 LLM 返回。

**Upload 接口：** **绝对不要同步等待 MinerU 解析完！**

1. Java 接收文件 -> 上传 OSS。
2. Java 发送一条消息到 **RabbitMQ / Redis List**：`{ "tenant_id": "A", "file_url": "..." }`。
3. Java 立刻返回给前端：`{ "status": "processing", "task_id": "123" }`。
4. **后台 Python Worker (Celery):** 慢慢消费队列里的消息，解析完后把结果写入库，并通过 WebSocket 通知前端。

###### **第三道防线：兜底超时 (Timeout)**

防止 Agent 死循环。

- **场景：** Agent 在思考时卡住了，或者试图调用外部 API 没响应。
- **实现：**
  - 在 Java 调用 Python Agent 的 HTTP 请求中，**强制设置 `ConnectTimeout` = 2s, `ReadTimeout` = 60s**。
  - 如果 60 秒没吐字，直接掐断，返回"服务繁忙，请重试"。

### **针对外部客户的差异化需求，你的架构如何适配？** (配置化, 插件化设计)

**问题描述：** "你们的客户有做金融的、有做法律的、有做科研的。金融客户要求表格解析精度极高，法律客户要求条款间的引用关系能被追踪，科研客户要求公式能被还原。你不可能给每个客户写一套代码。你的架构怎么做到'一套系统，多种场景'？"

**解决方案：**

我的架构在三个层面做了配置化和插件化：

1. **解析层配置化：** MinerU 本身提供规则管线和视觉模型管线两条路径。我在租户配置表中增加了 `parse_profile` 字段，不同客户可以选择不同的解析策略。比如金融客户强制走视觉模型管线（表格精度高），纯文本合同客户走规则管线（速度快、成本低）。解析策略的切换对下游 Agent 完全透明，因为输出格式统一为大纲 JSON 树。

2. **工具链插件化：** Agent 的工具链（`search_within_doc`、`read_section`、`search_kg`、`top_doc`）是模块化注册的。不同客户可以通过配置启用或禁用特定工具。比如没有建知识图谱的客户，`search_kg` 工具不会出现在 Agent 的可用工具列表中，避免 Agent 调用不存在的能力。新增工具只需实现统一的 Tool 接口并注册，不需要改 Agent 核心逻辑。

3. **Prompt 模板化：** 不同行业客户的 System Prompt 可以通过模板引擎动态组装。比如法律客户的 Prompt 会额外注入"注意条款间的引用关系"的指令，科研客户会注入"优先使用 read_section 阅读公式密集的章节"的策略提示。模板和租户配置绑定，运行时动态渲染。

> **面试口径对齐：** 这里的核心卖点是"Agent 的行为通过配置驱动，而不是硬编码"。简历里写的"多粒度工具链"天然支持这种插件化——每个工具是独立的，Agent 根据可用工具列表动态决策。

### 关于 ReAct 循环的"死锁与烧钱" (Deadlocks & Cost)

**问题描述**："你的 Agent 又是看地图，又是搜，又是读。如果一个文档很难懂，Agent 在里面像无头苍蝇一样转了 10 圈怎么办？ 每一次 `read_section` 都是 2000 token。转 10 圈就是 20k token，再加上 Input Context，这一通下来几块钱没了。 **你有熔断机制吗？如果 Agent 陷入了 'Search -> Read -> Search -> Read' 的死循环，你的系统怎么检测并强行终止？** 别跟我说 'LLM 很聪明'，我看过太多 LLM 在两个章节之间反复横跳了。"

**解决方案：**

这个问题我在状态机设计阶段就预判到了，所以做了三层防御，它们不是事后补丁，而是**状态机的一等公民设计**：

1. **Max Steps (最大步数)：** 状态机硬编码 `max_iterations = 5`。超过次数，状态机强制跳转到 `GENERATE_ANSWER` 状态，基于已收集到的信息尽力回答，而不是返回空结果。这样用户至少能拿到一个"部分答案"，体验远好于"未找到"。

2. **Visited Memory (访问记忆) —— 状态机核心字段：** 状态机维护 `visited_node_ids`（已读章节集合）和 `used_keywords`（已用关键词集合）。每轮决策时，这两个集合注入 Prompt：`你已经访问过以下章节: {visited_ids}，已使用过以下关键词: {used_keywords}。严禁重复。` 这从信息层面切断了"反复横跳"的可能性。如果 Agent 尝试调用 `read_section` 读取已访问的章节，Tool 层直接拦截返回 `"该章节已读取，请选择其他章节或尝试回答"`，**零 Token 消耗**。

3. **Cost Guard (成本卫士)：** 在状态机的每轮循环入口处累加 Token 消耗。单次 Session 设置 Token 上限（50k tokens），超限直接触发熔断，状态机跳转到终态输出当前最优答案。这个阈值是根据业务场景下"一个复杂问题通常需要 3-4 轮检索"的经验值反推出来的。

> **面试口径对齐：** 这里的 `visited_node_ids` 和 `used_keywords` 就是简历里写的"状态机维护 Action 执行历史及结果、已生成关键词集合、已读章节集合"。面试时要强调：这些防御机制是状态机 State 的核心字段，和 Action 历史一起驱动每轮决策，不是外挂的限流组件。

### 关于 MinerU 的"理想主义" (The Clean Data Fallacy)

**问题描述**："你的整个 Layer 2 和 Layer 3 导航，全部依赖于 MinerU 解析出的那个漂亮的 JSON Tree。 现实世界里，我上传一份 **扫描歪了的、甚至带有手写笔记的 PDF 合同**，OCR 根本识别不出 H1/H2 标签。 这时候 `generate_tree` 失败，或者生成了一棵只有 Root 节点的树。 **你的 Agent 此时会报错崩溃吗？如果没有树，你的 `Maps_tree` 工具怎么退化？你的系统鲁棒性在哪里？**"

**解决方案：**

我的数据层实现了 **'自适应 ETL' (Adaptive ETL)**。MinerU 解析后，我会校验目录树的**深度**和**节点密度**。如果发现解析失败（比如 OCR 乱码或无标题），系统会自动降级为 **'物理页码模式' (Physical Pagination)**。此时生成的 Tree 结构变为 `Page 1 -> Page 2`。对 Agent 来说，它依然可以通过 `Maps_tree` 进行翻页和查找，只是粒度从'章节'变成了'页'。这保证了系统的**高可用性 (High Availability)**，无论用户传什么烂文件，系统都能跑。

> **面试口径对齐：** 这和简历里写的"对于无标题结构的文档（扫描件、纯文本），按页码将固定段落数组成'虚拟章节'，确保所有文档都具备可导航的大纲"是同一个设计。面试时要强调：这不是一个 edge case 补丁，而是 ETL 管道的标准流程——每份文档解析完都会经过大纲质量校验，不合格的自动走 fallback 路径。

### 黑盒调试与可观测性 (The Black Box)

**问题描述：** "ReAct Agent 最恶心的地方在于它是不确定的。昨天用户问'A'，它答对了；今天问'A'，它去读了别的章节，答错了。运维跑来问你：'为什么这个 Case 错了？' 你怎么办？去翻几千行的 text log？ **请展示你的 Trace 设计。每一个 Step，每一个 Tool 的输入输出，每一次 Thought 的耗时，你是怎么记录的？你用了什么工具（LangSmith? Arize? Jaeger?）？** 如果没有完整的 Trace ID 链路追踪，你的系统就是个不可维护的黑盒。"

**解决方案：**

**核心逻辑：结构化 Tracing，和状态机天然结合。**

因为我的 Agent 是状态机驱动的，每一轮循环的 State 本身就是一个天然的 Trace 节点。我在状态机的每轮状态转移时，自动记录一个结构化的 Step Log：

```json
{
  "trace_id": "req-uuid-xxx",
  "state": "TOOL_EXECUTION",
  "action": { "tool": "read_section", "params": { "doc_id": "xxx", "section": "3.2" } },
  "result_summary": "返回 1200 tokens，包含表格数据",
  "token_usage": { "input": 3200, "output": 1200, "cumulative": 12400 },
  "latency_ms": 850,
  "visited_nodes": ["1.0", "2.1", "3.2"],
  "used_keywords": ["营收", "Q3", "同比增长"]
}
```

- **工具选型：** 开发期用 **LangSmith**（可视化好，能直接看到每轮 Thought/Action/Observation 的完整链路）；生产期用 **OpenTelemetry + Jaeger**（和公司现有的微服务监控体系打通，每个 Tool Call 是一个 Span，嵌套在 Agent Session 的父 Span 下）。
- **Trace ID 设计：**
  - `TraceID`: 全局唯一的 Request ID，从 API Gateway 生成，贯穿 Java -> Python Agent -> ES/KG 全链路。
  - `SpanID`: 每一个 Tool Call (Search, Read, KG) 都是一个 Span，记录输入参数、输出摘要、耗时、Token 消耗。
- **关键指标：**
  1. **Token Usage:** 每一步消耗多少 token（和 Cost Guard 的累加器共用同一个计数器）。
  2. **Latency Breakdown:** Search 耗时 vs. Read 耗时 vs. LLM Thinking 耗时，可以精确定位瓶颈在哪一步。
  3. **Feedback Loop:** 在前端加一个"点踩"按钮。用户点踩后，该 Trace ID 及其所有 Step Logs 自动推送到"Bad Case 审查队列"。运维拿到 Trace ID 后，可以在 LangSmith/Jaeger 里一键回放整个 Agent 的决策过程，精确到每一轮看了什么、搜了什么、为什么做出了错误判断。

> **面试加分点：** 强调状态机的 State 本身就是结构化日志的数据源。不需要额外埋点，状态机每轮转移自然产生完整的决策轨迹。这比在 LangChain 的 callback 里手动拼日志优雅得多。

### 模型延迟

**问题描述：** "你把一个 0.5B 的模型放在了 Search 的返回链路里？还是放在 `search_within_doc` 这种高频工具里？你知不知道 Search 是 **热路径 (Hot Path)**？通常 Search 的 SLA (服务等级协议) 是 <200ms。哪怕是 0.5B 模型，如果你没有做极端的工程优化，加载+推理+Token输出，随随便便就是 500ms+。如果 Agent 决定连续搜 3 次，用户就要等 1.5 秒才能看到下一步动作。**你的系统 P99 延迟是多少？你是怎么部署这个 0.5B 模型的？vLLM 还是 ONNX？你是串行处理 10 个 Snippet 还是并行？显存炸了怎么办？**"

**解决方案：**

首先要澄清一个关键点：**0.5B 的小模型在我的架构里不是放在 Search 热路径上的，而是作为 Agent 的决策 LLM（即状态机的"大脑"）。** Search 本身走的是 ES 的 BM25 + 向量检索，延迟在 50ms 以内，不涉及模型推理。小模型负责的是"看完搜索结果后决定下一步做什么"——这个环节的延迟容忍度比 Search 高得多。

具体的延迟优化策略：

1. **部署方案：** 0.5B 模型用 **vLLM** 部署，开启 Continuous Batching。vLLM 的 PagedAttention 机制在多并发场景下显存利用率远高于原生 HuggingFace 推理。单张 A10G（24GB 显存）可以同时服务 20+ 并发的 Agent Session。

2. **热路径与冷路径分离：**
   - **热路径（Search）：** ES BM25 + 向量检索，P99 < 100ms。重排序用的是轻量级的 Cross-Encoder（如 bge-reranker-base），ONNX Runtime 部署，单次推理 10 个 Snippet 并行打分，P99 < 80ms。
   - **冷路径（Agent 决策）：** 小模型推理，单轮 Thought + Action 输出约 200-500 tokens，P99 约 800ms-1.2s。这个延迟对用户来说是可接受的，因为 Agent 的每轮决策之间本身就有"思考"的语义预期。

3. **显存兜底：** vLLM 本身有请求队列机制，显存不够时新请求会排队而不是 OOM。我在上层加了一个并发控制：单个 vLLM 实例最多同时处理 N 个请求（N 根据显存容量动态计算），超出的请求走降级策略——要么排队等待，要么路由到备用的更小模型（如 0.2B 的 draft model）。

> **面试口径对齐：** 简历里写的"系统在小模型（低成本）场景下仍能稳定运行"，这里就是具体的工程实现。关键是要让面试官理解：小模型是 Agent 的决策大脑，不是 Search 的热路径组件，两者的延迟 SLA 完全不同。

### 小模型的格式异常与纠错机制 (Small Model Error Recovery)

**问题描述：** "你用 0.5B 的小模型做 Agent 决策？小模型的指令遵循能力很差的。它输出的 JSON 格式经常是坏的，工具名拼错，参数缺失。你的系统怎么处理这些情况？总不能每次格式错了就直接报错给用户吧？"

**解决方案：**

这是我在实际开发中花了大量精力解决的问题，也是简历里专门写的一个亮点。

1. **多层解析兜底：** Agent 每轮输出后，先尝试标准 JSON 解析。如果失败，走正则提取（从乱七八糟的输出里抠出 `{"action": ..., "params": ...}` 结构）。如果正则也失败，走 LLM 自修复——把错误输出和错误信息一起喂回去，让模型在下一轮修正。

2. **错误反馈写入状态机：** 这是关键设计。格式错误不是简单地 retry，而是把错误信息作为一条特殊的 Observation 写入状态机的 Action 历史：`"你上一轮的输出格式错误：{error_detail}。请严格按照以下 JSON Schema 输出：{schema}"`。这样 LLM 在下一轮决策时能看到自己犯了什么错，自我修正的成功率远高于简单 retry。

3. **工具名模糊匹配：** 小模型经常把 `search_within_doc` 写成 `search_doc` 或 `searchWithinDoc`。我在 Tool 路由层做了模糊匹配（编辑距离 + 别名表），只要能唯一匹配到一个工具就自动修正，不浪费一轮决策。

4. **熔断降级：** 如果连续 3 轮都格式错误（说明模型在这个 case 上彻底懵了），状态机直接跳转到 `GENERATE_ANSWER`，用已收集到的信息尽力回答，同时把这个 case 标记为 Bad Case 推送到审查队列。

> **面试口径对齐：** 简历里写的"实现面向小模型的全面异常处理与纠错机制"和"格式异常自动恢复率 95%+"就是这套方案的成果。面试时要强调：错误反馈是写入状态机的，不是简单的 retry loop，这和状态机驱动的 Agent Loop 设计是一体的。
