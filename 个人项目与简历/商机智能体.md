# 企微群商机智能体（数字员工）

**技术栈：** Java, Agent 编排, Skill 模块化, 企微消息回调 API, 企微表格 API, LLM, Redis 任务队列

**项目规模：** 3000+ 行代码，1-3 个月从 0 到 1 独立开发并上线

---

## 项目概述

做了一个"数字员工"——它像一个真实的团队助理一样，常驻在公司的企微群里。群成员在聊天中提到的商机信息，它能自动识别、结构化提取、录入商机表格，信息不全时会主动追问，录入完成后在群里 @ 发消息的人反馈结果。

这不是一个简单的 API 调用脚本——它是一个完整的 Agent：感知环境（监听群消息，带历史上下文窗口）、理解意图（NL 抽取）、自主决策（新增/修改/去重判断）、执行操作（写入表格）、与用户交互（追问和 @ 通知）。它模拟了一个真人助理在群里"看上下文→记商机→@回复反馈"的完整工作流。

开发过程中遇到的核心技术挑战是**多 Agent 实例并发写入企微表格导致数据覆盖丢失**，通过引入任务队列串行化写入方案彻底解决。

## Agent 核心能力

### 1. 实时消息感知与意图识别

Agent 通过企微消息回调机制实时接收群消息。**实现细节：在调用 LLM 之前，系统前置抓取表格当前全部数据，与消息上下文一同注入 LLM 调用。意图分类、结构化提取、去重判断在同一次 LLM 调用中完成**，不是串行三步。

关键设计：**回调传递给 Agent 的不只是当前消息，而是当前消息 + 前 N 条历史消息的上下文窗口（N 可配置）**。这意味着 Agent 看到的不是孤立的单条消息，而是带有对话上下文的信息流——它能理解"刚才说的那个项目"指的是哪一条消息里的内容，能关联前后分散在多个消息里的信息片段拼凑出完整商机。N 的默认值是 5-10 条，在上下文充分性和 Token 消耗之间平衡。

Agent 对窗口内的消息逐条分析，首先通过 LLM 理解消息语义，将消息分为三类：

- **闲聊：** "下午茶点什么"、"周末去哪玩" → 忽略
- **商机操作：** 新增/更新/删除商机记录 → 走 CRUD 流程
- **分析查询：** "昨天商机整体情况如何"、"本月商机来源分布" → 走只读分析流程

这里隐含了一个核心架构原则——**读/写分离**：

- **读操作（分析查询）：** 直接执行。查表格、LLM 生成分析回复，不走队列。延迟低，用户可以即时看到分析结果。
- **写操作（商机增/改/删）：** 走消息队列串行化执行。因为写操作会改变表格行号和内容，并发执行会产生数据覆盖。队列保证了同一时刻只有一个写入在执行，从根本上消除并发冲突。

简单说：不涉及改操作直接执行，改操作走队列。这是整个系统并发模型的基石。

```
上下文窗口示例 → Agent 逐条判断：

  [历史]  "昨天跟XX公司聊了一下"            → 非商机（仅提及公司名，无项目信息）
  [历史]  "他们那个智慧园区项目挺有搞头"      → 商机线索，标记关注
  [当前]  "预算大概500万"                    → 商机补充信息，关联上条中的商机
  ──────────────────────────────────────────
  综合判断：XX公司 - 智慧园区项目 - 预算500万 → 进入提取流程

  "大家下午茶要点奶茶吗？"                   → 非商机，忽略
  "上个月说的YY科技项目不做了"               → 商机状态变更，走删除流程
```

**设计决策：发消息触发 vs @数字员工触发**

团队在这个点上讨论了很久。两个方案：

| 方案 | 优点 | 缺点 |
|------|------|------|
| **@Agent 触发** | 意图明确，几乎不会有误识别 | 增加操作成本——群成员必须先想"这是不是商机"，再手动 @ 数字员工。从行为数据看，多这一步操作，商机录入量会大幅下降 |
| **全量消息触发（采用）** | 零操作成本，成员正常聊天即可。符合数字员工的愿景——主动感知环境，而非被动等待指令 | 需要准确区分商机和非商机消息，否则 Agent 在群里说"下午茶要点奶茶吗"是商机会显得很蠢 |

**最终选择全量消息触发。** 核心理由：这个数字员工的卖点就是"群成员不用改变工作习惯"——他们在群里正常聊商机，数字员工自动提取录入。如果还要 @ 一下，就跟在群里 @ 一个人说"帮我记一下"没有区别了，丧失了智能体的存在价值。

但这也意味着**意图识别的准确率是整个系统的生命线**——误识别一次（把闲聊当商机追问），群成员对数字员工的信任就打折扣。LLM 的意图识别能力是这个方案成立的前提：如果是关键词方案，全量触发会产生大量误触发，无法实用。幸好当前 LLM 的语义理解已经足够可靠，实测误触发率可控制在可接受范围。

### 2. 自然语言结构化提取

商机信息散落在非结构化聊天中——公司名叫法不统一（"腾讯" / "深圳市腾讯计算机系统有限公司"），项目名可能夹杂在长句子里，预算有时精确有时模糊（"500万左右" / "大概五六百万"）。

Agent 通过 LLM 将自然语言提取为结构化字段：

```json
{
  "company_name": "深圳市腾讯计算机系统有限公司",
  "project_name": "智慧园区安防升级二期",
  "budget": "500万",
  "contact": "王总",
  "department": "技术部",
  "source_message": "王总刚才说的腾讯那个智慧园区安防的项目，二期预算大概500万",
  "source_user": "张三"
}
```

### 3. 商机全生命周期管理（CRUD + 智能去重）

Agent 不只是"往里写"，而是模拟真人助理管理商机表格的完整逻辑：

- **新增：** 新商机。通过公司名 + 项目名模糊匹配，判断表格中是否已存在
- **更新：** 已有商机的新信息。比如之前记录了公司名和项目名，现在有人补充了预算和联系人，Agent 判断为同一商机并更新对应行
- **删除：** 商机失效。群里有消息说"XX 项目黄了"，Agent 找到对应行标注为失效或删除

**去重逻辑：** 企微群里的公司名不统一，"腾讯"和"深圳市腾讯计算机系统有限公司"可能指同一家公司。Agent 通过编辑距离模糊匹配 + 项目名包含关系来判断是否同一商机，而不是简单的精确匹配。

### 4. 多轮对话与 @ 反馈机制

Agent 的回复全部通过 **@ 发消息人** 的方​​式在群里输出，确保当事人能收到通知，群里其他人也知道数字员工处理了这条商机。根据识别结果，Agent 会走不同的回复路径：

**路径 A：信息不完整 → @ 追问补充**

商机信息往往不是一条消息就能说全的。群里有人随口提了一句"XX 公司有个项目"，公司名有了，但项目名、预算、联系人全缺。Agent 会主动 @ 对方追问：

```
用户A: XX科技有个项目机会
Agent:  @张三 收到，请问项目名称和大概预算是多少？
用户A: 智慧园区项目，预算大概500
Agent:  @张三 好的，还需要一个联系人信息，方便后续跟进
用户A: 联系他们技术部的王总就行
Agent:  @张三 已录入：XX科技 - 智慧园区项目 - 预算500万 - 联系人王总（技术部）。如需修改请随时告诉我。
```

**路径 B：商机已存在 → @ 提醒勿重复添加**

```
用户A: 腾讯那个智慧园区项目，二期预算500万
Agent:  @张三 该商机已存在（腾讯 - 智慧园区项目），请勿重复添加。如需更新信息请说明具体变更。
```

**路径 C：提取成功 → @ 确认录入完成**

```
用户A: YY科技数字化转型项目，预算300万，联系人李总
Agent:  @张三 已录入：YY科技 - 数字化转型项目 - 预算300万 - 联系人李总。如需修改请随时告诉我。
```

三种回复路径的核心目的：让群成员对数字员工的行为有**可预期感**——发了商机一定会被 @ 回复，信息不全会被追问，重复了会被提醒，成功了会确认。这避免了"消息石沉大海，不知道录入没有"的糟糕体验。

多轮追问不是为了炫技——真实场景下，群成员发商机就是随口一说，不会填表。Agent 如果不追问，录入质量会很差，表格里大量字段为空，老板看了也没用。

### 5. 商机分析查询（只读）

Agent 不只是录入工具，也是群里随叫随到的商机数据分析师。群成员可以像问同事一样向它提问：

```
用户A: @Agent 昨天咱们整体商机情况怎么样？
Agent:  @张三 昨日共新增商机 3 条，分别是：
       1. XX科技 - 智慧园区项目 - 预算500万
       2. YY公司 - 数字化转型 - 预算300万
       3. ZZ集团 - 数据中心建设 - 预算800万
       合计预算 1600 万，较前日增长 60%。

用户B: @Agent 本月商机来源分布？
Agent:  @李四 本月共 15 条商机，来源分布如下：
       - 技术部：6 条（40%）
       - 销售部：5 条（33%）
       - 管理层：4 条（27%）
       预算总额 8500 万，其中已中标 3 条共 2000 万。

用户C: @Agent 查一下腾讯那个项目的联系人
Agent:  @王五 腾讯 - 智慧园区项目，联系人：王总（技术部），预算500万，状态：跟进中。
```

与商机 CRUD 操作不同，分析查询**只读不写**——Agent 读取表格数据后直接生成分析回复，不经过任务队列，延迟更低。LLM 将自然语言问题转化为对表格数据的查询逻辑，然后以自然语言呈现结果，不需要群成员去翻表格。

## 系统架构

```
══════════════════════════════════════════════════════════════════
                        企微群
══════════════════════════════════════════════════════════════════
     │              │              │
  用户A消息       用户B消息       用户C消息
     │              │              │
     ▼              ▼              ▼
══════════════════════════════════════════════════════════════════
                 企微消息回调 (Webhook)
         每条新消息触发 → 抓取当前消息 + 前 N 条历史（N 可配置）
══════════════════════════════════════════════════════════════════
     │
     │  传入：上下文窗口 [msg_{t-N}, ..., msg_{t-1}, msg_t]
     │        + 群 ID + 表格 ID（Agent 无状态，状态全部在入参）
     ▼
┌─────────────────────────────────────────────────────────────┐
│                  Agent 实例（无状态，可多实例并发）            │
│                                                             │
│  ① 消费队列积压的写任务（BRPOP 非阻塞，有就执行写入）         │
│                                                             │
│  ② 前置抓取：读取表格当前数据（只读快照）                     │
│                                                             │
│  ③ 一次 LLM 调用（意图分类 + 结构化提取 + 去重判断）          │
│         │                │              │                   │
│    闲聊→忽略      分析查询→@回复    商机操作                 │
│                   (读路径，直接)      (增/改/删)              │
│                                       │                     │
│                                字段完整性检查                 │
│                                缺→@追问  全→LPUSH入队         │
│                                          │                  │
│  ④ 立即消费队列（BRPOP 执行刚入队的写任务）                   │
│     └─ 重新读表格 → 执行写入 → @ 通知 "已录入"               │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
══════════════════════════════════════════════════════════════════
              Redis List（写任务队列，按表格 ID 分 key）
══════════════════════════════════════════════════════════════════
```

**架构要点：**

- **Agent 无状态**——所有状态外置：群 ID 定位对话上下文，表格 ID 定位操作目标，聊天消息承载历史。Agent 实例本身不持有任何内存状态，任意实例拿到相同的入参就产生相同的行为
- **读/写分离**——读操作（分析查询）直接 @ 回复，不经过队列；写操作（增/改/删）入 Redis List，由 Agent 实例在触发时顺手消费。不做无谓的串行化
- **队列消费内嵌在 Agent 流程中**——没有独立的 Worker 线程或服务。每个 Agent 实例在流程首尾各消费一次队列（④ 执行刚入队的写任务，① 清理其他实例遗留的积压）。Redis List 的 BRPOP 原子操作天然保证多个实例不会同时消费同一条任务
- **失败重试 + 死信队列**——写入失败指数退避重试，超限次数移入死信队列并企微通知运维
- **每日子 Agent 异步整理**——每天第一条消息触发时 fork 一个子 Agent 异步维护表格：去重合并、格式规整、过期商机标记。子 Agent 独立执行不阻塞主流程

## Agent 决策流程

Agent 是无状态的——入参只有（当前消息 + 前 N 条历史 + 群 ID + 表格 ID），所有上下文来自这些参数。每次消息触发时，按以下流程执行：

```
消息到达（当前消息 + 前 N 条历史上下文 + 群 ID + 表格 ID）
  │
  ▼
◎ 每日首触发检查：如果今天是第一次触发 → fork 子 Agent 异步整理表格（去重合并、格式规整、过期商机标记）
  │   子 Agent 独立执行，不阻塞主流程
  │
  ▼
① 消费队列：BRPOP 取队列中积压的写任务（非阻塞，有就执行，无则跳过）
  │   Redis List 原子操作保证只有一个实例拿到任务
  │
  ▼
② 前置抓取：读取表格当前全部数据（只读快照）
  │
  ▼
③ 一次 LLM 调用（意图分类 + 结构化提取 + 去重判断，三合一）
  │
  ├── 闲聊 ──→ 忽略（不回复）
  │
  ├── 分析查询 ──→ @发消息人 回复分析结果（读路径，不走队列）
  │
  │ 商机操作（增/改/删）── LLM 在同一次调用里输出结构化字段 + 去重结论
  ▼
字段完整性检查
  │
  ├── 缺关键字段 ──→ @发消息人 追问补充 ──→ 等待回复 ──→ (下次触发 ③)
  │
  ├── 已有同商机 ──→ @发消息人 "该商机已存在，请勿重复添加"
  │
  │ 新商机 / 信息更新
  ▼
LPUSH 写任务入 Redis List → @发消息人 "已收到，正在录入..."
  │
  ▼
④ 立即消费队列：BRPOP 取出刚入队的任务 → 重新读表格 → 执行写入 → @发消息人 "已录入：XX公司 - XX项目"
  （如果当前实例抢不到——被其他并发实例的 ① 抢先消费了——对方也会正确执行写入并通知）
```

## 核心技术挑战：并发写入导致数据覆盖丢失

### 问题场景

上面 Agent 流程开发完成后，测试正常，但上线后发现**多人同时发商机时数据会丢**。

原因：企微群消息回调是并发的——A 的消息和 B 的消息会同时触发 Agent 实例。每个 Agent 实例执行"读表格 → 判断行号 → 写表格"的流程，但读和写之间有时间差：

```
Agent A: 读表格（10行）→ LLM判断：新增，插第11行 → 写第11行 ✅
Agent B: 读表格（10行）→ LLM判断：新增，插第11行 → 写第11行 ❌ 覆盖A的数据
```

B 读表格时 A 还没写完，B 也认为该插入第 11 行，结果 A 的商机静默丢失。更隐蔽的 case：

- **删除冲突：** A 删第 10 行后表格变成 9 行，B 此时更新"第 10 行"实际写到了原第 11 行
- **连续删除：** A 删第 10 行，B 也要删第 10 行——结果删了第 10 和第 11 两行

### 本质

企微表格 API 必须指定行号写入，没有原子追加接口。并发场景下"读取时的行号"和"写入时的行号"不是同一个状态，存在读-改-写窗口期。

### 为什么选择任务队列而不是分布式锁

| 方案 | 问题 |
|------|------|
| 分布式锁（Redis SETNX） | Agent 持锁期间要做 LLM 调用，持锁时间几百 ms 到几秒，阻塞所有其他请求；锁竞争失败后的重试策略难设计 |
| 企微 API 层解决 | 企微表格 API 不支持原子追加，也无法加行级锁，外部系统无能为力 |
| **Redis List 任务队列（采用）** | 写操作 LPUSH 入队，Agent 实例在处理消息时顺手 BRPOP 消费。Redis 单线程模型 + List 原子操作天然保证同一任务只被一个实例拿到，无需额外锁。没有独立的 Worker 进程，消费逻辑就嵌在 Agent 流程的首尾 |

### 实现细节

**1. 队列消费嵌在 Agent 流程中**

改造前 Agent 一条龙串行执行：识别商机 → 读表格 → 判断操作 → 写表格。改造后，写操作通过 Redis List 串行化，消费逻辑嵌在 Agent 流程的首尾：

```
每次 Agent 实例被消息触发：
  ① 先消费队列积压（BRPOP，非阻塞）
  ... LLM 调用，判断需要写入 ...
  ④ 再消费队列（BRPOP 执行刚入队的写任务，或由其他实例的 ① 抢先消费）
```

去掉了独立的 Worker 线程——队列的生产者和消费者都是 Agent 实例自身。Redis List 的 BRPOP 原子性自然保证并发安全：多个实例同时 BRPOP，Redis 单线程只会把任务交给其中一个。

**2. 执行时重新读表格**

消费队列中的写任务时，不是基于任务创建时的表格快照，而是**当场重新读取表格最新状态**，基于当前真实行数计算目标行号后写入。这消除了读-改-写窗口期。

**3. 幂等去重**

执行前基于公司名 + 项目名模糊匹配检查表格最新状态——即使同一个商机被两个并发 Agent 实例先后推入队列两条 insert 任务，消费第二条时会发现表格里已经有了，自动转为 update。

**4. 失败重试 + 死信队列**

写入失败时指数退避重试（2s → 4s → 8s），超过 5 次移入死信队列并企微通知运维。死信队列保留完整上下文，支持手动重放。

### 成果

商机录入成功率从约 **70%** 提升至 **99.9%+**，彻底消除并发覆盖导致的数据静默丢失。

## 面试深度问答

### Q1: 这跟一个简单的"关键词匹配 + 写表格脚本"有什么区别？

A: 三个本质区别：

1. **理解而非匹配：** 关键词匹配只能处理固定格式（"公司：XX，预算：500万"），但真实群消息是自然语言，表述千奇百怪。Agent 通过 LLM 理解语义，从"王总说的那个腾讯的项目大概 500 个"这种消息中提取出公司、项目、预算。
2. **决策而非执行：** Agent 自行判断是新增还是更新、信息是否完整、是否需要追问——这些不是写死的规则，而是基于上下文推理的决策。
3. **交互而非单向：** Agent 不只是"收到消息→写表格"，它会追问、反馈、@ 相关人，像一个真实的团队成员在群里协作。

### Q2: 多轮对话的状态怎么管理？用户半天不回怎么办？

A: 每个群维护一个对话上下文，记录当前是否在等待某个用户的补充信息。如果超过一定时间（比如 30 分钟）未回复，自动关闭对话上下文，下次消息当作新的独立消息处理。不会因为一个未完成的对话卡住整个群的商机识别。

### Q3: 为什么不用分布式锁解决并发写入问题？

A: 三个原因：① Agent 持锁期间要做 LLM 调用，持锁时间不可控，会阻塞所有其他请求；② 锁竞争失败后是重试还是排队？重试风暴会把 API 打爆，而 Redis List 的原子 BRPOP 天然就是"排队"，不需要手动实现锁竞争策略；③ 队列附带重试、死信等可靠性保障，不需要额外造轮子。而且这种方案没有引入独立的 Worker 进程，队列消费就是 Agent 流程的一部分，架构更简洁。

### Q4: 没有独立 Worker，每次 Agent 触发才消费队列，写入延迟会不会很高？

A: 不会。商机录入是低频场景——一个群一天 10-50 条商机，群消息的到达频率远高于写任务的产生频率。Agent 在流程首尾各消费一次队列：④ 基本当场把刚入队的任务消费掉，延迟在毫秒级；① 只是兜底消费其他实例遗留的积压。极端情况下（深夜没人发消息，上一个写任务没被消费），第二天第一条群消息触发 Agent 时会在 ① 中清理掉隔夜积压。对于商机录入这个场景，这种延迟完全可接受——比丢数据强一万倍。

### Q5: 去重用模糊匹配，会不会把两个不同公司判成同一家？

A: 阈值保守设置 + 实测验证，误判率 < 1%。极端 case（"北京ABC科技"和"上海ABC科技"）编辑距离差异明显，不会误判。如果后期误判率不可接受，可以升级为 embedding 语义匹配 + 人工确认兜底。
