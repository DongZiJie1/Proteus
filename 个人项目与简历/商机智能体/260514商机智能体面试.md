# 综合面试 Q&A 记录 — 2026-05-14（商机智能体专项）

## Q1：ReAct 容错兜底——错误分类器机制

**你的回答：** 独立的规则错误分类器（非 LLM），根据 API 返回的 errCode 分三大类：API 地址、请求头、请求参数（参数下再分子类如工作表 ID、行数校验等）。分类结果以自然语言建议注入 Observation，LLM 下轮修正。100 次测试中 5 次触发参数错误，经纠错后无一重复。连续 3 次同类错误熔断，推企微告警。恢复通过 MongoDB 记录消息处理状态，未完成消息下次 Agent 自动重处理。

**追问过程：**
- Q: 分类器独立还是 LLM 自纠？→ A: 独立规则模块 → ✅
- Q: 几类错误？注入格式？→ A: 三大类、拼 Observation → ✅
- Q: errCode 映射表——几个 errCode？写在哪？→ A: 前三轮未回答，第四轮答"Nacos 配置中心" → ⚠️ 答了位置但始终未给出 errCode 数量级
- Q: 崩溃恢复重复录入怎么防？→ A: LLM 去重 + 夜维兜底 → ⚠️ 链路完整
- 知识边界：errCode 映射表规模、分类树叶子节点数

**点评：** 错误分类→纠错→熔断→恢复完整链路清晰。弱项：对自己对接的外部 API 有多少错误码缺乏量化认知。LLM 去重可靠性过度自信，与文档 Q6 旧版自述矛盾（文档已修正）。

**评级：** `⚠️ 方向对但细节不足`

## Q2：Redis List 队列 vs 分布式锁——崩溃恢复

**你的回答：** BRPOP 消费写任务，崩溃恢复靠 MongoDB 消息状态兜底（未完成→下次重新生成任务）。去重在入队后 ReAct Thought 阶段通过实时 read_table 对比 LLM 判断，不依赖任务级幂等 ID。写入操作本质幂等（同操作做两次不影响结果）。

**追问过程：**
- Q: BRPOP 后崩溃任务丢失？→ A: MongoDB 消息状态兜底 → ✅
- Q: 重复任务入队？→ A: 前任务已删，本质同操作做两遍，LLM 实时对比防重复写入 → ✅
- 知识边界：write_table 端到端延迟量级未给出

**点评：** 崩溃恢复链路完整，Redis + MongoDB 双存储各司其职。"去重靠 LLM 实时对比而非幂等 ID"在低频场景下务实。但对自己系统的延迟特性没有量化数据。

**评级：** `✅ 基本正确`

## Q3：上下文爆炸事故

**你的回答：** 200K 上下文模型，20 多行表格，2-3 次 read_table 把上下文打到接近 200K。根因是企微 API 每个单元格附带大量格式元数据（字体/字号/颜色/对齐/边框等）+ 深层嵌套 JSON 结构。通过查日志看实际 prompt 发现。

**追问过程：**
- Q: 什么模型？上下文窗口多大？→ A: 外部 API 大模型，200K → ✅（经澄清模型非 Qwen2.5-7B）
- Q: 每个单元格多少 token？→ A: 未给出精确结构，但说明 JSON 嵌套 4-5 层 + 格式字段全量返回 → ⚠️
- 知识边界：API 返回的原始 JSON schema 精确结构未掌握

**点评：** 事故原因和排查路径正确。但对企微 API 的数据格式没有精确量化描述。

**评级：** `⚠️ 方向对但细节不足`

**评级：** `⚠️ 方向对但表述需更精确`

## Q5：上下文窗口限制机制

**你的回答：** 历史消息上下文用双重限制（30 条消息 + 2000 字符，先到先截），基于商机群经验（一条商机不跨太多消息），参数可动态调整无需重新部署。这只是历史消息的截取逻辑，不等于 Agent 总上下文大小。

**追问过程：**
- Q: 文档原来写的 2000 token → A: 纠正为 2000 字符 + 30 条限制 → ⚠️ 文档与真实实现不一致
- 知识边界：参数选择有业务分析支撑

**点评：** 纠正了文档中的表述错误。参数选择有经验依据。但"文档由 Claude Code 优化但与实际不符"是本场反复出现的 pattern。

**评级：** `⚠️ 方向对但表述需更精确`

## Q6：全量消息触发——误判率数据

**你的回答：** 保守策略偏重追问。漏识别率接近零（每周抽 100 条被忽略消息，几周复查无一遗漏），误触发率 1-4%（实测）。误触发后被 Agent 追问机制兜底。

**追问过程：**
- Q: 数据在哪？→ A: 漏识别率每周抽检零遗漏，误触发率 1-4% → ✅
- 知识边界：数据量级给出

**点评：** 追问后给出了采样方法和量化数据。设计闭环完整——宁可追问不漏 + 追问兜底误触发。

**评级：** `✅ 基本正确`

## Q7：多轮追问——状态管理

**你的回答：** 无外部状态管理——追问状态全在群聊历史消息里。Agent 的追问 + 用户回复形成天然问答对，LLM 从上下文中自己串起来。极端情况（追问被挤出窗口）下，Agent 把孤立消息当新线索追问补充缺失字段。

**追问过程：**
- Q: 追问状态存哪？→ A: 只在历史消息里，不存外部状态 → ✅
- Q: 追问被挤出窗口？→ A: Agent 会重追问 → ✅
- 知识边界：设计一致，极端边界有兜底

**点评：** "无状态"设计自洽——群聊历史即状态。极端边界处理务实但不完美（会重追问而非续接上下文）。

**评级：** `✅ 基本正确`

**追问过程：**
- Q: 通信方式？→ A: 进程内 SDK → ✅
- Q: 延迟？→ A: 1-2 秒正常，10 秒超时 → ✅
- 知识边界：清楚

**点评：** 架构选择务实（同进程避免网络开销），延迟数据合理。

**评级：** `✅ 基本正确`

## Q9：夜间商机整理

**你的回答：** 夜维每夜跑，首次清理积压脏数据，之后触发频率低。主要修正 LLM 白天漏判的重复商机（同公司 + 相似项目名）。不改商机状态（状态只能群消息触发）。

**追问过程：**

**点评：** 离线兜底设计合理。但缺少运营数据支撑。

**评级：** `⚠️ 方向对但细节不足`

## 下一轮重点打击方向

基于本轮暴露的弱项：

1. **外部 API 对接的量化认知** — 企微 API 错误码体系、回调 SLA、频率限制具体数值
2. **多租户扩展设计** — 从单群到 100 群的部署拓扑、连接池、队列架构变化
3. **评测体系设计** — 从手工抽检到自动化评测的迁移方案
4. **文档准确性** — 多处"文档写的 ≠ 实际做的"，面试前应核对文档

## 建议下一步

1. **梳理企微 API 对接的量化数据**：errCode 分类表、回调 SLA、频率限制、表格 API 响应格式精确 schema，全部落成文档

3. **设计自动化评测方案**：标注 200+ 条消息作为测试集，每次改 skill.md 后自动跑回归
