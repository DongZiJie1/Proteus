# 企微群商机智能体（数字员工）

**技术栈：** Java（工具层 + API 封装）, 中台内置 Agent（ReAct 范式）, Skill 模块化编排, 企微消息回调 API, 企微表格 API, LLM, Redis 任务队列

**项目规模：** 3000+ 行代码，1-3 个月从 0 到 1 独立开发并上线。核心由 Java 工具层 + Agent Skill 层两层组成

---

## 项目概述

做了一个群商机助手——从群成员的体感上，它像一个真实的团队助理，常驻在公司企微群里。群成员在聊天中提到的商机信息，它能自动识别、结构化提取、录入商机表格，信息不全时主动追问，录入完成后 @ 发消息人反馈结果。此外每天 9:00 自动推送前一天的商机汇总到群里，群成员不用翻表格就能了解整体情况。

**架构上分为两层：**

- **Java 工具层：** 企微官方 API 返回的表格数据太乱——字段名是机器可读的 key、嵌套结构深、空字段散落各处、数据类型不统一。我封装了两个 Java 工具：`read_table`（调用企微 API → 拿原始 JSON → 清洗规整 → 转 Markdown）和 `write_table`（接收结构化写入意图 → 转企微 API 调用）。工具层的职责是**向下屏蔽企微 API 的脏乱差，向上对 Agent 暴露干净的数据抽象**。
- **Agent Skill 层：** 整个商机识别与录入流程写成一个 Skill，运行在公司中台内置的 Agent 上。Skill 采用文件夹结构（`skill.md` + `reference/` + `prompts/`），本质就是完整的 Prompt 工程载体——Agent 本身无状态，全部行为由 Skill 驱动。Agent 运行在 ReAct 范式上，但 ReAct 在这里的核心价值不是多步推理，而是**容错兜底**——LLM 提取字段出错时，Observation 能发现问题，下一轮 Thought 修正，最差也能走到结束点给回复，而不是工作流那样一步错直接报错。Skill 与群/表格解耦，其他团队加载文件夹配置参数即可复用。

这不是一个简单的 API 调用脚本——Java 工具层向上暴露干净的数据抽象，Skill 层将业务逻辑封装为可复用的 Prompt 工程单元。开发过程中遇到的核心技术挑战是**多 Agent 实例并发写入导致数据覆盖丢失**，通过 Redis List 任务队列串行化写入方案解决。

**归属：** 中台提供 Agent 运行框架（ReAct 循环、工具路由）和 Skill 加载机制。我做了：Java 工具层封装、Skill 全部业务逻辑设计、并发写入问题的发现与解决。

## Java 工具层：向下屏蔽企微 API 的复杂性

企微官方表格 API 返回的原始数据对 LLM 极不友好：

```json
// 企微 API 原始返回（节选）—— 字段 key 是机器标识符，值散落在多层嵌套里
{
  "errcode": 0,
  "records": [{
    "record_id": "rpxxxx",
    "values": {
      "公司名称": [{ "type": "text", "text": "腾讯" }],
      "项目名称": [{ "type": "text", "text": "智慧园区" }],
      "预算": [{ "type": "text", "text": "500万" }],
      "状态": [{ "type": "text", "text": "跟进中" }],
      "_l": [{ "type": "text", "text": "" }],       // 空字段
      "_m": [{ "type": "text", "text": "" }],
      // ... 还有 20+ 个系统字段
    }
  }]
}
```

问题：① 每个单元格的值套在 `[{type, text}]` 数组里，LLM 看到的 token 有一半是 JSON 结构开销；② 空字段不省略，全量返回；③ 系统字段（`_l`, `_m`, `_o` 等）和业务字段混在一起，LLM 难以区分哪些是关键信息。

**真实踩坑：上下文爆炸事故。** 上线初期有一次 Agent 突然什么都不回了——所有请求超时。排查发现原因很简单：企微表格 API 返回的原始 JSON 里，每个单元格不只是 `[{type, text}]` 这么简单，还附带大量格式元数据——字体、字号、颜色、对齐方式、单元格位置坐标、边框样式、背景色……一个只写了"腾讯"两个字的单元格，API 实际返回了几百字节的格式化描述。当时表格只有几十行、十来列，但两次 `read_table` 调用就把 Agent 的上下文窗口直接打爆了，LLM 什么也处理不了。但对我来说，修改表格只需要知道单元格的**行号、列名、文本内容**三个字段，格式信息全部是噪音。

我封装了两个 Java 工具，职责就是**把脏数据洗干净，给 Agent 一个清爽的界面**：

**`read_table` 工具：**

```
企微 API → 原始 JSON → 清洗管道：
  1. 过滤空字段（"" / null / 仅空格）
  2. 剥离系统字段（_l, _m, _o 等）
  3. 将 [{type:"text", text:"腾讯"}] 扁平化为 "腾讯"
  4. 规整为 Markdown 表格输出
```

输入：表格 ID，输出：

```markdown
| 序号 | 公司名称 | 项目名称 | 预算  | 联系人 | 状态   | 更新时间   |
|------|---------|----------|-------|--------|--------|-----------|
| 1    | 腾讯    | 智慧园区  | 500万 | 王总   | 跟进中 | 05-13     |
| 2    | YY科技  | 数字化转型 | 300万 | 李总   | 已中标 | 05-12     |
```

Markdown 表格在 LLM 的 token 消耗上远低于原始 JSON（同样 10 行数据，Markdown 约 800 token vs 原始 JSON 约 2500+ token）。而且 Markdown 是 LLM 训练语料中的高频格式，理解准确率更高。

**`write_table` 工具：**

接收结构化的写入意图（操作类型 + 行号 + 字段内容），内部处理企微 API 的行号计算、字段格式回填（Markdown 值 → `[{type:"text", text:...}]` 结构），对外只是一个干净的调用。Agent 不需要知道企微 API 的行号机制和字段格式。

**为什么放在 Java 层？** 清洗逻辑需要稳定、高性能、可测试，不适合放在 LLM 的 prompt 里靠自然语言处理。而且 Java 工具可以直接集成公司的服务治理体系（限流、熔断、链路追踪）。

但更根本的原因是我对 Agent 上下文的一个认知：**上下文窗口就是 Agent 的内存，跟电脑 RAM 一样，是极度稀缺资源。** 给 LLM 的信息越多，它的注意力越容易被稀释——你喂它 2 万 token 的原始 JSON，它不一定能从噪音里挑出那 800 token 关键信息。所以必须在工具层做减法：只给模型它需要的东西（行号 + 列名 + 文本内容），用它最容易理解的格式（Markdown 表格）。这样不仅省 token，更重要的是让模型的注意力集中在商机数据本身，而不是跟 JSON 嵌套结构和格式元数据搏斗。

这个认知当时在团队里是有分歧的。领导和其他同事没有亲手做过 Agent，觉得"LLM 不就是处理文本的吗，多喂点数据不是更好"。但我因为日常就在跟 Agent 打交道，知道上下文污染对模型推理的破坏力——很多幻觉和漏判不是模型能力不够，是信息噪音太大。所以当时我力排众议坚持把 Java 工具层做了。上线后的上下文爆炸事故验证了这个判断，清洗管道从"锦上添花"变成了"系统能跑的前提"。

## Agent + Skill 架构：业务逻辑可复用

整个商机智能体的业务逻辑写成一个 **Skill**，运行在公司中台内置的 **Agent** 上。Skill 采用常见的 Skill 文件夹结构：

```
商机智能体/
├── skill.md          # Skill 主定义：行为描述、工具列表、调用规范
├── reference/        # 参考文档：企微表格 schema、字段映射规则
└── prompts/          # 对话模板：追问话术、回复模板
```

**Skill 本质就是 Prompt 工程。** Agent 本身是无状态的，全部行为由 Skill 驱动——`skill.md` 里写明了 Agent 应该怎么思考（Thought）、什么时候调用什么工具（Action）、拿到结果后怎么判断（Observation）。比如 `skill.md` 中的 Thought 指令大致是：

> 你是一个商机管理助手。当前表格数据如下（{read_table_output}）。请判断用户消息的意图：闲聊则忽略，分析查询则直接回复，商机操作则提取公司名、项目名、预算、联系人等字段，并判断是新增、更新还是重复。

这就是 prompt 工程的完整载体，不是一句"写了个 prompt 让它提取商机"能概括的。

```
┌──────────────────────────────────────────────┐
│          中台内置 Agent（ReAct 范式）          │
│   Thought → Action → Observation → Thought   │
│                                              │
│   ┌──────────────────────────────────────┐  │
│   │        商机智能体 Skill               │  │
│   │                                      │  │
│   │  skill.md 规范了：                    │  │
│   │  - 工具列表与调用条件                 │  │
│   │  - Thought 判断逻辑（闲聊/分析/商机）  │  │
│   │  - 工具调用顺序（read_table 先于      │  │
│   │    write_table）                     │  │
│   │  - 字段提取 schema                   │  │
│   │  - 追问与回复策略                     │  │
│   │                                      │  │
│   │  可用工具（Java 封装）：               │  │
│   │  - read_table | write_table          │  │
│   │  - send_message（企微消息回复）         │  │
│   └──────────────────────────────────────┘  │
└──────────────────────────────────────────────┘
```

- **为什么用 ReAct 而不是工作流：** 工作流方案固然可以覆盖正常路径（读表格 → 提取 → 写入），但 LLM 提取字段出错时怎么办——提取了不存在的公司名、漏了关键字段、格式不符合预期？工作流只能报错，然后呢？ReAct 有 loop 兜底——Observation 发现上一轮 Action 结果不对，下一轮 Thought 可以修正。最差情况也能走到结束点给用户一个回复，而不是静默失败。这个容错机制在生产环境中非常关键
- **Skill 与群/表格解耦**——`skill.md` 不硬编码群 ID 和表格 ID，作为入参传入。市场部新建一个商机群，只需配置新的群 ID + 表格 ID，Skill 代码零改动
- **Skill 跨团队直接复用**——Skill 是自包含的文件夹，工具接口标准化（`read_table(table_id)` → Markdown），其他团队加载文件夹即可。不需要改 prompt，不需要改工具链，不需要懂企微 API

**归属声明：** 中台提供的是 Agent 运行框架（ReAct 循环、工具路由）和 Skill 加载机制。我在这个项目里做的是：① Java 工具层的 `read_table`/`write_table` 封装；② Skill 的全部业务逻辑设计（`skill.md` 的 Thought 判断规则、工具调用时序、多轮追问状态流转、字段提取 schema）；③ 并发写入问题的发现和 Redis List 串行化方案。

**设计对比：为什么不用 LangChain？**

LangChain 的 Chain/Agent 抽象太重，且 Python 生态与公司 Java 微服务体系割裂。中台内置 Agent 已经是公司基础设施，Skill 模式天然支持多团队共用。这个项目的核心产出不是一段脚本，而是**一个可复用的 Skill 单元 + 一组标准化的 Java 工具**。

## Agent 核心能力

### 1. 实时消息感知与意图识别

Agent 通过企微消息回调机制实时接收群消息。**实现细节：在进入 ReAct Loop 之前，系统先通过 `read_table` Java 工具拿到表格当前数据的 Markdown 版本，与消息上下文一同注入 Agent 的 Thought 阶段**。Agent 在 Thought 中同时完成意图分类、结构化提取、去重判断——不需要多轮 LLM 调用来做这些，是在一次 Thought 中综合推理。

关键设计：**回调传递给 Agent 的不只是当前消息，而是当前消息 + 历史消息的上下文窗口，按双重条件限制（默认最近 30 条消息 + 2000 字符，先到先截）**。这意味着 Agent 看到的不是孤立的单条消息，而是带有对话上下文的信息流——它能理解"刚才说的那个项目"指的是哪一条消息里的内容，能关联前后分散在多个消息里的信息片段拼凑出完整商机。这是基于商机群的实际经验取的参数——一条商机通常不会跨越太多个消息，30 条/2000 字符足够覆盖绝大多数情况，且参数可动态调整无需重新部署。当前版本只处理文本消息——图片、语音、文件消息在回调阶段过滤掉（商机信息 90%+ 在文本里，ROI 尚不支持多模态接入）。

Agent 在 Thought 阶段将消息分为三类：

- **闲聊：** "下午茶点什么"、"周末去哪玩" → 忽略
- **商机操作：** 新增/更新/删除商机记录 → 走 CRUD 流程
- **分析查询：** "昨天商机整体情况如何"、"本月商机来源分布" → 走只读分析流程

这里隐含了一个核心架构原则——**读/写分离**：

- **读操作（分析查询）：** 直接执行。查表格、LLM 生成分析回复，不走队列。延迟低，用户可以即时看到分析结果。
- **写操作（商机增/改/删）：** 走消息队列串行化执行。因为写操作会改变表格行号和内容，并发执行会产生数据覆盖。队列保证了同一时刻只有一个写入在执行，从根本上消除并发冲突。

简单说：不涉及改操作直接执行，改操作走队列。这是整个系统并发模型的基石。

```
上下文窗口示例 → Agent 逐条判断：

  [历史]  "昨天跟XX公司聊了一下"            → 非商机（仅提及公司名，无项目信息）
  [历史]  "他们那个智慧园区项目挺有搞头"      → 商机线索，标记关注
  [当前]  "预算大概500万"                    → 商机补充信息，关联上条中的商机
  ──────────────────────────────────────────
  综合判断：XX公司 - 智慧园区项目 - 预算500万 → 进入提取流程

  "大家下午茶要点奶茶吗？"                   → 非商机，忽略
  "上个月说的YY科技项目不做了"               → 商机状态变更，走删除流程
```

**设计决策：发消息触发 vs @数字员工触发**

团队在这个点上讨论了很久。两个方案：

| 方案 | 优点 | 缺点 |
|------|------|------|
| **@Agent 触发** | 意图明确，几乎不会有误识别 | 增加操作成本——群成员必须先想"这是不是商机"，再手动 @ 数字员工。从行为数据看，多这一步操作，商机录入量会大幅下降 |
| **全量消息触发（采用）** | 零操作成本，成员正常聊天即可。符合数字员工的愿景——主动感知环境，而非被动等待指令 | 需要准确区分商机和非商机消息，否则 Agent 在群里说"下午茶要点奶茶吗"是商机会显得很蠢 |

**最终选择全量消息触发。** 核心理由：这个数字员工的卖点就是"群成员不用改变工作习惯"——他们在群里正常聊商机，数字员工自动提取录入。如果还要 @ 一下，就跟在群里 @ 一个人说"帮我记一下"没有区别了，丧失了智能体的存在价值。

但这也意味着**意图识别的准确率是整个系统的生命线**——误识别一次（把闲聊当商机追问），群成员对数字员工的信任就打折扣。LLM 的意图识别能力是这个方案成立的前提：如果是关键词方案，全量触发会产生大量误触发，无法实用。幸好当前 LLM 的语义理解已经足够可靠，实测误触发率可控制在可接受范围。

### 2. 自然语言结构化提取

商机信息散落在非结构化聊天中——公司名叫法不统一（"腾讯" / "深圳市腾讯计算机系统有限公司"），项目名可能夹杂在长句子里，预算有时精确有时模糊（"500万左右" / "大概五六百万"）。

Agent 在 ReAct 的 Thought 阶段，基于消息上下文和 `read_table` 返回的表格 Markdown，将自然语言提取为结构化字段：

```json
{
  "company_name": "深圳市腾讯计算机系统有限公司",
  "project_name": "智慧园区安防升级二期",
  "budget": "500万",
  "contact": "王总",
  "department": "技术部",
  "source_message": "王总刚才说的腾讯那个智慧园区安防的项目，二期预算大概500万",
  "source_user": "张三"
}
```

### 3. 商机全生命周期管理（CRUD + 智能去重）

Agent 不只是"往里写"，而是模拟真人助理管理商机表格的完整逻辑：

- **新增：** 新商机。通过公司名 + 项目名模糊匹配，判断表格中是否已存在
- **更新：** 已有商机的新信息。比如之前记录了公司名和项目名，现在有人补充了预算和联系人，Agent 判断为同一商机并更新对应行
- **删除：** 商机失效。群里有消息说"XX 项目黄了"，Agent 找到对应行标注为失效或删除

**去重逻辑：** 企微群里的公司名不统一，"腾讯"和"深圳市腾讯计算机系统有限公司"可能指同一家公司。去重判断由 LLM 完成——Thought 阶段将提取到的公司名+项目名与 `read_table` 返回的已有商机比对，通过语义理解综合判断是否同一商机，而非编辑距离或精确匹配等规则方案。

### 4. 多轮对话与 @ 反馈机制

Agent 的回复全部通过 **@ 发消息人** 的方​​式在群里输出，确保当事人能收到通知，群里其他人也知道数字员工处理了这条商机。根据识别结果，Agent 会走不同的回复路径：

**路径 A：信息不完整 → @ 追问补充**

商机信息往往不是一条消息就能说全的。群里有人随口提了一句"XX 公司有个项目"，公司名有了，但项目名、预算、联系人全缺。Agent 会主动 @ 对方追问：

```
用户A: XX科技有个项目机会
Agent:  @张三 收到，请问项目名称和大概预算是多少？
用户A: 智慧园区项目，预算大概500
Agent:  @张三 好的，还需要一个联系人信息，方便后续跟进
用户A: 联系他们技术部的王总就行
Agent:  @张三 已录入：XX科技 - 智慧园区项目 - 预算500万 - 联系人王总（技术部）。如需修改请随时告诉我。
```

**路径 B：商机已存在 → @ 提醒勿重复添加**

```
用户A: 腾讯那个智慧园区项目，二期预算500万
Agent:  @张三 该商机已存在（腾讯 - 智慧园区项目），请勿重复添加。如需更新信息请说明具体变更。
```

**路径 C：提取成功 → @ 确认录入完成**

```
用户A: YY科技数字化转型项目，预算300万，联系人李总
Agent:  @张三 已录入：YY科技 - 数字化转型项目 - 预算300万 - 联系人李总。如需修改请随时告诉我。
```

三种回复路径的核心目的：让群成员对数字员工的行为有**可预期感**——发了商机一定会被 @ 回复，信息不全会被追问，重复了会被提醒，成功了会确认。这避免了"消息石沉大海，不知道录入没有"的糟糕体验。

多轮追问不是为了炫技——真实场景下，群成员发商机就是随口一说，不会填表。Agent 如果不追问，录入质量会很差，表格里大量字段为空，老板看了也没用。

### 5. 商机分析查询（只读）

Agent 不只是录入工具，也是群里随叫随到的商机数据分析师。群成员可以像问同事一样向它提问：

```
用户A: @Agent 昨天咱们整体商机情况怎么样？
Agent:  @张三 昨日共新增商机 3 条，分别是：
       1. XX科技 - 智慧园区项目 - 预算500万
       2. YY公司 - 数字化转型 - 预算300万
       3. ZZ集团 - 数据中心建设 - 预算800万
       合计预算 1600 万，较前日增长 60%。

用户B: @Agent 本月商机来源分布？
Agent:  @李四 本月共 15 条商机，来源分布如下：
       - 技术部：6 条（40%）
       - 销售部：5 条（33%）
       - 管理层：4 条（27%）
       预算总额 8500 万，其中已中标 3 条共 2000 万。

用户C: @Agent 查一下腾讯那个项目的联系人
Agent:  @王五 腾讯 - 智慧园区项目，联系人：王总（技术部），预算500万，状态：跟进中。
```

与商机 CRUD 操作不同，分析查询**只读不写**——Agent 读取表格数据后直接生成分析回复，不经过任务队列，延迟更低。LLM 将自然语言问题转化为对表格数据的查询逻辑，然后以自然语言呈现结果，不需要群成员去翻表格。

### 6. 定时主动能力（早报 + 夜维）

商机智能体有两个不依赖消息触发的主动机制——早上推送汇总，晚上整理表格。

#### 6a. 每日商机早报（9:00）

每天早上 9:00，Agent 自动在群里推送前一天的商机汇总：

```
Agent: @all 昨日商机汇总（05-12）
       新增商机 3 条：
       1. XX科技 - 智慧园区项目 - 预算500万 - 联系人王总
       2. YY公司 - 数字化转型 - 预算300万 - 联系人李总
       3. ZZ集团 - 数据中心建设 - 预算800万
       合计预算 1600 万，较前日增长 60%。
       
       状态变更 2 条：
       - YY科技 数字化转型 → 已中标
       - ABC公司 ERP项目 → 已失效
```

这解决了三个问题：
- **信息不对称**：老板和销售总监不用翻表格就能掌握全局，群成员也不需要主动去问
- **数据纠错**：汇总推送到群里后，如果某条商机录入有误（比如预算不对），相关成员会直接在群里纠正——相当于一份每日推送触发了"众包数据校验"
- **Agent 存在感**：即使当天没人发商机消息（静默日），Agent 仍然推汇总，群成员不会"忘了还有这个助手"

实现上，这是一个独立的定时触发器（Cron `0 9 * * *`），复用 Agent 的 `read_table` 工具读取表格 + LLM 生成汇总报告 + `send_message` 推送。**不是完整的 Agent 决策循环，而是定时触发的单轮"读取→生成→推送"管道**——这样不需要维持常驻进程，也没有多实例并发问题。

#### 6b. 夜间商机整理（00:00）

每天零点，系统自动触发一次表格整理任务。与早报不同，整理任务**会改数据**，因此走的是写入队列：

1. `read_table` 读取全表数据
2. LLM 扫描表格，检测问题：重复行（同一公司+项目出现多次）、状态过期（超过 N 天未更新的"跟进中"商机）、字段异常（预算格式不一致、联系人缺失）、格式错乱
3. 执行修正——合并重复行、标记过期商机为"待确认"、统一预算格式
4. **如果有任何修改，将修改内容汇总推送到群里**：

```
Agent: @all 夜间商机整理报告（05-13 00:00）
       自动修正 2 条：
       - 合并重复：腾讯-智慧园区（第3行与第11行重复，已合并）
       - 状态更新：YY科技-数字化转型（30天未更新，标记为"待确认"）
       表格已整理完毕，请相关同事确认上述变更。
```

**设计要点：**
- 整理走写队列，即使零点有多条整理任务触发也不会并发冲突
- 只推变更通知不推全量——只有发生修改时才推送，避免每天深夜骚扰群成员
- 修正策略偏保守——宁可标记"待确认"不直接删除，把人拉回来看一眼比静默删数据安全
- LLM 扫描规则（去重阈值、过期天数）通过 Skill 配置文件可调，不同群可以有不同的整理策略

**架构要点：**

- **Java 工具层屏蔽 API 复杂性**——`read_table`（JSON → 清洗 → Markdown）、`write_table`（写入意图 → 企微 API 调用）。Agent 看到的是干净的 Markdown 表格，不是原始 JSON
- **ReAct 提供容错兜底**——工作流方案在 LLM 提取出错时直接报错，ReAct 的 Observation→Thought 循环可以在下一轮修正错误，最差也能走到结束点给回复，不会静默失败
- **Skill 解耦，可跨团队复用**——Skill 不硬编码群 ID/表格 ID，配置入参传入。其他团队加载 Skill 即可用，零改动
- **Agent 无状态**——群 ID 定位对话上下文，表格 ID 定位操作目标，聊天消息承载历史
- **读/写分离**——读路径直接通过 read_table 工具执行；写路径通过 Redis List 串行化后走 write_table
- **队列消费内嵌在 Agent 流程中**——没有独立 Worker，Agent 实例触发时顺手消费队列。BRPOP 原子操作保证并发安全
- **定时主动能力**——9:00 早报（读路径，推送昨日汇总）+ 00:00 夜维（写路径，整理表格并推送变更），不依赖消息触发的主动性机制

## Agent 决策流程

Agent 是无状态的——入参只有（当前消息 + 历史上下文窗口 + 群 ID + 表格 ID），全部行为由 Skill 驱动。Agent 运行在 ReAct 范式上：正常路径一轮 Thought→Action 就结束，ReAct 的核心价值在异常路径——Observation 发现 LLM 提取出错时，下一轮 Thought 可以修正，兜底保证不会走到死胡同。

```
消息到达（当前消息 + 历史上下文窗口（30条/2000字符双重限制）+ 群 ID + 表格 ID）
  │
  ▼
① 消费队列：BRPOP 积压的写任务（write_table 执行写入，非阻塞跳过）
  │
  ▼
┌─ ReAct Loop ─────────────────────────────────────────────┐
│                                                          │
│  Thought: 分析当前状态（消息内容 + read_table 拿到的      │
│           表格 Markdown），判断意图，决定下一步 Action     │
│     │                                                    │
│     ├── 闲聊 → Action: 不回复，结束                       │
│     │                                                    │
│     ├── 分析查询 → Action: send_message @回复分析结果     │
│     │              （读路径，直接用 read_table 的数据）    │
│     │                                                    │
│     └── 商机操作 → Action: 先 send_message @回复用户      │
│                   "已收到/请补充/已存在"                   │
│                   → 如需要写入，write_table 意图入队       │
│                   → ④ 立即 BRPOP 消费队列执行写入          │
│                          │                               │
│              ┌───────────┘                               │
│              ▼                                           │
│  Observation: 观察上一轮 Action 结果，更新上下文           │
│     │                                                    │
│     ├── 追问后等待用户回复 → 新消息到达，重新进入 Loop     │
│     │                                                    │
│     └── 写入完成 / 分析完成 → 结束                        │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**关键：这个 ReAct Loop 不是 LangChain 的 AgentExecutor，而是中台内置 Agent 的原生能力。** Skill 层面只需声明"我有哪些工具、调用顺序是什么"，Agent 框架自动处理 Thought-Action-Observation 循环和工具路由。

## 核心技术挑战：并发写入导致数据覆盖丢失

### 问题

上线后发现群活跃时段（连续多条商机消息），偶尔出现数据丢失。原因很简单：企微消息回调是并发的，A 和 B 的消息同时触发两个 Agent 实例，都读到表格有 10 行、都判断新商机应插入第 11 行，后写进去的那个覆盖掉前面的。企微表格 API 没有原子追加接口，读-改-写窗口期内并发写入会互相覆盖。

### 方案

思路很直接：**读可以并发，写必须排队**。所有写操作 LPUSH 到 Redis List，Agent 实例在流程首尾顺手 BRPOP 消费——① 先消费积压的写任务，④ 写完后再消费自己刚入队的任务。Redis 单线程模型保证 BRPOP 的原子性，多个实例同时拉取也不会拿到同一条。

为什么不用分布式锁？Agent 持锁期间要做 LLM 调用，持锁时间不可控；锁竞争失败后的重试策略很难设计。Redis List 天然就是"排队"，不需要自己实现锁竞争逻辑。

消费时当场重新读表格获取最新状态，不信任任务创建时的快照。写入失败指数退避重试，超过 5 次移入死信队列并通知运维。

### 成果

录入成功率从约 **95%** 提升至 **99%+**。丢的 5% 集中在群最活跃的时段，解决后不再有静默丢失。

- 商机录入完整度：从随手记的 1-2 个字段 → 多轮追问后平均 4-5 个字段（公司、项目、预算、联系人、部门）
- 群成员使用率：从"没人主动填表"到群内商机讨论后数字员工自动响应覆盖率 100%（全量触发，零遗漏）

## 面试深度问答

### Q1: 跟直接调 LLM API + 写表格脚本有什么区别？看起来也能跑通。

A: Demo 确实能跑通，但三个工程问题解决不了：① Prompt 散落在代码字符串里，改一句提取逻辑要翻几百行，Skill 文件夹结构让 prompt 集中管理、可版本控制、可团队 review；② 企微 API 原始数据每个单元格套双层 `[{type, text}]` 嵌套，10 行数据 2500+ token，塞进 prompt 几下就爆上下文，Java 工具层清洗压缩到 800 token；③ 团队复用时别人要重新读你的代码、理解你的 prompt、对接企微 API——我的方案是加载 Skill 文件夹 + 配置群 ID 表格 ID，三分钟完成。

### Q2: 为什么不用分布式锁解决并发写入问题？

A: 三个原因：① Agent 持锁期间要做 LLM 调用，持锁时间不可控，会阻塞所有其他请求；② 锁竞争失败后是重试还是排队？重试风暴会把 API 打爆，而 Redis List 的原子 BRPOP 天然就是"排队"，不需要手动实现锁竞争策略；③ 队列附带重试、死信等可靠性保障，不需要额外造轮子。而且这种方案没有引入独立的 Worker 进程，队列消费就是 Agent 流程的一部分，架构更简洁。

### Q3: 没有独立 Worker，每次 Agent 触发才消费队列，写入延迟会不会很高？

A: 不会。商机录入是低频场景——一个群一天 10-50 条商机，群消息的到达频率远高于写任务的产生频率。Agent 在流程首尾各消费一次队列：④ 基本当场把刚入队的任务消费掉，延迟在毫秒级；① 只是兜底消费其他实例遗留的积压。极端情况下（深夜没人发消息，上一个写任务没被消费），第二天第一条群消息触发 Agent 时会在 ① 中清理掉隔夜积压。对于商机录入这个场景，这种延迟完全可接受——比丢数据强一万倍。

### Q4: 为什么要把 read_table 的返回值转成 Markdown 而不是直接用 JSON？

A: 三个原因：① Token 效率——同样 10 行表格数据，Markdown 约 800 token，企微原始 JSON 嵌套结构约 2500+ token，差了 3 倍。Agent 每轮 Thought 都要带表格数据，长期积累的差距非常大；② LLM 理解准确率——Markdown 表格是 LLM 训练语料中的高频格式，模型对它的结构理解远好于企微特有的嵌套 JSON schema；③ 清洗逻辑不该放在 prompt 里——过滤空字段、剥离系统字段、扁平化值结构这些操作是确定性的数据处理，放在 Java 工具层性能高、可测试，放在 prompt 里让 LLM 自己做不仅浪费 token 还不稳定。

### Q5: Skill 给别人复用，真的零改动吗？具体怎么操作？

A: 商机智能体 Skill 只依赖三个工具接口——`read_table`、`write_table`、`send_message`。工具的入参只有表格 ID/群 ID 这些配置项，没有硬编码。市场部新建一个商机群，只需要：① 在企微后台创建新表格，拿到表格 ID；② 在 Agent 中台配置 Skill 的参数（群 ID + 表格 ID）；③ 完成。Skill 代码、Prompt、工具调用逻辑一律不动。而且因为 Java 工具层是标准化的，即使换了一个表格 schema（比如市场部多了一个"预计签约日期"列），只需要在 `read_table` 的清洗配置里加一行字段映射，Skill 层同样零改动。

### Q6: 去重用模糊匹配，会不会把两个不同公司判成同一家？

A: 去重判断由 LLM 完成——将当前消息提取的公司名+项目名与表格中已有商机比对，综合判断是否同一商机。以当前大模型的能力，只要商机描述不是极其模糊（这种情况下人类也无法判断），LLM 的语义理解远优于编辑距离等规则方案。如果后期误判率不可接受，可以升级为 embedding 语义匹配 + 人工确认兜底。

### Q7: 全量消息触发，每条消息都调一次 LLM，成本扛得住吗？

A: 用的是外部 API 大模型（上下文 200K），单次 Thought 约 2000-3000 input + 200 output token。一个活跃商机枪群日均 300-500 条消息，按 500 条算，一个月约 5000 万 token，成本在百元级别。如果未来铺到 100 个群，会做两个优化：① 前置规则过滤——纯表情、单字回复（"好的""收到""OK"）、明确的闲聊特征词（"奶茶""外卖"）直接跳过，预计过滤 40% 的消息；② 同一秒内到达的并发消息合并成 batch 调用，减少 API 请求次数。目前的规模下，优化 ROI 不够，还没做。

### Q8: 意图识别这一步如果错了怎么办？LLM 把商机判成闲聊，不就静默丢失了？

A: 坦率说，意图识别确实是单点——如果 Thought 判定闲聊，Agent 直接结束，后续没有机制发现错误。我做的主要补偿：① 意图识别 prompt 偏向"宁可追问，不要漏"——对于模糊消息（"这个项目还可以""那个事情有进展"），Agent 倾向于追问而不是判为闲聊；② 每周从所有群随机抽 100 条被忽略的消息人工复查，计算漏识别率并校准 prompt。但这个单点风险客观存在，下一步计划是在 Thought 输出中加 confidence score，低于阈值的走"弱追问"路径（"你刚才提到的XX，是商机还是闲聊？"），把人拉回来确认。

### Q9: 你怎么评估 Agent 的质量？有标注数据集吗？

A: 坦白说没有完整的标注+评测体系——这需要业务方配合做大量标注工作。我在项目周期内做了三件事：① 每周从所有群随机抽样 100 条消息，人工复查 Agent 的处理结果（意图判断是否正确、字段提取是否完整），作为线上的质量抽检；② 用户纠错反馈自动积累——群里有成员说"这条你没记""这个不对"，这条消息和处理结果自动入 Bad Case 库，用于 prompt 校准；③ 保存了约 50 条典型消息作为回归集，每次改 skill.md 后跑一遍，确保改动不引入退化。这不是工业级 eval，但在项目阶段足够保证质量不下滑。坦白说如果要铺更多群，评测自动化是第一个要补的基础设施。

### Q10: 你这个系统有"事件驱动的主动性"吗？还是只能定时推送？

A: 有，而且不需要额外机制，就是 ReAct Loop 的自然延伸。场景：群里有人说"腾讯智慧园区项目预算确认了，800 万"。Agent 走正常流程——`read_table` 拿到表格，Thought 阶段发现腾讯-智慧园区已存在、原预算 500 万、新预算 800 万、涨幅 60%。于是同一次 Thought 输出两条 Action：① `write_table` 入队更新预算；② `send_message` @ 原录入人和发消息人："腾讯-智慧园区项目预算从 500 万变更为 800 万（涨幅 60%），如确认无误请忽略，如有疑问请核实。"

这不是我专门写了一套事件驱动引擎，而是 Skill prompt 里的一条规则——更新已有商机时，如果关键字段变动超过阈值（预算变化 > 50%、状态从跟进中变已失效等），除了更新表格，需额外 @ 相关人员推送变更预警。ReAct 的 Thought 本来就在综合判断"这条消息意味着什么 + 我应该怎么响应"，异常变动预警只是 Thought 中多一条判断分支。这也是为什么说 ReAct 在这里的价值不止容错——它天然支持"看到什么就响应什么"，包括看到异常数据时主动预警。
