# 多智能体长文档交叉分析系统（Agentic RAG → Multi-Agent 演进）

**技术栈：** Python, Claude API, MinerU, Elasticsearch, Threading, JSON 文件持久化

**项目规模：** 核心代码 3000+ 行，在项目一 Agentic RAG 基础上 1-2 个月完成架构演进的方案设计与实现。系统由 Planner Agent + Reader Agent（多实例并行）+ Synthesizer Agent 三层组成，Agent 间通过 JSON 文件系统的 TaskBoard 和 MessageBus 通信

---

## 项目概述

项目一的 Agentic RAG 上线后，用户频繁提出跨文档对比分析类问题："这三份合同在违约条款上有什么差异？""设计方案是否同时满足 A 标准和 B 标准的消防要求？""我们的方案和竞品方案在技术路线上有什么不同？"

单 Agent 处理这类问题时出现结构性瓶颈：需要串行读取多篇长文档的相关章节，上下文窗口膨胀严重（3 篇 200 页文档可达 200k+ tokens，单篇文档大纲标题就有上千个），且 LLM 在超长上下文中间部分注意力下降（Lost in the Middle），分析质量不可控。这本质上是"一个人同时精读多份长文档，脑子装不下"的问题。

我延续项目一"模拟真人认知过程"的思路去推导：真人遇到这种情况，自然是找几个同事分工——每人负责精读一份文档；读到一半发现"这里说参照 B 标准第 3.2 节"，直接在群里 @ 负责 B 文档的同事问一句，不用跑去找领导汇报再转达；最后大家把各自的提取结果汇总到一张表上，发现有空缺再让对应的人补一下。

这个"团队分工协作查资料"的类比，直接映射成了 **Planner-Reader-Synthesizer 三层多 Agent 架构**——多个 Reader 并行精读对应多个员工分工，P2P 跨文档引用对应群里直接沟通，Synthesizer 反馈闭环对应汇总时发现空缺让人补充。

**架构上分为三层：**

- **Planner Agent：** 接收用户问题 + 所有文档的大纲 JSON 树，生成每篇文档的 extraction_goal。只做规划不做阅读，职责清晰。少数存在依赖关系的场景标注 DAG 依赖，系统据此控制启动顺序
- **Reader Agent（多实例并行）：** 每篇文档分配一个独立 Reader，内部复用项目一的 Agentic RAG 状态机。多个 Reader 在独立线程中并行执行，各自拥有隔离的上下文窗口
- **Synthesizer Agent：** 汇总所有 Reader 的结构化提取结果，构建对比维度矩阵，自动发现维度缺口并驱动 Reader 补充提取，最终生成交叉分析报告

开发过程中遇到的核心技术挑战：Reader 间跨文档引用如何路由（P2P vs Hub-and-Spoke 的选择）、Synthesizer 补充提取的收敛控制、多 Agent 并发写入共享状态的安全性。

**归属：** 我独立完成了：① Planner-Reader-Synthesizer 三层架构设计；② P2P 跨文档引用机制（含 Subagent 隔离设计）；③ Synthesizer 反馈闭环（维度缺口发现 → 定向补充 → 再汇总）；④ TaskBoard + MessageBus 共享状态层设计；⑤ Reader 容错与上下文压缩。

## Planner-Reader-Synthesizer 三层架构

### Planner：只看大纲，不做阅读

Planner 的核心原则是"轻量、快速、不做阅读"。它接收用户问题和所有文档的大纲 JSON 树（不加载全文），分析问题后为每篇文档生成 extraction_goal（提取目标）。提取目标是自然语言描述的，精确定义该 Reader 应该从文档中提取什么信息。

```
问题：三份合同在违约条款上有什么差异？

Planner 输出：
  Document A（销售合同）→ extraction_goal: 提取违约金比例、触发条件、赔偿上限、免责条款
  Document B（采购合同）→ extraction_goal: 提取违约金比例、触发条件、赔偿上限、免责条款
  Document C（框架协议）→ extraction_goal: 提取违约金比例、触发条件、赔偿上限、免责条款
```

绝大多数场景下各文档的提取任务互相独立，系统按 task 列表一对一创建 Reader 线程全部并行启动。少数存在依赖关系的场景（如"先总结 A 的结论，再用该结论评估 B"），Planner 会标记 DAG 依赖，系统据此控制启动顺序。

### Reader：复用项目一的状态机

每个 Reader 内部复用项目一的 Agentic RAG 状态机设计：

- 大纲驱动：文档大纲 JSON 树注入 Reader 上下文
- 多粒度工具链：`search_within_doc` + `read_section`
- 已读章节/已用关键词状态维护（独立于其他 Reader）
- 按需精读，不预加载全文

多个 Reader 在独立线程中并行执行，各自拥有隔离的上下文窗口。单 Agent 场景下 3 份文档串行阅读的上下文累积问题被彻底解决——每个 Reader 只需要装下自己那份文档的内容。

### Synthesizer：汇总交叉分析

所有 Reader 完成后提交结构化提取结果（不是原始文档，是提取后的结构化信息）。Synthesizer 汇总所有结果，构建对比维度矩阵，发现缺口后驱动补充，最终生成交叉分析报告。

## Reader 间 P2P 跨文档引用机制

### 场景驱动

文档 A 中出现"违约金按照 B 标准第 3.2 节执行"。Reader A 正在精读 A，它需要 Reader B 提供 B 标准第 3.2 节的内容才能完成自己的提取任务。

### 为什么是 P2P 而不是 Hub-and-Spoke

两个方案：

| 方案 | 路由方式 | 问题 |
|------|---------|------|
| Hub-and-Spoke | Reader A → Planner → Reader B → Planner → Reader A | Planner 上下文被大量跨文档内容撑爆，延迟翻倍 |
| **P2P（采用）** | Reader A → Reader B → Reader A | 直接通信，Planner 完全不知情，延迟最小 |

选择 P2P 的核心理由：跨文档引用是 Reader 之间的工作协调，不是规划层面的问题。Planner 的任务是"分配工作"，不是"协调工作"。如果把协调也走 Planner，Planner 的上下文会被大量跨文档内容碎片撑爆，而且每次中转延迟翻倍。

### 异步非阻塞设计

Reader A 发出跨文档请求后不等待——继续阅读其他章节，每轮 Agent Loop 开头检查收件箱，收到回复后注入 messages 继续分析：

```
Reader A 的 Agent Loop：
  ① 检查收件箱 → 没有回复
  ② 继续读下一章 → 发现需要 B 标准 3.2 节 → 发送 P2P 请求给 Reader B
  ③ 检查收件箱 → 没有回复 → 继续读其他章节
  ④ 检查收件箱 → 收到 Reader B 的回复 → 注入 messages → 继续分析
  ⑤ 所有章节读完，信息充分 → 提交提取结果
```

当 A 无其他章节可读时，进入轮询等待模式（带超时兜底），避免提前退出丢失关键信息。

### Subagent 隔离

Reader B 收到跨文档请求后，启动独立子 Agent（Subagent）处理：子 Agent 拥有独立的 messages 上下文，根据 B 的大纲自主定位章节、多轮阅读后提取所需信息回复 A。处理完成后子 Agent 的中间过程全部丢弃，B 的主 Loop 上下文不受任何污染。

这个设计很关键——如果不隔离，Reader B 处理跨文档请求时读入的大量内容会混入自己的主 Loop，影响 B 对自己文档的理解和提取质量。

## Synthesizer 反馈闭环：信息缺口发现与补充提取

### 问题根源

各 Reader 独立提取时，由于 extraction_goal 的描述粒度有限，加上不同文档的章节组织方式差异很大，各 Reader 提取出的信息维度经常不对齐。例如：

```
Reader A 提取了：违约金比例、触发条件、免责条款
Reader B 提取了：违约金比例、赔偿上限
Reader C 提取了：违约金比例、触发条件、争议解决方式
→ 直接拼对比表格：A 缺赔偿上限/争议解决，B 缺触发条件/免责条款/争议解决，C 缺赔偿上限/免责条款
→ 大量空缺，用户体验很差
```

### 四步闭环流程

**第一步：维度矩阵构建与缺口检测。** Synthesizer 拿到所有 Reader 的结构化提取结果后，先遍历所有提取项收集出现过的全部对比维度（违约金、触发条件、免责条款、赔偿上限、争议解决……），然后逐文档检查覆盖情况，生成一个"维度 × 文档"的矩阵，标记出每个文档缺失的维度。维度发现由 LLM 通过 `compare_dimension` 工具完成，不是硬编码规则——因为对比维度本身是从提取结果中动态发现的，不同问题的维度完全不同。

**第二步：定向补充请求。** 对于矩阵中的每个缺口，Synthesizer 通过 `request_supplementary` 工具向对应 Reader 发送补充提取请求。请求内容是具体的自然语言描述："请补充提取文档 B 中关于违约触发条件和免责条款的信息"，而非泛泛的"再读一遍"。消息写入对应 Reader 的收件箱（`supplement_request` 类型）。

**第三步：Reader 高效补充。** Reader 收到补充请求后，不需要从头阅读文档——它的状态机中保留了已读章节集合和大纲信息，LLM 可以直接判断"触发条件可能在 4.3 节，我之前没读过这个章节"，精准定位后读取并提交补充结果。如果之前已经读过相关章节但没提取该维度的信息，Reader 会从已有上下文中直接提炼，无需重复读取。补充完成后回传 `supplement_response` 给 Synthesizer。

**第四步：Synthesizer 再汇总。** 收到补充结果后，Synthesizer 将新信息填入矩阵，再次检查是否还有缺口。如果仍有缺失（Reader 确认文档中确实没有该信息），则在对比表格中标注"该文档未涉及此条款"，与"未提取"做明确区分——这对用户判断很重要。

### 收敛控制

补充请求最多触发一轮（Synthesizer 的 Agent Loop 中硬限制），避免 Synthesizer 和 Reader 之间无限来回。实测一轮补充即可将矩阵完整度从约 60% 提升至 95%，剩余 5% 是文档中确实不存在的信息。

### 具体示例

用户问："A 标准和 B 标准在消防要求上的差异，我们的设计方案是否同时满足？"

三个 Reader 分别精读 A 标准、B 标准、设计方案后提交提取结果。Synthesizer 构建矩阵时发现：

```
对比维度       A 标准     B 标准     设计方案
耐火等级       ✅ 已提取   ✅ 已提取   ✅ 已提取
疏散通道宽度   ✅ 已提取   ❌ 缺失     ✅ 已提取
喷淋系统规格   ❌ 缺失     ✅ 已提取   ❌ 缺失
```

Synthesizer 向三个 Reader 分别发送定向补充请求。Reader A 查大纲发现第 7.3 节"自动灭火系统"之前没读过，读取后补充了喷淋要求；Reader B 的 LLM 判断疏散通道信息在已读的第 5 章中，直接从已有上下文提炼，无需重复读取；Reader C 读取了设计方案附录中的消防设备清单。

补充后矩阵填满，Synthesizer 才能准确判断："设计方案的疏散通道满足 A 标准但不满足 B 标准的宽度要求，喷淋系统同时满足两个标准"——如果没有反馈闭环，这个结论根本得不出来，因为关键的对比维度在第一轮提取中被遗漏了。

## TaskBoard + MessageBus：基于文件系统的共享状态层

选择文件系统（JSON 文件）而非 Redis/DB 做共享状态，是因为这个系统的规模不重——3-6 个 Agent 实例，一次分析任务几分钟完成。文件系统足够简单、无需额外依赖、天然持久化，且所有中间状态可以直接审计。

### TaskBoard：任务状态持久化

Planner 生成任务后，每个 task 持久化为独立 JSON 文件：

```json
{
  "task_id": "task-001",
  "document_id": "doc-a-xxx",
  "extraction_goal": "提取违约金比例、触发条件、赔偿上限、免责条款",
  "status": "completed",
  "result": {
    "违约金": {"比例": "20%", "触发条件": "逾期超过30天", "赔偿上限": "合同总额"},
    ...
  },
  "error": null,
  "created_at": "2024-05-13T10:30:00Z",
  "updated_at": "2024-05-13T10:32:15Z"
}
```

系统按 task 列表一对一创建 Reader 线程——不存在"抢任务"的竞争，每个 Reader 只读写自己的 task 文件更新状态（pending → in_progress → completed）和写入提取结果。Synthesizer 轮询所有 task 文件判断是否全部完成后启动汇总。

TaskBoard 的核心价值是状态持久化：Reader 崩溃后可基于 task 文件恢复进度（重新读取文档，从上次的 visitedNodeIds 继续），所有中间状态支持过程审计——运维拿到 Trace ID 后可以直接翻看每个 task 的 JSON 文件看"这个 Reader 读了什么、提取了什么"。

### MessageBus：Agent 间异步消息通信

每个 Reader 有一个独立的 JSONL 收件箱文件。消息采用 append-only 写入 + drain-on-read 读取（读后清空防重复处理）：

```jsonl
{"type":"p2p_request","from":"reader_a","to":"reader_b","content":"请提供B标准第3.2节的违约金触发条件","timestamp":"..."}
{"type":"supplement_request","from":"synthesizer","to":"reader_b","content":"请补充提取文档B中关于免责条款的信息","timestamp":"..."}
```

由于多个 Agent 可能同时向同一个 Reader 的收件箱写入消息（如 Reader A 的 P2P 请求和 Synthesizer 的补充请求同时写入 Reader B 的收件箱），通过文件锁保证并发写入安全。

### 为什么不用消息队列

Redis/RabbitMQ 当然可以做得更优雅。但三点考量：① 这个系统只有 3-6 个 Agent 实例，不是几百个——文件系统的并发压力完全可控；② JSON 文件天然就是"持久化的 Trace"，TaskBoard 和 MessageBus 的所有内容就是系统行为的审计日志，不需要额外接 tracing 系统就能排查问题；③ 零外部依赖，部署简单。

**架构要点：**

- **Planner 只看大纲，不做阅读**——与 Reader 职责严格分离，Planner 上下文不膨胀
- **Reader 一对一绑定文档**——每个 Reader 独立上下文窗口，消除单 Agent 串行阅读的上下文爆炸问题
- **P2P 跨文档引用**——Reader 间直接通信，不走 Planner 中转，延迟最小
- **Subagent 隔离**——跨文档请求用独立子 Agent 处理，不污染主 Reader 的上下文
- **Synthesizer 反馈闭环**——自动发现维度缺口，定向补充，不是一次性汇总
- **文件系统共享状态**——TaskBoard + MessageBus 零外部依赖，天然持久化，支持审计

## 全局决策流程

```
用户问题 + 所有文档大纲 JSON 树
  │
  ▼
① Planner：分析问题，为每篇文档生成 extraction_goal
   写入 TaskBoard（一个文档 = 一个 task JSON 文件）
  │
  ▼
② 系统：创建 Reader 线程（一个 task → 一个 Thread）
   每个 Reader 读取 TaskBoard 中自己的 task
  │
  ├── Reader A (Thread) ────┐
  ├── Reader B (Thread) ────┤ 并行执行
  └── Reader C (Thread) ────┘
       │
       │  每个 Reader 内部 = 项目一的 Agent Loop：
       │  ┌─ Loop ─────────────────────────────┐
       │  │  ① 检查收件箱（P2P 请求/补充请求）   │
       │  │  ② 大纲引导阅读章节                  │
       │  │  ③ 发现跨文档引用 → P2P 发送请求     │
       │  │  ④ 更新 TaskBoard 中 task 状态      │
       │  │  ⑤ 上下文压缩检查                    │
       │  │  ⑥ 信息充分 → 提交提取结果到 task    │
       │  └────────────────────────────────────┘
       │
       ▼
③ Synthesizer：轮询 TaskBoard，所有 task completed
  │
  ├── ④ 收集所有 Reader 提取结果
  ├── ⑤ 构建对比维度矩阵，标记缺口
  ├── ⑥ 定向补充请求（request_supplementary → MessageBus → Reader）
  ├── ⑦ 等待补充结果（带超时兜底）
  ├── ⑧ 再汇总，标注"未涉及"vs"未提取"
  └── ⑨ 生成最终交叉分析报告 → 输出给用户
```

---

## 核心技术挑战

### 挑战一：单 Agent 上下文膨胀（为什么需要多 Agent）

场景：三份 200 页的合同对比分析。单 Agent 需要串行读取所有合同的相关章节——读完 A 的违约条款，再读 B 的，再读 C 的，然后交叉对比。所有读取的章节内容堆在同一个上下文窗口里，3 篇文档轻松超过 200k tokens。

多 Agent 方案：每个 Reader 只装自己那篇文档的内容（约 60-80k tokens），三个 Reader 并行工作。Synthesizer 接收的是压缩后的结构化提取结果（每个 Reader 输出约 500-1000 token 的结构化数据），而非原始文档。Token 总消耗降低 50-60%。

### 挑战二：P2P vs Hub-and-Spoke 的路由选择

P2P 的风险是每个 Reader 的收件箱可以被多个 Agent 同时写入（并发安全需要文件锁）。Hub-and-Spoke 的风险是 Planner 上下文被中继的大量跨文档内容撑爆。

最终选择 P2P。核心理由：跨文档引用是 Reader 之间的工作协调，不属于规划层面。Planner 的任务是分配工作，不是协调工作。如果所有协调都走 Planner，Planner 的上下文会变成所有文档碎片的汇集点——这恰恰是我们引入多 Agent 要解决的问题。

### 挑战三：Synthesizer 补充提取的收敛控制

Synthesizer 发现维度缺口后，理论上可能和 Reader 之间无限来回——补充了一个维度后发现又缺了另一个。解决方案是**硬限制一轮补充**：Synthesizer 的 Agent Loop 中补充请求只触发一次。实测一轮补充即可覆盖绝大多数遗漏，剩余 5% 是文档中确实不存在的信息，在报告中标注"该文档未涉及此条款"，与"未提取"做明确区分。

### 挑战四：Reader 崩溃恢复

Reader 可能在 API 超时、格式异常时崩溃。解决方案：① Reader 内部有项目一的异常处理机制（格式纠错闭环 + 熔断）；② Reader 崩溃后重试机制——基于 TaskBoard 中 task 文件的 `status` 和 `updated_at` 判断 Reader 是否还活着，超时未更新则自动重试；③ 如果重试也失败，标记 task 为 `failed`，通知 Synthesizer 在汇总时标注"该文档分析失败"，不阻塞其他 Reader。

---

## 成果总结

- 3 篇文档的跨文档分析场景，多 Agent 并行方案相比单 Agent 串行方案耗时降低约 **45%**
- Token 总消耗降低约 **50-60%**（每个 Reader 只读需要的章节，Synthesizer 处理的是压缩后的结构化结果而非原始文档）
- Synthesizer 反馈闭环使对比表格维度完整度从约 **60%** 提升至 **95%**
- 异步非阻塞的 P2P 跨文档引用机制避免了 Reader 间互相等待导致的串行化退化
- TaskBoard + MessageBus 文件系统方案零外部依赖，所有中间状态天然可审计
- 架构从项目一直接演进，Reader 内部复用 Agentic RAG 状态机，不重复造轮子

## 面试深度问答

### Q1: 三个 Agent 之间怎么通信？为什么不用消息队列？

A: 见上文"为什么不用消息队列"章节。核心逻辑：系统只有 3-6 个 Agent 实例，文件系统的并发压力完全可控；JSON 文件天然是审计日志，不用额外接 tracing；零外部依赖，部署简单。如果未来扩展到几十个 Agent 的规模，可以平滑迁移到 Redis Pub/Sub——MessageBus 的接口是抽象的，底层存储可替换。

### Q2: Planner 怎么知道该给每个 Reader 分配什么提取目标？

A: Planner 拿到的是用户问题 + 所有文档的大纲 JSON 树（标题层级结构）。它看到的是每份文档的目录，不是全文。基于目录信息，Planner 的 LLM 判断"这份文档的哪些章节可能跟用户问题相关"，然后将判断转化为 extraction_goal。比如看到文档 B 大纲里有"第 3 章 违约责任"，用户问的是违约条款差异，那 extraction_goal 就是"提取第 3 章及相关章节中的违约金比例、触发条件、免责条款等信息"。Planner 不需要精确到"读 3.2 节的第 5 段"——那是 Reader 自己的工作。

### Q3: 如果 Planner 分配错了怎么办？比如漏掉了关键章节？

A: 两个兜底机制。① Reader 内部有自主扩展能力——Reader 的 Agent Loop 不限于只读 Planner 指定的范围。如果 Reader 在阅读过程中发现"这个章节引用了另一个相关章节"，它可以自主扩展阅读范围。extraction_goal 是引导，不是牢笼。② Synthesizer 反馈闭环——即使 Planner 漏掉了某个维度，Reader 第一轮也没覆盖到，Synthesizer 在构建对比矩阵时会发现缺口并通过补充请求捞回来。这是"Plan 可以 imperfect，闭环兜底"的设计思路。

### Q4: P2P 跨文档引用，如果 Reader B 还没读到那个章节怎么办？

A: P2P 请求是发给 Reader B 的收件箱，Reader B 的 Agent Loop 每轮开头检查收件箱。Reader B 收到请求后启动独立 Subagent 处理，Subagent 拥有独立的上下文，可以根据 B 的大纲自主定位到请求的章节，不受 B 当前阅读进度的影响。所以不存在"B 还没读到那个章节就回复不了"的问题——Subagent 是新开一个阅读会话。

### Q5: Subagent 会不会让 Reader B 的上下文也膨胀？

A: 不会。Subagent 拥有独立的 messages 上下文，处理完成后 Subagent 的中间过程全部丢弃，只有最终的提取结果回复给请求方。Reader B 的主 Loop 上下文不受任何污染。这是刻意设计的隔离机制——如果不隔离，Reader B 处理 P2P 请求时读入的大量内容会混入自己的主 Loop，影响 B 对自己文档的理解质量。

### Q6: 多 Agent 架构增加了系统复杂度，凭什么说比单 Agent 好？

A: 单 Agent 处理跨文档分析的上限很低——上下文窗口是硬约束，3 篇 200 页文档轻松突破 200K tokens，而且 Lost in the Middle 效应导致长上下文中间部分分析质量下降。增加 Agent 数量确实增加了编排复杂度，但这个复杂度换来的是每个 Agent 的上下文窗口不再随文档数量线性膨胀，可以处理任意数量的文档对比。而且 Reader 是并行执行的，耗时不会随文档数量线性增长。

### Q7: Synthesizer 的补充提取为什么只做一轮？如果一轮后还有缺口呢？

A: 硬限制一轮是实践中的 pragmatic choice。实测一轮补充即可将矩阵完整度从 60% 提升至 95%，剩余 5% 是文档中确实不存在的信息——比如某个合同的违约条款里就是没写免责条款。如果继续第二轮、第三轮补充，投入产出比极速下降，反而可能导致无限循环。对剩余缺口，Synthesizer 在报告中标注"该文档未涉及此条款"，与"未提取"做明确区分。这对用户判断很重要——"文档没写"和"Agent 漏了"是两回事。

### Q8: 跟中心化编排的多 Agent 方案有什么区别？

A: 中心化编排方案（如用一个 StateGraph 编排所有 Agent）本质上所有通信都要经过中央节点。我的方案是去中心化的——Planner 只做初始分配，Reader 之间直接 P2P 通信，Synthesizer 读取 TaskBoard 判断完成状态。去中心化的好处是：Reader 之间的协调不经过中央节点，单点瓶颈少；TaskBoard 和 MessageBus 是文件系统，天然持久化，Agent 崩溃后状态可恢复。坏处是：需要自己处理并发安全（文件锁），调试复杂度更高。选择去中心化是因为跨文档引用的通信模式就是去中心化的——真人协作也是这样，不需要一个"领导"中转每句话。
