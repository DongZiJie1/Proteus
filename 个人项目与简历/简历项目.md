## 项目名称：企业级长文档智能问答系统（Agentic RAG 架构重构）

**技术栈：** Java, Python, MinerU, Elasticsearch, 知识图谱, LLM (大模型 + 小模型), vLLM

**项目背景：**
公司原有 RAG 系统采用业界标准的"Chunk 切分 + 向量/关键词/知识图谱多路召回 + 重排序"线性管道架构。在长文档（5000k+ tokens）和复杂问题场景下，存在严重的检索质量和推理能力瓶颈。我在排查问题时意识到：传统 RAG 模拟的是一个"没有目录、只能在碎纸堆里靠关键词翻找"的人——Chunk 切分把文档打碎，向量检索在碎片里做相似度匹配，模型拿到的是几段脱离上下文的片段。但真人查一份 200 页的合同，绝不会这样做：一定是先翻目录建立全局认知，根据问题判断该看哪个章节，读完后评估信息是否充分，不够再去读下一个相关章节。这个"模拟真人查资料认知过程"的思路，直接驱动了我将系统从线性管道向 Agentic 架构重构的方向。

**核心职责与亮点：**

- **诊断老架构的根本缺陷并推动重构：**
  - 定位了老方案的三个核心问题：① Chunk 切分破坏文档逻辑结构（如表头与表格分离），导致跨片段推理失败；② 关键词检索依赖 LLM 在未见文档的情况下盲猜，检索偏移严重；③ 复杂问题的"预拆分子问题"机制是伪 Agent——子任务之间无信息传递，一步拆错则全链路失败。
  - 明确了重构方向：从"预计算索引（离线智能）"转向**"运行时认知（Runtime Cognition）"**。

- **砍掉 Rerank 层的决策与论证（面试话术）：**
  - 老架构在多路召回后接了一层 cross-encoder rerank（BGE-Reranker），上线初期看指标确实涨了几个点，但我在深入排查 bad case 后判断：rerank 在我们的场景里是在给上游的烂召回擦屁股，不是一个值得保留的环节。
  - 核心论据：① **rerank 解决不了召回阶段的根本问题**——如果相关内容在 top-k 召回里根本没出现（Chunk 切碎了、关键词盲猜偏了），rerank 再牛也排不出来，garbage in garbage out；② **延迟代价不值**——cross-encoder 是 query-doc pair 逐对打分，O(n) 复杂度，召回 50 个 chunk 过一遍 rerank 延迟直接翻倍，在我们的生产环境里这个延迟换来的精度提升 ROI 很低；③ **pointwise 打分逻辑不适用于我们的核心场景**——我们大量问题需要多章节综合推理，rerank 只能给单个 chunk 打相关性分，不考虑文档间的互补性和信息完整性，排到第一的 chunk 未必是回答问题最需要的那个；④ **开源 rerank 模型泛化能力存疑**——BGE-Reranker 在 BEIR 上跑分好看，但在我们的垂直领域（工程标准、合同条款）实测效果打折严重，不 fine-tune 基本是安慰剂。
  - 我的替代思路：与其在碎片召回后加一层重排序补救，不如从源头解决问题——大纲驱动让 Agent 具备文档全局感知能力，按需精读完整章节而非在碎片里做相似度匹配。实测砍掉 rerank 后，配合新架构的大纲粗筛 + 章节精读机制，准确率反而更高（65% → 90%+），同时省掉了 rerank 模型的推理延迟和 GPU 资源。
  - **面试追问预备**：如果面试官问"什么场景下 rerank 还是有用的"——事实型精确查询（"XX 公司 2024 年营收多少"）、延迟预算充足且上游已经优化到位的场景下，rerank 作为最后一层 polish 有边际价值。但对于我们的长文档多跳推理场景，投入产出比远不如优化 query 理解和文档导航策略。

- **主导文档解析层技术选型：从 RAGFlow + 阿里云迁移到 MinerU：**
  - 项目初期解析层采用 RAGFlow 内置 DeepDoc 解析 + 阿里云文档智能（Document Mind）的组合方案。随着业务规模扩大，暴露三个结构性问题：① RAGFlow 的 DeepDoc 与框架深度耦合，解析中间产物（BBox 坐标、标题元素）无法精细控制，二次开发成本极高——它是面向终端用户的产品，不是面向工程师的可编排组件；② 阿里云文档智能涉密文档出网的合规红线 + 按页计费在日均万页量级下成本陡增；③ 两套引擎并行维护意味着双倍异常处理和不一致的输出格式，下游 Agent 的 Prompt 依赖稳定结构化输出时，这种不一致被放大为检索质量波动。
  - 核心选型诉求：私有化部署（满足合规）、模块化可编排（满足 Agentic RAG 精细控制需求）、多模态元素高质量结构化输出（为 VLM 扩展预留空间）。横向压测 MinerU、Unstructured.io、Marker 等开源方案后收敛到 MinerU：MinerU 在插图精确裁剪保存、公式还原（UniMERNet）、表格识别（StructEqTable）上工程完成度更高，且输出的 `文本 + 裁剪图片 + BBox 坐标` 富结构数据为多模态 RAG 留足扩展空间。对简单文档走 MinerU 规则管线（纯 CPU，低延迟），复杂文档（双栏、无边框表格、公式密集）走视觉模型管线（DocLayout-YOLO 版面检测），按文档复杂度动态路由，平衡解析质量与计算成本。
  - 解决 MinerU 的工程缺陷：视觉管线延迟高，将解析层设计为异步事件驱动微服务解耦用户请求；MinerU 的 `text_level` 字段无法区分标题层级深度（实测经常把所有标题打成 level 1），自研后处理模块基于标题元素 BBox 坐标高度（反映渲染字号）做容差聚类，高度降序映射为 H1/H2/H3，重构出大纲 JSON 树的层级结构；对标题折行导致 BBox 高度异常的情况，通过 `middle.json` 中 lines 数量做归一化修正。
  - 迁移过程设计灰度切换机制：同一批文档同时走新旧两条管线，用 ROUGE-L + 人工抽检对比解析质量，确认 MinerU 在核心文档类型上达到或超过原方案精度后逐步切流。

- **设计大纲驱动的文档理解方案（核心创新）：**
  - 基于上述 MinerU 解析层输出的结构化数据，在文档上传时提取标题层级结构，生成**文档大纲 JSON 树**。对于无标题结构的文档（扫描件、纯文本），按页码将固定段落数组成"虚拟章节"，确保所有文档都具备可导航的大纲。
  - 将备选文档的大纲 JSON 树注入 Agent 的提示词，使 Agent 在决策时能像人翻目录一样看到文档全貌，**从根本上消除了"盲猜关键词"的问题**。
  - 大纲注入做了基于标题数量的自适应裁剪：标题总数 < 100 时全量注入（Token 占比可控），≥ 100 时裁剪所有叶子节点只保留父级标题（200+ 节点裁剪后通常只剩 40-60 个）。Agent 不需要理解层级结构，只需输出它认为相关的多个标题行号，系统侧自动展开到叶子节点并去重后精读，将 LLM 决策负担从"精确选择层级和章节"降到"粗筛相关标题"，对小模型尤其友好。
  - 大纲序列化采用**行号索引**优化：每个标题前加 `#行号` 前缀（如 `#12 3.2 消防设计要求`），Agent 工具调用时只需输出行号列表而非完整章节标题文本，同时省 Token、防止小模型输出标题时漏字/改字导致匹配失败、且行号为整数解析确定性高；兜底机制支持模型输出标题文本时 fallback 到模糊子串匹配，保障鲁棒性。

- **设计并实现状态驱动的 Agent Loop（原生 Java 实现，非框架依赖）：**
  - 未采用 LangChain/LangGraph 等 Python Agent 框架，而是基于 Java 原生实现了完整的状态驱动 Agent Loop。代码层面是一个 while 循环 + 每轮的结构化输出解析，核心在于每轮循环中对 State 对象的精确维护和注入。State 对象是一个 Java POJO，维护以下细粒度字段：
    - `visitedNodeIds`（Set<String>）：已读章节集合，防止重复阅读，工具层硬拦截已读 section_id
    - `usedKeywords`（Set<String>）：已用关键词集合，配合语义去重（embedding 相似度 > 0.85 视为重复）防止关键词穷举
    - `actionHistory`（List）：Action 执行历史及结果摘要，记录每轮做了什么、得到了什么
    - `currentStep`（int）：当前轮数，配合最大轮次（15 轮）硬限制防止无限循环
    - `cumulativeTokens`（long）：累计 token 消耗，触发上下文压缩的判断依据
    - `outlineContext`：当前注入的大纲片段（经粗筛裁剪后的相关子树）
  - 每轮 Prompt 构建时，从 State 对象实时序列化出结构化状态快照注入 Prompt（"你已阅读章节：[列表]，已用关键词：[列表]，当前第 N/15 轮"），而非简单地把历史 messages 全量塞给 LLM。这跟纯粹的"对话历史堆叠"式 Agent Loop 有本质区别：LLM 每轮看到的是经过筛选和压缩的决策依据，而不是原始对话流水。
  - 选择原生实现的核心考量：① 需要在工具执行层面做拦截（如 read_section 返回前检查 visitedNodeIds，已读则直接返回摘要而非实际执行），框架的抽象层反而成为定制障碍；② 小模型的格式纠错需要在每轮解析环节插入自定义逻辑，框架的 callback 机制无法满足；③ 核心 Agent 服务跑在 Java 微服务体系中，与公司现有的服务治理（限流、熔断、链路追踪）天然集成，避免引入 Python 框架带来的跨语言调用开销和运维复杂度。
  - Agent 先尝试回答，根据回答的完整性评估动态决定是否需要补充检索，而非预先拆分问题。

- **设计多粒度工具链，解耦 Search 与 Read：**
  - **search_within_doc：** 在已定位文档内部进行关键词（ES全文检索）搜索 ，返回命中的精确片段。
  - **read_section：** 基于大纲 JSON 树精确定位章节，LLM 输入子目标（goal）描述需要获取的信息，返回完整章节内容。
  - 该设计思路与北大 2026 年发表的 DocDancer 论文（arXiv:2601.05163）不谋而合：该论文同样将文档问答建模为 agentic information-seeking 问题，采用 MinerU 解析 + Search/Read 双工具 + ReAct Agent Loop 的架构，在 MMLongBench-Doc 和 DocBench 两个基准上超越了传统 RAG 和多 Agent 方案，验证了这一技术路线的有效性。我的工作在此基础上进一步延伸：一是扩展到多 Agent 多文档交叉分析场景（论文仅处理单文档），二是在状态机中维护已读章节/已用关键词等细粒度状态防止重复探索，三是实现了小模型场景下的格式纠错闭环（论文依赖 GPT-4o 等强模型）。

- **实现面向小模型的全面异常处理与纠错机制：**
  - 针对小模型作为决策 LLM 时频繁出现的格式问题（非 JSON 输出、错误工具名、参数缺失），设计了完整的异常捕获与纠错反馈链路。系统将错误信息写回状态机，LLM 在下一轮自动修正输出格式，实现"错误反馈 → 自我纠正"的闭环，保障小模型场景下的系统稳定性。

**成果：**
- 复杂多跳推理问题的回答准确率从 65% 提升至 **90%+**。
- 通过"大纲粗筛 + 按需阅读"机制替代全量 Chunk 检索，Token 平均消耗降低 **60%**。
- 系统在小模型（低成本）场景下仍能稳定运行，格式异常自动恢复率 **95%+**。

## 项目名称：多智能体长文档交叉分析系统（Agentic RAG → Multi-Agent 演进）

**技术栈：** Python, Claude API, MinerU, Elasticsearch, Threading, JSON 文件持久化

**项目背景：**
在上述 Agentic RAG 系统的生产运行中，发现用户频繁提出跨文档对比分析类问题（如"这三份合同在违约条款上有什么差异？"）。单 Agent 处理此类问题时，需要串行读取多篇长文档的相关章节，上下文窗口膨胀严重（3 篇 200 页文档可达 200k+ tokens，单篇文档大纲标题就有上千个），且 LLM 在超长上下文中间部分注意力下降（Lost in the Middle），分析质量不可控。这本质上是"一个人同时精读多份长文档，脑子装不下"的问题。我延续项目一"模拟真人认知过程"的思路去推导：真人遇到这种情况，自然是找几个同事分工，每人负责精读一份文档；读到一半发现"这里说参照 B 标准第 3.2 节"，直接在群里 @ 负责 B 文档的同事问一句，不用跑去找领导汇报再转达；最后大家把各自的提取结果汇总到一张表上，发现有空缺再让对应的人补一下。这个"团队分工协作查资料"的类比，直接映射成了 Planner-Reader-Synthesizer 的多 Agent 架构——多个 Reader 并行精读对应多个员工分工，P2P 跨文档引用对应群里直接沟通，Synthesizer 反馈闭环对应汇总时发现空缺让人补充。

**核心职责与亮点：**

- **设计 Planner-Reader-Synthesizer 三层多 Agent 架构：**
  - **Planner Agent**：接收用户问题和所有文档的大纲 JSON 树（不加载全文），分析问题后为每篇文档生成 extraction_goal（提取目标）。绝大多数场景下各文档的提取任务互相独立，系统按 task 列表一对一创建 Reader 线程全部并行启动；少数存在依赖关系的场景（如"先总结 A 的结论，再用该结论评估 B"），Planner 会标记 DAG 依赖，系统据此控制启动顺序。Planner 只做规划不做阅读，职责清晰。
  - **Reader Agent**：每篇文档分配一个独立 Reader，内部复用 Agentic RAG 的状态机设计（大纲驱动 + 多粒度工具链 + 已读章节/已用关键词状态维护）。多个 Reader 在独立线程中并行执行，各自拥有隔离的上下文窗口，解决了单 Agent 上下文膨胀问题。
  - **Synthesizer Agent**：汇总所有 Reader 的结构化提取结果（非原始文档），构建对比维度矩阵，生成交叉分析报告。

- **实现 Reader 间 Peer-to-Peer 跨文档引用机制：**
  - 场景驱动：文档 A 中出现"违约金按照 B 标准第 3.2 节执行"，Reader A 需要 Reader B 提供该章节内容。
  - 选择 P2P 直接通信而非 Hub-and-Spoke（全部经 Planner 中转），避免 Planner 上下文被大量跨文档内容撑爆、延迟翻倍。
  - 异步非阻塞设计：Reader A 发出请求后继续阅读其他章节，每轮 Agent Loop 开头检查收件箱，收到回复后注入 messages 继续分析。当 A 无其他章节可读时，进入轮询等待模式（带超时兜底），避免提前退出丢失关键信息。
  - Reader B 收到跨文档请求后，启动独立子 Agent（Subagent）处理：子 Agent 拥有独立的 messages 上下文，根据 B 的大纲自主定位章节、多轮阅读后提取所需信息回复 A。处理完成后子 Agent 的中间过程全部丢弃，B 的主 Loop 上下文不受污染。

- **实现 Synthesizer 反馈闭环（信息缺口发现与补充提取）：**
  - **问题根源**：各 Reader 独立提取时，由于 extraction_goal 的描述粒度有限，加上不同文档的章节组织方式差异很大，各 Reader 提取出的信息维度经常不对齐。例如 Reader A 提取了违约金比例、触发条件、免责条款，Reader B 只提取了违约金比例和赔偿上限，Reader C 提取了违约金比例、触发条件和争议解决方式——直接拼成对比表格会有大量空缺，用户体验很差。
  - **第一步：维度矩阵构建与缺口检测**。Synthesizer 拿到所有 Reader 的结构化提取结果后，先遍历所有提取项收集出现过的全部对比维度（违约金、触发条件、免责条款、赔偿上限、争议解决……），然后逐文档检查覆盖情况，生成一个"维度 × 文档"的矩阵，标记出每个文档缺失的维度。这一步是 Synthesizer 的 LLM 通过 `compare_dimension` 工具完成的，不是硬编码规则——因为对比维度本身是从提取结果中动态发现的，不同问题的维度完全不同。
  - **第二步：定向补充请求**。对于矩阵中的每个缺口，Synthesizer 通过 `request_supplementary` 工具向对应 Reader 发送补充提取请求。请求内容是具体的自然语言描述（如"请补充提取文档 B 中关于违约触发条件和免责条款的信息"），而非泛泛的"再读一遍"。消息通过 MessageBus 写入对应 Reader 的收件箱（`supplement_request` 类型）。
  - **第三步：Reader 高效补充**。Reader 收到补充请求后，不需要从头阅读文档——它的状态机中保留了已读章节集合和大纲信息，LLM 可以直接判断"触发条件可能在 4.3 节，我之前没读过这个章节"，精准定位后读取并提交补充结果。如果之前已经读过相关章节但没提取该维度的信息，Reader 会从已有上下文中直接提炼，无需重复读取。补充完成后通过 MessageBus 回传 `supplement_response` 给 Synthesizer。
  - **第四步：Synthesizer 再汇总**。收到补充结果后，Synthesizer 将新信息填入矩阵，再次检查是否还有缺口。如果仍有缺失（Reader 确认文档中确实没有该信息），则在对比表格中标注"该文档未涉及此条款"，与"未提取"做明确区分——这对用户判断很重要。
  - **收敛控制**：补充请求最多触发一轮（Synthesizer 的 Agent Loop 中硬限制），避免 Synthesizer 和 Reader 之间无限来回。实测一轮补充即可将矩阵完整度从约 60% 提升至 95%，剩余 5% 是文档中确实不存在的信息。
  - **具体示例**：用户问"A 标准和 B 标准在消防要求上的差异，我们的设计方案是否同时满足？"。三个 Reader 分别精读 A 标准、B 标准、设计方案后提交提取结果。Synthesizer 构建矩阵时发现：Reader A 提取了耐火等级要求和疏散通道宽度，Reader B 提取了耐火等级要求和喷淋系统规格，Reader C（设计方案）提取了耐火等级和疏散通道设计——矩阵中 A 缺喷淋系统、B 缺疏散通道、C 缺喷淋系统设计。Synthesizer 向三个 Reader 分别发送定向补充请求。Reader A 查大纲发现第 7.3 节"自动灭火系统"之前没读过，读取后补充了喷淋要求；Reader B 的 LLM 判断疏散通道信息在已读的第 5 章中，直接从已有上下文提炼出来，无需重复读取；Reader C 读取了设计方案附录中的消防设备清单。补充后矩阵填满，Synthesizer 才能准确判断"设计方案的疏散通道满足 A 标准但不满足 B 标准的宽度要求，喷淋系统同时满足两个标准"——如果没有反馈闭环，这个结论根本得不出来，因为关键的对比维度在第一轮提取中被遗漏了。

- **设计基于文件系统的共享状态层（TaskBoard + MessageBus）：**
  - **TaskBoard**：Planner 生成任务后，每个 task 持久化为独立 JSON 文件（含 document_id、extraction_goal、status、result 等字段），系统按 task 列表一对一创建 Reader 线程——不存在"抢任务"的竞争，每个 Reader 只读写自己的 task 文件更新状态（pending → in_progress → completed）和写入提取结果。Synthesizer 轮询所有 task 文件判断是否全部完成后启动汇总。TaskBoard 的核心价值是状态持久化：Reader 崩溃后可基于 task 文件恢复进度，且所有中间状态支持过程审计。
  - **MessageBus**：基于 JSONL 的 Agent 间收件箱（每个 Reader 一个收件箱文件），append-only 写入 + drain-on-read 读取（读后清空防重复处理）。由于多个 Agent 可能同时向同一个 Reader 的收件箱写入消息（如 Reader A 的 P2P 请求和 Synthesizer 的补充请求同时写入 Reader B 的收件箱），通过文件锁保证并发写入安全。
  - 所有中间状态（任务状态、提取结果、通信记录）持久化到磁盘，支持故障恢复和过程审计。

- **实现 Reader 容错与上下文压缩：**
  - Reader 崩溃（API 超时、格式错误）时自动重试，失败后标记任务状态并通知 Planner，不阻塞其他 Reader 的执行。
  - 长文档场景下 Reader 上下文接近 token 上限时，触发上下文压缩：完整历史落盘保留可追溯性，LLM 摘要中间过程后重建精简 messages，已读章节集合独立于压缩逻辑单独维护，防止压缩后重复阅读。

**成果：**
- 3 篇文档的跨文档分析场景，多 Agent 并行方案相比单 Agent 串行方案耗时降低约 **45%**，Token 总消耗降低约 **50-60%**（每个 Reader 只读需要的章节，而非全量加载）。
- Synthesizer 反馈闭环使对比表格维度完整度从约 60% 提升至 **95%**。
- 异步非阻塞的跨文档引用机制避免了 Reader 间互相等待导致的串行化退化。

## 简历精简版（直接贴简历用）

### 项目一：企业级长文档智能问答系统（Agentic RAG 重构）

**技术栈：** Java, Python, MinerU, Elasticsearch, 知识图谱, LLM, vLLM

- 主导 RAG 系统从传统"Chunk + 多路召回"线性管道向 Agentic 架构的重构，解决长文档场景下 Chunk 切分破坏上下文逻辑、关键词盲猜导致检索偏移、子任务无状态传递等核心缺陷。
- 论证并砍掉老架构的 Rerank 层：cross-encoder rerank 本质是给上游烂召回擦屁股，在长文档多跳推理场景下 ROI 极低——召回阶段没命中的内容 rerank 排不出来，pointwise 打分不考虑多章节互补性，开源模型在垂直领域泛化差。替代方案是从源头解决：大纲驱动全局感知 + 按需精读完整章节，砍掉 rerank 后准确率反而更高，同时省掉推理延迟和 GPU 资源。
- 主导文档解析层从 RAGFlow DeepDoc + 阿里云文档智能迁移到 MinerU：解决原方案框架耦合无法精细控制、涉密文档出网合规风险、按页计费成本陡增三大问题。按文档复杂度动态路由规则管线/视觉模型管线；针对 MinerU 标题层级不分的缺陷，自研基于 BBox 坐标高度容差聚类的后处理模块重构大纲树层级；设计灰度切换机制保障平滑迁移。
- 设计大纲驱动的文档理解方案：通过 MinerU 提取文档标题层级生成 JSON 大纲树（无标题文档按页码生成虚拟章节），注入 Agent 提示词，使其具备文档全局感知能力，从根本上消除盲猜检索。大纲注入做自适应裁剪（标题数 ≥ 100 时裁剪叶子节点只保留父级标题），Agent 只需输出多个相关标题行号，系统侧自动展开到叶子节点并去重后精读，将 LLM 决策负担降到"粗筛相关性"；序列化采用行号索引——每个标题加 `#行号` 前缀，Agent 输出行号列表而非完整标题文本，省 Token、防小模型输出标题漏字/改字、解析确定性高，兜底支持标题文本模糊匹配。
- 基于 Java 原生实现状态驱动的 Agent Loop（未依赖 LangChain/LangGraph 等框架），核心是一个 while 循环 + State 对象精确维护：已读章节集合（visitedNodeIds，工具层硬拦截防重复阅读）、已用关键词集合（usedKeywords，配合语义去重防穷举）、Action 执行历史、当前轮数（硬限制 15 轮），每轮从 State 实时序列化结构化快照注入 Prompt 驱动 LLM 决策；设计多粒度工具链（文档内搜索、章节精读、图谱遍历、全局文档检索），解耦 Search 与 Read。该"将文档问答建模为 agentic information-seeking + MinerU 解析 + Search/Read 双工具"的技术路线，与北大 2026 年发表的 DocDancer 论文（arXiv:2601.05163）核心思路一致，该论文在 MMLongBench-Doc 等基准上验证了这一方向的有效性；我的工作在此基础上进一步扩展到多 Agent 多文档交叉分析、细粒度状态防重复探索、以及小模型格式纠错闭环。
- 实现面向小模型的异常处理与纠错闭环，将格式错误反馈写入状态机供 LLM 下轮自我修正，保障低成本小模型场景下的系统稳定性。
- 复杂多跳问题准确率从 65% 提升至 90%+，Token 平均消耗降低 60%。

### 项目二：多智能体长文档交叉分析系统（基于项目一演进）

**技术栈：** Python, Claude API, MinerU, Elasticsearch, Threading

- 针对单 Agent 处理跨文档对比分析时上下文膨胀严重（多篇 200 页文档超出窗口限制）的瓶颈，设计 Planner-Reader-Synthesizer 三层多 Agent 协作架构。Planner 基于文档大纲为每篇文档生成提取目标，系统按任务一对一创建 Reader 线程并行精读（内部复用项目一的状态机与工具链），Synthesizer 汇总交叉分析。
- 实现 Reader 间 Peer-to-Peer 跨文档引用机制：异步非阻塞通信，Reader 发出请求后继续阅读其他章节；接收方启动独立子 Agent 处理请求，主 Loop 上下文不受污染。选择 P2P 而非 Hub-and-Spoke 中转，避免 Planner 上下文膨胀和延迟翻倍。
- 实现 Synthesizer 反馈闭环：自动发现各 Reader 提取结果的维度缺口，驱动 Reader 补充提取，对比表格完整度从 60% 提升至 95%。
- 设计基于文件系统的共享状态层（TaskBoard + MessageBus）：每个 task 独立 JSON 文件持久化状态，支持 Reader 崩溃恢复与过程审计；JSONL 收件箱实现 Agent 间异步消息通信（drain-on-read），文件锁保证多 Agent 并发写入同一收件箱的安全性。
- 多 Agent 并行方案相比单 Agent 串行，耗时降低约 45%，Token 消耗降低 50-60%。

### 项目三：企微群商机自动录入系统

**技术栈：** Python, 企微表格API, 任务队列（Redis）, LLM

- 针对企微群内多人同时发送商机时 Agent 并发写入企微表格导致数据覆盖丢失的问题，设计基于任务队列的串行化写入方案：所有表格操作封装为独立任务入队，消费端单线程串行执行，从根本上消除并发冲突
- 重构 Agent 执行流程为「识别商机 → 提交写入任务 → 异步通知」，识别与写入完全解耦，解决识别完成到写入之间表格已被他人修改的窗口期问题
- 实现任务幂等性（基于公司名+项目名去重）+ 失败重试 + 死信队列，保障数据最终一致性
- 商机录入成功率从约 70% 提升至 99.9%+，老板能实时查看完整商机表格，不再遗漏
