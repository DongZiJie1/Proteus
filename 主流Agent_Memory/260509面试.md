## Q1：Claude Code 的 query loop 中，工具并行调度具体是怎么实现的？模型需要自己区分哪些工具可以并行吗？

**你的回答：** 不知道。

**正确答案：** 模型**不需要**关心并行/串行。源码中 `toolOrchestration.ts` 的 `partitionToolCalls()` 按工具的 `isConcurrencySafe` 属性自动分批：只读工具（Grep / Glob / Read 等）合并为一批**并行执行**，写操作工具（Edit / Write / Bash 等）**串行执行**。模型可以在一轮中请求多个工具调用，loop 自动处理调度。这是 query loop 内部的实现细节，模型完全不感知。`❌ 未掌握`

## Q2：Hermes 的 Error Classifier 具体是怎么工作的？体现了什么 Harness Engineering 思想？请举一个有/无 Error Classifier 的对比例子。

**你的回答：** 根据工具的返回错误进行直接分类，从对应的预设去处理，避免了 LLM 的过度思考。体现了 harness 的系统配置降低模型压力的思想。如果没有 Error Classifier，工具结果返回 xxx 内部中间变量，模型不知道具体咋回事，从而浪费思考。

**点评：** 核心逻辑抓住了——预编码常见失败模式，避免弱模型浪费 token 去"猜"错误原因。可补细节：①分类器返回给 LLM 的是结构化信息 `{type: PERMISSION_ERROR, recoverable: true, suggestion: "..."}` 而不是原始错误；②例子选"内部中间变量"很好，但可以更具体。

**更好的对比例子：**

> 场景：Agent 调用 `shell_exec("npm install")` 返回 `EACCES: permission denied, mkdir '/usr/local/lib/node_modules'`

❌ **没有 Error Classifier**：LLM 看到这个错误，可能猜"是 npm 坏了？""是网络问题？""要不要换个包管理器？" → 换 yarn → 又失败 → 换 pnpm → 还是失败 → 三轮浪费 token 和时间

✅ **有 Error Classifier**：分类器识别 `EACCES + mkdir /usr/local` → 归类为 `SYSTEM_PATH_PERMISSION` → 直接返回 `"系统目录写入权限不足，建议用 --prefix 指定用户目录或加 sudo"` → LLM 一次搞定

`⚠️ 方向对，细节和例子待补`

## Q3：OpenClaw 的子 Agent 完成任务后，结果是怎么传回父 Agent 的？是同步返回还是异步？具体机制是什么？

**你的回答：** 同步返回，主 Agent 需等待获取子 Agent 的结果。

**正确答案：** 是**异步 push**，不是同步返回。

具体流程：
1. 子 Agent 完成任务
2. `subagent-announce.ts` 把结果以**"用户消息"形式**注入父 Agent 的消息队列
3. 父 Agent 在下一轮 LLM 推理中感知到完成事件
4. 父 Agent 等所有期待的子 Agent 都 announce 后，才输出最终回复

**关键细节：** spawn 时立刻返回 `accepted`（非阻塞），父 Agent 不轮询、不阻塞线程，等消息触发即可。源码注释写得很明确：

> *"After spawning children, do NOT call sessions_list, sessions_history, exec sleep, or any polling tool. Wait for completion events to arrive as user messages."*

**和 CC 的区别：** CC 默认是同步返回（子代理执行完直接返回结果），OpenClaw 是异步 push-based（spawn 立刻返回，完成后通过消息通知）。这个区别在面试对比题中是加分项。`❌ 未掌握`

## Q4：Hermes 的 Skill 两阶段检索是怎么做的？为什么要分两阶段而不是一次性全部加载？

**你的回答：** 第一次只会注入所有 Skill 的元信息，后续命中才会加载全文，防止 Skill 太长压垮上下文，不用的 Skill 全文注入是污染上下文。

**点评：** 核心逻辑完全正确。补充：两阶段具体实现是先用每个 Skill 的精简描述（SKILL.md 的 YAML 头部）和当前任务做匹配，粗排命中后才把 SKILL.md body + scripts/ 加载进上下文。和 `memory_search → memory_get` 的两步走是同一个思路：先轻量匹配、再按需加载。`✅ 优秀`

## Q5：MCP 的传输协议，本地和远程分别用什么？2025 年 3 月的远程协议有什么重大变化？

**你的回答：** 本地是 STDIO，不同进程；远程以前是 SSE，现在是 HTTP。

**点评：** 方向对但不够精确，面试官追问"那 SSE 完全不用了？"会站不住。

**完整回答：**
- **本地 MCP Server**：通过 **STDIO**（标准 I/O）通信，不涉及网络，不需要认证和长连接管理
- **远程 MCP Server**：2025 年 3 月有重大更新——
  - **旧版**：有状态的 HTTP 长连接 + **强制 SSE**（Server-Sent Events）推送
  - **新版**：改为**无状态协议**，用标准 **GET/POST HTTP 请求**，SSE 变为**可选**（仅需要实时推送时才启用）。更接近 REST 风格，降低了实现复杂度

`⚠️ 方向对但不够精确`

## Q6：OpenClaw 的压缩前记忆冲刷（Pre-Compaction Memory Flush）具体是怎么触发的？冲刷的是什么内容？和 Hermes 的做法有什么区别？

**你的回答：** 当上下文长度达到了压缩阈值，会将上下文中值得记忆的东西存储下来。Hermes 不止会存记忆，还可能会判断上下文中有没有可以沉淀的 Skill。

**点评：** 核心差异说对了——Hermes 多走一步。但漏了 OpenClaw 的具体机制。

**OpenClaw 的冲刷机制：** 系统**静默触发一轮 Agent 回合**（用户不可见，`NO_REPLY`），提醒当前 Agent 把重要信息写入 `memory/YYYY-MM-DD.md` 文件。每个压缩周期只触发一次。

**Hermes vs OpenClaw 精确对比：**

| | OpenClaw | Hermes |
|---|---|---|
| 触发时机 | 仅在压缩前触发一次 | 每 ~15 次 tool call 周期性触发 |
| 模式 | 事后归档 | 持续反思 |
| 更新范围 | 只写 memory | memory + skill + Honcho **三路** |
| 核心关注 | "别让重要信息被压缩丢了" | "值不值留 + 失败要避免 + 流程能抽象" |

一句话：OpenClaw 是**被动归档**，Hermes 是**主动反思**。`⚠️ 核心差异对，机制细节待补`

## Q7：Harness Engineering 和传统的 CI/CD 有什么区别？表面很像，本质差在哪？

**你的回答：** 传统的 CI/CD 是固定的、提前设计好的流水线，核心其实还是人去操作；Harness Engineering 核心是大模型预测，路线不是固定的，只是系统会兜底 LLM，让它走偏的时候拉回来。

**点评：** 核心方向抓住了——CI/CD 是确定性流水线，Harness 是不确定 Agent 在确定性约束里跑。表述可更精确。

**一句话区分：** CI/CD 是**确定性流水线跑确定性代码**；Harness 是**确定性约束跑不确定 Agent**。

**三个维度：**

| | 传统 CI/CD | Harness Engineering |
|---|---|---|
| 流水线里跑的 | 确定性代码，同样代码跑 100 次结果一样 | LLM 的不确定输出，同样 prompt 跑 100 次结果可能不同 |
| 流水线本身 | 静态的，yaml 写好步骤固定 | 动态的，Agent 根据当前状态自主决定下一步 |
| 主体 | 人是主体，CI/CD 只是执行人定义好的流程 | AI 是主体，环境是约束 |

**面试加分点：** CI/CD 里测试挂了，你修代码重新提交，流水线本身不需要改。Harness 里 Agent 反复失败，你可能需要**调整 Harness 本身**——任务粒度太粗？验证标准太严？Harness 本身也需要迭代优化。`⚠️ 方向对，表述可更精确`

## Q8：Hermes 的压缩时，磁盘上的原始对话数据会被删除吗？压缩后上下文结构是什么样的？

**你的回答：** 不会，压缩完之后会多一条压缩的记录。

**点评：** 核心对了。补充完整机制：①原始消息在发生时已实时 append 到 `.jsonl`，压缩不会删任何已有行；②压缩只是在 `.jsonl` 中**追加**一条 `compaction` 类型条目（包含摘要文本），同时设置 `firstKeptEntryId` 标记；③下次加载会话时从 `firstKeptEntryId` 之后开始读。

**压缩后上下文结构：** `[压缩摘要] + [保留区原文（最近 ~20K tokens）]`

`✅ 基本正确，细节待补`

## Q9：OpenClaw 选择中心化架构的根本原因是什么？和 Claude Code 选择中心化的原因一样吗？

**你的回答：** 一样，是因为一问一答的交互形式，接收回复的 Agent 和发出消息的 Agent 应该是同一个。

**点评：** "一进一出"这个核心逻辑抓住了，这是 OpenClaw 选中心化的根本原因。但说"一样"不完全准确——两者的出发点**不完全相同**，是**殊途同归**。

**OpenClaw 的出发点：** 对话结构约束——有确定的入口（用户发消息），有确定的出口（给用户一条完整回复），责任链不能断。

**CC 的出发点更丰富：**
1. **安全是第一驱动力**——中心化 = 单一审计点，保证所有行为在用户可监督范围内，权限链条不断裂。Boris 用"瑞士奶酪模型"形容安全设计
2. **简单是第二驱动力**——不需要处理 agent 间的消息路由、协商协议、冲突解决
3. **任务特征**——工具密集 + 顺序推理 + 需要质量门控

**理论支撑（Google 论文）：** 独立多体错误放大 17.2 倍，中心化只有 4.4 倍；去中心化通信开销 O(D·N²)，coding 低熵场景不划算。

**一句话总结：** OpenClaw 从对话结构出发，CC 从安全和任务特征出发，殊途同归选了中心化。面试时两面都答会更有深度。`⚠️ 核心对了，但"一样"太绝对`

## Q10：MCP Server Tool 的 `annotations` 字段是做什么的？为什么需要它？

**你的回答：** 是用来标记这个 Tool 的一些属性。

**点评：** 太简略了，需要展开。

**完整答案：** annotations 的作用是**声明 Tool 的副作用，让 Client 做安全决策**。四个关键属性：①是否只读（ReadOnlyHint）；②是否有破坏性（DestructiveHint）；③是否幂等（IdempotentHint）；④是否与外部世界交互（OpenWorldHint）。

**为什么需要它：** MCP 是 model-controlled——LLM 自己决定调什么工具。但有些工具是只读查询（安全），有些是删除操作（危险）。annotations 让 Client 有权决定是否需要弹窗确认：只读工具自动执行，有破坏性的工具让用户二次确认。

**一句话：** annotations 不影响执行逻辑，但影响 Client 的**信任策略**。`⚠️ 方向对但太简略`

## Q11：Hermes 的自进化循环中，Skill 是怎么防止越积越多爆炸的？每次成功都新建一个 Skill 吗？

**你的回答：** 它察觉到可以固化一个 Skill 时会通过和现有的 Skill 对比，来判断是修复还是新增。

**点评：** 方向接近，但核心机制是 **updating vs creating**，不是"对比修复"。

**正确答案：** 关键词是**覆盖更新（updating）**，不是新建（creating）。执行同类任务发现更好做法时，直接覆盖更新已有 SKILL.md，只有确实是全新类别才新建。这样保证 Skill 库精简，每个类别只有一个最新版本。`⚠️ 方向接近，表述需更精确`

## Q12：Claude Code 的记忆召回为什么不用 RAG，而用 Agentic Search？Boris 原话怎么说的？

**你的回答：** 因为 coding 场景，代码随时有可能会变，那就要重建索引。

**点评：** 核心原因抓住了。补全完整答案：

**CC 拒绝 RAG 的两个原因：** ①代码修改后索引不同步——新写的函数搜不到，删掉的函数还在索引里；②索引的权限管控复杂——谁能访问、如何防止越权。

**Boris 原话：** *"agentic search just outperformed everything, and when I say agentic search, this is a fancy word for glob and grep."*

**本质取舍：** 每次多花点 token 调 glob + grep 实时搜索 vs 建索引省 token 但要维护一致性。coding 场景代码变化极频繁，索引一致性成本高，所以 CC 选了不建索引。

**对比 OpenClaw：** OpenClaw 走了 RAG 路线（SQLite 本地索引，向量语义 + BM25 混合检索），因为面向更通用的 Agent 场景，记忆文件相对稳定，RAG 收益更明显。`✅ 核心对，可补完整`
