## Q6：s06 的上下文压缩有几层？每一层分别做什么？为什么要分层而不是直接做摘要？

**你的回答：** 三层压缩，第一层将旧的工具结果替换成占位符，第二层全量压缩摘要，保存原始对话。

**点评：** 方向对了，但漏了第三层，且没回答"为什么分层"。

**完整答案：**

| 层级 | 名称 | 做什么 | 代价 | 触发频率 |
|------|------|--------|------|----------|
| Layer 1 | micro_compact | 旧 tool result → 占位符 `[Previous: used {tool_name}]` | 几乎为零（字符串替换） | 每轮 |
| Layer 2 | auto_compact | token > 50k 时，保存 transcript 到磁盘，LLM 做摘要，替换所有消息 | 高（一次 LLM 调用） | 偶尔 |
| Layer 3 | manual compact | 模型主动调用 `compact` 工具，触发同样的摘要机制 | 高 | 极少 |

**为什么分层而不是直接做摘要：**
- 压缩有代价：每次摘要都要调用 LLM，既花时间又花钱，而且摘要必然丢失细节
- 三层设计的核心思想是**"能省则省，能晚则晚"**
- micro_compact 能处理 80% 的膨胀问题（tool result 是 token 大户），只有它兜不住时才上更重的手段

**补充：** 完整历史通过 transcript 保存在磁盘上。信息没有真正丢失，只是移出了活跃上下文。

`⚠️ 漏了第三层，没回答为什么分层`

## Q7：s08 的后台任务完成后，结果怎么通知主 Agent？主 Agent 会阻塞等待吗？

**你的回答：** 通过主动推消息，主agent按时去信箱里看看有没有。

**点评：** 方向对了，但表述不够精确。

**精确表述：**
- 不是"按时去看"，而是**"每轮 LLM 调用前排空通知队列"**
- 具体机制：`drain_notifications()` 只拿队列里已有的结果，队列空就跳过，LLM 照常调用
- **主 Agent 永远不会被卡住**，最坏情况就是"还没好，先不管"

**流程示例：**
```
第1轮: spawn "npm install" → drain → 队列空 → LLM正常回答 → 去建配置文件
第2轮: drain → 队列还是空(还在装) → LLM正常回答 → 继续干活
第3轮: drain → 队列有了(装完了) → 结果注入给LLM → LLM知道装好了
```

**类比：** 厨师（主 Agent）把汤丢到灶台上炖着（后台任务），转身去切菜炒别的。每次做完一道菜回头看一眼灶台（`drain_notifications()`），汤好了就处理，没好就继续干活。

`⚠️ 方向对，表述需更精确`

## Q8：s09 的 Agent Teams 用什么通信机制？为什么用 JSONL 文件而不是内存队列？

**你的回答：** 不知道。

**正确答案：**

**通信机制：** JSONL 文件收件箱
- 每个队友有一个收件箱：`.team/inbox/alice.jsonl`、`.team/inbox/bob.jsonl`
- `send()` 追加一行 JSON 到收件箱文件
- `read_inbox()` 读取全部并清空（drain-on-read）

**为什么用 JSONL 文件：**
1. **持久化**：消息写入磁盘，进程重启后不丢失
2. **简单**：不需要引入外部依赖（如 Redis）
3. **可调试**：可以直接查看文件内容，方便调试
4. **教学目的**：JSONL 文件更容易理解

**设计取舍：**
- drain-on-read 的竞态问题：先 `read_text()` 再 `write_text("")` 不是原子操作，但每个收件箱只有对应的队友自己会读，所以实际上不会冲突
- 无消息确认机制：消息发出去就完了，没有 ACK
- 单机限制：线程模型意味着所有队友跑在同一个进程里

**生产环境应该怎么做：**
- 用 Redis Streams 替代 JSONL 文件
- `XADD` 是原子操作，天然并发安全
- 用 consumer group 保证消息只被消费一次

## Q9：s10 的 Team Protocols 为什么要引入 request_id？关机协议和计划审批的方向是反的，但 FSM 完全一样，这说明什么？

**你的回答：** 这个不太清楚。

**正确答案：**

**为什么需要 request_id：**
- 在异步消息系统里，消息到达顺序不确定
- 如果领导同时给 3 个队友发了关机请求，收到的响应必须能对应到具体是谁的、哪个请求的
- request_id 就是这个关联键，它把一来一回绑定成一个完整的事务

**类比：** HTTP 请求-响应的关联、数据库事务的 transaction ID、甚至快递单号的作用完全一样。

**关机协议 vs 计划审批：**
```
关机协议:   Lead → Teammate  (领导发起, 队友响应)
计划审批:   Teammate → Lead  (队友发起, 领导响应)
```

**但 FSM 完全一样：** `pending -> approved | rejected`

**这说明什么：**
- request-response 模式跟"谁是老板"无关，它是一个**对称的通信原语**
- 任何一方都可以发起请求，任何一方都可以响应
- 类比：公司里不只是老板给员工派活，员工也会给老板提审批。表单格式一样，只是发起方和审批方换了

**核心洞察：** 好的协议不需要改底层基础设施，只需要在现有通信通道上定义新的消息格式和状态机。就像 HTTP 协议不需要改 TCP，s10 的协议跑在 s09 的消息总线上，总线本身一行代码没动。

`❌ 未掌握`

## Q10：s11 的 Autonomous Agents 怎么实现"自己找活干"？身份重注入的阈值为什么是 len(messages) <= 3？

**你的回答：** 自动去任务文件里认领任务。

**点评：** 方向对了，但不够完整。

**完整答案：**

**"自己找活干"的机制：**

1. **双阶段循环：WORK 和 IDLE**
   - WORK 阶段：执行任务
   - IDLE 阶段：轮询收件箱和任务看板

2. **空闲阶段轮询：**
   - 每 5 秒轮询一次，最多 60 秒
   - 先检查收件箱有没有新消息
   - 再扫描任务看板找未认领的任务

3. **任务看板扫描：**
   - 找 pending 状态、无 owner、未被阻塞的任务
   - 认领任务：写入 owner 字段，推进状态到 in_progress

**身份重注入的阈值为什么是 len(messages) <= 3：**
- 回顾 s06 的 auto_compact：压缩后 messages 被替换为两条消息：
  ```python
  [
      {"role": "user", "content": "[Compressed]\n\n{summary}"},      # 第 1 条
      {"role": "assistant", "content": "Understood. Continuing."},   # 第 2 条
  ]
  ```
- 然后 IDLE 阶段唤醒时会追加一条新消息（认领的任务）：第 3 条
- 所以 `len(messages) <= 3` 的含义是："如果消息列表只有压缩摘要 + 一条新任务，说明刚经历过压缩，身份信息可能丢了"

**为什么需要显式的 idle 工具：**
- LLM 停止调用工具有两种情况：真的没活干了 vs 只是想说句话
- `idle` 工具是一个明确的信号："我确认没有更多工作要做了，请让我进入空闲状态"
- 这把"是否空闲"的决策权交给了 LLM 自己，而不是靠外部推断

`⚠️ 方向对，细节待补`

## Q11：s12 的 Worktree 任务隔离解决了什么问题？"控制面"和"执行面"分别是什么？

**你的回答：** 同时开发多个功能，本地git worktree 可以同时申请多个agent在不同分支干各自的需求。

**点评：** 方向对了，但不够完整。

**完整答案：**

**解决的问题：**
- 到 s11，智能体已经能自主认领和完成任务
- 但所有任务**共享一个目录**
- 两个智能体同时重构不同模块——A 改 `config.py`，B 也改 `config.py`，未提交的改动互相污染，谁也没法干净回滚

**一句话：** 任务板管"做什么"但不管"在哪做"。解法：给每个任务一个独立的 git worktree 目录，用任务 ID 把两边关联起来。

**"控制面"和"执行面"：**

| 面 | 目录 | 职责 |
|---|---|---|
| 控制面 (Control plane) | `.tasks/` | 任务 JSON 文件，记录状态、依赖、worktree 绑定 |
| 执行面 (Execution plane) | `.worktrees/` | 实际的 git worktree 目录，每个任务独立的工作目录 |

**关键设计：**
- 任务管"做什么"，worktree 管"在哪做"
- 按 ID 绑定：task_id 和 worktree name 关联
- 收尾时可以 `keep`（保留目录）或 `remove`（删除目录 + 完成任务）
- 崩溃后从 `.tasks/` + `.worktrees/index.json` 重建现场

`⚠️ 方向对，细节待补`
