## Q1：Claude Code 的四层权限管道中，第四层 AI 分类器为什么分两个 Stage？token 预算分别是多少？分类器输入为什么要排除主 AI 的文本输出？

**你的回答：** 分两阶段是为了快速初筛+深度精筛，Stage 1 偏保守，安全直接放行，危险/无法判断走 Stage 2。Stage 2 先写推理过程再给结论。Token 预算 64 和 4096，大部分操作在 Stage 1 就能判断，为了成本考虑。排除主 AI 文本输出是为了防 prompt injection。

**追问过程：**
- Q: Token 预算分别多少？ → A: 64 和 4096 → ✅ 正确
- Q: 为什么不直接用 4096？ → A: 大部分操作 Stage 1 足够判断，成本考虑 → ⚠️ 正确但太简略
- Q: 为什么排除主 AI 文本输出？ → A: 防 prompt injection → ✅ 正确，但没展开原理

**点评：** 核心机制都答到了，两阶段设计、token 预算、防注入三个点都对。但两个问题：①成本优化的表述太笼统，没有具体说明"大部分操作在 Stage 1 就结束了（64 token，快且便宜），只有被标记为可疑的才需要 Stage 2"这个关键逻辑；②防注入的原理没有展开，只说了结论没说为什么——应该解释"恶意代码可能让主 AI 输出特定文本来说服分类器，所以排除主 AI 文本只看工具调用记录，从源头阻断注入路径"。

**完整答案：** Stage 1 用 64 token 做快速粗筛，提示词要求"Err on the side of blocking"（宁可错拦），大部分安全操作在这里就完成了。Stage 2 用 4096 token 做深度精筛，要求 AI 先写下推理链再给结论，给被误拦的操作一次"翻案"机会。如果 Stage 2 也无法判断，默认阻止（fail closed）。为什么不直接用一个阶段——省钱，大部分操作在 Stage 1 就被处理了。防注入设计的关键：分类器输入故意排除主 AI 的文本输出，只保留用户消息和工具调用记录。原因是恶意代码文件可能包含 prompt injection 指令，让主 AI 输出特定文本来"说服"分类器放行危险操作，排除主 AI 文本从源头阻断注入路径。分类器还有独立的系统提示词，与主 AI 的提示词体系完全隔离。

`⚠️ 方向对但细节不足`

## Q2：OpenClaw 和 Hermes 的记忆读取策略有一个关键区别——Hermes 有"系统自动预检索"机制，OpenClaw 没有。具体怎么工作的？为什么对弱模型友好？

**你的回答：** 不知道

**追问过程：**
- 主问题直接答不上来

**点评：** 这是 Hermes 相对 OpenClaw 记忆读取的核心差异点。OpenClaw 的记忆检索完全依赖 LLM 自己判断是否调用 `memory_search`，而 Hermes 多了一层 Context Engine 的自动预检索。

**完整答案：** Hermes 的 memory/ 笔记有两条检索通道：①系统自动（Hermes 特色）：Context Engine 在每轮推理前，根据当前用户消息自动检索 memory/ 目录，命中相关片段后直接注入上下文，对 LLM 完全透明；②Agent 主动：LLM 自己判断需要回忆时调用 `memory_search` → `memory_get`。OpenClaw 只有第 2 条通道。对弱模型友好的原因：弱模型的 tool call 能力较弱，不一定会主动判断"我需要搜记忆"，系统自动预检索兜底了这个能力缺陷——即使模型不知道该搜记忆，Context Engine 已经把相关片段塞进来了。本质是**用系统层面的确定性行为弥补模型层面的不确定性判断**。

## Q3：Harness Engineering 的核心实践"原子任务 + 上下文刷新"，为什么要"用完即销毁"？对应哪三个具体问题？

**你的回答：** 每个原子任务用一个 Agent 是为了保证上下文干净不稀释注意力。持续工作的 Agent 上下文会被已完成任务占用空间，稀释注意力影响当前任务执行效果。只知道 Context Rot 一个具体问题。

**追问过程：**
- Q: 对应哪三个具体问题？ → A: 只知道上下文腐化 → ❌ 缺两个

**点评：** "上下文被占用稀释注意力"方向正确，但 Harness Engineering 对这个问题有精确的三症状定义，只答出一个说明对 Harness 的问题域理解不完整。

**完整答案：** Harness Engineering 要解决长时任务中的三个核心问题：①**上下文腐化（Context Rot）**——上下文窗口被垃圾信息淹没，模型丧失对原始指令的遵循；②**幻觉完成（Hallucinated Completion）**——模型不堪重负时伪造"任务完成"的假象，输出看起来完成但实际错误的结果；③**模型漂移（Model Drift）**——多步执行中逐渐偏离初始目标。"用完即销毁"直接解决这三个问题：新 Agent 从干净上下文启动 → 不存在 Context Rot；每个 Agent 只做一件小事、目标明确 → 不会迷失导致 Hallucinated Completion；任务边界清晰、不跨越多个目标 → 不会 Model Drift。Anthropic 的 Harness 哲学原话："每个 Agent 只活一次，干完就死"（each agent lives once, does its job, and dies）。

`⚠️ 方向对但细节不足`

## Q4：Claude Code 的子 Agent 三种派生方式：Fork、Teammate、WorkTree。分别解决什么问题？是互斥还是可组合？

**你的回答：** Fork 是子 Agent 作为工具被主 Agent 调用，Teammate 是多个 Agent 并发执行各自任务，WorkTree 是在不同分支上开发各自需求。可以组合。

**追问过程：**
- Q: Fork 核心特点 + prompt cache 作用？ → A: 同进程、共享 prompt cache → ✅ 正确
- Q: Teammate 的关键特点？用户能做什么 Fork 做不到？ → A: 不知道 → ❌

**点评：** Fork 答对了核心，但三种派生方式的定位没搞清楚。三者是正交维度，不是互斥类型，分别解决三个独立问题。

**完整答案：**

| 方式 | 关键参数 | 解决什么问题 | 核心特点 |
|------|----------|-------------|----------|
| **Fork** | 省略 `subagent_type` | 上下文效率 | 同进程派生，共享 API prompt cache，继承父上下文。成本最低 |
| **Teammate** | `name` + `team_name` | 人机协作 | 独立终端进程，通过文件信箱通信。用户可在独立终端窗格中**观察和干预**每个 worker |
| **WorkTree** | `isolation: 'worktree'` | 代码隔离 | 独立 Git worktree，可叠加在任何模式上。并行改代码不冲突 |

关键洞察：Fork 和 WorkTree **可以组合**（正交维度），OpenClaw 的 `subagent` 和 `acp` 是互斥的（同一参数不同值），Claude Code 设计粒度更细。模型怎么知道该用哪种？靠**工具参数描述 + 系统 prompt 中的决策指南**，本质是"把选择标准写进 prompt，信任模型判断"。

`❌ 未掌握 Teammate 特点`

## Q7：MCP Server Tool 的 annotations 字段有哪四个属性？它们影响谁的行为？

**你的回答：** 忘了具体属性，知道有"只读/可写"属性，认为 annotations 是给大模型看的。

**追问过程：**
- Q: 给谁看的？ → A: 大模型 → ❌

**点评：** 核心误解：annotations 不影响执行逻辑，也不直接给大模型看，是**声明 Tool 的副作用让 Client 做安全决策**。

**完整答案：** 四个属性：ReadOnlyHint（是否只读）、DestructiveHint（是否有破坏性）、IdempotentHint（是否幂等）、OpenWorldHint（是否与外部世界交互）。作用对象是 **Client**，不是 LLM。Client 拿到这些信息后决定信任策略：只读工具自动执行，有破坏性的工具弹窗让用户二次确认。一句话：annotations 是 MCP 的安全层——Server 声明"我这个工具会做什么"，Client 决定"信不信你、要不要问用户"。

`⚠️ 方向对但表述需更精确`

## Q15：OpenClaw 子 Agent 完成任务后，结果同步返回还是异步 push？父 Agent 怎么知道所有子 Agent 都完成了？

**你的回答：** 异步 push。父 Agent 每次推理前查看消息队列。结果以用户消息形式注入父 Agent 上下文。

**追问过程：**
- Q: 同步还是异步？ → A: 异步 push → ✅
- Q: 主动轮询还是被动等待？结果以什么形式注入？ → A: 每次推理前看队列，用户消息形式 → ⚠️ 接近但不精确
- Q: 工具结果还是用户消息？ → A: 用户消息 → ✅

**点评：** 核心机制答对了——异步 push、用户消息形式注入。但"每次推理前看队列"表述不精确，应该是"消息到达后**触发**父 Agent 的下一轮 LLM 推理"，不是父 Agent 主动轮询。

**完整答案：** 子 Agent 完成后，`subagent-announce.ts` 把结果以"用户消息"形式注入父 Agent 消息队列。父 Agent 不轮询、不阻塞线程，等消息触发即可。spawn 时立刻返回 `accepted`（非阻塞）。源码注释："*After spawning children, do NOT call sessions_list, sessions_history, exec sleep, or any polling tool. Wait for completion events to arrive as user messages.*"
