# MCP & SKILL 面试 Q&A 记录 — 2026-05-08

## Q1：MCP 是什么？核心架构是怎样的？

**你的回答：** MCP 是一个标准，限定了 LLM 调用外部工具的标准。核心架构有 MCP Client（用户）和 MCP Server（远程）。

**点评：** 三个问题：①MCP 是"标准化接口协议"，不只是"一个标准"；②架构是三个组件，漏了 Host；③Client 不是"用户"，是 Host 内部的通信组件；④Server 可以是本地也可以是远程。`⚠️ 不完整`

**正确答案：** MCP 是一种**标准化接口协议**，统一了 AI 模型与外部资源（数据、工具、提示词）的连接方式。核心架构有三个组件：**Host**（运行 LLM 的应用程序）、**Client**（在 Host 中运行，负责与 Server 通信）、**Server**（暴露工具、数据资源、提示模板，供 LLM 使用）。典型优点是动态能力更新（工具参数变更时无需重写客户端代码）和接口标准化（不同工具统一接入，减少开发成本）。

---

## Q2：MCP 和 Function Calling 的核心区别是什么？

**你的回答：** FC 调用同进程，MCP 调用非本进程。远程工具修改了不需要重启。MCP 一家做好了工具，所有 client 都能调用。

**点评：** 方向对，但表述不够规范。"不需要重启"对应动态发现，"所有 client 都能用"对应跨平台复用。要用术语表达。`⚠️ 部分正确`

**正确答案：**

| 特性 | Function Calling | MCP |
|------|------------------|-----|
| 标准化程度 | 厂商专有实现，不同 LLM 格式不同 | 开放标准协议，推动互操作 |
| 系统架构 | LLM 与工具的点对点直接交互 | 客户端-服务器架构，支持连接多种 Server |
| 发现机制 | 会话上下文中静态配置工具列表 | 支持工具的动态发现和查询 |
| 可复用性 | 与特定应用和 LLM 平台深度绑定 | Server 可独立部署、跨平台复用 |

## Q3：MCP 的传输协议，本地和远程分别用什么？

**你的回答：** 不知道。

**正确答案：** **本地 MCP Server**：通过 **STDIO**（标准 I/O）通信，不涉及网络，不需要认证和长连接管理。**远程 MCP Server**：2025 年 3 月有重大更新——旧版是有状态的 HTTP 长连接 + 强制 SSE（Server-Sent Events）推送；新版改为**无状态协议**，用标准 **GET/POST HTTP 请求**，SSE 变为可选（仅需要实时推送时才启用）。更接近 REST 风格，降低了实现复杂度，提高了兼容性。`❌ 未掌握`

## Q4：MCP Server Tool 的入参设计原则？

**你的回答：** 不知道。

**正确答案：** 核心原则：**参数最少化、类型严格、约束内联**。①必填参数尽量少，能给默认值就给默认值；②用枚举收敛可选项，不让 LLM 自由发挥；③类型要明确（string/number/boolean）；④约束写在 schema 里（最小值、最大值、默认值），Server 端校验方便，LLM 传参准确率高。`❌ 未掌握`

---

## Q5：MCP Server Tool 的出参设计原则？

**你的回答：** 不知道。

**正确答案：** 核心原则：**结构化返回，LLM 友好**。①用 `outputSchema` 声明返回结构，让 Client 和 LLM 提前知道数据长什么样；②向后兼容——结构化返回的同时附带纯文本版本，保证老 Client 也能读；③不要丢裸 JSON 给 LLM——转成带标题、来源引用、置信分数的可读文本，LLM 读自然语言比读嵌套 JSON 效率高得多。`❌ 未掌握`

---

## Q6：MCP Server Tool 的错误处理原则？

**你的回答：** 不知道。

**正确答案：** 核心原则：**双层错误机制，错误信息对 LLM 友好**。①协议级错误：用标准 JSON-RPC error code 处理参数非法、tool 找不到等问题；②业务级错误：通过 `isError: true` 加 error message 报告。关键原则：错误信息要告诉 LLM 下一步该怎么办，而不是丢堆栈。比如返回"集合不存在，可用的集合有 A、B、C"，而不是 Python 异常堆栈。`❌ 未掌握`

---

## Q7：MCP Tool 的 annotations 字段是做什么的？

**你的回答：** 不知道。

**正确答案：** annotations 的作用是**声明 Tool 的副作用，让 Client 做安全决策**。四个关键属性：①是否只读；②是否有破坏性；③是否幂等；④是否与外部世界交互。这些标注不影响执行逻辑，但 Client 会据此决定是否需要弹窗确认。只读的查询工具自动执行，有破坏性的删除工具让用户二次确认。`❌ 未掌握`

## Q8：MCP Tool 的粒度应该怎么控制？

**你的回答：** 应该是原子的。

**点评：** 方向对但不完整。只说了"原子"，漏了"可组合"。`⚠️ 部分正确`

**正确答案：** 核心原则：**单一职责，工具间可组合**。太粗 → LLM 无法灵活组合；太细 → 调用链路过长容易出错。正确做法是每个工具做一件事，但上游的输出可以直接作为下游的输入参数。举例：RAG 项目拆了三个 Tool（列出集合 → 查看文档详情 → 查询知识库），形成"先发现 → 再了解 → 最后查询"的自然工作流。

## Q9：SKILL 是什么？它和 MCP 的本质区别是什么？

**你的回答：** Skill 是一套复用的工作流程，用自然语言编写，可以内置一些资源。Skill 里面包含了 MCP。

**点评：** 前半部分对，但"Skill 里面包含了 MCP"理解有误。Skill 和 MCP 是互补关系，不是包含关系。`⚠️ 部分正确`

**正确答案：** Skill 是一组打包了指令的文件夹，包含 `SKILL.md`（核心指令）+ 可选的 `scripts/`、`references/`、`assets/`。它教会 AI 如何处理特定的任务和流程。Skill 和 MCP 的本质区别：MCP 本质是**独立运行的程序**（工具箱），Skill 本质是**文档**（带目录的说明书）。用 Anthropic 官方原话：**"MCP connects Claude to data, Skills teach Claude what to do with data"**。Skill 可以编排多个 MCP 的使用流程，但不包含 MCP。

## Q10：SKILL 和传统 Prompt 的核心差别是什么？

**你的回答：** Skill 方便复用，且 Skill 支持两阶段加载。

**点评：** 答了两点，但不够完整。核心差别在于形态和生命周期。`⚠️ 不完整`

**正确答案：** 核心差别在于**形态**和**生命周期**：①**形态**：Prompt 是一次性对话内容，需手动保存复制粘贴；Skill 是工程化包装的文件夹，有结构、可版本管理、易于协作共享。②**加载方式**：Prompt 一次性全部塞进上下文，占用大量 Token；Skill 采用渐进式披露，按需加载，Token 消耗低。③**调用方式**：Prompt 需用户自己记住用哪个；Skill 自动匹配调用。总结：Skill 不是更复杂的 Prompt，而是把复杂的 Prompt 变成了有结构、可复用、可自动调用的模块。

## Q11：SKILL 的运行模式是怎样的？

**你的回答：** 不知道。

**正确答案：** 四步：①**用户发起任务**——Claude Code 把请求连同可用 Skills 列表交给 AI；②**AI 选择 Skill**——自动匹配，不需要用户手动指定；③**按需加载指令层**——选定后才把 SKILL.md 的 body 交给 AI（渐进式披露）；④**按需加载资源层**——需要时才读取 references/assets 喂给 AI，完成任务输出。关键：指令层和资源层都是按需加载的。`❌ 未掌握`

## Q12：SKILL 有哪些典型应用范式？

**你的回答：** 不知道。

**正确答案：** 五种范式：①**顺序工作流编排**——多步骤按指定顺序执行；②**多 MCP 协作**——任务流包含多个 MCP Server；③**迭代优化**——通过多次迭代提升输出质量；④**上下文感知工具选择**——根据场景动态选择不同工具；⑤**领域专业智能**——使用特定领域知识完成任务。`❌ 未掌握`

## Q13：SKILL 的测试和评估可以从哪几个角度进行？

**你的回答：** 不知道。

**正确答案：** 三个角度：①**触发测试**——保证在正确时机被加载，不误触发也不漏触发；②**功能测试**——输出符合预期、API 正确调用、错误被处理、边缘情况被覆盖；③**性能测试**——对比使用前后，用户介入次数、API 调用失败次数、Token 消耗是否降低。`❌ 未掌握`

## Q14：SKILL 的设计理念是什么？解决了哪些行业痛点？

**你的回答：** 方便可复用，把专家写的提示词给更多人使用，Skill 本质是能力，类似 workflow。

**点评：** 方向对但不够深入。需要结合行业痛点来讲。`⚠️ 不完整`

**正确答案：** 三个行业痛点：①模型常识弱，业务场景频繁踩雷，Agent 需要质检返工；②重复造轮子，同一业务规则在多个 Agent 里各写一遍；③Prompt 膨胀，Token 太大，模型不能精准执行，成本上升。SKILL 的设计理念：从"堆砌提示词迭代调试做 Agent"变成做**可治理、可迭代、可复用的产品**，通过 Skills 把开发模式变成写**可复用、可管理的技能资产（Reusable skills）**，结合通用内核解决痛点。

## Q15：SKILL 和 MCP 的关系是什么？

**你的回答：** 互补。

**点评：** 对，但只答了一个词，需要展开。`⚠️ 不完整`

**正确答案：** 二者是**互补关系**，不是替代关系。区别：MCP 侧重工具调用（本质是程序），Skill 侧重提示词（本质是文档）；MCP Token 消耗高，Skill Token 消耗低（渐进式披露）；MCP 开发难度高，Skill 开发难度低。最佳实践：在 MCP Server 文档中附上 SKILL 的链接，让模型调用 MCP 时取得更稳定的结果；多个 MCP Server 可用 SKILL 编排工作流。MCP 提供数据和工具，SKILL 教模型怎么用。

## 建议下一步

1. **重点背诵 MCP 传输协议**：本地 STDIO + 远程无状态 HTTP（GET/POST），SSE 已被替代，这是面试常考且容易被挑战的点
2. **掌握 MCP Tool 设计六原则**：入参（参数最少化/类型严格/约束内联）、出参（结构化+纯文本兼容）、错误（双层+LLM 友好）、annotations（声明副作用）、粒度（单一职责+可组合）、安全（输入校验+输出脱敏）
3. **熟记 SKILL 运行四步**：用户发起 → AI 选择 → 按需加载指令层 → 按需加载资源层
4. **掌握 SKILL 五种范式**：顺序编排、多 MCP 协作、迭代优化、上下文感知选择、领域专业智能
5. **把 SKILL vs MCP vs Prompt 的三角对比记熟**：本质、Token 消耗、开发难度、适用场景
