## Q1：AutoDream 触发条件和执行阶段

**你的回答：** 触发条件说了两个（>24h, >5 个会话），第三个没答出来。执行阶段完全不知道。

**追问过程：**
- Q: 第三个触发条件？→ 不知道
- Q: 执行阶段分几步？→ 不知道

**点评：** 触发条件 2/3，执行阶段 0/4。上次面试正确答案里写过四阶段，说明没有回去复习。

**正确答案：**

三个触发门（全部通过才执行）：
1. 距上次整合 > 24 小时（最先检查，成本最低——一次 stat 调用）
2. 距上次整合 > 5 个会话
3. 获取 consolidation lock（PID 文件锁，防并发；超过 1 小时且 PID 已死则可回收）

四个执行阶段：
1. **Orient**：读取 MEMORY.md 索引，浏览现有主题文件，了解当前状态
2. **Gather**：扫描每日日志和会话记录（JSONL），用 grep 范围搜索，不穷举
3. **Consolidate**：新信息合并到现有主题文件，修复矛盾，相对日期转绝对日期
4. **Prune**：保持 MEMORY.md < 200 行，删除过时条目，精简冗余描述

整合期间只读代码、只写记忆目录；UI 底栏显示 "dreaming" 标签。

`⚠️ 方向对但细节严重不足`

## Q2：第四层 AI 分类器完整机制

**你的回答：** 快速路径提到零副作用工具白名单。两阶段分类器：Stage 1 保守 64 token，Stage 2 是第二次机会 4096 token。

**追问过程：**
- Q: 第二条快速路径？→ 不知道
- Q: Stage 2 三种结果 + fail closed？→ 不知道

**点评：** 两阶段核心逻辑理解正确。漏了第二条快速路径和 Fail Closed 原则。

**正确答案：**

快速路径（两条）：
1. 安全工具白名单：Read/Grep/Glob/Sleep/TodoWrite 等零副作用工具 → 直接放行
2. 工作目录内编辑：项目目录内编辑文件（模拟 acceptEdits 行为）→ 直接放行

两阶段分类器：
- Stage 1：64 token，保守（"Err on the side of blocking"）→ 安全直接放行，危险/不确定进入 Stage 2
- Stage 2：4096 token，要求 AI 写推理链再给结论 → 安全放行，危险阻止，不确定→阻止

**Fail Closed（默认拒绝）**：贯穿整个管道——不确定就拒绝。分类器解析失败→阻止，API 不可用→退回用户确认，Stage 2 无法判断→阻止。

`⚠️ 框架正确但缺第二条快速路径和 Fail Closed`

## Q4：CLAUDE.md 加载顺序和 recency bias

**你的回答：** 加载顺序 outer→inner，项目级排最后。模型倾向听项目级。原理解释有误（说成自回归特性）。

**追问过程：**
- Q: 专门术语？→ 不知道
- Q: 四个层级？→ 不知道

**点评：** 结论正确但原理说错，四层级不掌握。

**正确答案：**

四个层级（排在越后面模型越关注）：
1. 管理员全局：`/etc/claude-code/CLAUDE.md`
2. 用户全局：`~/.claude/CLAUDE.md`
3. 项目级：`CLAUDE.md`、`.claude/CLAUDE.md`、`.claude/rules/*.md`
4. 本地私有：`CLAUDE.local.md`

原理：**recency bias（近因偏差）**——模型对 prompt 末尾的内容天然给予更高注意力。源码注释："Files are loaded in reverse order of priority, ie. the latest files are highest priority with the model paying more attention to them."

所有层级拼接注入，不会互相覆盖。支持 `include` 指令，上限 40,000 字符。

`⚠️ 结论正确但原理有误，四层级未掌握`

## Q5：为什么不存代码事实

**你的回答：** 代码事实可以通过读文件得到，代码随时会变。人的偏好和判断一般不会变。

**追问过程：**
- Q: 存了代码事实会怎样？→ 过时记忆成为干扰，导致幻觉或影响决策 ✓

**点评：** 准确抓住了核心设计理由。

**正确答案：**

代码会变但记忆不会自动更新，过时的代码记忆会变成**危险的误导**。代码相关的事实永远用 grep/glob 实时搜索，只有人的偏好和判断才值得持久化。

`✅ 优秀`

## Q7：第一层 Tool Result Budget

**你的回答：** 工具结果太大，用占位符替换减少上下文消耗。完整内容存磁盘，模型可通过文件路径读取。渐进式披露：模型需要时替换占位符获取原文。

**追问过程：**
- Q: 占位符多大？完整内容存哪里？→ 约 100 字符前缀（有误，约 2KB），存磁盘 ✓

**点评：** 核心理解正确。

**正确答案：**

触发条件：单条消息工具结果累计超过 ~200K 字符预算。把最大的工具结果存到磁盘，原位替换成约 **2KB 的预览 + 文件路径**。模型后续需要完整内容时，用 Read 工具去读磁盘上的完整文件。本质是**渐进式披露**——先给模型足够判断的预览，让它自己决定要不要加载全量。

`✅ 基本正确`

## Q8：Microcompact cache_edits 机制（第三次考）

**你的回答：** 第一种（缓存过期直接改）正确。第二种不清楚。

**追问过程：**
- Q: 缓存有效时怎么办？→ 不知道

**点评：** 第三次仍未掌握。

**正确答案：**

两种策略（按优先级）：
1. **缓存已过期（>60min）**：直接改客户端消息，替换为 `[Old tool result content cleared]`。缓存没了，改了没损失。
2. **缓存仍有效（≤60min）**：客户端消息**完全不改**，额外发一条 `cache_edits` 指令，让服务端在缓存副本里删。**前缀不变，缓存继续有效，省钱。** 本地消息数组保持完整（用于持久化、resume），但模型看到的上下文被缩减。

```
传统做法：改客户端消息 → 前缀变了 → 缓存失效 → 贵
cache_edits：不改客户端消息 → 前缀不变 → 缓存有效 → 省
```

本质：**协议级操作**——在不破坏缓存的前提下，让服务端"假装"旧内容不存在。

`⚠️ 第三次仍未掌握`

## Q9：Auto-compact 轻量 vs 重量

**你的回答：** 轻量不需要额外 LLM 调用，因为之前已经由 Fork Agent 按 token 增量生成了 Session Memory。全量压缩是把全部上下文做全量摘要。

**点评：** 理解正确。

**正确答案：**

轻量（Session Memory Compact）：用已有的 Session Memory 文件做摘要，保留最近 10K~40K tokens 原始对话（至少 5 条含文本消息）。不需要额外 LLM 调用，很便宜。

重量（Full LLM Compact）：把整段对话发给 LLM 做结构化摘要，输出上限 20K tokens，包含 9 个固定段落（核心请求、关键概念、文件列表、错误修复、问题解决、用户消息、待办事项、当前工作、下一步计划）。

`✅ 基本正确`

## Q10：瑞士奶酪模型和 Fail Closed

**你的回答：** 瑞士奶酪：每层都可能有问题（没说完）。Fail Closed：如果权限管道认为不行就问用户（部分正确）。

**追问过程：**
- Q: 叠在一起效果？→ 不知道
- Q: Fail Closed 含义？→ 部分正确

**点评：** 两个概念都只说了一半。

**正确答案：**

**瑞士奶酪模型**：不依赖单一安全机制，叠加多层防御，每层都可能有漏洞，但叠在一起时一层的孔被另一层覆盖——漏洞被叠加消除。

**Fail Closed（默认拒绝）**：不确定就拒绝，不是"问用户"。贯穿整个管道：
- 分类器解析失败 → 阻止
- API 不可用 → 退回用户确认（不是放行）
- dontAsk 无规则覆盖 → 拒绝
- Stage 2 无法判断 → 阻止

`⚠️ 两个概念都只说了一半`

## Q11：bypass 和 dontAsk 的常见误解

**你的回答：** 知道不是"跳过一切"和"全部允许"，但具体机制记不清。

**追问过程：**
- Q: bypass 跳过什么？跳不过什么？→ 记不清
- Q: dontAsk 语义？→ 记不清

**点评：** 记不清具体机制。

**正确答案：**

**bypass 模式**：排在第三层。跳过的是**本层的确认弹框**。跳不过的：第一层 deny 规则（不可覆盖）+ 第二层 bypass-immune（`.git/`、`.claude/` 等关键目录）。**bypass 不是无敌模式**。

**dontAsk 模式**：不是"全部允许"，是**"全部拒绝"**。语义："不要问我，直接拒绝所有没有被规则明确放行的操作"。适合无人值守场景。

`⚠️ 记不清`

## Q14：extractMemories 两轮模式和权限约束

**你的回答：** 会查所有记忆和上下文比对，看有没有新增或需要完善的。权限：只能操作记忆目录，不能调 MCP。

**追问过程：**
- Q: 两轮模式？→ 不知道
- Q: 核心原因？→ 不知道

**点评：** 功能描述对了，执行机制和权限原因未掌握。

**正确答案：**

两轮模式：
- **第一轮**：并行读取所有需要更新的文件
- **第二轮**：并行写入所有变更

权限：只能读文件 + 只写记忆目录。不能执行 bash、不能调 MCP、不能启动子代理。

核心原因：**它是后台自主运行的，没有 human-in-the-loop**。主 Agent 执行 bash 时用户能看到、能审批，但 extractMemories 在后台静默执行，用户看不到中间步骤。如果能执行 bash，出了问题用户来不及阻止。CC 安全设计核心是"人在回路中"，后台 Agent 绕过了这个机制，必须用沙箱物理约束。

`⚠️ 功能描述对，执行机制和权限原因未掌握`

## Q15：子代理不能再派生子代理的机制

**你的回答：** CC 通过不给子 Agent 调用子 Agent 的工具实现。OpenClaw 通过参数限制（子 Agent 数量、深度）。

**追问过程：**
- Q: OpenClaw 叶子节点和 CC 的共同点？→ 不知道

**点评：** 框架正确，共同点未掌握。

**正确答案：**

CC：子代理启动时工具列表**不包含 Agent 工具**，模型自然无法派生。

OpenClaw：用 `maxSpawnDepth` + `maxChildrenPerAgent` 限制。但**叶子节点也物理移除 spawn 工具**。

共同点：两者最终都用**物理移除工具**的手段。设计意图：防止递归失控，成本和复杂度指数级爆炸。

`✅ 框架正确，共同点未掌握`
