## Q1：Microcompact 的 cache_edits 机制

**你的回答：** Microcompact 保留最近 5 轮工具结果，cache 命中时客户端上下文不变、服务端过滤。不知道 `cache_read_input_tokens` 如何反推 breakpoint 位置。

**追问过程：**
- Q: cache_edits 怎么判断哪些消息对应 cache breakpoint → A: 从工具结果位置得来 → 方向对但太模糊
- Q: `cache_read_input_tokens` 告诉你什么？怎么反推 breakpoint 精确位置 → A: 不知道 → 知识边界在此

**点评：** 你理解了 cache_edits 的"效果"（客户端不变、服务端过滤），但不理解"实现机制"。核心缺失是：不知道 API response 中的 cache token 数如何映射到 context 中的具体位置。

**正确/完整答案：** API response 中的 `cache_read_input_tokens` 告诉你前 N 个 token 被缓存。Microcompact 从 context 头部按 token 计数到 N 的位置就是 breakpoint。Breakpoint 之前的内容保持原样（保护 cache），之后的内容可以激进压缩。

`⚠️ 方向对但细节不足`

## Q5：Uncorrelated context windows

**你的回答：** 子代理看不到主代理对话历史，主代理看不到子代理推理过程。关键信息会体现在子代理执行结果中。安全（无权限）和效率（不稀释注意力）两个维度原因分析到位。

**追问过程：**
- Q: context 隔离的含义 + 原因 → A: 正确 → ✅
- Q: 隔离是否是问题？ → A: 不会，任务描述不清才是问题 → 偏理想化但核心观点成立

**点评：** 比上次有明显进步。安全和效率两个维度都答对了。对"隔离是否是问题"的回答偏理想化（忽略了重复探索的 tradeoff），但核心观点成立。

**正确/完整答案：** Uncorrelated context windows 的三个原因：1）安全——子代理不继承权限上下文；2）效率——注意力集中在任务上；3）简洁性——实现简单。已知 tradeoff：子代理可能重复探索主代理已知的信息。CC 通过 task prompt 写入足够上下文来缓解。

`✅ 基本正确`

## Q13：Session Memory vs AutoDream

**你的回答：** Session Memory 触发于 token 阈值（auto-compact），存储在 JSONL 文件，会话结束后仍存在但新会话不自动加载（除非 resume）。AutoDream 触发于时间和对话次数，修改 memory 目录文件。两者都 fork agent 执行。

**追问过程：**
- Q: Session Memory 会话结束后还存在吗？新会话加载吗？ → A: 存在，新会话不自动加载，除非 resume → ✅

**点评：** 核心区别理解到位——Session Memory 是会话级，AutoDream 是跨会话级。

**正确/完整答案：** Session Memory：触发于 context 达 token 阈值 → fork agent 生成结构化笔记 → 存储在 JSONL → 用于压缩 context 和恢复会话 → 新会话不自动加载。AutoDream：触发于对话结束/空闲/token 阈值 → fork agent 整合记忆 → 修改 memory 目录文件 → 跨会话持久存在 → 每次新会话自动加载。

`✅ 基本正确`

## Q14：WorkTree agent 的创建机制

**你的回答：** Fork 是同进程异步，Teammate 是多进程并行，WorkTree 是 git worktree 隔离。WorkTree 修改文件主 agent 看不到（不同工作目录）。可自动 merge。三模式正交，可组合。

**追问过程：**
- Q: WorkTree 完成后怎么合并？三种模式互斥吗？ → A: 自动 merge，正交可组合 → ✅

**点评：** 三种模式的区别和正交关系都答对了，比上次有明显进步。

**正确/完整答案：** Fork（同进程、共享文件系统、同分支）→ Teammate（不同进程、共享文件系统、同分支）→ WorkTree（不同进程、git worktree 隔离、不同分支）。正交关系：Fork vs WorkTree 是文件系统隔离维度，Teammate 是并行执行维度，可自由组合。

`✅ 基本正确`
