# Harness Engineering 面试 Q&A 记录 — 2026-05-08

## Q1：什么是 Harness Engineering？核心要解决什么问题？三个病理分别是什么？各举一个例子。

**你的回答：** Harness Engineering 是围绕大模型构建运行环境与管控基础设施的工程，核心要解决 Agent 在长时间、复杂、多步骤任务中稳定可靠运行的问题。三个病理：①上下文腐化——工具结果返回大量垃圾信息占据上下文；②幻觉完成——Agent 在长链推理中跳步骤并伪造输出，比如第三步之后直接说文件已生成但实际没有；③模型漂移——比如装 OpenCloud，先装 A 失败、转而装 B 失败、再试 C，逐渐偏离最初目标。

**点评：** 定义准确，三个病理都答到了。幻觉完成和模型漂移的例子举得真实具体，来自你的实际使用经验，很有说服力。小瑕疵：Context Rot 的例子只说"工具结果太大"，漏了更典型的情况——长对话中错误尝试、修正、绕路的历史堆积把原始指令挤出上下文窗口。`✅ 优秀`

## Q2：Prompt Engineering、Context Engineering、Harness Engineering 三者是什么关系？替代还是叠加？

**你的回答：** 三者是叠加/递进关系。Prompt Engineering 解决"对 AI 说什么"；Context Engineering 解决"让 AI 知道什么"；Harness Engineering 解决"让 AI 在什么环境里做事"。Harness 内部的每次 session 仍需 Context Engineering，每次与模型交互仍需 Prompt Engineering。

**点评：** 叠加关系判断准确，三层递进的逻辑说清楚了，也点出了关键——Harness 不是替代前两者而是把它们包进更大的系统。`✅ 优秀`

## Q3：Harness Engineering 的核心实践原则有哪些？挑 3-5 个最重要的说。

**你的回答（初始）：** 不知道。

**提示后补充：** ①状态外部化——用文件系统/Git 持久化记忆；②确定性验证轨道——Linter/类型检查/单元测试做门禁；③原子任务 + 上下文刷新——每个 Agent 只做一个小任务就销毁；④Skill 按需加载；⑤Human-in-the-loop 关键阶段人工审查。

**追问"原子任务和确定性验证是什么关系"：** 原子任务让上下文注意力集中，也方便做确定性验证，因为任务范围小。

**点评：** 提示后能罗列原则，但初始不会独立展开。追问时说了前半句（原子任务方便验证），没说到后半句——确定性验证让原子任务的交接可信：验证通过后才更新状态，下一轮 Agent 接手时不需怀疑上一步是否真的完成。两个原则互相支撑。`⚠️ 部分正确`

**正确答案补充：**
- 两个原则的关系：原子任务缩小验证范围 → 验证通过率提升；验证通过保证任务真正 closed → 交接可信。缺一不可。
- 完整十条原则：外部持久记忆、确定性验证轨道、原子任务+上下文刷新、子 Agent 集群、技能文件、Guard Rails+Checkpoints、Handoffs、Human-in-the-loop、架构约束、垃圾回收 Agent

## Q4：Anthropic Harness 架构的核心循环是什么样的？从 PRD 到项目完成，完整描述。

**你的回答（初始）：** 不知道。

**提示 Initializer 职责后：** 将需求拆分成一堆原子任务。

**追问 Coding Loop 后：** 不知道，要求面试官讲一遍。

**面试官讲解后追问"为什么上下文被丢弃是 feature 不是 bug"：** 因为任务是原子的，做完就销毁 Agent 丢掉上下文，不影响下一个原子任务。

**再追问"如果上一个 Agent 上下文里有踩坑经验，丢掉不可惜吗"：** 有长期记忆，上一个 Agent 会把经验教训存到本地，下一个 Agent 启动时会读取。

**点评：** 架构细节记忆薄弱——Initializer 只说了拆任务，漏了"定义验证标准"和"初始化项目骨架/代码仓库"。Coding Loop 六步生命周期（读取状态→选任务→实现→验证→交接→销毁）完全无法独立描述。不过"上下文丢弃是 feature"的追问中理解到了本质，Handoff 机制也说出了概念（虽然术语"长期记忆"不够精确）。`❌ 未掌握`

**正确答案补充：**
- **Initializer 三件事：** 读需求 → 拆 feature list + 定义验证标准 → 初始化项目骨架和代码仓库
- **Coding Loop 六步：** ①创建全新 Agent（干净上下文）；②读取外部状态文件（feature_list.json、progress.md、代码库）；③选一个明确的原子任务；④只实现这一个任务；⑤确定性验证（测试/lint/类型检查）；⑥验证通过→更新状态文件+写 Handoff + git commit→Agent 退出销毁；下一轮新 Agent 从①开始
- **为什么上下文丢弃是 feature：** 项目记忆存在外部状态文件，不在任何 Agent 上下文窗口。上下文是易失 RAM，状态文件是持久存储。Handoff（交接文件）传递经验教训，不依赖上下文窗口

## Q5：ClawCode 与 Anthropic Harness 架构有什么异同？分别适用于什么场景？

**你的回答（初始）：** 不清楚。

**提示后：** Anthropic 从需求文档开始，ClawCode 多了一步分析源码的过程。

**追问"为什么 ClawCode 要建 Python 中间层"：** 不知道。

**点评：** 在提示下能说出最核心差异——起点不同。但不能独立展开，对 ClawCode 五阶段流水线完全不了解，Python 中间层的作用（JSON 是"有什么"，Python 中间层是"怎么运转的"）也不知道。`❌ 未掌握`

**正确答案补充：**

| | **Anthropic Harness** | **ClawCode** |
|---|---|---|
| 场景 | 从零写新项目 | 重写已有 51 万行 TS → Rust |
| 起点 | PRD/需求文档 | 已有庞大代码库 |
| Initializer | 读需求 → 拆 feature | 扫描源码 → JSON 清单 + Python 中间层 |
| 验证方式 | 测试/lint/类型检查 | Mock API 行为对等性测试 |
| 编排 | 串行 Coding Loop | 9-lane 并行 + OmX/clawhip/OmO 三工具 |
| 核心 insight | Agent 用完即销毁避免上下文腐化 | 建中间层降低 Agent 认知成本 |
| 相同原则 | 状态外部化、任务原子化、确定性验证、人只做关键决策 | |

- **ClawCode 五阶段：** ①归档+快照（TS→JSON 清单）；②Python 镜像工作区（1:1 骨架+架构模型）；③Parity Tracking（量化覆盖率）；④Mock Parity Harness（确定性端到端测试）；⑤多 Agent 并行执行
- **Python 中间层作用：** JSON 只有静态结构（"有什么文件"），Python 中间层用 5 个可运行文件建模动态行为（启动流程、命令路由、查询引擎 turn-loop），Agent 先查这个轻量沙盘就能快速定位，不需在 51 万行代码里大海捞针

## Q6：如果让你用 Harness 思想从零设计一个小项目（如命令行 TODO 工具），完整流程怎么走？

**你的回答（初始）：** 先做项目分析得到 PRD → 解构成原子任务 → 执行时做好验证。

**提示补充后：** ①用 JSONL 存储任务信息和状态；②Initializer 拆任务时同时定义验证方法；③无关任务并发执行，可用 git worktree 隔离，有依赖的任务顺序执行；④全程验证通过人不需要介入，但很吃验证准确性。

**追问"相关性怎么判断，谁来定义"：** Initializer 可以判断，但生产中人在初始拆解时必须介入检查，保证后续流程稳定性。

**点评：** 初始回答框架对但太简略。补充后方案完整——JSONL 存状态、Initializer 同时定义验证、并发用 worktree 是亮点。最后承认"全自动很吃验证准确性"和人必须审 Initializer 输出，务实判断比背答案更有说服力。`✅ 良好`

## Q7：Harness Engineering 目前有哪些局限性？是技术问题还是方法本身固有？

**你的回答（初始）：** 不知道。

**提示后：** ①Initializer 是单点高杠杆操作——拆错了后面全错；②Handoff 质量不稳定——有时总结得好有时漏关键问题；③单步 99%，多步后整体可靠性累积衰减；④生产角度看还是需要人介入。

**点评：** 提示后四个局限都覆盖到了，累积衰减的量化理解很好。但没有展开回答是技术问题还是方法固有的判断。`⚠️ 基本正确（需提示）`

**正确答案补充：**
- 四个局限性本质上是**方法本身的固有挑战**，技术可以缓解但无法根除——Initializer 拆解质量有上限（AI 理解需求总有歧义），多步累积衰减是数学规律（0.99^n），Handoff 质量波动源于自然语言传递信��的固有限制
- Anthropic 官方也承认这些局限，目前的结论是：高度自动化，但不完全放手，关键节点仍需 Human-in-the-Loop
