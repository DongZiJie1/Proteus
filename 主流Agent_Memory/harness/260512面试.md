## Q3：Harness Engineering vs 传统 CI/CD

**你的回答：** "前者的核心是 LLM，后者的核心是人写的代码，CI/CD 并不会随着人写的代码迭代而迭代，而 Harness 会。"

- Q: CI/CD 挂了 vs Harness 失败，修的对象有何不同？→ A: "CI/CD 修代码，Harness 优化自身" → **方向对但笼统**
- Q: 具体调 Harness 什么？三个例子 → A: 改提示词 / 改工具 / 改 loop 参数 → **三个维度都对，具体可操作**
- 知识边界：初始表述模糊，追问后追回精确表述

**点评：** 初始回答太模糊，追问后补上了"确定性流水线跑确定性代码 vs 确定性约束跑不确定 Agent"这个精确表述（从上一轮面试学到的）。三个具体调整例子答得实用。但第一反应没有命中核心表述，说明还没内化成直觉。

## Q6：原子任务与确定性验证的关系

**你的回答：** "原子任务执行结束之后，会去验证结果。原子任务缩小验证范围，验证逻辑保证 Agent 正确执行判断。"

**追问过程：**

- Q: 没验证就交接会怎样？→ A: "在屎山上雕花" → **直觉对但表述不精确**
- 知识边界：方向一说了一半，方向二（验证保证交接可信）完全没理解

**点评：** 第三次考这道题。方向一"原子任务缩小验证范围"记住了，但方向二"验证让交接可信"仍然没理解，甚至反问"交接验证是什么意思"——说明对 Handoff/交接的机制理解仍有空白。

**正确/完整答案：** 不是先后关系，是互相依赖。原子任务→验证：任务小→验证精准，定位成本低。验证→原子任务：验证通过=任务真正完成（不是 Agent 自述），下一个 Agent 信任上一步结果不需重新检查。没有验证，原子任务的交接就是空中楼阁——效率优势全部被抵消。两个原则缺一个另一个就失效。

**评级：** `⚠️ 方向对但细节不足`
