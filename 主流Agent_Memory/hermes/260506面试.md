## Q1：Hermes 总体架构是几层？和 OpenClaw 的核心差异在哪？

**你的回答：** 三层，触发层→网关层→Agent 层，核心差异在 Agent 层多了记忆自反馈，可以自己进化升级成 skill。

**点评：** 漏了"1 个闭环"——标准答法是"主干 3 层 + 1 个自进化闭环"。自进化循环是三路沉淀（memory / skill / Honcho），Skill 只是其中一路，不是全部。`⚠️ 薄弱`

**正确答案：** 标准答法是**"主干 3 层 + 1 个自进化闭环"**。主干三层：触发层（Message/Heartbeat/Cron/Webhook/Hooks）→ 网关层（Gateway，连接管理+协议转换+路由分发+安全控制）→ Agent 层（Think→Act→Remember 循环）。1 个闭环：Agent 层额外挂的自进化回路（Reflect → 记忆/Skill 沉淀 → 反哺下一次触发），每 ~10 次 tool call 触发，分三路沉淀——事实性知识 → memory/、程序性流程 → skills/、用户偏好 → Honcho。和 OpenClaw 的核心差异就在这个闭环上。

## Q3：压缩时被压掉的旧消息是不是永久丢失了？

**你的回答：** 不会消失，会保存在 Session Transcript 中，压缩后的内容也保存在这里。

**点评：** 方向对但不精确。关键机制是原始消息在发生时已实时 append 到 `.jsonl`，压缩不删任何内容，只是在 `.jsonl` 中追加一条 `compaction` 摘要条目 + 设 `firstKeptEntryId` 标记，下次从该标记之后开始读。压缩前数据在磁盘上一行没少。`⚠️ 部分正确`

**正确答案：** 不会永久丢失。Session Transcript 的存储分两部分：`sessions.json`（索引表，存所有 session 的元信息）+ `<sessionId>.jsonl`（对话正文，每条消息/工具调用/压缩摘要占一行 JSON，按时间顺序 append）。压缩时的精确机制：①原始消息在发生时已实时 append 到 `.jsonl`，压缩不会删任何已有行；②压缩只是在 `.jsonl` 中**追加**一条 `compaction` 类型条目（包含摘要文本），同时设置 `firstKeptEntryId` 标记；③下次加载会话时从 `firstKeptEntryId` 之后开始读，旧消息虽然不在上下文里但磁盘上一行没少。

## Q5：网关层（Gateway）做什么？调不调 LLM？

**你的回答：** 不会调 LLM，会接收所有外部事件，由它统一分发给对应 Agent。

**点评：** 答出"不调 LLM"和"路由分发"两点，但漏了连接管理和协议转换。Gateway 做 4 件事：连接管理 + 协议转换 + 路由分发 + 安全控制。一句话：只回答"消息该给哪个 Agent？""回复该送回哪个渠道？" `⚠️ 不完整`

**正确答案：** Gateway 不调 LLM，做 4 件事：①**连接管理**——同时维持多个渠道的长连接（CLI、Telegram、Discord、Slack、WhatsApp、Email 等，一进程多渠道）；②**协议转换**——把不同平台协议（Telegram JSON、Discord WebSocket 帧、Email MIME、Webhook HTTP POST）统一转换成内部标准事件格式；③**路由分发**——根据事件里的 agentId/sessionId 投递给正确的 Agent 会话，回复按原路送回对应渠道；④**安全控制**——入口认证（token/签名校验）、限流、沙箱隔离。关键边界：Gateway 只回答两个问题——"这条消息该给哪个 Agent？""Agent 的回复该送回哪个渠道？"

## Q6：Error Classifier 是什么？体现什么工程思想？

**你的回答：** 不知道。

**你的追问：** 分类了然后呢？

**正确答案：** 把常见失败模式（网络错误、权限错误、文件不存在、rate limit 等）预编码成分类器。失败时直接归类走预设恢复路径，不让弱模型硬猜。体现 **harness engineering**——用系统层预处理降低对模型推理能力的依赖。`❌ 未掌握`

**举例——有无 Error Classifier 的区别：**

> 场景：Agent 调用 `shell_exec("rm /tmp/cache")`，返回 `Permission denied`。

❌ **没有 Error Classifier（弱模型硬猜）**：
```
LLM 看到: "Error: Permission denied"
LLM 想:  这是权限不够？还是文件不存在？还是命令拼错了？
LLM 做:  再试一次 → 加 sudo → 又失败 → 换个写法 → 还是失败 → 放弃
```
问题：弱模型把 token 浪费在"猜错误原因"上，猜错了反复重试，对话轮次膨胀、成本浪费。

✅ **有 Error Classifier（Hermes 做法）**：
```
1. Error Classifier 捕获 "Permission denied" → 匹配到预设类别: PERMISSION_ERROR
2. 预设恢复路径: "告知用户权限不足，建议检查文件所有权或使用 sudo"
3. 返回给 LLM 的不是原始错误，而是已被归类的结构化信息:
   {type: PERMISSION_ERROR, recoverable: true, suggestion: "请检查文件权限或尝试 sudo"}
4. LLM 直接按 suggestion 回复用户，不再自己猜
```
效果：弱模型不用推理错误原因，走预设路径即可。5 次重试 → 1 次搞定，token 省了、速度快了。

## Q7：自进化循环每多少次 tool call 触发？自问哪三件事？

**你的回答：** 约 10 次。事实性记忆需不需要修改/新增；从会话能不能固化出程序性流程；需不需要更新用户画像。

**点评：** 三件事里漏了最关键的一个——"有没有失败的步骤需要记下来避免？"。你提到的用户画像是三路沉淀中的一路（Honcho），不是第 4 步自问的内容。`⚠️ 部分正确`

**正确答案：** 触发频率约每 10 次 tool call。自问**三件事**：①这次学到的东西值不值得保留？②有没有失败的步骤需要记下来避免？③成功的流程能不能抽象成可复用 skill？回答"是"之后，分**三路沉淀**：事实性知识 → `memory/YYYY-MM-DD.md`（更新 Retrieval Index）；程序性流程 → `skills/xxx/SKILL.md`（更新 Skill Library）；用户偏好 → Honcho User Model（更新用户画像）。注意：自问的三件事和三路沉淀是两步——先自问三件事判断"值不值留"，再按类别分三路沉淀。用户画像（Honcho）是沉淀的目标之一，不在自问三件事里。

## Q8：主模型 + 辅助模型分工，辅助模型干什么？

**你的回答：** 辅助模型做总结、摘要等轻活，主模型推理。

**点评：** 辅助模型实际做三件事：压缩/摘要、错误归类、轻量分类。好处是在成本、速度、质量之间取得平衡，按任务难度分层匹配模型。`⚠️ 不完整`

**正确答案：** 主模型做长链推理、规划、复杂任务（用强模型如 Claude Opus/GPT-5）；辅助模型（Auxiliary Client）做三件轻活：①**压缩/摘要**——把对话历史压缩成摘要，没必要让 Opus 来干；②**错误归类**——配合 Error Classifier 把工具失败归类到已知模式走预设恢复路径；③**轻量分类**——如判断任务类型、意图识别等。本质是 harness engineering 思路：按任务难度分层匹配模型，在成本、速度、质量之间取得平衡。

## Q9：压缩时必须保留的内容有哪些？举 3 个以上。

**你的回答：** 运行中的任务状态、TODO、已作出的决策及理由。

**点评：** 答了 3 个，合格。但实际有 7 类，常被忽略的是：**不透明标识符**（UUID/hash/token/API key/URL/文件名，硬性保护）、用户最后请求 + 正在做的事、承诺的后续行动。`✅ 基本正确`

**完整 7 项：** ①进行中的任务及其状态（进行中/阻塞/待办）；②批量操作进度（如"17 项中已完成 5 项"）；③用户最后请求 + 正在做的事；④已做出的决策及理由；⑤TODO、开放问题、约束条件；⑥承诺的后续行动；⑦所有**不透明标识符**（UUID、hash、token、API key、URL、文件名）——由 `identifierPolicy: "strict"` 硬性保护，绝对不能丢。
