# Hermes 面试 Q&A 记录 — 2026-05-07

## Q1：Hermes 的记忆体系是"5 层 + 1 个用户画像"，这 6 块分别是什么？按"陈述性记忆/程序性记忆/用户建模"三类归类。

**你的回答：** Bootstrap Files（永久身份与规则）、Session Transcript（完整对话历史存档）、Context Window（上下文窗口）、Retrieval Index（记忆文件的全文索引）、Skill Library、Honcho User Model（独立用户画像）。前 4 个属于陈述性记忆，Skill 属于程序性记忆，Honcho 属于用户建模。

**点评：** 6 块分类正确、三大归类也正确。小补充：Context Window 严格说不是"记忆层"，是记忆的消费端——它是模型 RAM 中的瞬态容器，每轮拼装、满了就压缩，面试时可以多提一句这个定性。`✅ 优秀`

## Q2：Hermes 的压缩流程分几步？压缩后上下文结构是什么？压缩前数据在磁盘上有没有被删除？

**你的回答：** 分 4 步：①划分保留区和压缩区，最近 ~20K tokens 进保留区原封不动，之前旧消息进压缩区；②分块总结，压缩区分多块，用模型对每块生成摘要；③合并摘要，多个摘要合并成连贯总摘要；④写入磁盘，在 JSONL 中写入一条 compaction 类型的条目。必须保留的内容包括：进行中的任务及其状态、用户最后请求和正在做的事、已做出的决策及理由、TODO、承诺的后续行动等。

**点评：** 前三步和保留项答得不错。但漏了三点：①压缩后上下文结构没直接说——是 `[压缩摘要] + [保留区原文]`；②磁盘数据是否被删除——没有删除，压缩只是在 `.jsonl` 中追加一条 compaction 条目 + 设 `firstKeptEntryId` 标记，原始消息在磁盘上一行没少，下次加载从标记之后开始读；③保留项还漏了批量操作进度（"17 项中已完成 5 项"）和不透明标识符（UUID/hash/token/URL/文件名，`identifierPolicy: "strict"` 硬性保护）。`⚠️ 部分正确`

**正确答案补充：**
- 压缩后上下文结构：`[压缩摘要] + [保留区原文]`
- 磁盘数据：不删。`.jsonl` 追加 compaction 条目 + 设 `firstKeptEntryId`，下次从标记之后开始读
- 完整保留 7 项：①进行中的任务及其状态；②批量操作进度；③用户最后请求+正在做的事；④已做出的决策及理由；⑤TODO、开放问题、约束条件；⑥承诺的后续行动；⑦不透明标识符（UUID/hash/token/API key/URL/文件名）

## Q3：Hermes 的 Pre-Compaction Memory Flush 和 OpenClaw 的有什么不同？

**你的回答：** 预压缩指压缩前把上下文给另外一个模型反思有没有重要内容需要写入记忆，用户无感知，这是 OpenClaw 做法。Hermes 在此基础上多走了一步，不仅提醒 Agent 写入记忆，还会先调用自进化反思判断有没有新 skill 可以沉淀。

**点评：** 核心差异说对了。补两个细节：①触发时机不同——OpenClaw 的 flush 仅在压缩前触发一次，Hermes 的自进化循环每 ~15 次 tool call 就触发，压缩前只是其中一次；②关系是包含——自进化循环 $\supseteq$ Pre-Compaction Flush，Hermes 在压缩前的反思是 flush 的超集，更新范围从 memory 一路扩展到 skill + Honcho 三路。一句话记：OpenClaw 是事后归档（别丢了），Hermes 是持续反思（值不值留+能不能抽象+有没有坑要记住）。`✅ 优秀`

## Q4：Hermes 的子 Agent（spawn_subagent）和父 Agent 之间的通信模式是什么？子 Agent 的中间推理过程父 Agent 能看到吗？

**你的回答：** 不知道。

**正确答案：** 和 CC/OpenClaw 一样：①通信模式是 **Tool Call**（非 Handoff）——父 Agent 把子 Agent 当工具调用，传参数进去、等返回值，自己保留控制权继续决策，子 Agent 完成后不直接回复用户；②消息传递是**仅共享最终结果**——子 Agent 的完整推理链（每步 thinking + tool call）留在自己的私有 session 里，父 Agent 看不到，只收到蒸馏后的最终回复。Hermes 在这里没有创新，完全沿用了业界主流做法。`❌ 未掌握`

## Q5：Hermes 的 Retrieval Index 索引什么内容？聊天里说过但没写进文件的内容能搜到吗？

**你的回答：** 可以索引保存下来的事实性记忆文件以及会话存档记录。聊天里说过但没写进文件的内容可以搜到，因为属于会话记录。

**点评：** 前半句对——`memory/` 下的 md 文件会被索引。但后半句不对：Retrieval Index **只索引文件**（`MEMORY.md` + `memory/**/*.md`），聊天里说过但没写进文件的内容搜不到。Session Transcript（`.jsonl`）和 Retrieval Index 是两码事——Transcript 是对话日志用于续会话，但不是被 `memory_search` 索引的对象。Index 底层是 `memory/<agentId>.sqlite`，数据来源是 md 文件，通过 chokidar 监听文件变化后自动重建。所以如果聊天里说过但 Agent 没把它写进 memory 文件，下次 `memory_search` 搜不到。这也是为什么自进化循环和压缩前 flush 要把重要内容写入 memory 文件——不写进去索引就找不到。`⚠️ 有错误`

## Q6：Hermes 的 Skill 两阶段检索是怎么做的？为什么不用一阶段直接加载？

**你的回答：** 一阶段直接加载会导致所有 skill 全文进入上下文，造成上下文爆炸。很多情况下不需要 skill 或只需要其中一个，所以先通过每个 skill 的简单描述判断需要加载哪个，再去加载对应 skill 的全部内容，能减少大量 token 消耗。

**点评：** 核心逻辑说清楚了——先描述粗排命中、再加载完整内容，和 memory 的 `memory_search → memory_get` 两步走是同一个思路（先轻量匹配、再按需加载）。`✅ 优秀`

---

## Q7：Hermes 的自进化循环和 OpenClaw 的 extractMemories（Flush 到 memory）更新范围有什么不同？触发频率呢？

**你的回答：** 不知道。

**正确答案：**

| | **OpenClaw extractMemories** | **Hermes 自进化循环** |
|---|---|---|
| 更新范围 | 只写 `memory/YYYY-MM-DD.md` | memory + skill + Honcho **三路** |
| 触发频率 | 仅在压缩前触发一次 | 每 ~15 次 tool call 周期性触发 |
| 模式 | 事后归档 | 持续反思 |
| 关注维度 | "别让重要信息被压缩丢了" | "值不值留 + 失败要避免 + 流程能抽象" |

一句话：OpenClaw 是单路、被动、归档式；Hermes 是三路、主动、反思式。`❌ 未掌握`

## Q8：Hermes 的 memory/ 笔记和 Bootstrap Files 在读取策略上有什么本质区别？

**你的回答：** 前者在每次模型推理前系统会主动通过索引去找，找到了就放入上下文，没找到就不管。推理过程中如果模型认为需要找记忆，前者也有可能被找到从而在下次推理时进入上下文。后者一直放在上下文的系统提示词中。

**点评：** 核心区别正确——Bootstrap Files 无条件自动注入，memory/ 笔记按需检索。补两个细节：①memory/ 的两条检索通道说全了（Context Engine 系统自动预检索 + LLM 主动调 `memory_search`），但 LLM 主动检索是在 Think 阶段判断"信息不够"→ Act 阶段调用工具 → 结果回灌下一轮 Think，是 ReAct 循环的一部分，不是"推理之后"再找；②Bootstrap Files 每轮都占空间但免疫 Compaction——因为每次从磁盘重读，不经过 `.jsonl`，压缩压不到它，所以 Bootstrap 文件要尽量精简。`✅ 基本正确`

# 总结

## 擅长领域

- **记忆体系静态结构**：6 块记忆 + 三大分类归类准确，能分清陈述性/程序性/用户建模
- **压缩策略**：4 步压缩流程清晰，知道保留区/压缩区的划分和保留项
- **Skill 两阶段检索**：能说清精排→加载的设计逻辑和上下文防膨胀动机
- **Pre-Compaction Flush 对比**：能抓住 Hermes 多一步自进化反思的核心差异
- **Bootstrap vs memory 读取策略**：能分清自动注入和按需检索的本质区别
