**正确/完整答案：**

按 `prompt_builder.py` + `context_engine.py` 顺序：

| 顺序 | 内容 | 数据源 |

## Q2：Hermes 的 Skill Library 和 `memory/` 笔记的本质区别是什么？从认知心理学角度分别对应哪类记忆？

**你的回答：** Skill Library 是程序性记忆，memory/ 是事实性记忆。

**点评：** 心理学分类正确，两阶段检索和防膨胀到位。漏了系统预置路径。

**正确/完整答案：**

## Q5：Hermes 的 Error Classifier 是什么？体现什么工程思想？举一个具体例子。

**你的回答：** 工具结果的错误分类器，体现 harness 思想——通过系统兜底预设避免大模型不稳定性。例子：调用远程 API 缺 token，有分类器能直接走预设恢复路径。

**追问过程：**
- Q: 这个工程思想叫什么？ → A: Harness → ✅

**点评：** 比 260506 Q6（完全不知道）进步显著。核心理解到位。补一个关键细节：Error Classifier 返回的不是原始错误，而是结构化信息 `{type: PERMISSION_ERROR, recoverable: true, suggestion: "..."}`，LLM 直接按 suggestion 行动。

**正确/完整答案：**

Error Classifier（`agent/error_classifier.py`）：把常见失败模式（网络错误、权限错误、文件不存在、rate limit 等）预编码成分类器。失败 → 归类 → 走预设恢复路径。

体现 **harness engineering**——用系统层面预处理，降低对模型推理能力的依赖。效果：弱模型不用猜错误原因，走预设路径即可，5 次重试变 1 次搞定。

## Q7：Hermes 的 Retrieval Index 索引什么内容？聊天里说过但没写进文件的内容能搜到吗？搜索权重怎么算的？

**你的回答：** 只索引 memory/ 下的文件，聊天内容搜不到。权重 70% 向量 + 30% 关键词。存储在 .sqlite。

**追问过程：**
- Q: 索引存储格式？ → A: 向量数据库 → 提示后承认是 sqlite → ⚠️
- Q: 时间衰减？所有文件都衰减吗？ → A: 不知道 → ❌

**点评：** 比 260507 Q5（以为能搜 transcript）正确了——核心概念纠正了。但时间衰减和常青文件例外不知道。

**正确/完整答案：**

- 索引范围：`MEMORY.md` + `memory/**/*.md`，可通过 `memorySearch.extraPaths` 扩展
- 存储：`~/.hermes/memory/<agentId>.sqlite`
- 搜索权重：`finalScore = 0.7 × vectorScore + 0.3 × textScore`（向量语义 + BM25 关键词）
- 文件监听：chokidar，变化后 1.5 秒内自动重索引
- MMR 去重 + 时间衰减（半衰期 30 天）
- **例外**：常青文件（`MEMORY.md`、非日期命名的 `memory/*.md`）不受时间衰减影响

`✅ 基本正确`

## Q12：线上 Agent 的上下文突然溢出了，你怎么排查？从触发机制、恢复流程、压缩后上下文结构三个层面说。

**你的回答：** 手动 `/compress`，划分保留区/压缩区，压缩后结构为压缩摘要 + 保留原文。参数：20000 / 4000 / 176K。

**追问过程：**
- Q: 阈值公式参数？ → A: 20000 / 4000 / 176K → ✅ 全部正确

**点评：** 排查思路清晰，参数全部正确。

**正确/完整答案：**

- 自动触发公式：`contextWindow - reserveTokensFloor(20000) - softThreshold(4000)`，200K 模型约 176K 时触发

- 压缩过程：①划分保留区（~20K tokens）和压缩区 ②分块总结 ③合并摘要 ④写入 JSONL
- 压缩后结构：`[压缩摘要] + [保留区原文]`
- 磁盘数据：不删，追加 compaction 条目 + 设 `firstKeptEntryId` 标记，下次从标记后开始读

## Q15：如果让你从零设计一个 Agent 的记忆系统，你会怎么做？结合 Hermes 的设计，从静态结构到动态更新策略完整说一遍。

**你的回答：** 去 Honcho，memory 强制分用户偏好/反馈等类 + 绝对时间存相对时间显。动态更新：自进化只看 skill 不看 memory，每 3 次 Agent loop 异步触发；事实性记忆靠用户说"记住"或写记忆工具；小模型做相关性判断+时间转换；定期子 Agent 整理记忆；不提取文件中已有的东西。

**追问过程：**
- Q: 为什么不能提取文件中已有的？ → A: 代码文件随时可能变 → ✅ 洞见好
- Q: 事实性记忆全依赖用户"记住"，如果用户忘了怎么办？ → A: （未答但隐含方案——每 3 轮自动触发写记忆）

**点评：** 有独立设计判断，不是照搬 Hermes。几个好思路：①去 Honcho 减少外部依赖；②记忆分类结构化；③绝对→相对时间转换；④不提取文件已有知识（避免 stale）。可商榷：①"每 3 轮"太激进，Hermes 的 10-15 次阈值是实验平衡点；②去掉 memory 自动沉淀，单靠用户"记住"可能漏重要长期知识。

**参考设计要点：**

静态结构：工作空间文件（身份/规则）+ 对话归档（.jsonl）+ 可检索事实知识（memory/）+ 可复用流程（skills/）+ 索引层（sqlite）

动态更新：
- 写入：实时 append transcript + 会话结束归档 + 周期性反思沉淀
- 读取：自动注入（Bootstrap/Transcript）+ 按需检索（memory/Skill）
- 压缩：分保留区/压缩区 + 保留 7 项关键内容
- 反思：周期性自问 → 三路沉淀

`✅ 优秀`

1. **Gateway 4 件事必须背熟**：这是固定考点，两次面试都没答全。记一句口诀——"连接管理、协议转换、路由分发、安全控制" + 一句话边界"只做路由，不调 LLM"。

2. **自问三件事 vs 三路沉淀画一张图**：连续两次都混淆。三步走：触发（~10次 tool call）→ 自问（值不值留？有没有失败？能不能抽象为 skill？）→ 三路沉淀（memory/skill/Honcho）。把这两个阶段物理分清楚。

3. **整理 Act 层工具表格**：6 种终端后端（local/Docker/SSH/Daytona/Singularity/Modal）+ Error Classifier + 工具列表。从本地到云端画一条线，每个后端说出一条特点即可。

4. **压缩保留 7 项变成肌肉记忆**：口诀——"任务状态、批量进度、最后请求、决策理由、TODO、承诺、标识符"。面试追到这里基本必考，连续两次都没背全。

5. **补齐记忆模块细节**：Honcho 是独立服务不是文件、Retrieval Index 时间衰减和常青文件例外、Bootstrap 子 Agent 权限、Session Transcript 6 种读取场景。这些都是"记了就得分"的知识点。

6. **继续保持设计思维**：Q15 的回答展示了独立判断，这在面试中加分很大。下次可以结合自己的 Proteus 项目具体说说怎么落地。
